# Дебаты: Овердрафтная модель vs Покупка-расход

#дебаты #кредиты #мультиметрика

**Дата:** 16 апреля 2026
**Цель:** Определить, какая модель учёта кредитных операций лучше подходит для мультиметрической системы (Поток, Капитал, DTI, Запас)
**Статус:** Завершён

## Участники
- **МОДЕЛЬ A** — текущая «Покупка = расход» (защитник)
- **МОДЕЛЬ B** — предлагаемая «Овердрафтная» (защитник)
- **АРБИТР** — нейтральный анализ, итоговые вердикты

## Определения моделей

### Модель A: «Покупка = расход»

Разные правила для разных типов кредитов:

| Тип кредита | Покупка | Тело погашения | Проценты |
|---|---|---|---|
| Кредитная карта (режим «покупка = расход») | Расход в категории | Перевод | Расход |
| Кредитная карта (режим «платёж = расход») | Только категорийная метка, НЕ расход | Credit_payment (расход) | В составе платежа |
| Карта рассрочки | Расход в момент покупки | Перевод | — (0%) |
| Потребкредит | Деньги на счёт; траты = обычные расходы | Перевод | Расход |
| Автокредит / ипотека | Нерегулярный расход (через счёт) | Перевод | Расход |

**Ключевая особенность:** для кредитных карт пользователь выбирает один из двух режимов при создании счёта.

### Модель B: «Овердрафтная»

Единая схема для ВСЕХ типов кредитов:

| Шаг | Что происходит | Классификация |
|---|---|---|
| 1. Получение кредитных средств | Disbursement | Нерегулярное поступление |
| 2. Трата средств | Расход в реальной категории | Следует регулярности категории* |
| 3. Погашение тела | Перевод | Не в потоке |
| 4. Уплата процентов | Расход | В базовом потоке |

Для кредитной карты / карты рассрочки: шаги 1 и 2 происходят **одновременно** (аналогия с овердрафтом — банк «выдал» деньги и тут же оплатил покупку).

**Ключевая особенность:** нет выбора режима. Все кредиты работают одинаково.

> *\*Критический вопрос о регулярности:* промпт описывает расход как «нерегулярный». Ниже (Раунд 2, Сценарий 2) показано, что буквальное прочтение «все кредитные расходы = нерегулярные» катастрофически ломает кешбэк-сценарий. Разумная интерпретация: расход следует регулярности своей категории, «нерегулярный» описывает типичный случай (крупная покупка), а не принудительную классификацию.

## Базовые параметры стресс-тестов (если не указано иное)

- Зарплата: 200 000 ₽/мес (регулярный доход)
- Регулярные расходы: 120 000 ₽/мес (с дебетового счёта)
- Начальный кэш на дебете: 300 000 ₽
- Кредитных долгов нет

## Раунд 1: Единообразие и простота

### Анализ Модели A

**Количество правил:** 5 строк в таблице (по числу типов кредитов) + 2 режима для кредитной карты.

**Формулы базового потока (две!):**
1. Режим «покупка = расход»: `Базовый поток = рег. доходы − рег. расходы − проценты по кредитам`
2. Режим «платёж = расход»: `Базовый поток = рег. доходы − рег. расходы − кредитные платежи (тело + %)`

**Ветвления в коде:**
- При записи транзакции: `if (account.type == 'credit_card' && account.mode == 'purchase_is_expense')` → расход; `else if (mode == 'payment_is_expense')` → только категорийная метка
- При погашении: `if (purchase_was_expense)` → transfer; `else` → credit_payment
- При расчёте базового потока: условие по режиму
- Отдельная логика для каждого типа: installment_card (InstallmentPurchase-таблица), credit (disbursement на дебет), mortgage (проходная операция)

**Пользовательская сложность:** при создании кредитной карты — вопрос «Как вы используете эту карту?» с двумя вариантами. Неправильный выбор даёт ложные метрики (подробнее в Раунде 2).

### Анализ Модели B

**Количество правил:** 1 строка (универсальная для всех типов).

**Формула базового потока (одна!):**
`Базовый поток = рег. доходы − рег. расходы − проценты по кредитам`

**Ветвления в коде:**
- При покупке: создать disbursement + расход (для кредитки/рассрочки — одновременно; для потребкредита — последовательно)
- При погашении: **всегда** transfer (тело) + расход (проценты)
- Нет условий по типу счёта, нет условий по режиму

**Пользовательская сложность:** при создании счёта — стандартные поля (лимит, ставка). Никаких вопросов о режиме.

### Сравнительная таблица

| Критерий | Модель A | Модель B |
|---|---|---|
| Типов правил | 5 | 1 |
| Формул базового потока | 2 | 1 |
| Пользовательских выборов | 1 (режим кредитки) | 0 |
| Ветвлений по типу счёта | Да (credit_card, installment_card, credit, mortgage) | Нет |
| Объяснение пользователю | «Зависит от типа карты и режима» | «Покупка = расход, платёж = перевод. Всегда» |
| Документация для разработчика | 5 правил + 2 формулы + логика переключения | 1 правило + 1 формула |

| | Модель A | Модель B |
|---|---|---|

## Раунд 2: Стресс-тесты на 6 сценариях

### Сценарий 1: MacBook 350к на кредитку (12 мес, 15%)

**Параметры:** доход 200к, расходы 120к, MacBook 350к — нерегулярная категория «Электроника». Аннуитетный платёж ≈ 32к/мес. Проценты (месяц 1) ≈ 4,4к.

**Месяц покупки:**

| Метрика | Модель A (покупка=расход) | Модель A (платёж=расход) | Модель B |
|---|---|---|---|
| Базовый поток | 200 − 120 = **+80к** | 200 − 120 = **+80к** | 200 − 120 = **+80к** |
| Полный поток | **+80к** | **+80к** | **+80к** |
| Капитал | 380к − 350к = **+30к** | **+30к** | **+30к** |
| DTI | **0%** | **0%** | **0%** |
| Запас | 380/120 = **3,2 мес** | **3,2 мес** | **3,2 мес** |

Месяц покупки: **все три варианта идентичны.** MacBook — нерегулярный расход, не влияет на базовый поток. Долг +350к виден в Капитале.

**Месяц 1 (первый платёж):**

| Метрика | Модель A (покупка=расход) | Модель A (платёж=расход) | Модель B |
|---|---|---|---|
| Базовый поток | 200 − 120 − 4,4 = **+75,6к** | 200 − 120 − 32 = **+48к** | 200 − 120 − 4,4 = **+75,6к** |
| Полный поток | 200 − 120 − 32 = **+48к** | **+48к** | **+48к** |
| Капитал | 428к − 322к = **+106к** | **+106к** | **+106к** |
| DTI | 32/200 = **16%** | **16%** | **16%** |
| Запас | 428/152 = **2,8 мес** | **2,8 мес** | **2,8 мес** |

**Ключевое расхождение: базовый поток.**

- Модель A (платёж=расход): +48к. Включает **полный** платёж 32к (тело + %). Отражает реальные обязательства: «после всех платежей банку у тебя остаётся 48к».
- Модели A (покупка=расход) и B: +75,6к. Включают **только** проценты 4,4к. Тело 27,6к = перевод, невидим в базовом потоке.

**Что точнее?** Реально у пользователя остаётся 48к свободных (200 − 120 − 32). Модель A (платёж=расход) ближе к реальности в базовом потоке. Но Модель B (и A purchase=expense) компенсирует через DTI 16% — пользователь видит, что 16% дохода уходит банку.

**Контраргумент за Модели B / A (purchase=expense):** включение полного платежа 32к в базовый поток нарушает правило «каждый рубль — один раз». MacBook уже учтён как расход (нерегулярный, не в базовом потоке, но в аналитике). Если тело тоже в потоке — 350к MacBook стоит 350к (расход) + 350к (тело) = 700к в суммарной аналитике. Двойной счёт.

> **Вердикт сценария 1:** Все модели показывают корректные DTI, полный поток, Капитал, Запас. Расхождение только в базовом потоке. Модель B = Модель A (покупка=расход).

### Сценарий 2: Кешбэк-пользователь (все 120к через кредитку, полное погашение)

**Параметры:** ВСЕ ежедневные расходы 120к/мес (продукты, транспорт, коммуналка) оплачиваются кредитной картой. Погашение 100% в грейс-период. Процентов нет.

| Метрика | Модель A (покупка=расход) | Модель A (платёж=расход) | Модель B |
|---|---|---|---|
| Базовый поток | 200 − 120 = **+80к** | 200 − 120 = **+80к** | 200 − 120 = **+80к** |
| Полный поток | **+80к** | **+80к** | **+80к** |
| Капитал | стабилен | стабилен | стабилен |
| DTI | **0%** | 120/200 = **60%** ❌ | **0%** |
| Запас | стабилен | стабилен | стабилен |
| Категории | ✅ Видны (продукты 40к, транспорт 10к...) | ❌ Всё = «Кредитные платежи» | ✅ Видны |

**Модель A (платёж=расход) катастрофически ломается:**

1. **DTI 60%** у человека с нулевым долгом. Система сигнализирует «критическая кредитная нагрузка» при идеальном финансовом поведении. Для школы финансовой грамотности — подрыв доверия.

2. **Категории невидимы.** 120к в одном сером секторе «Кредитные платежи». Круговая диаграмма бесполезна. Дашборд — чёрный ящик. Это проблема, задокументированная в стресс-тесте (Сценарий А).

3. **Масштаб:** кешбэк через кредитку — массовый сценарий для ЦА 30–45. Вероятно, большинство активных пользователей.

**Модели A (покупка=расход) и B: работают идентично.** DTI 0% (погашение = перевод, нет credit_payment). Категории видны. Базовый поток точен.

**Критический вопрос для Модели B: регулярность расходов.**

Если буквально «расход = нерегулярный» (как в описании Модели B):
- Все 120к расходов на кредитку → нерегулярные → **не в базовом потоке**
- Базовый поток = 200 − 0 = **+200к** → **абсурд**. Система говорит «весь доход свободен», хотя человек тратит 120к/мес

Если расход следует регулярности категории (разумная интерпретация):
- Продукты (регулярная) → в базовом потоке. Электроника (нерегулярная) → не в базовом потоке
- Базовый поток = 200 − 120 = +80к → **корректно**

**Вывод:** «нерегулярный» в описании Модели B НЕ МОЖЕТ быть принудительной классификацией. Расход ОБЯЗАН следовать регулярности категории. Иначе кешбэк-сценарий уничтожает метрику. Далее в дебатах используется разумная интерпретация.

> **Вердикт сценария 2:** Модель B = Модель A (покупка=расход) — оба работают идеально. Модель A (платёж=расход) — DTI 60% при нулевом долге + категории невидимы. Это **главный аргумент** против режима «платёж=расход» и, следовательно, против необходимости пользовательского выбора в Модели A.

### Сценарий 3: Карта рассрочки — телефон 60к, 12 мес, 0%

**Параметры:** покупка телефона на Халве. Платёж 5к/мес. Без процентов.

| Метрика | Модель A | Модель B | Разница? |
|---|---|---|---|
| **Месяц покупки** | | | |
| Базовый поток | +80к (телефон нерегулярный) | +80к | Нет |
| Полный поток | +80к (дебет не затронут) | +80к | Нет |
| Капитал | 300 − 60 долга = +240к | +240к | Нет |
| DTI | 5/200 = 2,5% | 2,5% | Нет |
| Запас | 300/120 = 2,5 мес | 2,5 мес | Нет |
| **Месяцы 1–12** | | | |
| Базовый поток | +80к (тело = перевод, % = 0) | +80к | Нет |
| Полный поток | 200−120−5 = +75к | +75к | Нет |
| DTI | 2,5% | 2,5% | Нет |

**Внутренняя механика различается:**

- Модель A: покупка → расход −60к на installment_card. Одна транзакция.
- Модель B: disbursement +60к + расход −60к на installment_card. Две транзакции, результат тот же.

В обоих случаях:
- Долг на installment_card: +60к
- Дебетовый счёт: не затронут
- Платежи: 5к/мес перевод с дебета

> **Вердикт сценария 3:** Полностью идентичные метрики. Разница только во внутреннем представлении (1 vs 2 транзакции).

### Сценарий 4: Потребкредит 500к — ремонт квартиры

**Параметры:** кредит 500к, 12 мес, 15%. Деньги на дебете. Тратятся на ремонт за 3 месяца (~150к/мес). Аннуитетный платёж ≈ 45к/мес. Категория «Ремонт» — нерегулярная.

**Проблема, общая для обеих моделей:** деньги кредита на дебете, смешались с зарплатой. Система не может автоматически отличить «купил плитку на кредитные деньги» от «купил продукты на зарплату».

| Метрика (если ремонт помечен нерегулярным) | Модель A | Модель B | Разница? |
|---|---|---|---|
| **Месяцы 1–3 (траты)** | | | |
| Базовый поток | 200 − 120 = +80к | +80к | Нет |
| Полный поток | 200 − 270 = **−70к** | **−70к** | Нет |
| Капитал | дебет тает, долг стабилен | то же | Нет |
| **Месяцы 4+** | | | |
| Базовый поток | 200 − 120 − ~5к(%) = +75к | +75к | Нет |
| Полный поток | 200 − 120 − 45 = +35к | +35к | Нет |
| DTI | 45/200 = 22,5% | 22,5% | Нет |

**Проблема двойного счёта (если ремонт НЕ помечен):**

Если пользователь не пометил ремонтные расходы как «из кредита» — они считаются обычными расходами (150к/мес). Плюс потом кредитные платежи (тело = перевод). Двойной счёт:
- Расходы за 3 мес: 500к (ремонт) + 360к (регулярные) = 860к
- Кредитные платежи: 540к (45к × 12, тело = перевод, только % ≈ 40к в потоке)
- Суммарная стоимость ремонта: 500к (расходы) + 40к (проценты) = 540к. ✅ Нет двойного счёта!

Но в **аналитике расходов**: ремонт 500к отображается как реальные расходы, хотя оплачен заёмными средствами. Это искажает картину.

**Решение (в обеих моделях):** тегирование / кредитный конверт — пользователь помечает расходы как «из кредита», система помечает их нерегулярными. Обе модели одинаково зависят от действия пользователя.

> **Вердикт сценария 4:** Метрики идентичны. Проблема смешивания денег на дебете одинакова в обеих моделях. Ни одна из них не решает её автоматически.

### Сценарий 5: Ипотека 5 млн, платёж 45к/мес

**Параметры:** ипотека 5 000к, 20 лет, ~10%. Платёж 45к/мес. Проценты (мес. 1) ≈ 42к, тело ≈ 3к. Квартира 5 000к — неликвидный актив (не в Капитале).

**Месяц покупки:**

Обе модели:
1. Disbursement +5 000к на дебет (нерегулярное поступление)
2. Покупка квартиры −5 000к с дебета (нерегулярный расход)
3. Нетто на дебете: 0. Появился долг −5 000к на кредитном счёте.

| Метрика | Модель A | Модель B | Разница? |
|---|---|---|---|
| **Текущие месяцы** | | | |
| Базовый поток | 200 − 120 − 42(%) = **+38к** | **+38к** | Нет |
| Полный поток | 200 − 120 − 45 = **+35к** | **+35к** | Нет |
| Капитал | дебет − 5 000к долга = **глубоко отрицательный** | то же | Нет |
| DTI | 45/200 = **22,5%** | **22,5%** | Нет |
| Запас | дебет / (120+45) = зависит от накоплений | то же | Нет |

**Примечание:** Капитал сильно отрицательный, потому что квартира — неликвидный актив (не учитывается). Это осознанное решение мультиметрики: Капитал = ликвидная позиция. Ипотечный заёмщик «беден ликвидно», но владеет активом. Виджет «Активы» покажет полную картину.

**Базовый поток: только проценты или полный платёж?**

В обеих моделях покупка квартиры = нерегулярный расход → тело = перевод → только проценты в базовом потоке.

Замечание: это расходится с решением дебатов о базовом потоке (Критика 4.1), где для ипотеки принято «полный платёж в потоке, т.к. покупка = балансовая операция». Если принять то решение, базовый поток = 200 − 120 − 45 = +35к (только в Модели A). Разница невелика (+38к vs +35к), т.к. на ранних сроках ипотеки почти весь платёж — проценты.

> **Вердикт сценария 5:** Метрики идентичны в обеих моделях. Нюанс с «балансовой операцией» vs «нерегулярным расходом» для ипотеки — отдельный открытый вопрос, но он не зависит от выбора между Моделями A и B.

### Сценарий 6: Накопление рассрочек — 60к + 40к + 80к за 3 месяца

**Параметры:**
- Месяц 1: покупка A (60к, 6 мес, 0%) → платёж 10к/мес
- Месяц 2: покупка B (40к, 4 мес, 0%) → платёж 10к/мес
- Месяц 3: покупка C (80к, 8 мес, 0%) → платёж 10к/мес

| Месяц | Суммарный платёж | DTI | Базовый поток | Полный поток |
|---|---|---|---|---|
| 0 | 0 | 0% | +80к | +80к |
| 1 | 10к (A) | 5% | +80к | +70к |
| 2 | 20к (A+B) | 10% | +80к | +60к |
| 3–5 | 30к (A+B+C) | 15% | +80к | +50к |
| 6 | 20к (A+C) | 10% | +80к | +60к |
| 7–10 | 10к (C) | 5% | +80к | +70к |

**Идентично в обеих моделях.** Базовый поток стабилен (+80к), т.к. все рассрочки 0% → проценты = 0, тело = перевод. Но:
- DTI растёт: 0% → 5% → 10% → **15%** — виден кумулятивный эффект
- Полный поток падает: +80к → **+50к** — виден реальный кэш-дрейн
- Запас ухудшается: отток растёт → запас падает

**Проблема (общая):** базовый поток **не чувствует** рассрочки 0%. Это «слепое пятно» — человек набирает обязательства, а базовый поток стабилен. Компенсация: DTI и полный поток.

> **Вердикт сценария 6:** Полностью идентичны. Слепое пятно базового потока к 0%-рассрочкам — свойство обеих моделей, компенсируется DTI.

## Раунд 3: Полный поток — точность

Полный поток = Δ кэша на ликвидных счетах за период. Это чисто балансовый показатель: разница остатков дебетовых/кассовых счетов.

### Проверка всех сценариев

**Принцип:** полный поток зависит ТОЛЬКО от того, сколько реальных денег пришло и ушло с дебетового счёта. Он не зависит от классификации транзакций (расход/перевод/disbursement).

| Событие | Дебет Δ | Модель A | Модель B |
|---|---|---|---|
| Зарплата +200к | +200к | +200к | +200к |
| Расходы с дебета −120к | −120к | −120к | −120к |
| Покупка на кредитку | 0 (дебет не затронут) | 0 | 0 |
| Покупка на рассрочку | 0 (дебет не затронут) | 0 | 0 |
| Погашение кредитки −32к | −32к | −32к | −32к |
| Платёж по рассрочке −5к | −5к | −5к | −5к |
| Получение потребкредита +500к | +500к | +500к | +500к |
| Покупка квартиры −5 000к | −5 000к | −5 000к | −5 000к |

**Вывод:** полный поток — тождественная функция от реальных банковских операций. Он **не может** различаться между моделями, потому что реальное движение денег одинаково.

### Точность обеих моделей

Полный поток точно отражает Δ кэша **по определению**. Единственный источник расхождения — если модель «забывает» какое-то движение:

- Модель A: покупка на кредитку = расход (или категорийная метка). В любом случае, дебет не меняется → полный поток не искажён.
- Модель B: disbursement + расход на кредитном счёте. Дебет не меняется → полный поток не искажён.

Обе модели точны на 100%.

### Есть ли расхождения с полным потоком?

**С базовым потоком — да.** Базовый поток исключает нерегулярные операции. Если у пользователя много нерегулярных (ремонт, крупные покупки), базовый и полный поток расходятся:

- Полный поток (мес. ремонта): −70к
- Базовый поток: +80к

Расхождение 150к — это стоимость ремонта. Оно **одинаково** в обеих моделях и является ФИЧЕЙ, а не багом. Два числа отвечают на два разных вопроса.

| | Модель A | Модель B |
|---|---|---|
| Точность полного потока | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Расхождение с базовым потоком | Идентично | Идентично |

**Вердикт раунда: Ничья.** Полный поток — балансовый показатель, не зависящий от модели учёта.

## Раунд 5: Операционные сложности

### 5.1 Импорт банковских выписок

**Модель A:**

Выписка по **дебетовой карте:** обычные транзакции. Нет разницы между моделями.

Выписка по **кредитной карте:** содержит покупки (merchant charges) и платежи (repayments).
- Режим «покупка = расход»: покупки → расход в категории. Погашение → перевод.
- Режим «платёж = расход»: покупки → категорийная метка (не расход). Погашение → credit_payment (расход).
- При импорте система должна знать режим счёта и применить правильную логику.

Выписка по **installment_card:** содержит только ежемесячные списания (платежи). Покупки в этой выписке не фигурируют — они создаются пользователем через InstallmentPurchase.

**Модель B:**

Выписка по **кредитной карте:** покупки → disbursement + расход (одновременно). Погашение → перевод. Нет условий по режиму.

Выписка по **installment_card:** те же ежемесячные списания → credit_payment.

| Аспект импорта | Модель A | Модель B |
|---|---|---|
| Нужно ли знать режим счёта при импорте? | Да (для кредитки) | Нет |
| Создание двух записей из одной строки выписки? | Нет | Да (disbursement + expense для кредитки) |
| Сложность парсера | Средняя (ветвления по режиму) | Средняя (логика создания парных записей) |
| Вероятность ошибки пользователя | Выбрал не тот режим → ложные метрики | Нет выбора → нет ошибки |

**Вывод:** сложность парсера примерно одинакова. Но Модель B исключает ошибку выбора режима.

### 5.2 Досрочное погашение

**Модель A:**
- Режим «покупка = расход»: досрочное погашение = credit_early_repayment (перевод, не расход). Долг обнуляется. Капитал: кэш ↓, долг ↓ → нетто 0 (для 0%).
- Режим «платёж = расход»: досрочное погашение = credit_early_repayment. НЕ расход (по правилу: досрочное не в потоке). DTI пересчитывается.

**Модель B:**
- Досрочное погашение = перевод (тело) + расход (проценты, если есть). Если остаток тела выплачен полностью → долг закрыт.

**Различие:** в Модели A (платёж=расход) нужна отдельная логика для досрочного погашения (credit_early_repayment ≠ credit_payment). В Модели B: досрочное — просто бо́льший перевод тела. Один механизм для регулярного и досрочного.

### 5.3 Возврат товара

**Покупка 30к на кредитке → возврат.**

**Модель A (покупка=расход):**
1. Исходная транзакция: расход −30к.
2. Возврат: +30к на кредитную карту → отрицательный расход (или доход) +30к в той же категории.
3. Баланс кредитки: долг уменьшился на 30к.
4. В аналитике: расходы и возврат видны, нетто = 0.

**Модель B:**
1. Исходная транзакция: disbursement +30к + расход −30к.
2. Возврат: «обратный disbursement» −30к + обратный расход +30к.
3. Баланс кредитки: долг уменьшился на 30к.
4. В аналитике: 4 записи вместо 2, но нетто = 0.

**Модель B сложнее:** при возврате нужно отменить и disbursement, и расход. Два действия вместо одного. Это не критично, но добавляет сложности в обработке возвратов.

| | Модель A | Модель B |
|---|---|---|
| Записей при возврате | 1 (обратный расход) | 2 (обратный disbursement + обратный расход) |
| Логика | Простая | Средняя |
| Результат | Идентичный | Идентичный |

### 5.4 Конвертация покупки по кредитке в рассрочку

**Сценарий:** купил ТВ 80к на кредитке. Через неделю банк конвертировал в рассрочку 6 × 14к.

**Модель A (покупка=расход):**
1. Покупка 80к = расход. Уже в аналитике.
2. Конвертация: помечаем транзакцию `converted_to_installment`, исключаем из расходов. Создаём InstallmentPurchase.
3. Если в том же месяце → чисто. Если в прошлом → меняем историю (или ручная коррекция).
4. Без коррекции: двойной счёт (80к расход + 84к платежей = 164к вместо 84к).

**Модель A (платёж=расход):**
1. Покупка 80к = только категорийная метка (не расход).
2. Конвертация: долг переходит с кредитки на рассрочку. Капитал: нетто 0.
3. Никаких проблем — покупка не была расходом, двойного счёта нет.

**Модель B:**
1. Покупка 80к = disbursement + расход. Расход в аналитике.
2. Конвертация: то же, что Модель A (покупка=расход). Нужно отменить disbursement + расход, создать InstallmentPurchase.
3. Та же проблема с кроссмесячной конвертацией.

**Интересный момент:** конвертация — единственный сценарий, где Модель A (платёж=расход) работает ЛУЧШЕ обеих альтернатив. Но частота этого сценария низкая.

| Карта рассрочки | Лимит, monthly_payment, остаток | Лимит, monthly_payment, остаток |
| Потребкредит | Сумма, ставка, срок, monthly_payment | Сумма, ставка, срок, monthly_payment |

Единственная разница: кредитная карта в Модели A требует выбора режима. Модель B — нет. Для новичка — одним вопросом меньше, одной возможной ошибкой меньше.

Обоснование:
1. Метрики идентичны с лучшим режимом Модели A — ничего не теряется.
2. Худший режим Модели A (платёж=расход) создаёт серьёзные проблемы (DTI 60% без долга, невидимые категории) для самого массового сценария (кешбэк).
3. Модель B радикально проще: 1 правило, 1 формула, 0 выборов.
4. Единственное преимущество режима «платёж=расход» — более точный базовый поток при активном долге — компенсируется DTI (который показывает долю дохода, уходящую банкам).

### Что теряется при отказе от «платёж=расход»

| Потеря | Серьёзность | Компенсация |
|---|---|---|
| Базовый поток не показывает полный кредитный платёж | Средняя | DTI показывает долю; Полный поток показывает реальный кэш |
| Конвертация в рассрочку требует коррекции | Низкая | Редкий сценарий; ручная коррекция в MVP |

### Критический нюанс: регулярность расходов

При принятии Модели B необходимо зафиксировать:

**Расход на кредитные средства следует регулярности своей КАТЕГОРИИ, а не принудительно помечается нерегулярным.**

Продукты на кредитку = регулярный расход (категория «Продукты» — регулярная).
MacBook на кредитку = нерегулярный расход (категория «Электроника» — нерегулярная).

Без этого правила Модель B ломается на кешбэк-сценарии (базовый поток = 200к при расходах 120к).

### Практическая реализация Модели B

Если принята:

1. **Кредитная карта:** убрать выбор режима. Покупки = расход в категории. Погашение = перевод. Единственная логика.
2. **Installment card:** без изменений (уже работает по аналогичному принципу).
3. **Потребкредит:** без изменений (disbursement → расходы → платежи).
4. **Ипотека/автокредит:** без изменений.
5. **Формула базового потока:** `рег. доходы − рег. расходы − проценты по кредитам`. Одна формула. Тело всегда = перевод.
6. **DTI:** `Σ(monthly_payment) / рег. доход`. Без изменений — DTI считается из Account, не из типа транзакции.

Объём изменений в коде: **удаление** логики выбора режима для credit_card, **удаление** ветвлений по режиму в расчёте потока. Модель становится проще, а не сложнее.

Основания для уверенности:
- 6 из 6 сценариев дают идентичные метрики (Model B = Model A purchase=expense)
- Единственное потерянное преимущество (точный базовый поток при долге) компенсируется DTI

Вычеты:
- **−5:** Парные записи (disbursement + расход) при каждой покупке на кредитку — усложнение данных. В Модели A одна запись, в B — две. Если disbursement не несёт самостоятельной аналитической ценности, его можно реализовать как «неявный» (не создавать отдельную транзакцию, а пометить расход флагом `source: credit`).
- **−4:** Вопрос регулярности не формализован в описании Модели B. Без чёткого правила «расход следует регулярности категории» модель ломается.
- **−3:** Конвертация покупка→рассрочка из прошлого месяца — нет элегантного решения в обеих моделях. Это открытый операционный вопрос.
