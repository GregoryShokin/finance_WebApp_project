# Стресс-тест: Классификация расходов по регулярности + двухслойный Поток

#стресс-тест #мультиметрика #архитектура

> 12 сценариев. Роль: КРИТИК. Цель — найти все способы сломать систему.

**Дата:** апрель 2026

**Тестируемая система:**
- Базовый поток = регулярные доходы − регулярные расходы − платежи по кредитам
- Полный поток = все доходы − все расходы (честный кэшфлоу)
- Флаг регулярности на уровне категории (бинарный, не ретроактивный)

## Сценарий 1: Кешбэк-пользователь

**Исходные данные:**
- Доход: 200к (регулярный, «Зарплата»)
- Расходы: 130к (регулярные категории, все через кредитку)
- Кешбэк: ~3к/мес (категория «Кешбэк», регулярная)
- Кредитка погашается полностью каждый месяц

**Расчёт метрик:**
- Базовый поток: (200к + 3к) − 130к − 0 = **+73к**
- Полный поток: (200к + 3к) − 130к = **+73к** (совпадает — всё регулярное)
- Капитал: кэш на счетах − 0(карта погашена) = кэш
- Нагрузка (DTI): 0 / 203к = **0%** (кредитка — не кредит, погашение = перевод)
- Запас: при кэше 500к → 500к / 130к = **3.85 мес**

**Атака 1:** Погашение кредитки — перевод или расход?
Проверяю: в мультиметрике покупка на кредитку = расход (130к, в реальных категориях). Погашение = перевод (с дебетового на кредитку). 130к расхода считаются один раз — при покупке. Погашение НЕ удваивает. ✅

**Атака 2:** Кешбэк как доход — не завышает ли Поток?
3к/мес при расходах 130к — это возврат 2.3%. Включён в базовый поток как регулярный доход. Корректно: кешбэк стабилен и предсказуем. Если пользователь пометит «Кешбэк» как нерегулярный — выпадет из базового. Его выбор, оба варианта допустимы.

**Атака 3:** DTI = 0% при активной кредитке — не обманывает ли?
Кредитка, погашаемая полностью — не долговое обязательство. Нет процентов, нет минимального платежа. DTI = 0% корректно. Если бы пользователь НЕ погашал полностью → минимальный платёж появился бы в DTI.

**Результат: ✅ Держится.** Базовый = полный (всё регулярное). DTI = 0. Двойного счёта нет.

## Сценарий 2: Ремонт за потребкредит (Проект)

**Исходные данные:**
- Доход: 150к (регулярный)
- Обычные расходы: 80к (регулярные)
- Потребкредит 500к, платёж 15к/мес (тело ~12к + проценты ~3к)
- Ремонт: 150к/200к/150к за 3 месяца (30 транзакций/мес, категории нерегулярные)
- Создан Проект: 500к, 3 месяца

**Расчёт метрик (регулярность):**
- Базовый поток = 150к − 80к − 15к = **+55к** (ремонт исключён — нерегулярный)
- Полный поток (мес 1): 150к − 80к − 150к − 15к = **−95к**
- Полный поток (мес 2): 150к − 80к − 200к − 15к = **−145к**
- Полный поток (мес 3): 150к − 80к − 150к − 15к = **−95к**
- DTI: 15к / 150к = **10%**
- Запас (мес 1, кэш ~505к): 505к / (80к + 15к) = **5.3 мес**

**Атака 1 (КРИТИЧЕСКАЯ): Проект + флаг нерегулярности = двойная коррекция.**

Механизм Проекта даёт: Базовый = Полный + коррекция = Полный + 500к/3 = Полный + 167к.
Месяц 1: −95к + 167к = **+72к** (Проект-базовый)

Флаг нерегулярности даёт: Базовый = 150к − 80к − 15к = **+55к** (регулярность-базовый)

Два механизма — два разных числа: +72к vs +55к.

Если система применит ОБА: +55к + 167к = **+222к**. Абсурд. Ремонт исключён дважды — и через нерегулярную категорию, и через коррекцию Проекта.

**Диагноз:** флаг нерегулярности и коррекция Проекта — **альтернативные механизмы**, не аддитивные. Оба решают одну задачу: убрать проектные расходы из «базового уровня жизни». Применять нужно ОДИН из них.

**Решение: приоритет флага нерегулярности.**

| Ситуация | Механизм | Базовый поток |
|---|---|---|
| Ремонтные категории = нерегулярные | Флаг (основной) | 150к − 80к − 15к = +55к |
| Ремонтные категории = регулярные (забыл пометить) | Проект (fallback) | Полный + 167к |
| Нерегулярные + Проект | Только флаг, Проект = UX | +55к (коррекция НЕ применяется) |

Проект при нерегулярных категориях становится **UX-механизмом**: иконка «📋 Ремонт» на дашборде, школьные моменты, прогресс расходования, объяснение разницы базового и полного. Метрики не корректирует.

**Правило: если ВСЕ проектные расходы в нерегулярных категориях → `Project.metric_correction = false`.** Система проверяет автоматически. Если часть расходов в регулярных — коррекция применяется пропорционально.

**Результат: ⚠️ Нюанс — двойная коррекция.** Решаемо приоритетом флага над Проектом. Требует явного правила в архитектуре.

## Сценарий 3: Покупка машины за свои деньги

**Исходные данные:**
- Доход: 200к (регулярный), расходы: 120к (регулярные)
- Кэш: 1.5М
- Купил машину: 800к (категория «Авто покупка», нерегулярная)
- Добавил машину в активы: оценка 800к

**Расчёт метрик:**
- Базовый поток: 200к − 120к = **+80к** (покупка исключена — нерегулярная) ✅
- Полный поток: 200к − 120к − 800к = **−720к** (один красный месяц)
- Капитал до: 1.5М. После: (1.5М + 200к − 120к − 800к) кэш + 800к актив = 780к + 800к = **1.58М** (+80к)
- DTI: **0%**
- Запас до: 1.5М / 120к = **12.5 мес**. После: 780к / 120к = **6.5 мес**

**Атака 1:** Капитал не просел при покупке за 800к?
Кэш −800к, но актив +800к. Чистый эффект = 0. Плюс месячный профицит +80к. Капитал вырос на 80к. ✅

**Атака 2:** Запас упал с 12.5 до 6.5 мес — корректно?
Запас считает только ликвидный кэш, не активы. Машину нельзя быстро продать для оплаты аренды. Падение запаса **корректно** — пользователь стал менее ликвидным. ✅

**Атака 3:** Полный поток −720к — один красный месяц в истории.
Среднее полное за 6 мес: (−720к + 5 × 80к) / 6 = −320к / 6 = −53к/мес. Среднее остаётся отрицательным 9 месяцев. Если какой-то показатель использует среднее полного потока — будет искажён. Но FI-score использует базовый поток → не затронут. ✅

**Результат: ✅ Держится.** Базовый чист. Капитал корректен (кэш → актив). Запас честно падает.

## Сценарий 4: Покупка машины в кредит

**Исходные данные:**
- Доход: 200к (регулярный), расходы: 120к (регулярные)
- Автокредит 1.2М, платёж 25к/мес (тело ~21к + проценты ~4к)
- Машина = актив (оценка 1.2М)
- Кэш до покупки: 500к

**Расчёт метрик:**
- Базовый поток: 200к − 120к − 25к = **+55к** ✅
- DTI: 25к / 200к = **12.5%**
- Капитал: кэш + авто − долг. Кэш ≈ 555к, авто 1.2М, долг ~1.179М → **576к** (+76к от начальных 500к)
- Запас: 555к / (120к + 25к) = **3.83 мес**

**Атака (КРИТИЧЕСКАЯ): Полный поток — асимметрия получения и покупки.**

Физика операции:
1. Банк перечисляет 1.2М на счёт (credit_disbursement)
2. Пользователь оплачивает машину 1.2М (расход)

В мультиметрике: disbursement ≠ доход (балансовая операция). Покупка = расход.

Полный поток: 200к + 0(disbursement не доход) − 120к − 1.2М(покупка) − 25к = **−1.145М**

Но кэш пользователя: 500к + 200к + 1.2М(поступление) − 120к − 1.2М(покупка) − 25к = **555к**. Реальное изменение кэша: +55к. Полный поток показывает −1.145М при реальном кэшфлоу +55к. **Семикратное искажение.**

Причина: disbursement = деньги приходят на счёт (кэш ↑), но не считается доходом. Покупка = деньги уходят (кэш ↓), считается расходом. Асимметрия.

**Корень проблемы: определение полного потока.**

Промпт говорит: «Полный поток = все доходы − все расходы (честный кэшфлоу)». Но «честный кэшфлоу» и «доходы − расходы» — это разные вещи:
- **Честный кэшфлоу** = все поступления на счета − все списания со счетов = +1.2М(disbursement) + 200к − 120к − 1.2М − 25к = +55к ✅
- **Доходы − расходы** = 200к − 120к − 1.2М − 25к = −1.145М ❌ (disbursement не «доход»)

**Решение: полный поток = честный кэшфлоу (все движения денег), а не «доходы минус расходы».**

Credit_disbursement — это кэш-поступление. Оно нейтрально для Капитала (кэш ↑, долг ↑), но это реальное движение денег. Если полный поток = «сколько реально прошло через счета», disbursement включается как поступление.

Тогда: полный поток = 200к + 1.2М − 120к − 1.2М − 25к = +55к. Disbursement и покупка взаимогасятся. ✅

Альтернативно: покупка на кредитные деньги = балансовая операция (как и disbursement). Оба исключаются. Полный поток = 200к − 120к − 25к = +55к. Тот же результат.

**Итого: нужно уточнить определение полного потока.** «Все доходы − все расходы» неточно. Правильно: «все денежные поступления − все денежные списания» ИЛИ «disbursement-linked покупки = балансовые операции».

**Результат: ⚠️ Нюанс — определение полного потока.** Базовый поток корректен (+55к). Полный поток требует уточнения: disbursement и кредитная покупка должны взаимогаситься. Решаемо определением «полный поток = чистое движение кэша».

## Сценарий 5: Семья «Первый шаг» (DTI 42%)

**Исходные данные:**
- Доход семьи: 120к (регулярный)
- Ипотека: 30к/мес, потребкредит: 15к/мес, кредитка мин. платёж: 5к/мес
- Общие платежи по кредитам: 50к/мес
- Обычные расходы: 65к (регулярные)
- Кэш: 50к
- Квартира: оценка ~4М, ипотечный долг ~3М, потребкредит ~200к, кредитка ~100к

**Расчёт метрик:**
- Базовый поток: 120к − 65к − 50к = **+5к** 🟡
- Полный поток: 120к − 65к − 50к = **+5к** (всё регулярное → базовый = полный)
- Капитал: 50к(кэш) + 4М(квартира) − 3.3М(долги) = **750к**
- DTI: 50к / 120к = **42%** 🔴
- Запас: 50к / (65к + 50к) = 50к / 115к = **0.43 мес ≈ 13 дней** 🔴

**FI-score (оценка):**
- Поток: +5к/120к ≈ 4% нормы → ~2/10
- DTI: 42% → зона «опасно» → ~2/10
- Запас: 0.43 мес → ~1/10
- Норма сбережений: 5к/120к = 4% → ~1/10
- **FI-score: ~1.5–2.5/10** 🔴

**Что видит семья:**
```
ПОТОК       КАПИТАЛ     НАГРУЗКА    ЗАПАС       FI
+5к 🟡      750к 🟡     42% 🔴      13 дн 🔴    2/10 🔴
```

**Атака 1:** Поток +5к — честно?
120к доход − 65к жизнь − 50к кредиты = 5к. На счёте реально остаётся +5к/мес. ✅

**Атака 2:** Капитал 750к — не завышен ли?
Зависит от оценки квартиры. Если квартира оценена в 4М, а ипотечный долг 3М → нетто 1М минус другие долги 300к + кэш 50к = 750к. При точной оценке — корректно. При завышенной — обманчиво. Но это не проблема классификации расходов — это проблема оценки активов. ✅ (для данного теста)

**Атака 3:** DTI 42% — кредитка с минимальным платежом.
Минимальный платёж 5к при долге ~100к. Проценты ~2к/мес. Платёж 5к = 2к проценты + 3к тело. При таком темпе долг 100к закроется за ~33 месяца. Если семья платит только минимум — спираль не видна в DTI (DTI стабилен). Нужен «Здоровье долга».

Это **известное ограничение** (из дебатов раунд 4): долговая спираль невидима без отдельного слоя. Не проблема классификации.

**Атака 4:** Запас 13 дней — семья видит реальность?
Да. 50к кэша при оттоке 115к/мес = 13 дней. Школа: «Потеря дохода — катастрофа через 2 недели. Первый приоритет — подушка.» Жёстко, но честно. ✅

**Результат: ✅ Держится.** Семья видит четыре числа: поток еле положителен, нагрузка критическая, запас почти нулевой, FI-score в зоне риска. Школа: «Фокус — нагрузка 42%».

## Сценарий 6: Фрилансер с волатильным доходом

**Исходные данные:**
- Доход: янв 150к, фев 80к, мар 300к, апр 120к (категория «Фриланс», регулярная)
- Расходы: 100к/мес (регулярные)
- Кредитов нет
- Кэш: 200к на начало

**Расчёт метрик:**

| Мес | Доход | Расходы | Базовый | Полный | Кэш | Запас |
|---|---|---|---|---|---|---|
| Янв | 150к | 100к | +50к | +50к | 250к | 2.5 мес |
| Фев | 80к | 100к | −20к 🔴 | −20к | 230к | 2.3 мес |
| Мар | 300к | 100к | +200к | +200к | 430к | 4.3 мес |
| Апр | 120к | 100к | +20к | +20к | 450к | 4.5 мес |

Средний базовый (4 мес): (50 − 20 + 200 + 20) / 4 = **+62.5к**
DTI: 0%

**Атака 1:** Поток скачет — февраль красный. Проблема?
Честно: в феврале фрилансер заработал меньше, чем потратил. Поток −20к — это правда. Школьный момент: «Февральский поток отрицательный. Для нерегулярных доходов важен запас — он у тебя 2.3 мес, хватит пережить слабый месяц.» ✅

**Атака 2:** Если бы был кредит, DTI бы скакал?
DTI = платежи / **средний регулярный** доход. Средний = (150 + 80 + 300 + 120) / 4 = 162.5к. DTI стабилен. ✅ (используется среднее, не месячное)

**Атака 3:** FI-score использует скользящее среднее базового. Одноразовый всплеск (300к) завышает среднее?
Скользящее среднее за 6 мес сглаживает. При 4 месяцах данных: +62.5к среднее. Разумно. Один хороший месяц не создаёт иллюзии. ✅

**Атака 4:** Должен ли фрилансер пометить доход как нерегулярный?
«Фриланс» = основной доход. Пометка «нерегулярный» обнулит базовый поток (базовый = 0 − 100к = −100к каждый месяц). Это абсурд — фриланс нерегулярен по суммам, но регулярен по факту (приходит каждый месяц). Правильно: регулярный. Волатильность сглаживается средним. ✅

**Результат: ✅ Держится.** Волатильность в месячном потоке — норма для фрилансера. Среднее сглаживает. DTI по среднему — стабилен.

## Сценарий 7: Пользователь игнорирует переключатель

**Исходные данные:**
- Доход: 200к, расходы: 100к (всё регулярное по умолчанию)
- Купил мебель 200к, **не пометил** категорию как нерегулярную

**Расчёт метрик:**
- Базовый поток: 200к − 100к − 200к = **−100к** 🔴
- Полный поток: 200к − 100к − 200к = **−100к** 🔴
- Базовый = полный (нет нерегулярных → нет разницы)

**Атака:** Система ломается?
Нет. Система **менее информативна** — базовый не отфильтровал разовую покупку. Но не врёт: пользователь действительно потратил 200к. Это верно в рамках «всё регулярное».

Школьный момент при резком падении потока:
> «Твой поток упал на 200к. Если это разовая покупка — пометь категорию "Мебель" как нерегулярную. Базовый поток покажет твой обычный уровень расходов отдельно.»

Graceful degradation: без настройки — менее полезно, но не сломано. ✅

**Результат: ✅ Держится.** Не ломается — просто не раскрывает потенциал. Школьный момент подталкивает к правильной настройке.

## Сценарий 8: Множественные нерегулярные расходы в один месяц

**Исходные данные:**
- Доход: 200к (регулярный), обычные расходы: 100к (регулярные)
- В один месяц: мебель 150к + техника 80к + подарки 50к = 280к (все нерегулярные)
- Кэш до: 600к

**Расчёт метрик:**
- Базовый поток: 200к − 100к = **+100к** 🟢
- Полный поток: 200к − 100к − 280к = **−180к** 🔴
- Кэш после: 600к + 200к − 100к − 280к = **420к**
- Запас до: 600к / 100к = **6 мес**. После: 420к / 100к = **4.2 мес** 🟡

**Атака: Зелёный базовый + падающий Запас = противоречие?**

Пользователь видит:
```
Прошлый месяц:  ПОТОК +100к 🟢    ЗАПАС 6 мес 🟢
Этот месяц:     ПОТОК +100к 🟢    ЗАПАС 4.2 мес 🟡
```

Поток зелёный, но Запас упал на 1.8 мес. Пользователь: «Почему Запас падает, если Поток зелёный?»

Ответ: «Базовый поток показывает регулярный уровень жизни — он в порядке. Запас упал из-за разовых покупок на 280к (мебель + техника + подарки). Это не проблема — покупки одноразовые. Запас восстановится через 3 месяца при текущем потоке.»

**Проверяю:** Запас восстанавливается? При потоке +100к/мес (кэш +100к ежемесячно): 420к → 520к → 620к → 720к. Через 2 месяца Запас = 620к / 100к = 6.2 мес. Восстановился. ✅

**Нюанс:** нужен школьный момент, связывающий падение Запаса с нерегулярными расходами. Без объяснения пользователь не поймёт расхождение. Полный поток −180к — туда и смотреть.

**Результат: ⚠️ Нюанс.** Система не ломается, но UX обязан связать падение Запаса с нерегулярными расходами. Без школьного момента — путаница. С ним — мощный педагогический момент: «Разовые траты не изменили твой уровень жизни, но съели кэш. Запас восстановится.»

## Сценарий 9: Продажа актива + покупка нового

**Исходные данные:**
- Доход: 200к (регулярный), расходы: 100к (регулярные)
- Продал старую машину: 500к (категория «Продажа авто», нерегулярный доход)
- Купил новую: 800к (категория «Авто покупка», нерегулярный расход)
- Доплата из своих: 300к

**Расчёт метрик:**
- Базовый поток: 200к − 100к = **+100к** (обе операции исключены — нерегулярные) ✅
- Полный поток: (200к + 500к) − (100к + 800к) = 700к − 900к = **−200к** (отражает доплату 300к + профицит 100к − 200к... нет)

Проверяю: реальное изменение кэша = +200к(зп) − 100к(расходы) + 500к(продажа) − 800к(покупка) = −200к. Полный поток = −200к. ✅ Верно: кэш уменьшился на 200к (300к доплата − 100к профицит).

- Капитал: старая машина (−500к) → новая (+800к) = +300к активов. Кэш: −200к. Нетто: **+100к** (= месячный профицит)
- Запас: кэш снизился → Запас снизился. ✅

**Атака:** Продажа и покупка в разные месяцы.
Месяц 1: продажа 500к. Полный = 200к + 500к − 100к = +600к. Кэш: +600к.
Месяц 2: покупка 800к. Полный = 200к − 100к − 800к = −700к. Кэш: −700к.

Дикие качели в полном потоке: +600к → −700к. Но базовый стабилен: +100к → +100к. ✅ Базовый поток корректно фильтрует. Полный поток честно показывает кэшфлоу.

Среднее полного за 2 месяца: (+600 − 700) / 2 = −50к/мес. Корректно: суммарно кэш уменьшился на 100к за 2 месяца (300к доплата − 200к профицит). ✅

**Результат: ✅ Держится.** Базовый чист от обеих операций. Полный честен. Капитал корректен.

## Сценарий 10: Рассрочка на телефон (мелкая)

**Исходные данные:**
- Доход: 200к, расходы: 100к (регулярные)
- Телефон 60к, рассрочка 0%, 12 мес, платёж 5к/мес
- Рассрочка = кредитный счёт с monthly_payment

**Расчёт метрик (мультиметрика из дебатов):**

В принятой мультиметрике: покупка в рассрочку = **расход в момент покупки** (60к). Платежи по рассрочке = **переводы** (тело, процентов нет).

- Месяц покупки:
  - Базовый поток: 200к − 100к − 60к(телефон, если регулярная) = +40к. Или если «Электроника» нерегулярная: 200к − 100к = +100к.
  - Полный поток: 200к − 100к − 60к = +40к
- Месяцы 2–12:
  - Платёж 5к = перевод (тело). Проценты = 0 (рассрочка 0%).
  - Базовый: 200к − 100к = +100к
  - Полный: 200к − 100к = +100к (платёж = перевод, не расход)

**Атака (КРИТИЧЕСКАЯ): Формула из промпта vs модель из дебатов.**

Промпт определяет: «Базовый поток = рег. доходы − рег. расходы − **платежи по кредитам**».

Если «платежи по кредитам» = полный платёж (тело + проценты):
- Рассрочка 5к/мес → в базовом потоке. Базовый = 200к − 100к − 5к = +95к (мес 2–12).
- Итого в потоке: 60к(покупка) + 12 × 5к(платежи) = **120к за 60к телефон. Двойной счёт!**

Если «платежи по кредитам» = только проценты (как в дебатах):
- Рассрочка 0% → 0 процентов. Базовый = 200к − 100к = +100к (мес 2–12).
- Итого в потоке: 60к(покупка) + 0(проценты) = **60к. Корректно.** ✅

**Диагноз:** формула в промпте неточна. «Платежи по кредитам» нужно читать как «**проценты** по кредитам + обязательный минимальный платёж (для кредиток)». Тело = перевод. Иначе — двойной счёт для любой рассрочки и для любого кредита, где покупка уже учтена как расход.

**Проверяю для потребкредита:** кредит 500к, покупка ремонта за 500к = расход. Платёж 15к/мес (12к тело + 3к проценты). Если полный платёж в потоке: 500к + 36 × 15к = 500к + 540к = 1.04М за 500к кредит. Двойной счёт тела.

Если только проценты: 500к + 36 × 3к = 500к + 108к = 608к. Корректно: стоимость покупки + стоимость кредита. ✅

**Но есть исключение: ипотека и автокредит в сценарии 4.**

Для автокредита, если покупка машины = балансовая операция (не расход), то платёж (тело + проценты) = единственный «расход» по кредиту. Тогда полный платёж корректен в базовом потоке. Двойного счёта нет, потому что покупка не была расходом.

**Итого — два режима в зависимости от того, была ли покупка расходом:**

| Тип кредита | Покупка = расход? | В базовом потоке |
|---|---|---|
| Рассрочка (Сплит, 0%) | Да (покупка в момент оформления) | Только проценты (= 0) |
| Потребкредит + расходы | Да (расходы из кредитных денег) | Только проценты |
| Автокредит (банк → дилер) | Нет (балансовая операция) | Полный платёж (тело + %) |
| Ипотека | Нет (балансовая операция) | Полный платёж (тело + %) |

Правило: **если покупка уже учтена как расход → тело платежа = перевод. Если покупка = балансовая операция → тело платежа = расход в потоке.**

Это согласуется с принципом: каждый рубль считается ОДИН раз.

**Результат: ⚠️ Нюанс — двойной счёт при рассрочках.** Решаемо правилом: «тело = перевод, если покупка была расходом». Для ипотеки/автокредита (покупка = баланс): полный платёж в потоке. Нужно формализовать.

## Сценарий 11: Бонус + крупная покупка в один месяц

**Исходные данные:**
- Обычный доход: 200к (регулярный, «Зарплата»)
- Бонус: 300к (нерегулярный, «Бонус»)
- Техника: 250к (нерегулярный, «Крупная техника»)
- Обычные расходы: 200к (регулярные)

**Расчёт метрик:**
- Базовый поток: 200к − 200к = **0к** (бонус и техника исключены — нерегулярные)
- Полный поток: (200к + 300к) − (200к + 250к) = **+50к**
- Капитал: кэш +50к (если техника — потребление, не актив). **+50к**
- DTI: 0%

**Атака:** Базовый = 0 (break even), полный = +50к. Базовый < полного — необычно.
Обычно полный ≤ базового (нерегулярные расходы тянут полный вниз). Здесь наоборот: нерегулярный доход (бонус 300к) > нерегулярного расхода (техника 250к). Нетто нерегулярных = +50к.

Это нормально? Да. Пользователь получил бонус больше, чем потратил на технику. Полный поток +50к — честный чистый кэшфлоу. Базовый 0 — его регулярный уровень (тратит всё, что зарабатывает регулярно). Оба числа осмысленны. ✅

**Атака:** Базовый = 0. FI-score: поток-компонент низкий. Но пользователь в этом месяце в плюсе (+50к).
FI-score по скользящему среднему базового. Если lifestyle = 200к/200к = 0, то FI-score справедливо показывает: «регулярно ты на нуле». Бонусы не гарантированы. ✅

**Результат: ✅ Держится.** Оба потока осмысленны. Базовый < полного — допустимо при нетто-положительных нерегулярных.

## Сценарий 12: Пользователь создаёт и архивирует категории

**Исходные данные:**
- Создал «Ремонт кухни» (нерегулярная), использовал 3 месяца (50 транзакций)
- Заархивировал категорию
- Создал «Ремонт ванной» (нерегулярная)

**Атака 1:** Транзакции старой категории сохраняются?
Архивация = категория скрыта из выбора, но существует. Транзакции привязаны → сохраняются в истории. ✅

**Атака 2:** Базовый поток за прошлые месяцы корректен?
«Ремонт кухни» была нерегулярной → расходы исключены из базового. После архивации: категория по-прежнему нерегулярная → прошлые расчёты не меняются. ✅

**Атака 3 (КРИТИЧЕСКАЯ): Нерретроактивность — как обеспечить?**

Промпт: «Переключение НЕ ретроактивно». Но если базовый поток **вычисляется динамически** (query-time), то текущий флаг категории определяет расчёт за ВСЕ месяцы.

Сценарий: пользователь создал «Подарки» как регулярную. За 6 мес — 60 транзакций по 5к = 30к/мес в базовом потоке. Потом переключил на нерегулярную. Если расчёт динамический → все 6 месяцев базового потока пересчитаны: +30к/мес выше. Исторические данные **изменились задним числом**.

Проблемы ретроактивного пересчёта:
- Среднее за 6 мес скачком меняется
- FI-score пересчитывается за все прошлые месяцы
- Пользователь не понимает, почему «вчерашние» графики изменились

**Решение: хранить флаг на уровне транзакции (snapshot).**

При создании транзакции: `is_regular = category.is_regular` (копируем текущий флаг). Расчёт базового потока использует `transaction.is_regular`, а не `category.is_regular`.

Переключение категории влияет только на **будущие** транзакции. Прошлые — заморожены.

Альтернатива: хранить историю флага на категории (`flag_changed_at`). При расчёте за месяц март — использовать флаг, актуальный в марте. Сложнее, но не дублирует данные.

**Рекомендация:** snapshot на транзакции (`is_regular` boolean). Проще, надёжнее, один JOIN.

**Результат: ⚠️ Нюанс — ретроактивность.** Если не хранить snapshot флага — переключение категории пересчитывает историю. Решение: `is_regular` на транзакции. Критично для реализации, не для архитектуры.

### Три найденные архитектурные проблемы

#### 1. Проект + нерегулярность = двойная коррекция (сценарий 2)

**Проблема:** два механизма решают одну задачу (убрать проектные расходы из базового). При совместном применении — двойной вычет.

**Решение:** приоритет нерегулярности. Если проектные категории нерегулярные → Проект = UX-только (иконка, прогресс, школьные моменты), метрики не корректирует. Коррекция Проекта применяется только к расходам в **регулярных** категориях.

**Правило:** `project.metric_correction_active = any(project_expenses in regular_categories)`

#### 2. Тело платежа: расход или перевод? (сценарий 10)

**Проблема:** если покупка уже учтена как расход + тело платежа тоже в потоке → двойной счёт. 60к телефон = 120к в потоках.

**Решение:** правило «каждый рубль — один раз».

| Покупка была расходом? | Тело платежа | Проценты |
|---|---|---|
| Да (рассрочка, потребкредит) | **Перевод** (не в потоке) | Расход (в потоке) |
| Нет (ипотека, автокредит через банк) | **Расход** (в потоке) | Расход (в потоке) |

Критерий: есть ли matching покупка-расход. Если кредит = disbursement → покупка с дебета (расход) → тело = перевод. Если кредит = прямая оплата банком дилеру (нет покупки-расхода) → тело = расход.

#### 3. Нерретроактивность: snapshot на транзакции (сценарий 12)

**Проблема:** при динамическом расчёте переключение флага категории пересчитывает историю.

**Решение:** `Transaction.is_regular: bool` — snapshot флага категории на момент создания транзакции. Расчёт базового потока использует `transaction.is_regular`, а не `category.is_regular`.

---

### Дополнительная находка: определение полного потока (сценарий 4)

**Проблема:** «все доходы − все расходы» некорректно при кредитных покупках. Disbursement ≠ доход, но покупка = расход → асимметрия.

**Решение:** определить полный поток как **чистое движение кэша**: все поступления − все списания, включая disbursement как поступление. Тогда disbursement (+1.2М) и покупка (−1.2М) взаимогасятся. Полный поток = 200к − 120к − 25к = +55к. Честный кэшфлоу.

Формальное определение: **Полный поток = Σ(кэш-приходы) − Σ(кэш-расходы)** за месяц, включая ВСЕ движения по ликвидным счетам. Это не «доходы − расходы», а «delta кэша за период».
