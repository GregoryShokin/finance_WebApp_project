# Дебаты: Операционные слабости модели рассрочки

## Участники
- **АРХИТЕКТОР** — предлагает решения. Помнит 14 отвергнутых подходов и не повторяет их.
- **КРИТИК** — атакует каждое предложение конкретными сценариями с цифрами.

## Контекст

Модель рассрочки из `Дебаты — Карта рассрочки.md` прошла 12 сценариев — метрики сходятся. Три операционные проблемы не были затронуты:

1. **Ввод данных:** откуда InstallmentPurchase? Выписка показывает только «Халва 5000₽».
2. **Разорванные категории:** покупки за кэш — в аналитике, покупки в рассрочку — нет.
3. **Конвертация:** покупка на кредитке → банк предложил рассрочку → двойной учёт.

Релевантные принятые решения:
- Кредитка: выбор пользователя «покупка = расход» / «платёж = расход» (✅ принято)
- Budgetpeer: «Кредитные платежи» → при клике разбивка по категориям (✅ интегрировано)
- Тегирование потребкредита: система спрашивает «Это покупка из кредитных денег?» (✅ принято)
- Ретроспективная трансформация (❌ отвергнута): прошлое не меняется
- Фантомные транзакции is_financed (❌ отвергнуты): усложнение без пропорциональной выгоды

## Раунд 1: Ввод данных — откуда берутся InstallmentPurchase

### Архитектор

**Предлагаю: Гибридная модель (Вариант В).**

Суть: агрегатные поля на Account — первичные (пользователь вводит), покупки — опциональное обогащение.

**Минимальный вход (всё, что нужно для метрик):**
1. Создать счёт `installment_card`
2. Ввести: `monthly_payment` (ежемесячный платёж), `credit_current_amount` (остаток долга)
3. Опционально: `credit_limit_original` (лимит)

Всё. Метрики работают: DTI считает monthly_payment, Капитал считает долг, Поток включает credit_payment, Запас учитывает отток.

**Обогащение покупками (когда пользователь хочет):**
- Добавляет InstallmentPurchase (описание, сумма, срок, ставка, категория)
- Система валидирует: «Ваши покупки дают платёж 15к/мес, но на счёте указано 20к. Возможно, есть неучтённые рассрочки.»
- Если сумма покупок ≈ агрегату → автоматический пересчёт агрегатов из детей
- Если нет покупок → агрегаты = введённые пользователем значения

**Изменение модели данных по сравнению с прошлыми дебатами:**

Было (из `Дебаты — Карта рассрочки.md`):
```
Account.monthly_payment      — COMPUTED: Σ purchase.monthly_payment
Account.credit_current_amount — COMPUTED: Σ purchase.remaining_amount
```

Стало:
```
Account.monthly_payment      — USER-ENTERED (primary), validated vs purchases if they exist
Account.credit_current_amount — USER-ENTERED (primary), validated vs purchases if they exist
```

Покупки остаются в модели, но из обязательных стали опциональными. Поля Account — не вычисляемые, а вводимые пользователем (как для обычного кредита).

**Школьные моменты — два уровня:**

| Уровень | Данные | Пример |
|---------|--------|--------|
| Базовый (без покупок) | Только агрегаты | «Ваш платёж по рассрочкам 20к/мес = 13% дохода» |
| Детальный (с покупками) | Покупки + агрегаты | «5 рассрочек: телефон, стиралка... Общий платёж 20к/мес. Могли бы накопить за 2 мес» |

Базовый уровень работает ВСЕГДА. Детальный — только когда пользователь добавил покупки.

### Критик

**Атака 1: Сценарий 13 — 5 рассрочек, только выписка по дебету**
- Пользователь добавил Халву, импортировал выписку с дебетового счёта
- В выписке: 5 строк «Халва, 4 000₽» (списания за месяц)
- Покупок не знает (не помнит, когда брал)

С гибридной моделью:
- Import распознаёт 5 транзакций как `credit_payment` на счёт Халвы
- Система: monthly_payment = 20к (сумма за месяц). Пользователь подтверждает или корректирует.
- credit_current_amount: пользователь вводит вручную (смотрит в приложении банка) или оставляет пустым.

Метрики:
- DTI = 20/150 = 13.3% ✅
- Поток: −20к credit_payment ✅
- Капитал: −долг (если ввёл credit_current_amount) ✅
- Школьный момент (базовый): «Рассрочки = 20к/мес, 13% дохода» ✅
- Школьный момент (детальный): ❌ невозможен — нет данных о покупках

**Что теряет пользователь:** не видит разбивку «на что», не получает совет «мог бы накопить». Но метрики работают, базовый школьный момент есть.

**Критично ли?** Для семьи «Первый шаг» с DTI 42% главный вопрос — не «на что рассрочка», а «сколько я плачу банкам». Базовый уровень это покрывает.

✅ **Сценарий 13 — работает с гибридом на базовом уровне.**

**Атака 2: Сценарий 17 — Новичок добавил Халву, не знает про покупки**
- Открыл приложение, создал счёт «Халва» (installment_card)
- Система спрашивает: monthly_payment? credit_current_amount?
- Новичок: «А откуда мне знать?»

**Два пути:**
1. Пользователь смотрит в приложении банка → вводит (трение, но одноразовое)
2. Пользователь импортирует выписку с дебета → система видит credit_payment → предлагает: «Похоже, ваш ежемесячный платёж по Халве ~20к. Верно?»

Путь 2 снижает трение: не нужно лезть в другое приложение. Но нужен хотя бы один импорт с дебетового счёта.

**Что показывает дашборд, если пользователь ничего не ввёл?**
- monthly_payment = 0 → DTI не учитывает Халву → показатели завышены
- credit_current_amount = 0 → Капитал не знает про долг → завышен

**Это проблема.** Пустой счёт = молчащие метрики. Но это общая проблема для ЛЮБОГО кредитного счёта (если добавить ипотеку без monthly_payment — тот же эффект).

**Решение:** при создании installment_card — monthly_payment обязательное поле. «Сколько вы платите по этой карте в месяц? (Посмотрите последнее списание в банковском приложении.)»

✅ **Сценарий 17 — работает. monthly_payment обязательно при создании.**

**Атака 3: Мёртвая фича? Будет ли кто-то вбивать покупки?**
- Гибрид означает: InstallmentPurchase таблица существует, код написан, но возможно никто не добавляет покупки.
- Если < 5% пользователей добавляют покупки — стоит ли вообще реализовывать InstallmentPurchase?

**Контраргумент:**
- Стоимость: InstallmentPurchase — одна таблица, один CRUD-сервис. Не rocket science.
- Ценность: даже если 5% добавляют — для них школьные моменты 11 и 12 работают.
- Будущее: импорт выписок по карте рассрочки (Вариант Г) может заполнять покупки автоматически. Инфраструктура уже будет готова.

**Но:** реализовывать InstallmentPurchase в MVP или нет — это вопрос приоритетов. Если MVP — можно начать с aggregate-only (по сути Вариант В из старых дебатов), а InstallmentPurchase добавить в следующей итерации. Модель данных это позволяет: поля Account уже не вычисляемые.

### Решение раунда
- **Принято:** Гибридная модель. Агрегаты на Account — первичные (пользователь вводит), покупки — опциональное обогащение.
- **Принято:** monthly_payment — обязательное поле при создании installment_card.
- **Принято:** Импорт дебетовой выписки может подсказать monthly_payment из credit_payment транзакций.
- **Принято:** Школьные моменты в двух уровнях: базовый (всегда) и детальный (с покупками).
- **Принято:** Валидация: если покупки добавлены, сверять их сумму с агрегатами. Предупреждение при расхождении.
- **Уточнено:** Account.monthly_payment и credit_current_amount — НЕ вычисляемые (меняет решение прошлых дебатов). Вычисляемые — только когда покупки полностью покрывают агрегат.
- **Отложено (post-MVP):** Импорт выписки по самой карте рассрочки (Вариант Г) — отдельная интеграция для каждого банка. Инфраструктура InstallmentPurchase будет готова.
- **Пользователь теряет (без покупок):** детальные школьные моменты 11 и 12, разбивку «на что рассрочка», совет «мог бы накопить».

## Раунд 2: Разорванные категории

### Архитектор

Проблема: покупки за кэш — в аналитике расходов, покупки в рассрочку — в отдельном разделе. Нет единой картины «сколько потратил на электронику».

**Контекст:**
- Фантомные транзакции (is_financed) — отвергнуты (❌)
- Для кредитки: принят «выбор пользователя» (покупка = расход / платёж = расход)
- Открытый вопрос: логично ли применять «покупка = расход» к рассрочке?

**Предлагаю: Вариант А (оставить как есть) + аннотация в аналитике.**

Почему НЕ «покупка = расход» для рассрочки:
1. Физическая разница с кредиткой: на кредитке транзакция СУЩЕСТВУЕТ (merchant charge). На рассрочке — нет. Для «покупка = расход» пришлось бы создавать транзакцию из воздуха → фантомная транзакция → отвергнуто.
2. Shock-month: 60к телефон в расходах месяца ломает аналитику. Для кредитки это принято как «пользователь должен почувствовать решение». Но для рассрочки вся модель построена на аналогии «банк → магазин» (как ипотека). Менять аналогию — ломать консистентность.

**Аннотация:**

Когда у пользователя есть installment_card с добавленными покупками (InstallmentPurchase с category_id), аналитика показывает:

```
Электроника      5 000₽
  + в рассрочку   60 000₽ (Халва: iPhone 15)
```

Без добавленных покупок:

```
Электроника      5 000₽

ⓘ Также: рассрочки 20 000₽/мес (без категорий — добавьте покупки для детализации)
```

Аннотация — не фантомная транзакция. Это подпись в интерфейсе, которая ссылается на InstallmentPurchase. Не влияет на Поток, не меняет формулы.

### Критик

**Атака 1: Сценарий 14 — Всё покупает в рассрочку**
- Доход 200к, коммуналка 8к (дебет), продукты 15к (дебет)
- В рассрочку за месяц: телефон 60к, стиралка 40к, одежда 30к, мебель 70к = 200к
- Аналитика расходов: **23к** (коммуналка + продукты)
- Реальные «приобретения»: **223к**

Аналитика показывает 23к. Пользователь в шоке: «Я же на 200к набрал!»

С аннотацией (если покупки добавлены):
```
Коммуналка           8 000₽
Продукты            15 000₽
Электроника              0₽
  + в рассрочку     60 000₽ (телефон)
  + в рассрочку     40 000₽ (стиралка)
Одежда                   0₽
  + в рассрочку     30 000₽
Дом                      0₽
  + в рассрочку     70 000₽ (мебель)
───────────────────────────
Итого расходов:     23 000₽
Итого в рассрочку: 200 000₽
```

Без покупок:
```
Коммуналка           8 000₽
Продукты            15 000₽
───────────────────────────
Итого расходов:     23 000₽

ⓘ Также: рассрочки на Халве (20к/мес платёж)
```

**Проблема:** «Итого расходов: 23к» при реальных тратах 223к — это технически верно (23к реально ушло с дебета), но **психологически ложь**. Пользователь чувствует, что приложение скрывает правду.

**Аннотация помогает?** Да, если пользователь её прочитает. Но «23к» в заголовке кричит громче, чем мелкий шрифт аннотации.

**Альтернатива — два заголовка:**
```
Расходы за месяц:           23 000₽
Новые обязательства:       200 000₽
```

Это честнее. Пользователь видит обе цифры одинакового веса.

✅ **Аннотация работает, но нужен второй заголовок для новых рассрочек.**

**Атака 2: Сценарий 16 — «Первый шаг» + Халва + кредитка**
- Доход 120к
- Кредитка (режим «покупка = расход»): продукты 40к, бензин 10к, рестораны 5к = 55к
- Халва (3 рассрочки): платёж 10к/мес
- Ипотека: 30к, потребкредит: 15к

Аналитика расходов:
- Продукты 40к ✅ (кредитка, «покупка = расход»)
- Бензин 10к ✅
- Рестораны 5к ✅
- Коммуналка, подписки... допустим ещё 10к с дебета
- **Итого расходов: ~65к** (видны категории)

Кредитные платежи:
- Ипотека 30к, потребкредит 15к, рассрочки 10к = 55к

**Что видит пользователь:**
- Расходы: 65к (категории видны, кредитка в «покупка = расход»)
- Кредитные платежи: 55к (из них рассрочки 10к)
- Базовый поток: 120 − 65 − 55 = 0 → Показатель 0% → 🟡
- DTI = 55/120 = 46%

**Где пробелы:**
- Покупки по Халве (на что рассрочка) — не в основной аналитике. Только в аннотации или на карточке Халвы.
- Если покупки не добавлены — даже аннотация не покажет «на что».

**Критично?** Для «Первого шага» главное — видеть DTI 46% и Показатель 0%. Детализация рассрочек — nice to have, но не критична для осознания масштаба проблемы.

✅ **Мультиметрика ловит главное. Детализация рассрочек — вторична.**

**Атака 3: Не стоит ли пересмотреть отказ от фантомных транзакций?**
- Фантомная транзакция (is_financed) решила бы проблему: покупка 60к появляется в аналитике с флагом, исключена из Потока.
- Отвергнута как «усложнение без пропорциональной выгоды».

Но теперь видно: аналитика пустая для пользователя, который ВСЁ берёт в рассрочку (сценарий 14). Это не «непропорциональная выгода» — это дыра в UX.

**Контраргумент:** фантомная транзакция — это транзакция без реального движения денег. Она ломает принцип «Поток = реальное движение кэша». Даже с флагом is_financed — это прецедент: в системе появляются транзакции, которым нет соответствия в банковской выписке. Это размывает доверие к данным.

**Аннотация + двойной заголовок = тот же результат без ломания принципа.** Пользователь видит обе суммы (расходы + рассрочки). Данные чистые. Принцип сохранён.

Да, аннотация — это UI-хак, а не архитектурное решение. Но она ДЕШЕВЛЕ фантомных транзакций и не создаёт прецедентов.

### Решение раунда
- **Принято:** Аннотация в аналитике — InstallmentPurchase (если добавлены) показываются под соответствующими категориями с пометкой «в рассрочку».
- **Принято:** Двойной заголовок на странице аналитики: «Расходы: X₽» и «Новые обязательства (рассрочки): Y₽» — оба видны одинаково.
- **Принято:** Без добавленных покупок — общая пометка: «Также: рассрочки N₽/мес платёж».
- **Принято:** Школьный момент: «Аналитика показывает реальные денежные расходы. Покупки в рассрочку — это новые обязательства, не расход в момент покупки. Но они увеличивают ваш платёж и DTI.»
- **Отвергнуто (повторно):** Фантомные транзакции — ломают принцип «Поток = реальное движение кэша», создают прецедент.
- **Отвергнуто:** «Покупка = расход» для рассрочки — нет реальной транзакции на счёте, ломает аналогию «банк → магазин», ломает консистентность с ипотекой/автокредитом.
- **Отложено (post-MVP):** Переключатель «Показывать рассрочки в аналитике» — объединённый режим для продвинутых пользователей.
- **Пользователь теряет:** единую картину «сколько всего потратил на электронику» в одном числе. Компенсируется двойным заголовком и аннотациями.

## Раунд 3: Конвертация покупки в рассрочку

### Архитектор

Сценарий: пользователь купил ТВ за 80к на кредитке («покупка = расход»). Через неделю банк конвертировал в рассрочку 6×14к.

**Ключевой анализ — зависит от режима кредитки:**

**Режим «платёж = расход»:**
- Покупка 80к — НЕ в расходах (только в категорийной аналитике)
- Конвертация: долг переходит с кредитки на рассрочку
- Пользователь: уменьшает баланс кредитки на 80к, создаёт InstallmentPurchase 80к
- Капитал: кредитный долг −80к, рассрочный долг +80к → **нетто 0**
- DTI: кредитный платёж уменьшается, рассрочный +14к → DTI обновляется
- Поток: credit_payment по кредитке уменьшается, credit_payment по рассрочке +14к → **корректно**
- **Никаких проблем.** ✅

**Режим «покупка = расход»:**
- Покупка 80к — УЖЕ в расходах (категория «Электроника», −80к в Потоке)
- Кредитная карта: погашение 80к было бы transfer (не влияет на Поток)
- После конвертации: рассрочка 14к/мес → credit_payment → ВХОДИТ в Поток
- **Двойной учёт в Потоке:** 80к (расход) + 84к (6×14к платежей) = 164к вместо корректных 84к.
- **Двойной учёт в Капитале:** 80к ушло как расход (уменьшило кэш кредитки → долг уже учтён). Создание InstallmentPurchase добавляет ещё 80к долга → Капитал −160к вместо −80к.

Для режима «покупка = расход» нужен механизм коррекции.

**Предлагаю:** конвертация как действие пользователя в два шага.

**Шаг 1:** На оригинальной транзакции (80к на кредитке) — кнопка «Конвертировано в рассрочку». Транзакция получает флаг `converted_to_installment = true` → исключается из расходной аналитики и Потока.

**Шаг 2:** Пользователь создаёт InstallmentPurchase (80к, 6 мес, 14к/мес). Стандартная логика: долг +80к, ежемесячные credit_payment.

**Результат:**
- Поток: −80к расход убран, +14к/мес credit_payment → суммарно 84к. ✅
- Капитал: расход убран → кэш восстановлен на 80к. InstallmentPurchase → долг +80к. Нетто: корректно. ✅
- DTI: +14к/мес. ✅

**Ограничение:** флаг `converted_to_installment` меняет прошлую транзакцию. Это нарушает принцип «прошлое не меняется»?

**Аргумент «нет»:**
1. Это не системный ретроспективный пересчёт (отвергнутый подход #5). Это точечная коррекция одной транзакции пользователем.
2. Финансовый факт реально изменился — банк реструктурировал долг. Транзакция «покупка за 80к» перестала быть расходом и стала рассрочкой.
3. Инициировано пользователем, не системой.

### Критик

**Атака 1: Сценарий 15 — Двойной учёт в Капитале**
- Кредитка «покупка = расход». Купил ТВ 80к.
- Состояние ДО конвертации: кредитный долг 80к. Расход 80к в аналитике.
- Банк конвертирует в рассрочку 6 × 14к.

Без действий пользователя:
- Капитал: кредитный долг 80к (старый) + рассрочный долг 80к (новый) = **160к долга** вместо 80к
- DTI: кредитный платёж по-прежнему считается + рассрочный 14к = завышен

С действиями пользователя (по предложению Архитектора):
1. Маркирует транзакцию 80к → `converted_to_installment`
2. Корректирует баланс кредитки (−80к долга, банк убрал)
3. Создаёт InstallmentPurchase (80к, 6 мес, 14к)

Результат:
- Кредитный долг: −80к. Рассрочный долг: +80к. Капитал: нетто 0 от конвертации. ✅
- DTI: кредитный платёж уменьшился, рассрочный +14к. ✅
- Поток: расход 80к убран, credit_payment 14к/мес добавлен. ✅

**Работает, но требует 3 осознанных действия от пользователя.** Семья «Первый шаг» справится? Это не тривиальная операция.

**Контраргумент:** конвертация — нечастое событие. И пользователь, который конвертирует покупку в рассрочку, по определению взаимодействует с банковским приложением. Он уже в контексте «управление долгом». Три шага — приемлемо для нечастого сценария.

✅ **Работает при участии пользователя.**

**Атака 2: Сценарий 18 — Конвертация в прошлом месяце**
- Январь: купил ТВ 80к на кредитке. Расход в январской аналитике.
- Февраль: банк конвертировал в рассрочку. Пользователь маркирует транзакцию.

Что происходит:
- Январская аналитика: расход 80к → пропадает (транзакция marked as converted)
- Январский Поток: был −80к → стал 0 (без этого расхода)
- **Прошлое изменилось.**

**Проблема:** пользователь в январе видел −80к и, возможно, принял решения на основе этого. Теперь январь другой. Это ретроспективная трансформация?

**Контраргумент Архитектора:** это не то же, что подход #5 (массовый пересчёт задним числом). Это одноразовая коррекция конкретного факта, который реально изменился. Разница:
- Подход #5: система сама пересчитывает прошлое (FIFO-очереди, ретро-распределение)
- Здесь: пользователь вручную корректирует одну транзакцию, потому что её природа объективно изменилась

**Но всё равно неприятно.** Если пользователь в январе увидел «расходы 180к» и подумал «нужно экономить», а потом январь стал «расходы 100к» — его решение экономить было основано на неправильных данных? Нет — решение экономить было правильным. Просто 80к из этих расходов стали обязательством, растянутым на полгода. Это ещё серьёзнее.

**Прагматичная альтернатива:** не менять прошлый месяц. Конвертировать начиная с текущего.

Тогда:
- Январь: расход 80к — остаётся
- Февраль: InstallmentPurchase создаётся с флагом `source: credit_card_conversion`, `pre_expensed: true`
- Для покупок с `pre_expensed`: credit_payment → классифицируется как transfer (не credit_payment)
- Поток: не получает дополнительных −14к/мес (потому что payment = transfer)
- DTI: monthly_payment на installment_card всё равно включает 14к → DTI корректен

**Проблема:** для этой конкретной рассрочки платежи НЕ входят в Поток (transfer), но входят в DTI (через monthly_payment). Это нестандартное поведение — для всех остальных рассрочек платежи = credit_payment.

**Ещё одна альтернатива:** не поддерживать конвертацию из прошлого месяца в MVP.

### Архитектор (уточнение)

Три подхода по временны́м рамкам:

| Конвертация | Действие | Прошлое |
|------------|----------|---------|
| В том же месяце | Маркировать + создать InstallmentPurchase | Не меняется (тот же период) |
| В прошлом месяце | **Вариант А:** маркировать (меняет прошлое) | Меняется |
| В прошлом месяце | **Вариант Б:** не маркировать, InstallmentPurchase с `pre_expensed` | Не меняется, но нестандартное поведение |
| В прошлом месяце | **Вариант В:** не поддерживать. Инструкция: «Удалите расход вручную и создайте рассрочку» | Меняется (ручная коррекция) |

**Предлагаю для MVP: Вариант В (ручная коррекция).**

Обоснование:
- Конвертация из прошлого месяца — редкий сценарий
- Ручная коррекция (удалить/изменить транзакцию + создать рассрочку) — понятна и прозрачна
- Не требует специального кода (pre_expensed флаги, нестандартные transfer'ы)
- Пользователь видит, что именно он поменял

Для текущего месяца — `converted_to_installment` флаг: дешёвая реализация, чистый результат.

### Решение раунда
- **Принято:** Режим «платёж = расход» — конвертация работает без проблем. Пользователь корректирует баланс кредитки и создаёт InstallmentPurchase.
- **Принято:** Режим «покупка = расход», конвертация в том же месяце — флаг `converted_to_installment` на транзакции (исключает из расходов/Потока) + создание InstallmentPurchase.
- **Принято (MVP):** Режим «покупка = расход», конвертация из прошлого месяца — ручная коррекция. Пользователь сам редактирует/удаляет старую транзакцию и создаёт рассрочку. Документируем инструкцию.
- **Отвергнуто:** Автоматическая конвертация из прошлого месяца — слишком сложно для MVP, требует нестандартного поведения платежей.
- **Отложено (post-MVP):** Автоматизированный flow конвертации с коррекцией прошлых периодов — когда будут данные о частоте этого сценария у реальных пользователей.
- **Пользователь теряет (при ручной коррекции из прошлого месяца):** автоматическую обработку. Должен знать, что нужно поправить данные. Компенсируется инструкцией и школьным моментом.

## Раунд 4: Перекрёстные эффекты и финальная сводка

### Архитектор

Три решения взаимодействуют. Проверим сценарий 16, который требует все три механизма одновременно.

**Сценарий 16: «Первый шаг» + Халва + кредитка**

Исходные данные:
- Доход 120к
- Кредитка (режим «покупка = расход»): продукты 40к, бензин 10к, рестораны 5к = 55к
- Халва: 3 рассрочки, платёж 10к/мес. Покупки не добавлены (пользователь не вбивал).
- Ипотека: 30к/мес
- Потребкредит: 15к/мес
- Другие расходы с дебета: 10к (коммуналка, подписки)

**Что пользователь ввёл при настройке:**
- Кредитку — выбрал «покупка = расход» (вопрос при онбординге)
- Халву — ввёл monthly_payment = 10к и credit_current_amount = 45к (обязательные поля)
- Покупки на Халву — не вбивал (гибридная модель: необязательно)

**Аналитика расходов:**

```
Продукты              40 000₽   ← кредитка (покупка = расход)
Бензин                10 000₽   ← кредитка
Рестораны              5 000₽   ← кредитка
Коммуналка             8 000₽   ← дебет
Подписки               2 000₽   ← дебет
───────────────────────────────
Расходы:              65 000₽

ⓘ Также: рассрочки Халва (10к/мес платёж)
```

Нет аннотации по категориям (покупки не добавлены). Но общая пометка есть.

**Метрики:**
| Метрика | Значение | Расчёт |
|---------|---------|--------|
| Базовый поток | 120 − 65 − 55 = 0₽ | рег.доход − рег.расходы − кредитные платежи |
| Показатель | 0% → 🟡 | 0/120 |
| DTI | (30+15+10)/120 = 46% | все credit_payment / доход |
| Капитал | зависит от активов/долгов | ипотечный долг + потребкредит + Халва (45к) + кредитка |

**Где пробелы:**
1. Рассрочки Халвы — без категорий (покупки не вбиты). Пометка есть, но «на что» — неизвестно.
2. Кредитка — категории видны (режим «покупка = расход»). ✅
3. DTI 46% — правильно учитывает и ипотеку, и потребкредит, и рассрочки. ✅
4. Показатель 0% — критическая зона. ✅

**Главное:** пользователь видит DTI 46% и Показатель 0%. Это ГЛАВНЫЙ сигнал. Детализация рассрочек — вторична. Все кредитные платежи (30+15+10=55к) видны в DTI. Школьный момент DTI: «46% дохода уходит на кредиты. Из них 10к/мес — рассрочки.»

### Критик

**Атака: Обновлённая модель — не противоречит ли прошлым дебатам?**

В `Дебаты — Карта рассрочки.md` принято:

3. Аналитика — плюс аннотация

**Нет критического противоречия.** Модель эволюционировала: из «всё обязательно» в «базовое обязательно, детали опциональны». Это прагматичная корректировка.

## Реестр решений

### Принято
1. **Гибридная модель ввода данных.** Агрегаты на Account — первичные (user-entered), покупки — опциональное обогащение. Школьные моменты в двух уровнях.
2. **monthly_payment — обязательное поле** при создании installment_card.
3. **Аннотация в аналитике.** InstallmentPurchase (если добавлены) показываются под категориями с пометкой «в рассрочку». Без покупок — общая пометка.
4. **Двойной заголовок.** «Расходы: X₽» и «Новые обязательства (рассрочки): Y₽» — оба видны.
5. **Конвертация в том же месяце.** Флаг `converted_to_installment` на транзакции + создание InstallmentPurchase.
6. **Конвертация из прошлого месяца (MVP).** Ручная коррекция с инструкцией.
7. **Конвертация в режиме «платёж = расход».** Работает без специальной логики.
8. **Валидация покупок vs агрегатов.** Если покупки добавлены — сверять суммы, предупреждать при расхождении.

### Уточнения к прошлым дебатам
9. **Account.monthly_payment и credit_current_amount — больше НЕ вычисляемые.** Первичные, вводимые пользователем. Вычисляемые — только когда покупки полностью покрывают агрегат.

### Отвергнуто
1. **Фантомные транзакции (повторно).** Ломают принцип «Поток = реальное движение кэша».
2. **«Покупка = расход» для рассрочки.** Нет реальной транзакции на счёте. Ломает аналогию с ипотекой/автокредитом.
3. **Автоматическая конвертация из прошлого месяца.** Слишком сложно для MVP, нестандартное поведение платежей.

### Отложено (post-MVP)
1. **Импорт выписки по карте рассрочки (Вариант Г).** Автозаполнение InstallmentPurchase. Нужна отдельная интеграция для каждого банка.
2. **Переключатель «Показывать рассрочки в аналитике».** Объединённый режим для продвинутых пользователей.
3. **Автоматизированная конвертация с коррекцией прошлых периодов.** Когда будут данные о частоте сценария.

### Что теряет пользователь
| Ограничение | Что теряет | Критичность |
|------------|-----------|------------|
| Покупки не добавлены | Детальные школьные моменты (11, 12), разбивку «на что», совет «мог бы накопить» | Средняя. Базовые школьные моменты работают. |
| Рассрочки не в аналитике расходов | Единую цифру «потратил на электронику» | Низкая. Двойной заголовок компенсирует. |
| Конвертация из прошлого месяца — ручная | Автоматическую обработку | Низкая. Редкий сценарий. |
