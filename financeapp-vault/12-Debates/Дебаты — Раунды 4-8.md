## Раунд 4 — Дашборд, edge cases проекта

### ЗАЩИТНИК:

Отвечаю на три открытых вопроса из раунда 3.

**1. Макет дашборда в ремонтные месяцы.**

Четыре карточки — четыре ответа на четыре вопроса. Никакой лишней информации.

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  ПОТОК       │  │  КАПИТАЛ     │  │  НАГРУЗКА    │  │  ЗАПАС       │
│  +95к  ✅    │  │  ↓ −60к/мес  │  │  0%     ✅   │  │  0.9 мес 🔴  │
│  (−55к всего)│  │  ⚠️ плановый │  │              │  │              │
│  📋 Ремонт  │  │              │  │              │  │              │
└──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘
```

Логика: главное число на карточке «Поток» — **базовый** поток (+95к), выделен зелёным. Мелко под ним — полный поток (−55к). Иконка проекта кликабельна.

FI-score считается по базовому потоку → ~6/10 (запас тянет вниз, остальное ок). Без проекта было бы ~4/10.

Школьный момент при активном проекте + запас < 1 мес:
> «У тебя активный проект "Ремонт". Базовый поток +95к — уровень жизни не изменился. Но запас 0.9 мес — следи за ним. Когда ремонт закончится, запас начнёт расти.»

После ремонта (месяц 4+): все 4 карточки честные, без пометок. Нагрузка 22.5% (жёлтая), запас восстанавливается.

**2. Edge cases проекта.**

- **Бюджет перерасходован:** проект не закрывается автоматически. Школьный момент: «Расходы превысили план. Пересмотреть бюджет?» Пользователь решает.
- **Закончился раньше:** пользователь закрывает вручную. Или система предлагает закрыть, если 0 расходов за месяц.
- **Несколько проектов:** раздельные бюджеты и сроки. Базовый поток = полный − Σ(бюджет_i / срок_i). При >2 проектах — школьное предупреждение о точности.

**3. Школа** — предлагаю Критику провести поурочный анализ (у него доступ к vault).

**Педагогический аргумент:** Эта семья — «Первый шаг» (DTI 42%, доход 120к). Что она видит?

| Метрика | Текущая модель | Мультиметрика |
|---|---|---|
| СМО / Поток | −35к 🔴 (одно красное число, страшно) | +5к 🟢 (уровень жизни еле-еле, но устойчив) |
| DTI / Нагрузка | 42% 🔴 | 42% 🔴 (та же правда) |
| Подушка / Запас | 0 🔴 | 0.3 мес 🔴 |
| — / Капитал | нет эквивалента | −380к 🔴 (долг больше активов) |

Текущая модель: СМО = −35к, потому что кредитные платежи 50к целиком идут в расход. Для семьи с реальным lifestyle-потоком +5к это **демотивирующая ложь** — они не тратят 35к больше, чем зарабатывают. Они еле сводят концы, но сводят.

Мультиметрика: Поток +5к честно говорит «ты на грани, но жив». Нагрузка 42% кричит — вот где проблема. Капитал показывает масштаб долга. Школа: «Твоё слабое звено — нагрузка. Сфокусируйся на снижении DTI.»

### КРИТИК:

Прежде чем соглашаться с дашбордом — атакую по чеклисту.

**Атака 1: Фиксированное распределение — шум в базовом потоке.**

Проект «Ремонт», бюджет 500к, 3 месяца. Коррекция = 167к/мес.

Но реальные расходы по месяцам: 150к, 200к, 150к (неравномерно).

| Месяц | Полный поток | Коррекция | Базовый поток | «Истинный» lifestyle |
|---|---|---|---|---|
| 1 | 200к − 255к = −55к | +167к | **+112к** | +95к |
| 2 | 200к − 305к = −105к | +167к | **+62к** | +95к |
| 3 | 200к − 255к = −55к | +167к | **+112к** | +95к |

Базовый поток скачет: 112к → 62к → 112к. «Истинный» lifestyle стабилен (+95к). **Разброс ±30к.** Пользователь видит жёлтый месяц 2 без причины — его обычная жизнь не изменилась, просто электрик пришёл раньше.

Это не фатально (среднее за 3 месяца = +95к, точно), но **помесячный базовый поток ненадёжен**. Если школьные триггеры или алерты завязаны на месячные значения — будут ложные срабатывания.

**Сценарий-вердикт:** атака частично успешна. Базовый поток точен **в среднем**, но **помесячно шумит**. Нужно или (а) использовать скользящее среднее базового потока для алертов, или (б) не привязывать алерты к месячным значениям при активном проекте.

**Атака 2: «Первый шаг» — три красных из четырёх.**

Семья: доход 120к, расходы 105к, кредитные платежи ~50к.

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  ПОТОК       │  │  КАПИТАЛ     │  │  НАГРУЗКА    │  │  ЗАПАС       │
│  +5к   🟡    │  │  −380к  🔴   │  │  42%    🔴   │  │  0.3 мес 🔴  │
└──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘
```

Три красных, один жёлтый. Защитник говорит «это мотивирует — видно, что чинить». Проверяю: человек с DTI 42% и нулевой подушкой **уже в стрессе** (принцип «За каждой цифрой — эмоция»). Три красных карточки — **усиливают** стресс или дают контроль?

Сценарий-вердикт: атака **не** проходит. В текущей модели семья видит СМО = −35к — одно число, непонятно что делать. В мультиметрике — три конкретных проблемы с конкретными действиями. Школа говорит: «Фокус — нагрузка 42%. Снижай её.» Это лучше, чем «СМО отрицательный, анализируй категории» (при том что категории в порядке — давит не lifestyle, а долг). **Соглашаюсь с Защитником** — мультиметрика точнее диагностирует проблему.

**Атака 3: FI-score и «проектный» gaming.**

Если FI-score считается по базовому потоку, пользователь может завысить его, создав «проект» с раздутым бюджетом. Берёт кредит 500к, заявляет проект «Инвестиции, 500к, 12 месяцев». Базовый поток корректируется на +42к/мес. FI-score растёт. Реальный поток не изменился.

Сценарий-вердикт: атака **теоретически валидна**, но **практически маловероятна**. Пользователь должен: (а) понять внутреннюю механику, (б) намеренно обмануть себя. ЦА (семья с DTI 42%) не будет этим заниматься. Для продвинутых — их проблема, система предупреждает честными метриками (капитал, нагрузка не обманешь). **Не блокер.**

**Атака 4: Пользователь НЕ создал проект.**

Берёт кредит 500к. Система предлагает: «Это проект?» Пользователь пропускает. Тратит 150к/мес на ремонт. Поток: −55к, −105к, −55к. Все три месяца красные. Алерты срабатывают. FI-score падает.

Какой fallback? Из bridge-cowork.md: «школьное напоминание, если поток упал после получения кредита». Но это **реактивный** механизм. Поток уже красный, алерт уже сработал, FI-score уже упал. Пользователь уже напугался.

Сценарий-вердикт: атака **проходит частично**. Fallback работает (напоминание через месяц), но **с задержкой**. Нужно усилить: при credit_disbursement система должна **настоятельно** предложить проект, не просто подсказку. Формулировка: «Вы получили кредит 500к. Чтобы аналитика учитывала это правильно — укажите, на что и на какой срок.» С предзаполнением из данных кредита.

**Атака 5: Два кредита + рассрочка одновременно.**

Кредит A: ремонт 500к / 3 мес. Кредит B: образование 200к / 6 мес. Рассрочка: холодильник 60к.

Коррекция = 167к + 33к = 200к/мес. При доходе 200к и расходах 105к:
- Полный поток с ремонтом + образованием: 200к − 105к − 150к(рем) − 30к(обр) − 60к(холод) = **−145к**
- Базовый поток: −145к + 200к(коррекция) = **+55к**

Но холодильник на рассрочке — в мультиметрике это расход в момент покупки (−60к), а платежи = переводы. Проект не покрывает рассрочку. Корректнее:

- Полный поток (месяц покупки холодильника): 200к − 105к − 150к(рем) − 30к(обр) − 60к(холод) = **−145к**
- Базовый: −145к + 200к = **+55к**

Это завышено — «истинный» lifestyle = 200к − 105к = +95к, а не +55к. Холодильник не вычтен. Базовый поток врёт на −40к.

Стоп. Холодильник — разовая покупка, не проект. Он ударит по потоку один раз. В следующем месяце (без холодильника): полный = 200к − 105к − 150к − 30к = −85к. Базовый = −85к + 200к = +115к. Опять шум.

Сценарий-вердикт: при множественных проектах + разовых кредитных покупках базовый поток **становится ненадёжным**. Но это **крайний** случай. Школьное предупреждение при >2 проектах корректно. **Не блокер**, но нужно документировать ограничение.

**Атака 6: Кешбэк-пользователь — проверяю.**

100% расходов через кредитку. Поток = 200к − 100к(покупки) = +100к. Погашение = перевод. Все категории видны. Обязательные/второстепенные работают.

Сценарий-вердикт: **чисто**. Проблема #1 не существует. Атака не нашла уязвимости.

**Атака 7: Долговая спираль — проверяю.**

Долг 350к, проценты 10к/мес, платёж 5к. Поток: +85к (хороший — проценты вычтены). Нагрузка: 2.5%. Капитал: растёт (но медленнее). Все четыре метрики «нормальные».

Сценарий-вердикт: **подтверждаю** — долговая спираль невидима без «Здоровья долга». Это было известно с раунда 1. Не новое, но **проект не решает эту проблему**. Здоровье долга обязательно в обеих архитектурах.

### Результат раунда 4:
- **Закрыто:** Дашборд — концепт убедителен. Педагогическое преимущество для «Первой шаг» подтверждено числами. Кешбэк — чисто.
- **Частично закрыто:** Базовый поток при проекте — точен в среднем, шумит помесячно. Решение: алерты не привязывать к месячным значениям при активном проекте. Fallback при отсутствии проекта — усилить промпт при credit_disbursement.
- **Подтверждено из раунда 1:** Долговая спираль — «Здоровье долга» обязательно в обеих архитектурах.
- **Баланс: 70/30 за мультиметрику** (дашборд убедителен, «Первый шаг» — сильный аргумент)

## Раунд 5 — Школа: поурочный анализ

### ЗАЩИТНИК:

Школа — один из оставшихся блокеров. Провожу поурочный анализ модулей 0–2 (модули 3–5 не затронуты — подушка, цели, инвестиции).

**Модуль 0 — Знакомство (бесплатный)**

| Урок | Суть | Затронут? | Что меняется |
|---|---|---|---|
| 0.1 «Где ты сейчас» | Пирамида как рамка | ✅ Да | Пирамида → Радар. «Четыре вопроса вместо одной лестницы» |
| 0.2 «Твои деньги, где они лежат» | Типы счетов | ❌ Нет | Типы счетов не меняются |
| 0.3 (импорт) | Техническая настройка | ❌ Нет | — |
| 0.4 (категории) | Настройка категорий | ❌ Нет | — |
| 0.5 «Твой финансовый портрет» | Первый ага-момент, воронка оплаты | ✅ Частично | Портрет показывает 4 метрики вместо СМО + DTI |

**Модуль 1 — Денежный поток → Поток и Капитал**

| Урок | Суть | Затронут? | Что меняется |
|---|---|---|---|
| 1.1 «Сколько ты зарабатываешь» | Классификация доходов | ❌ Нет | Регулярный/нерегулярный — без изменений |
| 1.3 «СМО — фундамент» | Объяснение СМО | ✅ **Полная переработка** | СМО → Поток. Новая формула. Другая философия учёта |
| 1.5 «Норма сбережений» | Адаптивная модель | ✅ Частично | Формула пересчитывается на основе Потока |
| 1.6 «Отрицательный СМО» | Что делать при минусе | ✅ Переработка | «Отрицательный поток» + контекст (проект, разовая покупка) |

**Модуль 2 — Кредитная нагрузка → Нагрузка**

| Урок | Суть | Затронут? | Что меняется |
|---|---|---|---|
| 2.1 «Сколько ты работаешь на банк» | DTI, метафора | ❌ Нет | Нагрузка = DTI. Те же зоны, та же метафора |
| 2.2 «Виды долгов» | Классификация | ❌ Нет | — |
| 2.4 «Высокая нагрузка» | Стратегии снижения | ❌ Нет | — |
| 2.5 «Кредитная карта — ловушка» | Грейс, проценты | ✅ Частично | Упрощается: нет двух режимов кредитки. Одна модель |

**Итого по модулям 0–2:**

| Статус | Количество | Уроки |
|---|---|---|
| Полная переработка | 2 | 0.1 (пирамида→радар), 1.3 (СМО→поток) |
| Частичная переработка | 3 | 0.5 (портрет), 1.5 (норма), 2.5 (кредитка) |
| Без изменений | 6+ | Все остальные |

**Главный аргумент:** из ~12 уроков модулей 0–2 полностью переписываются **два**. Три адаптируются. Остальные — без изменений. Школа в vault, не в коде. Стоимость: **2-3 дня работы**, не «переписать всю школу».

Педагогически радар **сильнее** пирамиды:
- Пирамида: линейная иерархия. «Сначала фундамент, потом этаж 1, потом этаж 2.» Но реальные люди не живут линейно — у кого-то хороший поток, но нулевой запас.
- Радар: «Вот твоя форма. Видишь перекос? Усиливай слабое звено.» Это уже описано в vault: «при нелинейных ситуациях контент адресует слабое звено, а не уровень пирамиды по порядку» ([[Пирамида финансовых приоритетов]]). Мультиметрика делает этот принцип **архитектурным**, а не исключением.

### КРИТИК:

Перед тем как согласиться с «2 урока — 3 дня работы» — атакую.

**Атака 1: Потеря метафоры пирамиды.**

Пирамида — мощная метафора. «Сначала фундамент, потом стены, потом крыша.» Любой человек понимает: без фундамента дом рухнет. Это интуитивно, не требует объяснений.

Радар — абстрактная геометрическая фигура. «Вот четыре оси, вот твоя форма.» Для человека с DTI 42% и финансовым стрессом — что сильнее: «построй фундамент» или «улучши форму радара»?

Проверяю на «Первом шаге»: семья, DTI 42%, СМО ~5к. Открывает приложение первый раз.

Пирамида: «Видишь? Фундамент красный. Давай сначала стабилизируем доходы-расходы. Потом займёмся долгами. Потом подушкой.»

Радар: «У тебя перекос — нагрузка и запас красные. Фокусируйся на нагрузке.»

Сценарий-вердикт: атака **частично проходит**. Радар информативнее, но пирамида **эмоционально проще** для тревожного человека. «Начни с фундамента» — одно действие. «У тебя три слабых звена» — паралич выбора.

**Но:** Защитник уже ответил на это — школа говорит «фокус: нагрузка» (одно действие). И текущая пирамида тоже использует принцип «слабое звено» для нелинейных случаев. Разница в том, что мультиметрика делает это **по умолчанию**, а не как исключение.

**Компромисс:** можно сохранить пирамидальную **метафору** для школы (модуль 0: «четыре этажа финансового здоровья: поток, долги, запас, рост»), но с радарной **механикой** (показатели равноправны, школа адресует слабейший). Пирамида как визуальный нарратив, радар как аналитический инструмент.

**Атака 2: Урок 1.3 — ключевой урок школы, не «один из двенадцати».**

Урок 1.3 «СМО — фундамент» — это не просто один урок. Это **центральное понятие**, на которое ссылаются все остальные уроки: «Помнишь СМО? Посмотри, как DTI влияет на твой СМО.» «Подушка — это X месяцев твоего СМО.» «Цель — 20% от СМО.»

При переходе на мультиметрику нужно заменить **все ссылки** на СМО во всех уроках на «Поток». Это не только урок 1.3 — это десятки перекрёстных ссылок.

Сценарий-вердикт: атака **валидна**, но масштаб переоценён. «Заменить СМО на Поток» — это поиск-и-замена в vault-файлах. Концепция та же (сколько остаётся в месяц), формула другая. Перекрёстные ссылки — механическая работа. **Не блокер**, но +1 день к оценке (итого 3-4 дня, не 2-3).

**Атака 3: Норма сбережений — формула меняется нетривиально.**

Текущая формула: Норма = СМО / Доход. В мультиметрике: Норма = Поток / Доход.

Но Поток в мультиметрике включает кредитные покупки как расход. У кешбэк-пользователя Поток = 200к − 100к(покупки) = +100к. Норма = 100к/200к = 50%. В текущей модели: СМО = 200к − 100к(кредитные платежи) = +100к. Норма = 50%. Результат тот же.

У пользователя с ремонтом: Поток месяц 1 = −55к. Норма = −55к/200к = −27.5%. Текущая: СМО = +90к (ремонтные расходы не считаются). Норма = 45%.

**В ремонтные месяцы норма сбережений обрушается.** Если школа триггерит на низкую норму — ложный алерт.

Сценарий-вердикт: атака **проходит**. Норма сбережений при активном проекте некорректна. Нужно: норму считать из **базового** потока при активном проекте, как и FI-score. Это ещё одна производная метрика, которая завязывается на механизм «Проект». Добавляет сложности, но решаемо.

---

### Результат раунда 5:
- **Закрыто:** Объём переработки школы конкретизирован: 2 полных переработки + 3 частичных + перекрёстные ссылки = 3-4 дня. Приемлемо.
- **Закрыто:** Пирамида не теряется — сохраняется как нарратив (четыре «этажа»), радар как аналитика.
- **Открыто:** Норма сбережений при проекте — считать из базового потока.
- **Баланс: 72/28 за мультиметрику** (школа — не блокер)

## Раунд 6 — Новичок без контекста

### ЗАЩИТНИК:

Пользователь открывает приложение впервые. Первый импорт выписки сделан. Кредитов нет. Что он видит?

**Текущая модель:**
```
┌───────────────────────┐
│  СМО: +47 000 ₽  🟢  │
│                       │
│  Доход: 150к          │
│  Обязательные: 65к    │
│  Второстепенные: 38к  │
└───────────────────────┘
```

Одно число. Понятно? Для кого-то да, для кого-то нет. «СМО» — аббревиатура. «Что такое среднемесячный остаток?» — первый вопрос новичка.

**Мультиметрика:**
```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  ПОТОК       │  │  КАПИТАЛ     │  │  НАГРУЗКА    │  │  ЗАПАС       │
│  +47к  🟢    │  │  320к  🟢    │  │  0%     ✅   │  │  3.2 мес 🟢  │
└──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘
```

Четыре числа, но каждое отвечает на **бытовой вопрос**:
- «Сколько осталось?» → Поток +47к
- «Я богатею?» → Капитал 320к
- «Сколько работаю на банк?» → 0%
- «Сколько продержусь?» → 3.2 месяца

Новичок без кредитов видит **четыре зелёных числа**. Нагрузка = 0, запас нормальный. Всё ок. Никакой перегрузки.

**Прогрессивное раскрытие:** Модуль 0 (бесплатный) объясняет каждую карточку. Урок 0.1: «Вот четыре вопроса. Сегодня разберём первый — сколько у тебя остаётся каждый месяц.» Не все четыре сразу. По одному, за 4 мини-урока.

**Ключевой аргумент:** Мультиметрика не сложнее, чем прогноз погоды. Температура, ветер, давление, влажность. Четыре числа. Никто не жалуется, что погоду трудно понять.

---

### КРИТИК:

**Атака 1: Четыре числа vs одно — когнитивная нагрузка.**

Прогноз погоды — плохая аналогия. Человек смотрит на температуру и решает, брать ли куртку. Не на четыре числа одновременно. Давление и влажность — для метеозависимых, не для всех.

Аналогично: новичок смотрит на дашборд и хочет **один ответ**: «У меня всё нормально?» FI-score даёт этот ответ (6.5/10 → «Нормально, есть куда расти»). Четыре карточки — это **детализация**, не главный экран.

Проверяю на «Первом шаге»: семья видит 3 красных + 1 жёлтый. Что делать? Починить нагрузку? Или запас? Или капитал? Пирамида давала чёткий приоритет: «сначала фундамент».

Сценарий-вердикт: атака **валидна наполовину**. Проблема не в четырёх числах, а в отсутствии **иерархии**. Если школа не скажет «фокус — нагрузка», пользователь потеряется.

Но: FI-score + школа решают эту проблему. FI-score = «общий ответ». Клик → 4 карточки = «детализация». Школа = «фокус на слабейшем». Это работает, если UX правильно расставляет приоритеты.

**Атака 2: «Капитал −380к» — демотивация.**

Семья «Первый шаг» открывает приложение. Видит: «Капитал: −380к». Первая реакция: «Я в минусе на 380 тысяч? Всё плохо!» Закрывает приложение. Эффект страуса (из [[За каждой цифрой — эмоция]]).

В текущей модели нет виджета «Чистый капитал» на дашборде — он спрятан на странице Assets. Мультиметрика выносит его на главный экран.

Сценарий-вердикт: атака **проходит**. Отрицательный капитал на главном экране = демотивация для ЦА с долгами. Решение: показывать не абсолютное число, а **траекторию**. «Капитал: ↑ +8к/мес» (вместо «−380к»). Пользователь видит: «Я расту, я двигаюсь вверх.» Абсолютное число — при клике, для тех, кто хочет.

Это уже предложено в стресс-тесте мультиметрики: «Главная ценность — не абсолютное число, а траектория.» Нужно зафиксировать как обязательное требование к дашборду.

**Атака 3: Первая неделя — данных мало.**

Пользователь импортировал 2 недели выписки. Поток = расчёт за полмесяца → неточен. Капитал = один снимок → нет траектории. Нагрузка = может быть 0 (платёж ещё не прошёл). Запас = нормальный.

Четыре карточки, из которых три «в процессе расчёта» — **хуже**, чем одно число «подождите месяц для точного СМО».

Сценарий-вердикт: атака **технически валидна**, но одинаково бьёт по обеим архитектурам. СМО тоже неточен за полмесяца. Текущий месяц не включается в СМО (правило из vault). Мультиметрика может использовать то же правило: Поток считается только по завершённым месяцам. В первую неделю — «данных недостаточно, подождите до конца месяца». Одинаковое ограничение.

---

### Результат раунда 6:
- **Закрыто:** Новичок без кредитов — 4 зелёных карточки, не сложнее текущего дашборда. Прогрессивное раскрытие через модуль 0.
- **Зафиксировано:** Капитал на дашборде — показывать **траекторию** (↑+8к/мес), не абсолютное число. Абсолютное — при клике.
- **Зафиксировано:** FI-score = «главный ответ», 4 карточки = «детализация».
- **Подтверждено:** Холодный старт (мало данных) — одинаковое ограничение в обеих архитектурах.
- **Баланс: 73/27 за мультиметрику**

## Раунд 7 — Техническая реализация

### ЗАЩИТНИК:

Бэкенд изменения конкретно. Что меняется, что остаётся.

**Что НЕ меняется:**
- Модели данных: Account, Transaction, Category, Counterparty — без изменений
- Типы счетов: 7 типов — без изменений
- Типы транзакций: credit_disbursement, credit_payment, credit_early_repayment — без изменений
- Импорт: import_service, import_normalizer, import_recognition_service — без изменений
- Страницы: Assets, Goals, Planning — без изменений
- Виджет «Долги» + «Здоровье долга» — без изменений
- Классификация поступлений — без изменений

**Что УДАЛЯЕТСЯ:**
- Два режима кредитной карты (покупка=расход / платёж=расход) → один режим: покупка=расход, платёж=перевод
- Счётчик кредитных денег (remaining_counter, linked_categories, is_credit_funded) → не нужен
- Тегирование кредитных денег на дебете → не нужно
- Сектор «Кредитные платежи» на дашборде → не нужен
- Разделение на категорийную и расходную аналитику → одна аналитика
- FIFO-очередь для карт рассрочки → не нужна (покупка = расход сразу)

**Что ДОБАВЛЯЕТСЯ:**

1. **Метрика «Поток»** (`analytics_service.py`):
```python
flow = regular_income - expenses - credit_interest
# expenses = ВСЕ покупки (дебет, кредитка, рассрочка)
# credit_interest = проценты по кредитам (вычисляемые)
# credit_payment (тело) = НЕ расход (перевод)
# credit_early_repayment = НЕ расход (перевод)
```

2. **Метрика «Капитал»** (`health_service.py`):
```python
capital = sum(liquid_accounts.balance) - sum(credit_accounts.debt)
# liquid = regular + cash + deposit
# trajectory = capital_this_month - capital_last_month
```

3. **Метрика «Запас»** (`health_service.py`):
```python
reserve = capital / avg_monthly_expenses  # в месяцах
# НЕ ликвидные/расходы, а капитал/расходы (учитывает долг)
```

4. **Проценты по кредиткам** (`analytics_service.py`):
```python
cc_interest = end_balance - start_balance + payments - purchases
# Если > 0 → расход в потоке
```

5. **Механизм «Проект»** — новая сущность:
```python
class Project:
    loan_id: FK → Account
    name: str
    budget: Decimal
    duration_months: int
    created_at: date
    closed_at: date | None
    status: active | closed
```

6. **Базовый поток** = полный_поток + Σ(project.budget / project.duration_months) для активных проектов.

7. **FI-score** (`health_service.py`) — новые веса:

| Компонент | Вес | Источник |
|---|---|---|
| Поток (базовый, сглаженный) | 0.25 | analytics_service |
| Траектория капитала | 0.30 | health_service |
| Нагрузка (инверсия) | 0.20 | health_service (= текущий DTI) |
| Запас | 0.25 | health_service |

**Фронтенд:**
- Дашборд: виджет СМО → 4 карточки (Поток, Капитал, Нагрузка, Запас)
- Аналитика: убрать разделение категорийная/расходная → одна аналитика
- Страница Health: FI-score с новыми компонентами, радар вместо пирамиды
- Карточка счёта: убрать переключатель модели кредитки

**API изменения:**
- `GET /api/v1/analytics/summary` — возвращает flow, capital, load, reserve вместо smo
- `GET /api/v1/health` — новые компоненты FI-score
- `POST /api/v1/projects` — CRUD для проектов
- Убрать: `is_credit_funded`, `credit_card_model`, `remaining_counter`, `linked_categories`

**Оценка объёма:** 3-5 дней бэкенд, 3-5 дней фронтенд. Удаление кода (счётчик, тегирование, два режима) компенсирует добавление нового. Чистый объём — **сопоставим с текущим**, но код проще.

### КРИТИК:

**Атака 1: Миграция данных.**

Текущая БД содержит транзакции с `is_credit_funded=True`, счётчики `remaining_counter`, привязки `linked_categories`. При переходе на мультиметрику — что с ними?

Проверяю: в текущем коде (`app/services/account_service.py`, `app/schemas/account.py`) — эти поля ещё не реализованы (counter v2 — в vault, не в коде). Модели кредитной карты (два режима) — тоже в vault.

Сценарий-вердикт: атака **снимается**. Кредитная логика (счётчик, тегирование, два режима) **не реализована в коде**. Она только спроектирована в vault. Миграция данных не нужна. Переключение с текущей архитектуры на мультиметрику — это выбор **направления реализации**, а не рефакторинг существующего кода.

Это **критически важное** наблюдение. Стоимость перехода на мультиметрику сейчас = стоимость реализации с нуля. Стоимость перехода **потом** (после реализации счётчика + тегирования + двух режимов) = стоимость реализации + стоимость удаления + миграция данных.

**Атака 2: Проценты по кредиткам — точность.**

Формула `cc_interest = end_balance - start_balance + payments - purchases`. Если пользователь делает покупки и погашения внутри месяца — порядок операций влияет на промежуточные балансы, но не на конечную формулу. Проверяю:

Начало: долг 100к. Покупки: +50к. Погашения: −80к. Проценты: +3к. Конец: 100 + 50 − 80 + 3 = 73к.

Формула: 73 − 100 + 80 − 50 = **3к** ✅ — точно совпадает с начисленными процентами.

Другой пример: начало 0, покупки 100к, погашение 100к, проценты 0. Конец: 0. Формула: 0 − 0 + 100 − 100 = **0** ✅.

Сценарий-вердикт: формула **математически корректна** при условии, что мы знаем все покупки и погашения за месяц (а мы знаем — они импортируются). **Атака снимается.**

**Атака 3: Проект — новая сущность усложняет модель.**

Текущая модель: Account, Transaction, Category. Мультиметрика добавляет Project. Это новый CRUD, новые связи, новые edge cases. Не нарушает ли это принцип «мультиметрика проще»?

Сценарий-вердикт: Project — одна новая сущность с 5-6 полями. Текущая кредитная логика (если бы была реализована) добавляла бы: `remaining_counter` на Account, `linked_categories` (связь многие-ко-многим), `is_credit_funded` на Transaction, `credit_card_model` на Account. Это **4 новых поля/связи на существующих сущностях** (опаснее — загрязняет основные модели) vs **1 новая чистая сущность**. Мультиметрика **чище** архитектурно.

**Атака 4: Удаление FIFO — потеря функциональности.**

FIFO-очередь для рассрочек давала: прогресс погашения конкретных покупок («Ноутбук 120к — оплачено 40к, осталось 80к»). В мультиметрике покупка = расход сразу, платёж = перевод. Прогресс погашения **не отслеживается** на уровне отдельной покупки.

Сценарий-вердикт: атака **валидна**. Пользователь теряет детальный прогресс по каждой рассрочечной покупке. Видит общий долг по карте, но не «ноутбук на 60% закрыт». Для минималистичного MVP — допустимо (общий долг достаточен). Для полного продукта — **можно вернуть FIFO опционально** на странице долга, не влияя на метрики. Это UX-фича, не архитектурная. **Не блокер**, но потеря функциональности зафиксирована.

---

### Результат раунда 7:
- **Ключевое открытие:** кредитная логика (счётчик, тегирование, два режима) **не реализована в коде**. Переход на мультиметрику сейчас = выбор направления, не рефакторинг. Это радикально снижает стоимость.
- **Закрыто:** Формула процентов по кредиткам — математически корректна. Проект — чище, чем загрязнение основных моделей.
- **Зафиксировано:** Потеря FIFO — допустимо для MVP, можно вернуть опционально.
- **Оценка объёма:** ~7-10 дней (бэкенд + фронтенд). Сопоставимо с реализацией текущей кредитной логики, но код проще и чище.
- **Баланс: 78/22 за мультиметрику** (код не написан → нет стоимости миграции)

## Раунд 8 — Финальный вердикт

### КРИТИК: финальная позиция

Провёл 5 раундов атак (4-8). Подвожу итог.

**Что я пытался сломать и не смог:**
1. Двойной счёт — невозможен по конструкции. Атака не нашла ни одного сценария.
2. Кешбэк-пользователь — проблема #1 не существует. Чисто.
3. Импульсная покупка — поток кричит сам. Чисто.
4. Яндекс Сплит — обрабатывается естественно. Чисто.
5. Формула процентов по кредиткам — математически корректна.
6. Семья «Первый шаг» — мультиметрика **точнее диагностирует** проблему (нагрузка, а не «отрицательный СМО»).
7. Объём рефакторинга — школа 3-4 дня, код ~7-10 дней. Кредитная логика не в коде → нет миграции.

**Что я сломал:**
1. **Базовый поток помесячно шумит** при неравномерных проектных расходах (±30к). Решение: алерты по скользящему среднему при активном проекте. Закрыто.
2. **Отрицательный капитал на дашборде — демотивация.** Решение: показывать траекторию (↑+8к/мес), абсолютное число при клике. Закрыто.
3. **Норма сбережений при проекте** — считать из базового потока. Открыто (нужна конкретная формула).
4. **Fallback при отсутствии проекта** — пользователь пропускает, поток 3 мес красный. Решение: усилить промпт при credit_disbursement. Частично закрыто.
5. **Потеря FIFO** — нет прогресса погашения по отдельным рассрочечным покупкам. Допустимо для MVP. Зафиксировано.

**Что осталось одинаковым:**
- Долговая спираль — невидима без «Здоровья долга» в обеих архитектурах.
- Холодный старт (мало данных) — одинаковое ограничение.

**Мой финальный вердикт:**

Мультиметрическая система **побеждает** по трём решающим критериям:

1. **Архитектурная простота.** 14 документов кредитной логики → 2-3. Убирается: два режима кредитки, счётчик, тегирование, FIFO, привязка категорий, конверт, батч-вопросы. Добавляется: одна сущность Project + формула процентов.

2. **Момент принятия решения.** Кредитная логика не реализована в коде. Сейчас — выбор направления. Через 3 месяца (после реализации счётчика + тегирования) — рефакторинг + миграция + потеря работы.

3. **Педагогическая честность.** СМО = +95к при реальном перерасходе 255к/мес (ремонт) — это ложь метрики. Поток −55к + контекст «плановый ремонт, базовый поток +95к» — это правда с объяснением. Школа финансовой грамотности должна учить видеть реальность.

**Оставшиеся риски:**
- Норма сбережений при проекте (нужна формула)
- Fallback при пропущенном проекте (нужен сильный промпт)
- Потеря FIFO-прогресса (допустимо для MVP)

Ни один из этих рисков не является архитектурным блокером. Все решаемы итеративно.

### ЗАЩИТНИК: финальный ответ

Критик провёл добросовестную работу. Его атаки выявили реальные edge cases, которые нужно решить, — но ни одна не нашла архитектурного порока в мультиметрике.

**Принимаю** все зафиксированные решения:
- Траектория вместо абсолютного капитала
- Скользящее среднее для алертов при проекте
- Усиленный промпт при credit_disbursement
- Норма сбережений из базового потока при проекте
- FIFO как опциональная UX-фича (не влияет на метрики)

**Главный аргумент, который не был опровергнут:**

Мультиметрика превращает **исключительные случаи в дефолтное поведение**. В текущей архитектуре каждый тип кредита требует своего механизма (счётчик, тегирование, FIFO, два режима). В мультиметрике один принцип — «покупка = расход, погашение тела = перевод» — покрывает **все** типы кредитов без исключений.

Единственное необходимое дополнение — Project для потребкредитов на общие нужды. Но это **одна** сущность для **одного** edge case, а не шесть механизмов для шести проблем.

**Консенсус достигнут.**

Оба участника согласны: мультиметрическая система — правильное направление. Критик не нашёл архитектурного блокера. Защитник принял все уточнения.
