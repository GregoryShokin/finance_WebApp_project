# Дебаты: Карта рассрочки в мультиметрике

## Участники
- **АРХИТЕКТОР** — предлагает модель данных, метрики, школьные моменты
- **КРИТИК** — атакует каждое предложение конкретными сценариями с цифрами

## Контекст

Мультиметрическая система (принята):
- **Поток** — базовый (рег. доходы − рег. расходы − кредитные платежи) + полный (Δ кэша)
- **Капитал** — ликвидные активы + вклады − долги
- **Нагрузка (DTI)** — платежи по кредитам / регулярный доход
- **Запас** — доступный кэш / средний ежемесячный отток (в месяцах)
- **FI-score** — композитный балл

Принцип: **Поток = только то, что на банковском счёте.** Если операция не прошла через счёт пользователя — её нет в Потоке.

Карта рассрочки (Халва, Свобода, Яндекс Сплит) — НЕ кредитная карта. Каждая покупка = отдельный график платежей. На счёте пользователя при покупке ничего не происходит. Списания начинаются через месяц.

## Раунд 1: Модель данных и базовое определение

### Архитектор

Предлагаю **Вариант А: один счёт + дочерняя таблица покупок**.

```
Account (type = installment_card):
    credit_limit_original: Decimal       — лимит карты (300 000)
    credit_current_amount: Decimal       — вычисляемое: Σ remaining активных покупок
    monthly_payment: Decimal             — вычисляемое: Σ monthly_payment активных покупок

InstallmentPurchase:
    id: PK
    account_id: FK → Account
    description: str                     — "iPhone 15 Pro"
    category_id: FK → Category           — для аналитики "на что"
    original_amount: Decimal             — 60 000
    remaining_amount: Decimal            — обновляется при каждом платеже
    interest_rate: Decimal               — 0.0 или 0.12
    term_months: int                     — 12
    monthly_payment: Decimal             — 5 000
    start_date: date
    status: active | completed | early_closed
```

**Почему Вариант А, а не Б или В:**

| Критерий | А (счёт + покупки) | Б (покупка = счёт) | В (один счёт) |
|----------|--------------------|--------------------|----------------|
| UX | Одна карта, детализация внутри | 15 счетов в списке | Чисто, но слепо |
| Аналитика «на что» | ✅ через category_id | ✅ через счёт | ❌ нет |
| Досрочное погашение | ✅ конкретной покупки | ✅ закрытие счёта | ❌ только всей карты |
| Автообновление DTI | ✅ покупка закрылась → пересчёт | ✅ счёт закрылся | ❌ ручное обновление |
| Сложность модели | Средняя (1 новая сущность) | Низкая (без новых) | Минимальная |
| Школьные моменты | ✅ «5 рассрочек на 20к/мес» | Возможно | ❌ «платёж 20к» без деталей |

**Вычисляемые поля на Account:** `credit_current_amount` и `monthly_payment` пересчитываются при каждом изменении дочерних покупок (создание, платёж, досрочное закрытие). Это НЕ независимые поля — это агрегаты.

### Критик

**Атака 1: Новая сущность — оправдана ли сложность?**
- InstallmentPurchase — это первая дочерняя сущность к Account. Раньше все кредиты были «плоскими» (один счёт = один долг).
- Это прецедент: если появятся другие кредитные инструменты с множественными долгами — нужно будет ещё таблицы?
- **Контраргумент:** карта рассрочки — уникальный инструмент. Ипотека, потребкредит, автокредит — это один долг = один счёт. Карта рассрочки — много долгов на одном счёте. Новая сущность оправдана уникальностью инструмента.
- ✅ Оправдано.

**Атака 2: Вариант В для MVP?**
- Можно ли начать с Варианта В (один счёт, без покупок), а потом мигрировать на А?
- MVP: пользователь создаёт installment_card, вводит monthly_payment и credit_current_amount вручную. Работает для метрик.
- Потом: добавляем InstallmentPurchase, мигрируем.
- **Проблема:** при миграции пользователь потеряет историю покупок (их не было). Данные не retroactive.
- **Проблема 2:** сценарий 11 (рассрочки накопились) — без покупок невозможно показать «5 рассрочек по 3-5к». Пользователь видит только «платёж 20к».
- **Проблема 3:** при закрытии рассрочки в Варианте В пользователь должен ВРУЧНУЮ уменьшить monthly_payment. Забудет → DTI завышен.
- Вариант В — функционален для метрик, но слеп для UX и ненадёжен для обновлений.

**Атака 3: Вычисляемые vs. хранимые агрегаты**
- Если credit_current_amount и monthly_payment вычисляются из детей — что если пользователь хочет вручную скорректировать? Например, платёж по выписке ≠ сумме покупок (из-за комиссий, страховок).
- **Решение:** агрегаты вычисляемые, но с возможностью ручной корректировки (override). При override показываем предупреждение: «Ручная сумма отличается от расчётной на X₽». Для MVP — только вычисляемые, без override.

### Решение раунда
- **Принято:** Вариант А — один счёт `installment_card` + дочерняя таблица `InstallmentPurchase`.
- **Принято:** `credit_current_amount` и `monthly_payment` на Account — вычисляемые агрегаты из активных покупок.
- **Принято:** Новая сущность InstallmentPurchase оправдана уникальностью инструмента.
- **Отвергнуто:** Вариант Б — 15 счетов в списке неприемлемо для UX.
- **Отвергнуто:** Вариант В как целевой — слеп для аналитики, ненадёжен для обновлений. Допустим только как промежуточный шаг, если сроки критичны.
- **Открыто:** Ручная корректировка агрегатов → post-MVP.

## Раунд 2: Метрики — Поток и DTI

### Архитектор

**Поток:**

Покупка на карту рассрочки:
- Банк платит магазину → на счёте пользователя НИЧЕГО не происходит
- Покупка НЕ создаёт транзакцию на банковском счёте
- Покупка создаёт только `InstallmentPurchase` (долг)
- **В Поток (базовый и полный) покупка НЕ попадает** — аналогично ипотеке/автокредиту

Ежемесячный платёж:
- Реальное списание с дебетового счёта → транзакция `credit_payment`
- **В базовый поток: да** (как любой кредитный платёж)
- **В полный поток: да** (реальный отток кэша)

Это полностью аналогично модели «банк → продавец» (ипотека, автокредит).

**DTI:**

DTI = Σ(кредитные платежи) / средний регулярный доход

Ежемесячный платёж по карте рассрочки входит в числитель DTI наравне с ипотекой, потребкредитом и т.д.

При закрытии рассрочки → monthly_payment пересчитывается автоматически → DTI снижается.

### Критик

**Атака 1: Сценарий 1 — Одна рассрочка 0%**
- Доход 150к, расходы 80к (рег.), кредитов нет
- Покупка: телефон 60к, 12 мес, 0%, платёж 5к/мес

До покупки:
- Базовый поток = 150 − 80 = 70к → Показатель = 70/150 = 47% → 🟢
- DTI = 0%

После покупки (со следующего месяца):
- Базовый поток = 150 − 80 − 5 = 65к → Показатель = 65/150 = 43% → 🟢
- DTI = 5/150 = 3.3%

Показатель почти не изменился. DTI минимальный. Одна мелкая рассрочка — капля в море.
✅ **Корректно.**

**Атака 2: Сценарий 2 — Три рассрочки одновременно**
- Доход 150к, расходы 80к
- Телефон: 60к / 12 мес / 0% → 5к/мес
- Стиралка: 40к / 6 мес / 0% → 6.7к/мес
- Мебель: 90к / 10 мес / 12% годовых → ~9.5к/мес (аннуитет)
- Общий платёж: 5 + 6.7 + 9.5 = **21.2к/мес**

Метрики:
- Базовый поток = 150 − 80 − 21.2 = 48.8к → Показатель = 48.8/150 = 33% → 🟢
- DTI = 21.2/150 = **14.1%**

Три рассрочки сжали свободный поток с 70к до 49к (−30%), DTI вырос до 14%. Но показатель всё ещё зелёный. Пользователь может не заметить ползучего роста обязательств.

**Вопрос:** достаточно ли DTI 14% для тревоги? Нет — это нормальный уровень. Но тренд важен: DTI вырос с 0% до 14% за несколько месяцев. Нужен ли «тренд-алерт»?

**Контраргумент самому себе:** показатель зелёный, DTI в норме — зачем тревожить? Пользователь справляется. Школьный момент («5 рассрочек на 20к/мес») уместен при бо́льшем объёме — см. сценарий 11.
✅ **Держится.**

**Атака 3: Сценарий 3 — Рассрочка закрылась**
Через 6 месяцев стиралка выплачена:
- Платёж: 5 + 0 + 9.5 = **14.5к/мес** (было 21.2к)
- DTI = 14.5/150 = **9.7%** (было 14.1%)
- Показатель = (150 − 80 − 14.5) / 150 = 37% → 🟢

С Вариантом А:
- InstallmentPurchase (стиралка) → `status: completed`
- Account.monthly_payment пересчитывается автоматически: 5 + 9.5 = 14.5
- DTI обновляется без участия пользователя

С Вариантом В:
- Пользователь должен вручную уменьшить monthly_payment с 21.2 до 14.5
- Если забудет → DTI завышен на 4.5 п.п. НАВСЕГДА (до следующей ручной корректировки)

✅ **Вариант А критичен для автоматического обновления DTI.**

**Атака 4: Покупка НЕ в Потоке — а в аналитике расходов?**
- Пользователь купил телефон за 60к на рассрочку. В категории «Электроника» за этот месяц — 0₽ (покупка не транзакция).
- Пользователь открывает аналитику: «Я же купил телефон! Где он?»
- Если покупка только в InstallmentPurchase — аналитика расходов её не видит.

**Варианты решения:**
1. Аналитика расходов включает InstallmentPurchase (JOIN с покупками) → правильно по сути, но усложняет аналитику
2. Создать «фантомную» транзакцию с флагом `is_financed = true` → видна в аналитике, исключена из Потока
3. Аналитика показывает только реальные транзакции. Покупки в рассрочку — отдельный раздел.

**Предложение:** вариант 3 — разделение ответственности. Аналитика расходов = реальные денежные потоки. Детализация рассрочек = отдельный раздел (на карточке installment_card). Это проще и честнее: «В этом месяце ты потратил X₽. Отдельно: взял в рассрочку Y₽.»

### Решение раунда
- **Принято:** Покупка на карту рассрочки НЕ создаёт транзакцию. Не в Потоке (ни базовом, ни полном).
- **Принято:** Ежемесячный платёж = `credit_payment` → в базовый и полный поток, в DTI.
- **Принято:** Модель аналогична ипотеке/автокредиту (банк → продавец, пользователь видит только платежи).
- **Принято:** При закрытии рассрочки — автоматический пересчёт monthly_payment и DTI (Вариант А).
- **Принято:** Аналитика расходов — только реальные транзакции. Рассрочки — отдельный раздел на карточке счёта.
- **Отвергнуто:** «Фантомные» транзакции (is_financed) — усложнение без пропорциональной выгоды.
- **Подтверждено:** Сценарии 1–3 проходят без проблем.

## Раунд 3: Метрики — Капитал и Запас

### Архитектор

**Капитал = ликвидные активы + вклады − долги**

Карта рассрочки создаёт долг:
- Покупка: долг +60к → Капитал −60к
- Каждый платёж: долг уменьшается
- Для рассрочки 0%: весь платёж = тело → долг −5к → Капитал +5к
- Для рассрочки с %: часть платежа = проценты (не уменьшают долг) → нужно разделение

**Разделение тело / проценты:**

Для 0% рассрочек: тривиально. Весь платёж = тело.

Для рассрочек с процентами: аннуитетный график. У каждой InstallmentPurchase есть `interest_rate`, `remaining_amount`, `monthly_payment`. Можно вычислить:
- `interest_portion = remaining_amount × (interest_rate / 12)`
- `principal_portion = monthly_payment − interest_portion`

Это стандартная аннуитетная формула, реализация простая.

**Запас = доступный кэш / средний ежемесячный отток**

Платежи по рассрочкам входят в отток. При закрытии рассрочки отток уменьшается → Запас улучшается. Скользящее среднее (3–6 мес) «помнит» старый платёж — это нормально, среднее сгладит.

### Критик

**Атака 1: Сценарий 4 — Рассрочка с процентами, Капитал**
- Мебель 90к, 10 мес, 12% годовых (1% в месяц)
- Платёж: ~9 500₽/мес (аннуитет)

Месяц 1:
- interest = 90 000 × 1% = 900₽
- principal = 9 500 − 900 = 8 600₽
- remaining = 90 000 − 8 600 = 81 400₽
- Капитал: долг −8 600 (улучшение), кэш −9 500 (ухудшение) → **нетто: −900₽** (потеря на проценты)

Месяц 10 (последний):
- interest ≈ 94₽
- principal ≈ 9 406₽
- remaining → 0
- Капитал: долг −9 406 (улучшение), кэш −9 500 → **нетто: −94₽**

За весь срок:
- Общая переплата: 10 × 9 500 − 90 000 = **5 000₽**
- Капитал потерял 5 000₽ на процентах
- Правильно: это реальная стоимость заимствования.

✅ **Разделение тело/проценты работает корректно.**

**Атака 2: Сценарий 5 — Смешанные 0% и с процентами**
Три рассрочки из сценария 2:
- Телефон 0%: весь платёж 5к = тело → Капитал нейтрален по платежу
- Стиралка 0%: весь платёж 6.7к = тело → Капитал нейтрален
- Мебель 12%: из 9.5к — часть проценты → Капитал теряет ~500₽/мес на процентах

**Суммарный платёж 21.2к разносится:**
- Тело: 5к + 6.7к + ~8.6к = 20.3к → уменьшает долг
- Проценты: ~0.9к → потеря

Для Капитала важно: −21.2к кэш + 20.3к долг = **−0.9к нетто** (проценты).

Без разделения (если весь платёж считать телом): Капитал нетто 0₽. Ошибка: +0.9к/мес. За 10 месяцев: +5к ошибка. На фоне 190к общего долга — 2.6% ошибка.

**Критично ли для MVP?** Для 12% годовых — нет (ошибка мала). Для 24%+ — ошибка удваивается. Для корректности нужно разделение, но ошибка не катастрофическая.

**Предложение:** реализовать разделение сразу — формула простая (`interest = remaining × rate/12`), и это избавляет от накопления ошибок.

**Атака 3: Покупка на рассрочку vs. покупка за кэш — Капитал**
- Телефон 60к за кэш: кэш −60к → Капитал −60к
- Телефон 60к в рассрочку 0%: долг +60к → Капитал −60к

Одинаково! Но разница:
- За кэш: Запас сразу падает (меньше кэша)
- В рассрочку: Запас меньше страдает (кэш цел, но отток вырос)

Посчитаем Запас (кэш 200к, расходы 80к):
- За кэш: Запас = 140к / 80к = **1.75 мес**
- В рассрочку: Запас = 200к / (80к + 5к) = **2.35 мес**

Рассрочка даёт ЛУЧШИЙ Запас! Это корректно — наличие кэша при растянутых платежах действительно даёт бо́льшую гибкость.

**Но:** не создаёт ли это ложный стимул «бери рассрочку — запас выше»?
- Нет. DTI растёт (с 0% до 3.3%). Капитал одинаков. Только Запас лучше.
- И он РЕАЛЬНО лучше: если потеряешь доход, с рассрочкой у тебя 200к кэша (на 2.35 мес), а без рассрочки — 140к (на 1.75 мес).
- Метрики отражают реальность. Стимул — задача школьных моментов, не метрик.

✅ **Капитал и Запас считаются корректно.**

**Атака 4: Запас и скользящее среднее оттока**
- Пользователь платил 21к/мес по рассрочкам 6 месяцев. Стиралка закрылась → платёж 14.5к.
- Скользящее среднее оттока (3 мес): ещё «помнит» 21к. Запас занижен.
- Через 3 мес: среднее обновится до 14.5к. Запас улучшится.

Это нормальное поведение среднего. Задержка ~1–2 месяца. Не баг — фича сглаживания.

✅ **Приемлемо.**

### Решение раунда
- **Принято:** Покупка → долг +X → Капитал −X. Платёж → долг −тело, кэш −платёж → нетто Капитала = −проценты.
- **Принято:** Разделение тело/проценты реализовать сразу. Формула простая: `interest = remaining × rate/12`.
- **Принято:** Для рассрочек 0% — весь платёж = тело. Разделение тривиально.
- **Принято:** Запас: платежи по рассрочкам входят в отток. При закрытии — отток снижается → Запас растёт.
- **Подтверждено:** Рассрочка vs. кэш — одинаковый Капитал, но разный Запас. Это корректное отражение реальности.
- **Подтверждено:** Сценарии 4 и 5 проходят.

## Раунд 4: Лимит и «доступные средства»

### Архитектор

Ключевой вопрос: пользователь с 100к на дебете и 300к лимитом на Халве ощущает «400к доступных». Как приложение реагирует?

**Предлагаю Вариант В (показывать отдельно):**

На дашборде, виджет «Доступные средства»:
- **Свободные средства: 100 000₽** (реальный кэш)
- Лимит рассрочки **НЕ** включён

На карточке installment_card:
- **Использовано: 150 000₽ из 300 000₽** (шкала использования)
- **Доступный лимит: 150 000₽**
- Подпись: *«Лимит рассрочки — это не ваши деньги. Каждое использование увеличивает ежемесячный платёж.»*

**Школьный момент при высокой утилизации (> 70%):**
> *«Вы использовали 93% лимита рассрочки. Свободно всего 20 000₽. Каждая закрытая рассрочка восстанавливает лимит.»*

### Критик

**Атака 1: Сценарий 6 — Лимит почти исчерпан**
- Лимит 300к, использовано 280к, свободно 20к
- Допустим 7 активных рассрочек, общий платёж ~35к/мес
- Доход 150к, расходы 80к

Метрики:
- Базовый поток = 150 − 80 − 35 = 35к → Показатель = 35/150 = 23% → 🟢
- DTI = 35/150 = 23%
- Капитал: допустим кэш 100к, долг 280к → Капитал = 100 − 280 = **−180к**

Показатель зелёный (23%), но Капитал глубоко отрицательный (−180к). Пользователь «может позволить текущий образ жизни» (поток положительный), но его чистое состояние в минусе из-за рассрочек.

**Это нормально?** Да. Показатель и Капитал — разные метрики. Показатель: «справляешься ли ты с платежами». Капитал: «сколько у тебя чисто». Семья может справляться с платежами и при этом иметь отрицательный Капитал — это реальная ситуация.

Но школьный момент нужен: *«Ваш Капитал отрицательный: −180 000₽. Вы должны больше, чем имеете. По мере погашения рассрочек Капитал будет восстанавливаться.»*

✅ **Мультиметрика корректно показывает обе стороны.**

**Атака 2: Лимит как «потенциальный DTI»**
- Лимит 300к, используется 0к. DTI = 0%.
- Но пользователь может ЗАВТРА использовать весь лимит → DTI вырастет.
- Должен ли показатель учитывать неиспользованный лимит как риск?

**Нет.** Это как учитывать потенциальный кредит, который можно взять. Метрики считают текущее состояние, не потенциальные будущие решения. Если пользователь использует лимит — метрики отреагируют. До этого — нет.

Аналогия: одобренный, но не взятый ипотечный кредит не входит в DTI.

✅ **Неиспользованный лимит не входит в метрики.**

**Атака 3: А если пользователь НЕ видит лимит на дашборде — он пойдёт в банковское приложение?**
- Вариант А (показывать лимит в доступных) — пользователю удобно, но нечестно.
- Вариант Б (не показывать вообще) — честно, но пользователь теряет полную картину.
- Вариант В (показывать отдельно) — компромисс, но требует объяснения.

Вариант В правильный: информация доступна (на карточке счёта), но НЕ смешана со свободными средствами. Пользователь может видеть лимит, но понимает, что это не его деньги.

### Решение раунда
- **Принято:** Лимит рассрочки НЕ включается в «Доступные средства» на дашборде.
- **Принято:** Лимит показывается на карточке installment_card: использовано / лимит + шкала.
- **Принято:** Школьный момент при утилизации > 70%: «Лимит почти исчерпан».
- **Принято:** Неиспользованный лимит НЕ влияет на метрики.
- **Отвергнуто:** Лимит в «Доступных средствах» — создаёт иллюзию богатства.
- **Отвергнуто:** Скрытие лимита полностью — пользователь теряет информацию.
- **Подтверждено:** Сценарий 6 — мультиметрика корректно показывает и поток (зелёный), и капитал (отрицательный).

## Раунд 5: Яндекс Сплит и досрочное погашение

### Архитектор

**Яндекс Сплит — гибрид:**
- Покупка 40к: первый платёж 10к сразу с дебета + рассрочка 30к на 3 мес по 10к.

Модель:
1. **Первый платёж 10к** — обычная транзакция-расход с дебетового счёта. Категория: «Электроника» (или что купил). Попадает в Поток как обычный расход.
2. **Рассрочка 30к** — InstallmentPurchase на карте Сплит. original_amount = 30к, term = 3, monthly_payment = 10к.

Общий учёт: 10к (расход) + 30к (долг) = 40к покупка. Без двойного счёта.

**Досрочное погашение:**

Одной покупки:
- Транзакция: `credit_payment` с дебета на installment_card. Сумма = remaining_amount покупки.
- InstallmentPurchase → `status: early_closed`, `remaining_amount: 0`
- Account.monthly_payment пересчитывается (минус закрытая покупка)
- Капитал: кэш −X, долг −X → нетто 0 (для 0%). Для процентных: экономия на будущих процентах.

Всей карты:
- credit_payment = Σ remaining всех активных покупок
- Все InstallmentPurchases → `status: early_closed`
- Account: monthly_payment = 0, credit_current_amount = 0

### Критик

**Атака 1: Сценарий 8 — Яндекс Сплит**
- Покупка 40к: 10к сразу, 30к рассрочка (3 × 10к)

Метрики в месяц покупки:
- Базовый поток: −10к расход (если категория регулярная) − 0 кредитный платёж (ещё не начался)
- Полный поток: −10к (реальное списание)
- Капитал: кэш −10к, долг +30к → −40к (полная стоимость покупки). ✅

Месяцы 2–4 (платежи по рассрочке):
- Базовый поток: −10к/мес (credit_payment)
- Капитал: кэш −10к, долг −10к → нетто 0

Суммарно за 4 мес: 10к + 3×10к = 40к. ✅ Всё сходится.

**Но:** первые 10к — расход в категории «Электроника». Остальные 30к — credit_payment (не привязан к категории). В аналитике расходов видно только 10к, а не 40к.

**Это проблема?** Нет, если InstallmentPurchase имеет category_id = «Электроника». На карточке рассрочки пользователь видит: «Электроника, 30к в рассрочку». В аналитике расходов: 10к. В рассрочках: 30к. Суммарно = 40к. Информация есть, просто в двух местах.

✅ **Яндекс Сплит работает без специальной логики.**

**Атака 2: Сценарий 7 — Досрочное погашение одной покупки**
- У пользователя три рассрочки (телефон 30к remaining, стиралка закрыта, мебель 37к remaining)
- Решил закрыть телефон досрочно: платит 30к

Шаги:
1. Транзакция: credit_payment 30к (дебет → installment_card)
2. InstallmentPurchase (телефон): status → early_closed, remaining → 0
3. Account.monthly_payment: было 14.5к (5 + 9.5), стало 9.5к
4. Account.credit_current_amount: было 67к (30 + 37), стало 37к

Метрики:
- Поток (месяц досрочного погашения): −30к credit_payment (разовый). Базовый поток временно просядет.
- Но: скользящее среднее за 3 мес сгладит.
- DTI: с 9.7% (14.5/150) до 6.3% (9.5/150). Улучшение.
- Капитал: кэш −30к, долг −30к → нетто 0 (0% рассрочка).
- Запас: кэш уменьшился, но отток тоже уменьшился (−5к/мес). В долгосрочной перспективе — улучшение.

✅ **Досрочное погашение работает.**

**Атака 3: Частичное досрочное погашение**
- Пользователь платит 15к сверх обычного. Не хватает на полное закрытие ни одной покупки.
- Куда идут 15к?
- **Вариант 1:** пользователь выбирает, на какую покупку направить.
- **Вариант 2:** пропорционально по всем.
- **Вариант 3:** не поддерживаем. Только полное закрытие конкретной покупки.

**Предложение:** Вариант 3 для MVP. Досрочное погашение = закрытие конкретной покупки целиком. Это проще в модели и соответствует реальной банковской практике (большинство банков не принимают частичное погашение отдельной рассрочки внутри карты).

Если пользователь сделал переплату (банк сам распределил) → пользователь вручную корректирует remaining у нужных покупок. Это edge case для post-MVP.

### Решение раунда
- **Принято:** Яндекс Сплит: первый платёж = обычный расход с дебета, остаток = InstallmentPurchase. Без специальной логики.
- **Принято:** Досрочное погашение: только целиком конкретной покупки. credit_payment = remaining → status: early_closed → пересчёт агрегатов.
- **Принято:** Досрочное погашение всей карты: сумма всех remaining.
- **Отвергнуто:** Частичное досрочное погашение для MVP — слишком сложно, редкий кейс.
- **Подтверждено:** Сценарии 7 и 8 проходят.

## Раунд 6: Совместимость с кредитной логикой + школьные моменты

### Архитектор

**Совместимость с существующей моделью:**

Тип `installment_card` уже существует в системе. Поля Account, которые переиспользуются:
- `credit_limit_original` → лимит карты
- `credit_current_amount` → общий остаток долга (вычисляемый)
- `monthly_payment` → общий ежемесячный платёж (вычисляемый)

Типы транзакций:
- `credit_payment` → ежемесячный платёж (уже есть, тот же тип, что для ипотеки/автокредита)

Что добавляется:
- Таблица `InstallmentPurchase` (новая сущность)
- Логика вычисления агрегатов Account из дочерних покупок
- Разделение тело/проценты по каждой покупке (для Капитала)

**Минимум изменений в существующем коде.** Метрики используют те же поля Account (monthly_payment, credit_current_amount), что и для других кредитов. Новая логика — только в сервисе управления покупками.

### Критик

**Атака 1: Сценарий 9 — Кешбэк-покупатель + рассрочка**
- Часть покупок — через кредитку (для кешбэка), часть — через Халву (рассрочка)
- Доход 200к, расходы 130к (через кредитку), рассрочки 15к/мес

Кредитная карта:
- Расходы 130к — реальные транзакции по категориям, попадают в базовый поток как регулярные расходы
- Погашение кредитки 130к — перевод с дебета на кредитку (не credit_payment, а transfer)
- DTI по кредитке: 0% (нет минимального платежа при полном погашении)

Карта рассрочки:
- Покупки — не в Потоке
- Платёж 15к — credit_payment → в базовый поток, в DTI

Метрики:
- Базовый поток = 200 − 130 − 15 = 55к → Показатель = 55/200 = 27.5% → 🟢
- DTI = 15/200 = 7.5% (только рассрочка, кредитка при полном погашении не добавляет)

Нет путаницы: кредитка и рассрочка — разные типы счетов с разной механикой.
✅ **Не пересекаются.**

**Атака 2: Сценарий 10 — «Первый шаг» + рассрочки**
- Доход 120к, ипотека 30к, потребкредит 15к, рассрочки 10к
- Расходы 65к (рег.)

Метрики:
- Базовый поток = 120 − 65 − 30 − 15 − 10 = 0к → Показатель = 0/120 = **0%** → 🟡 (граница)
- DTI = (30 + 15 + 10) / 120 = **45.8%**
- Капитал: зависит от активов и долгов. Допустим: квартира 5М (актив), ипотека 3М, потребкредит 200к, рассрочки 60к → Капитал = 5М − 3М − 200к − 60к = **1.74М** (положительный благодаря квартире)

**Видит ли семья, что рассрочки — часть проблемы?**
- DTI 45.8%. Без рассрочек: DTI = 45/120 = 37.5%.
- Рассрочки добавили 8.3 п.п. к DTI. Это видно на карточке DTI.
- Показатель ровно 0% — на грани. Без рассрочек: (120 − 65 − 45) / 120 = 8.3% — жёлтый, но с бо́льшим запасом.

Школьный момент: *«Ваш DTI 46% — это высокая нагрузка. Из них 10к/мес (8%) приходится на рассрочки. Когда рассрочки закроются, DTI снизится до 37.5%.»*

✅ **Мультиметрика чётко показывает вклад рассрочек.**

**Атака 3: Сценарий 11 — Рассрочки накопились незаметно**
- Пять мелких рассрочек:
  - Наушники: 6к / 3 мес → 2к/мес
  - Чехол: 3к / 3 мес → 1к/мес
  - Кроссовки: 12к / 4 мес → 3к/мес
  - Часы: 20к / 6 мес → 3.3к/мес
  - Курсы: 30к / 6 мес → 5к/мес
  - **Суммарно: 14.3к/мес ≈ 15к** (в промпте: 20к/мес; скорректирую покупки для 20к)

Пусть суммарно 20к/мес. Доход 150к, расходы 80к.
- Показатель = (150 − 80 − 20) / 150 = 33% → 🟢
- DTI = 20/150 = 13.3%

Показатель зелёный! DTI умеренный. Но 20к/мес — это серьёзно. Где пользователь это увидит?

С Вариантом А: карточка installment_card показывает **«5 активных рассрочек, общий платёж 20 000₽/мес»** с разбивкой по покупкам.

Школьный момент (при ≥ 3 активных рассрочках):
> *«У вас 5 активных рассрочек с общим платежом 20 000₽/мес. Каждая кажется мелкой, но вместе они занимают 13% вашего дохода. Подумайте, все ли покупки были необходимы.»*

С Вариантом В: пользователь видит только «платёж 20к/мес». Не знает, что это 5 покупок. Не осознаёт масштаб.

✅ **Вариант А критичен для этого сценария.**

**Атака 4: Сценарий 12 — Рассрочка vs. накопить**
- Пользователь берёт рассрочку: 60к, 12 мес, 12% → платёж ~5.3к/мес, переплата ~3.6к
- Базовый поток: +50к/мес
- Мог бы накопить 60к за 1.2 месяца!

Для этого школьного момента нужно:
1. Знать original_amount покупки (60к) — есть в InstallmentPurchase
2. Знать interest_rate (12%) — есть
3. Посчитать переплату (3.6к) — формула
4. Знать базовый поток (+50к) — метрика

Школьный момент (при interest_rate > 0 и base_flow > monthly_payment):
> *«Вы платите за рассрочку с процентами: 60 000₽ + 3 600₽ переплата. При вашем свободном потоке (50 000₽/мес) вы могли бы накопить на эту покупку за 1–2 месяца без переплаты.»*

С Вариантом В: невозможно — нет данных о конкретных покупках и их процентных ставках.

✅ **Вариант А критичен для образовательных моментов.**

### Решение раунда
- **Принято:** installment_card переиспользует существующие поля Account + стандартный credit_payment.
- **Принято:** Минимальные изменения в коде метрик — они уже работают с monthly_payment и credit_current_amount.
- **Принято:** Кредитка и рассрочка не пересекаются — разные механики, разные счета (сценарий 9).
- **Принято:** Школьный момент при ≥ 3 активных рассрочках: «N рассрочек на X₽/мес, это Y% дохода».
- **Принято:** Школьный момент «рассрочка vs. накопить» — при interest_rate > 0 и достаточном свободном потоке.
- **Принято:** На карточке DTI — показывать вклад рассрочек отдельной строкой (сценарий 10).
- **Подтверждено:** Вариант А критичен для сценариев 11 и 12 — без детализации покупок школьные моменты невозможны.

### Итоговая модель

**Модель данных: Вариант А**

```
Account (type = installment_card):
    credit_limit_original: Decimal       — лимит карты
    credit_current_amount: Decimal       — COMPUTED: Σ remaining_amount (active)
    monthly_payment: Decimal             — COMPUTED: Σ monthly_payment (active)

InstallmentPurchase:
    id: PK
    account_id: FK → Account
    description: str                     — что куплено
    category_id: FK → Category           — для аналитики
    original_amount: Decimal             — полная стоимость
    remaining_amount: Decimal            — остаток долга
    interest_rate: Decimal               — 0.0 или ставка
    term_months: int                     — срок в месяцах
    monthly_payment: Decimal             — ежемесячный платёж
    start_date: date
    status: active | completed | early_closed
```

**Влияние на метрики:**

| Событие | Поток (базовый) | Поток (полный) | Капитал | DTI | Запас |
|---------|----------------|---------------|---------|-----|-------|
| Покупка (не Сплит) | — | — | долг ↑ (−) | — | — |
| Покупка (Сплит, первый платёж) | расход ↓ | расход ↓ | долг ↑ + кэш ↓ | — | кэш ↓ |
| Ежемесячный платёж | credit_payment ↓ | credit_payment ↓ | долг ↓, кэш ↓ (нетто = −%) | ↑ (в числитель) | отток ↑ |
| Закрытие рассрочки | payment ↓ (авто) | payment ↓ | долг → 0 | ↓ (авто) | отток ↓ |
| Досрочное погашение | разовый payment ↓↓ | разовый payment ↓↓ | кэш ↓, долг ↓ (нетто 0) | ↓ (авто) | кэш ↓, отток ↓ |

**Ключевой принцип:** покупка на рассрочку = банк → магазин. На счёте пользователя — только платежи. Модель аналогична ипотеке/автокредиту.

**Разделение тело/проценты:**
```
interest_portion = remaining_amount × (interest_rate / 12)
principal_portion = monthly_payment − interest_portion
```
Для 0% рассрочек: весь платёж = тело.

## Реестр решений

### Принято
1. **Модель: Вариант А** — один счёт installment_card + дочерняя таблица InstallmentPurchase.
2. **Агрегаты вычисляемые** — credit_current_amount и monthly_payment на Account пересчитываются из активных покупок.
3. **Покупка НЕ в Потоке** — аналогично ипотеке (банк → магазин). Только платежи = реальные транзакции.
4. **Платёж = credit_payment** — в базовый поток, в DTI. Стандартный тип транзакции.
5. **Разделение тело/проценты** — `interest = remaining × rate/12`. Реализовать сразу.
6. **Лимит НЕ в «Доступных средствах»** — показывать отдельно на карточке счёта.
7. **Яндекс Сплит**: первый платёж = обычный расход, остаток = InstallmentPurchase. Без спецлогики.
8. **Досрочное погашение**: только целиком конкретной покупки (MVP).
9. **Аналитика расходов**: только реальные транзакции. Рассрочки — отдельный раздел на карточке счёта.
10. **Школьные моменты**: 4 триггера (утилизация, количество рассрочек, переплата, вклад в DTI).

### Отвергнуто
1. **Вариант Б** (каждая покупка = счёт) — 15 счетов в списке, UX неприемлем.
2. **Вариант В как целевой** (без детализации) — слеп для аналитики, ненадёжен для автообновления DTI. Допустим как промежуточный шаг.
3. **Лимит в «Доступных средствах»** — создаёт иллюзию богатства, противоречит философии приложения.
4. **«Фантомные» транзакции** для покупок в рассрочку — усложнение, расход не прошёл через счёт.
5. **Частичное досрочное погашение** для MVP — слишком сложно, редкий кейс.
6. **Учёт неиспользованного лимита** в метриках — метрики считают текущее состояние, не потенциальное.

### Открытые вопросы (post-MVP)
- Ручная корректировка агрегатов (override) при расхождении с банковской выпиской
- Частичное досрочное погашение
- Импорт выписок по карте рассрочки (формат и автораспознавание)
- Уведомления о приближающемся закрытии рассрочки

## Балл уверенности

**90/100**

Все 12 сценариев пройдены. Модель консистентна с существующей кредитной логикой. Основные неопределённости:

- **Вычисляемые агрегаты** (−4 балла): синхронизация Account с дочерними покупками требует аккуратной реализации. Расхождение вычисляемого и реального платежа (комиссии, страховки банка) — нужен механизм корректировки.
- **Банковские нюансы** (−3 балла): разные банки по-разному начисляют проценты, распределяют платежи, обрабатывают досрочное погашение. Модель покрывает типовые случаи, но edge cases конкретных банков могут потребовать доработки.
- **Яндекс Сплит и аналоги** (−3 балла): модель «первый платёж = расход, остальное = рассрочка» работает для Сплита. Но могут появиться другие гибриды с отличающейся механикой.

Архитектура устойчива. Риски — в деталях реализации и банковских edge cases, не в концептуальной модели.
