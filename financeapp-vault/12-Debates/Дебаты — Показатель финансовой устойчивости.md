## Раунд 1: Стартовое предложение — комбинированный порог

### Архитектор

Предлагаю **Направление C** (комбинированный порог из существующих метрик) как отправную точку. Обоснование:

1. **Новая метрика = новые точки отказа.** Запас удалён именно потому, что отдельная метрика имела фундаментальный баг (fungible cash). Комбинация проверенных метрик безопаснее.

2. **Три условия для Этажа 2:**

```
Устойчивость = ALL of:
  (a) DTI < 40%               — Этаж 1 пройден
  (b) Базовый поток > 0       — avg за ≥ 3 завершённых мес.
  (c) Капитал ≥ 0             — нет чистых долгов
```

3. **Каждое условие ловит свой класс рисков:**
   - (a) ловит долговую перегрузку
   - (b) ловит lifestyle inflation (доход 250к, траты 260к — DTI может быть 15%, но человек тонет)
   - (c) ловит ситуацию «долгов больше, чем активов» — нетто-должник

4. **Не обманывается credit_disbursement:**
   - DTI: корректно увеличивается (новый платёж)
   - Поток: disbursement — не доход, не расход → не влияет
   - Капитал: +500к кэш, −500к долг = нетто 0 → не влияет

5. **Graceful degradation:** все три метрики вычисляются автоматически из транзакций. Не требует создания целей или настройки от пользователя.

### Продуктолог

**UX-оценка:**

Плюсы:
- Три условия — это **чеклист**. Пользователи любят чеклисты. «3 из 3 выполнено → поздравляем!»
- Каждое условие привязывается к уроку школы:
  - DTI → Модуль 2 (Кредитная нагрузка)
  - Поток → Модуль 1 (Денежный поток)
  - Капитал → Модуль 3 (Финансовая устойчивость)
- Пользователь видит прогресс: «2 из 3 ✅ — осталось снизить DTI»

Риски:
- Нет **одного числа**. Пирамида предполагает ясный индикатор: «ты на этаже 2 или нет». Три условия сложнее воспринять, чем «Запас = 4.5 мес.»
- Но контраргумент: Запас давал ложное чувство безопасности (F3). Честный чеклист лучше красивого враньеца.

**Предварительный UI:**
```
╭─ Этаж 2: Устойчивость ──────────╮
│  ✅ Долговая нагрузка < 40%      │
│  ✅ Поток положительный           │
│  ❌ Нет чистых долгов             │
│                                   │
│  2 из 3 — осталось немного!      │
╰───────────────────────────────────╯
```

### Критик

**Стресс-тест (5 сценариев):**

**С1. Credit_disbursement 500к на дебете.**
Пользователь берёт потребкредит 500к. Деньги приходят на дебет.
- DTI: был 0%, стал ~6% (платёж 10к/мес при доходе 160к) → ✅ < 40%, корректно
- Базовый поток: disbursement не считается → не изменился → корректно
- Капитал: +500к кэш, −500к долг = 0 → если был ≥ 0, остался ≥ 0
- **ПРОБЛЕМА:** Пользователь ТОЛЬКО ЧТО взял кредит, но порог всё ещё «зелёный». Формально корректно (DTI < 40%, поток не изменился, капитал нетто тот же), но ощущение, что система «не заметила».
- Контраргумент: если DTI < 40% и поток > 0 после кредита — пользователь действительно может его обслуживать. Что плохого? Кредит не равен нестабильности.
- **Оценка: 8/10** — формально верно, но неинтуитивно при больших суммах.

**С2. Фрилансер с волатильным доходом.**
Доход: 300к, 50к, 200к, 0, 400к, 100к (среднее 175к/мес). Расходы стабильны 100к/мес.
- DTI: зависит от среднего → ок
- Базовый поток: avg за 6 мес. = 175к − 100к = +75к > 0 → ✅
- Капитал: растёт → ≥ 0 → ✅
- Корректно: средний поток положительный. Фрилансер с таким средним УСТОЙЧИВ.
- **Оценка: 9/10**

**С3. Семья «Первый шаг» (DTI 42%, нет накоплений).**
- DTI 42% > 40% → ❌ → мгновенный провал порога. Корректно.
- **Оценка: 10/10**

**С4. Кешбэк-пользователь (все через КК, полное погашение).**
- DTI: полное погашение КК → долг на конец месяца ≈ 0, monthly_payment = min. → DTI ≈ 0% ✅
- Поток: покупки = expense (в категориях), погашение КК = transfer → поток корректен ✅
- Капитал: нет долга по КК → корректен ✅
- **Оценка: 9/10**

**С5. Новичок (первый месяц).**
- DTI: можно посчитать сразу (из параметров кредитных счетов)
- Базовый поток: нет 3 мес. данных → **не проверяется**
- Капитал: можно посчитать из текущих балансов
- Порог не может быть пройден ≤ 3 мес. → пользователь «в ожидании»
- **ПРОБЛЕМА:** Первые 3 месяца порог всегда «не определён». Это 25% первого года. Пользователь не видит прогресса по Этажу 2.
- **Оценка: 6/10** — graceful degradation работает, но 3 мес. ожидания — долго.

**Средний балл: 8.4/10. Минимальный: 6/10 (новичок).**

**Промежуточный вердикт: дорабатываем.** Основная проблема — условие «Капитал ≥ 0» и поведение для новичков.

## Раунд 2: Проблема ипотеки и данных новичка

### Архитектор

Критик нашёл два бага. Исправляю оба.

**Баг 1: Ипотека убивает условие «Капитал ≥ 0».**

Профиль «Рост»: доход 200к, DTI 25% (ипотека 50к/мес), поток +30к, вклад 300к. Но ипотека 3М → Капитал = 200к (дебет) + 300к (вклад) − 3М (ипотека) = **−2.5М**. Условие (c) провалено навсегда. Пользователь не пройдёт Этаж 2, пока не закроет ипотеку (через 20 лет).

Это фундаментальная ошибка. Человек с ипотекой и хорошим потоком УСТОЙЧИВ, но Капитал ≥ 0 говорит «нет».

**Решение: заменить абсолютный порог Капитала на траекторию.**

```
(c) Капитал-траектория ≥ 0 за последние 3+ мес.
    (Capital_month_N ≥ Capital_month_N-1 в среднем)
```

Что это значит: вы не теряете в чистых активах. Даже с ипотекой −2.5М, если Капитал растёт (гасите тело + копите) → траектория положительная → ОК.

Верификация на ипотечнике: Capital месяц 1 = −2.50М, месяц 2 = −2.47М, месяц 3 = −2.44М → траектория +30к/мес → ✅

Верификация на credit_disbursement: Capital до = 800к. Кредит 500к: Capital = 800к (нетто). Следующий месяц: тратит кредитные деньги, Capital падает → траектория отрицательная → корректно ловит ❌

Это даже лучше! Траектория ЛОВИТ credit_disbursement через поведение после получения кредита: когда пользователь начинает тратить кредитные деньги, Капитал падает → красный.

**Баг 2: Три месяца ожидания для новичка.**

Не баг, а фича. Рефрейминг:

1. Устойчивость по определению = стабильность во времени. Нельзя быть «устойчивым» по одному месяцу данных.
2. DTI можно проверить с первого дня (из параметров счетов). Это уже Этаж 1.
3. Поток и траектория требуют 3 мес. — это честно.

**UI для новичка:**
```
╭─ Этаж 2: Устойчивость ──────────────────╮
│  ✅ DTI 28% — под контролем              │
│  ⏳ Поток — собираем данные (ещё 2 мес.) │
│  ⏳ Капитал — собираем данные (ещё 2 мес.)│
│                                           │
│  Продолжайте вести учёт — скоро увидим   │
│  полную картину                           │
╰───────────────────────────────────────────╯
```

**Обновлённая формула:**

```
Устойчивость = ALL of:
  (a) DTI < 40%
  (b) avg(Базовый поток, M мес.) > 0, где M = min(available, 6), M ≥ 3
  (c) Капитал-траектория ≥ 0 за последние 3+ мес.
```

### Продуктолог

Траектория вместо абсолютного значения — сильное решение. Оно ложится в философию проекта: **«направление важнее позиции»**. Пользователь с −2.5М по Капиталу, но с ростом +30к/мес — на правильном пути. Это мотивирует, а не обескураживает.

**Уточнение UI:**

Три условия переименованы в понятные человеческие фразы:
1. «Кредитная нагрузка под контролем» (DTI < 40%)
2. «Стабильный положительный поток» (avg Flow > 0)
3. «Финансы растут» (Capital trajectory ≥ 0)

Когда все три зелёные → анимация разблокировки + школьный момент:
> «Вы достигли финансовой устойчивости! Теперь можно переходить к финансовым целям — начните с подушки безопасности.»

Подушка безопасности становится **первой целью** на Этаже 3 (Goals), а не условием Этажа 2. Логично: чтобы начать копить на подушку, нужно быть устойчивым. А не наоборот.

### Критик

**Повторный стресс-тест:**

**С1. Credit_disbursement 500к.** (переоценка)
- DTI: вырос, но < 40% → ✅
- Поток: не изменился → ✅
- Капитал-траектория: в момент получения = 0 (нетто). Через месяц, когда тратит деньги → траектория падает → ❌ если потратил существенно
- Если НЕ тратит (деньги лежат): траектория = 0 → погранично. Но каждый месяц платит тело → Capital растёт (долг уменьшается, а кэш не тратится) → ✅
- Если тратит: Capital падает (кэш уходит, долг остаётся) → ❌ траектория. Корректно!
- **Оценка: 9/10** — ловит проблему через поведение после кредита.

**С2. Фрилансер.** Avg flow +75к > 0 ✅. Капитал-траектория: если средний поток положительный → капитал в среднем растёт → ✅.
- **Оценка: 9/10**

**С3. Первый шаг.** DTI 42% → ❌. Всё.
- **Оценка: 10/10**

**С4. Кешбэк-пользователь.** Без изменений.
- **Оценка: 9/10**

**С5. Новичок.** DTI проверяется сразу. Поток и траектория — через 3 мес. UI показывает прогресс.
- **Оценка: 7/10** — лучше (+1), но всё ещё 3 мес. ожидания.

**Средний балл: 8.8/10. Минимальный: 7/10 (новичок).**

**Промежуточный вердикт: дорабатываем.** Ипотека решена. Новичок — приемлемо, но есть нюанс.

## Раунд 3: Контрпредложение — убрать Этаж 2?

### Архитектор

Прежде чем финализировать, проверим **Направление E** — убрать уровень целиком.

**Аргументы «за» удаление:**

1. Пирамида — **педагогическая модель**, не техническая система. Она объясняет приоритеты. Необязательно каждый этаж привязывать к отдельной метрике.

2. После удаления Запаса, «Устойчивость» как отдельный этаж — это искусственное удержание пустого слота. Мы ищем метрику не потому, что она нужна, а потому что дыра в пирамиде.

3. **Четырёхуровневая пирамида** вполне работает:
```
    ╭───────────╮
    │  Вершина  │  Финансовая свобода
    ├───────────┤
    │  Этаж 2   │  Финансовые цели
    ├───────────┤
    │  Этаж 1   │  Контроль долгов (DTI < 40%)
    ├───────────┤
    │ Фундамент │  Осознанность (мультиметрика)
    ╰───────────╯
```

**Аргументы «против» удаления:**

1. Это оставляет **дыру в пользовательском пути**. Между «DTI < 40%» и «начинай ставить цели» — качественный скачок. Пользователь снижает DTI с 42% до 38%. Ура, этаж 1 пройден! И сразу: «ставь финансовые цели». Но он ещё нестабилен — поток может быть отрицательным, капитал падает.

2. **Пирамида Маслоу работает потому, что каждый уровень — отдельная потребность.** «Безопасность» (stability) и «принадлежность» (goals) — разные вещи. Убрать «безопасность» — нарушить логику.

3. **Школьные модули привязаны к этажам.** Модуль 3 = «Финансовая устойчивость». Если этажа нет — модуль повисает без привязки.

### Продуктолог

Соглашаюсь с аргументами «против». Этаж 2 нужен.

Ключевой **педагогический аргумент:** пользователь «Первый шаг» после снижения DTI с 42% до 38% нуждается в промежуточной победе. «Вы снизили долговую нагрузку — теперь убедитесь, что ваш финансовый фундамент стабилен». Без этого этажа — скачок от «контроль долгов» к «финансовые цели» слишком резкий.

Кроме того, **подушка безопасности как первая цель** работает ТОЛЬКО если есть этаж между долгами и целями. Иначе подушка — просто ещё одна цель наравне с «отпуск» и «машина».

### Критик

Согласен. Направление E **отвергнуто**. Этаж 2 нужен по структурным и педагогическим причинам.

**Промежуточный вердикт: Направление E отвергнуто. Возвращаемся к доработке комбинированного порога.**

## Раунд 4: Доработка формулы + конкретный UI

### Архитектор

Финализирую формулу. Два уточнения:

**Уточнение 1: формализация Капитал-траектории.**

```
Capital_trend = linear_slope(Capital_values, last N months)

Условие: Capital_trend ≥ 0
```

Но linear slope — overkill для порога. Упрощаем:

```
Capital_trend_positive = count(months where Capital_M > Capital_M-1) ≥ ceil(N * 0.5)
                         over last N completed months (N = min(available, 6), N ≥ 3)
```

Иными словами: в большинстве месяцев Капитал рос. Для 6 мес. → ≥ 3 мес. роста. Для 3 мес. → ≥ 2 мес.

Почему «большинство», а не «все»:
- Допускает 1-2 плохих месяца (нерегулярные траты, ремонт)
- Фрилансер с волатильным доходом не наказывается за один плохой месяц

**Уточнение 2: формализация Потока.**

```
Flow_positive = avg(Basic_Flow, last M completed months) > 0
                where M = min(available_months, 6), M ≥ 3
```

Среднее, а не медиана: если avg > 0, значит суммарно пользователь в плюсе. Один плохой месяц компенсируется хорошими → устойчив. Если avg ≤ 0, то СУММАРНО в минусе → неустойчив.

**Финальная формула:**

```
Порог устойчивости (Этаж 2) = ALL of:
  (a) DTI < 40%
  (b) avg(Базовый поток, M мес.) > 0
      M = min(available_completed_months, 6), минимум 3
  (c) Капитал рос в ≥ ceil(N/2) из последних N месяцев
      N = min(available_completed_months, 6), минимум 3
```

**В FI-score:** НЕТ. Обоснование:
- FI-score — непрерывная шкала (0-10) для страницы Health
- Порог устойчивости — бинарный гейт для пирамиды
- Разные инструменты для разных задач
- Добавление в FI-score вернёт проблему компенсации (один компонент маскирует другой)

### Критик

**Полный стресс-тест финальной формулы:**

**С1. Credit_disbursement 500к на дебете.**

Исходное: DTI 0%, поток +80к, капитал 800к (6 мес. истории, все растущие).

Получает потребкредит 500к. Кэш → 1300к, долг 500к.
- Месяц 0: Капитал = 1300к − 500к = 800к (без изменений, нетто = 0)
- DTI: +6% → 6% < 40% ✅
- Поток: без изменений → +80к ✅

Сценарий А: тратит 500к за 2 месяца (250к/мес сверх обычного):
- Месяц 1: Капитал = (1300 − 250) − 490 = 560к → падение −240к
- Месяц 2: Капитал = (800 − 250) − 480 = 70к → падение −490к
- Капитал рос в 0 из последних 2 мес. → ❌ условие (c) ломается
- **Корректно**: человек активно тратит кредитные деньги, нестабильность видна.

Сценарий Б: не тратит, деньги лежат, платит по графику:
- Месяц 1: Капитал = (1300 − 10 тело − расходы) − (500 − 10) = ~800к → рост ≈ 0
- Стабильный → ✅
- **Корректно**: кредит есть, но человек его обслуживает и финансы стабильны.

**Оценка: 9/10**

**С2. Фрилансер (доходы: 300к, 50к, 200к, 0, 400к, 100к; расходы 100к/мес).**
- DTI: зависит от кредитов, допустим 0% → ✅
- avg(поток, 6 мес.) = avg(200, −50, 100, −100, 300, 0) = +75к > 0 → ✅
- Капитал: рос в 3 из 6 мес. (мес. с доходом > расходов) → = ceil(6/2) = 3 → ✅
- **Корректно**: несмотря на волатильность, в среднем в плюсе.

А если доходы: 150к, 50к, 100к, 0, 200к, 80к (avg = 97к, расходы 100к)?
- avg(поток) = −3к < 0 → ❌ условие (b)
- **Корректно**: на грани, средний поток отрицательный — неустойчив.

**Оценка: 9/10**

**С3. Семья «Первый шаг» (DTI 42%).**
- DTI 42% > 40% → ❌
- Мгновенный отказ. Ни поток, ни капитал не проверяются.
- **Оценка: 10/10**

**С4. Кешбэк-пользователь.**
- DTI: full repayment → ~0% ✅
- Поток: expenses counted at purchase → correct avg > 0 ✅
- Капитал: no debt → growing ✅
- **Оценка: 10/10**

**С5. Новичок (первый месяц).**
- DTI: можно вычислить из Account.monthly_payment сразу → отображается

- Через 3 мес.: все три условия проверяемы
- **Оценка: 8/10** — 3 мес. ожидания оправданы (устойчивость = стабильность во времени), UI прозрачно объясняет.

**Средний балл: 9.2/10. Минимальный: 8/10.**

## Раунд 5: Edge cases + сходимость

### Архитектор

**Edge case 1: Механизм «Проект» (ремонт 3 мес.)**

Пользователь начинает ремонт: 150к/мес сверх обычного в течение 3 мес. Потрачены собственные накопления (не кредит).

- Поток: если ремонт = нерегулярные расходы (категория «Ремонт», `irregular`) → Базовый поток НЕ затронут → условие (b) ✅
- Капитал: падает 3 мес. подряд (тратит накопления) → условие (c) ❌
- Порог временно проваливается по капитал-траектории.
- **Это ПРАВИЛЬНО.** Во время крупных трат пользователь объективно менее устойчив. После завершения ремонта капитал начнёт восстанавливаться → через 2-3 мес. порог вернётся.
- UI: «Ваш капитал снижался последние 3 мес. Если это плановые траты (ремонт, переезд), они временно влияют на устойчивость.»

**Edge case 2: Пользователь только что закрыл последний кредит.**

- DTI: 0% → ✅
- Поток: улучшился (нет платежей) → ✅
- Капитал: долг исчез, скачок вверх → рост → ✅
- **Порог пройден. Корректно:** закрытие кредита = достижение устойчивости.

**Edge case 3: Депозит открыт/закрыт (перемещение между счетами).**

Пользователь переводит 300к с дебета на вклад.
- Капитал: дебет −300к, вклад +300к → нетто 0 → траектория не затронута.
- Корректно: перемещение между собственными счетами не влияет на устойчивость.

**Edge case 4: Рассрочка 0% (installment_card).**

Покупка 60к по рассрочке (12 мес. × 5к).
- Расход: 60к в категории «Электроника» → Поток: если irregular — не в базовом потоке
- DTI: +5к/мес → DTI растёт → корректно
- Капитал: +60к актив (уже потрачен), −60к долг → нетто 0 → траектория без изменений
- Через месяцы: тело рассрочки списывается (перевод), долг уменьшается → капитал растёт
- **Корректно.** Рассрочка увеличивает DTI (правильно) и не ломает капитал/поток.

### Продуктолог

**Финальные UX-решения:**

1. **Название этажа:** «Финансовая устойчивость» — без изменений. Метафора работает: не «запас», а «устойчивость» (стабильное состояние, а не накопленный буфер).

2. **Три условия = три навыка.** Педагогическая привязка:
   | Условие | Навык | Урок |
   |---------|-------|------|
   | DTI < 40% | Контроль долгов | Модуль 2 |
   | Поток > 0 | Управление расходами | Модуль 1 |
   | Капитал растёт | Накопление | Модуль 3 |

3. **Принцип «приложение растёт с пользователем»:** все три условия видны с первого дня, но заполняются по мере накопления данных. Это сама по себе мотивация: «вижу чеклист, хочу заполнить».

4. **Graceful degradation:**
   - Нет данных (< 3 мес.): «Собираем данные о вашем потоке и капитале»
   - Есть DTI, нет потока: «DTI под контролем ✅ Ожидаем данные о потоке (ещё X мес.)»
   - Все данные есть: полный чеклист

5. **Подушка безопасности → первая рекомендованная цель** на Этаже 3. Не условие устойчивости, а первый шаг после её достижения. Школьный момент: «Устойчивость достигнута! Рекомендуем: подушка безопасности = 3-5 мес. расходов.»

### Критик

**Финальная сводная оценка:**

## Рекомендация

**Решение:** Комбинированный порог из трёх существующих метрик (Направление C, доработанное). Не вводить новую метрику — использовать DTI, Базовый поток и Капитал-траекторию как гейт.

**Формула:**

```
Порог устойчивости = ALL of:
  (a) DTI < 40%
  (b) avg(Базовый поток, M мес.) > 0
      M = min(available_completed_months, 6), minimum M = 3
  (c) Капитал рос в ≥ ceil(N/2) из последних N завершённых месяцев
      N = min(available_completed_months, 6), minimum N = 3

**В пирамиде:**

Этаж 2: Финансовая устойчивость
Условия: контроль долгов + стабильный поток + растущий капитал
```

**Порог устойчивости:**
- DTI < 40% (кредитная нагрузка управляема)
- Средний базовый поток > 0 за 3-6 мес. (зарабатываете больше, чем тратите)
- Капитал растёт в большинстве месяцев (финансовая ситуация улучшается)

## Раунды 6–10: Поиск показателя буфера (расширенное обсуждение)

> Обсуждение вышло за рамки формата «три роли» и перешло в свободный аналитический диалог.

### Проблема комбинированного порога

Комбинированный порог (DTI + Поток + Капитал-траектория) использует **те же три входа**, что и FI-score (DTI inverse + Норма сбережений + Капитал-траектория). Разница — только в математике (AND vs взвешенная сумма). Это создаёт ощущение дублирования.

Кроме того, комбинированный порог не отвечает на ключевой вопрос устойчивости: **«Переживу ли я удар?»** Он говорит «твои потоки и долги в порядке», но не про буфер.

### Возврат к показателю буфера

Четыре параметра устойчивости человека (без контекста приложения):

1. Тратишь меньше, чем зарабатываешь (→ Поток)
2. Долги не душат (→ DTI)
3. **Переживёшь удар** (→ ???) ← это Этаж 2
4. Становится лучше, не хуже (→ Капитал-траектория)

Параметры 1, 2, 4 — про поведение. Параметр 3 — про накопленный буфер. Именно его не хватает.

### Исследование: мировая практика

Проанализированы подходы: YNAB (конверты), JP Morgan Chase Institute (медиана месячных минимумов), Plaid, PocketGuard, Financial Health Network, CFPB, Commonwealth.

**Вывод:** ни одно приложение не решило автоматическую декомпозицию баланса по намерениям. Все успешные подходы: (а) пользователь сам раскладывает деньги (YNAB), (б) поведенческие прокси (JPMCI — медиана минимумов), (в) не измеряют буфер напрямую (комбинированные пороги).

### Анализ медианы месячных минимумов (JPMCI)

JP Morgan Chase Institute используют медиану минимальных дневных балансов по месяцам за 6-12 мес. Медиана фильтрует выбросы (разовая покупка машины — один месяц из 12, не двигает медиану).

**Проблема:** по сути это **сглаженная версия текущего баланса**. Для статистики по 6 млн счетов — отлично. Для персонального показателя — та же проблема fungible денег. Человек с 1М на карте «на машину» → медиана минимумов ≈ 900к → метрика говорит «устойчив» → покупает машину → обвал.

### Анализ модального подхода (KDE)

Предложение: искать моду распределения дневных балансов (самый частый уровень = «равновесие»). Работает для стабильного зарплатника, но ломается при тренде (копит → распределение размазано, моды нет) и волатильности (фрилансер → несколько горбов).

### Анализ тегирования кредитных средств

Два варианта: FIFO-списание кредитного пула, вопрос пользователю «на что планируете потратить?»

**Вывод:** FIFO произволен (почему продукты — из «кредитных денег»?), а результат математически эквивалентен `баланс − долг` = Капитал. Вопрос пользователю добавляет friction и edge cases (передумал, два кредита, смешанные траты). Тегирование не работает, потому что **деньги на одном счёте — это одно число**, разливать воду обратно по кранам невозможно.

### Фундаментальный вывод

**Автоматический показатель буфера не существует.** Это не техническая задача, а фундаментальное ограничение fungible денег. Любая метрика на базе баланса счёта не может отличить:
- «накопил на машину» от «подушка безопасности»
- «кредитные деньги» от «заработанные»
- «скоро потрачу» от «трогаю только в крайнем случае»

**Точность и автоматизация — две крайности спектра.** Нельзя получить оба свойства одновременно.

### Решение: деньги на вкладах как прокси буфера

**Ключевое наблюдение:** тип счёта = естественный фильтр намерения.

- Деньги на картах (regular, cash) = операционные, на расходы
- Деньги на вкладах и накопительных (deposit) = **сознательно отложенные**

Перевод на вклад — это действие: человек решил «эти деньги я не трачу на ежедневные расходы». Не нужно создавать цель, не нужно отвечать на вопросы.

**Формула:**

```
Буфер устойчивости = Σ(balance, deposit accounts) / avg(monthly_expenses)
Порог: ≥ 3 мес.
```

### Сравнение: вклады vs ликвидный капитал

| Критерий | Ликв. капитал | Вклады |
|----------|:---:|:---:|
| Автоматичность | ✅ | ✅ |
| Credit_disbursement | ✅ нетто 0 | ✅ не затрагивает |
| Отделяет буфер от расходных | ❌ | ✅ |
| Учитывает долги | ✅ | ❌ (DTI уже учёл) |
| Работает без вкладов | ✅ | ❌ (обучающий момент) |
| Понятность для ЦА | ⚠️ абстрактно | ✅ просто |
| Учит правильному поведению | Нейтрально | ✅ мотивирует |

**Вклады выбраны** как основа порога устойчивости. Долговая проверка уже на Этаже 1 (DTI), дублировать её не нужно. Ложный негатив (всё на дебете) — обучающий момент школы.
