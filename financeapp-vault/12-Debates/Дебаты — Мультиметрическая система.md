# Дебаты — Мультиметрическая система

> Протокол дебатов Cowork (финансовый эксперт-партнёр) vs Claude Code (критический аналитик) по [[Гипотеза — Мультиметрическая система]].

## Участники
- **Cowork** — роль: финансовый консультант-методолог, эксперт-партнёр школы. Защищает гипотезу.
- **Claude Code** — роль: критический аналитик. Ищет слабые места и проверяет на прочность.
- **Григорий** — арбитр. Принимает финальные решения.

## Статус
✅ Завершены — 8 раундов. Консенсус: 80/20 за мультиметрику. Рекомендация: принять гипотезу.

## Предмет дебатов
Замена СМО как единого фундамента на четыре равноправные метрики (Поток, Капитал, Нагрузка, Запас) с переходом на индустриальный подход учёта кредитных операций.

## Раунд 1 — Первичный анализ

### Claude Code: анализ стресс-теста
**Сильные стороны мультиметрики (согласен):**
1. Убирает корневую причину, а не симптомы — 14 документов кредитной логики существуют ради хирургических исключений из СМО
2. Три проблемы буквально исчезают (кешбэк, импульсная покупка, Яндекс Сплит)
3. Двойной счёт невозможен по конструкции
4. Одна аналитика вместо двух
5. Педагогически сильнее — радар (слабое звено) vs пирамида (линейная)

**Уязвимости (5 штук):**
1. Аномалии в среднем потоке — крупные покупки искажают среднее на месяцы. Медиана, пометка «разовая покупка», MAD-фильтр — все ранее отвергались по другим причинам
2. Ремонтный сценарий — 3 месяца красного потока, медиана не спасёт (3 из 6 месяцев отрицательные)
3. Разделение тело/проценты — обязательное условие, скрытая сложность
4. «Доступные финансы» всё ещё врут — 600к на счёте при 500к кредитных
5. Рефакторинг школы — не нулевая стоимость, 25+ уроков спроектированы

**Главный вывод:** мультиметрическая возвращает исходную проблему — пользователь видит плохое число, не понимает почему.

### Cowork: ответ на раунд 1
**Ключевой контраргумент:** Claude Code путает две РАЗНЫЕ проблемы.
- Исходная проблема = **двойной счёт** (математическая ложь: 500к покупок + 500к погашений = 1000к при реальных 500к)
- Аномалия в потоке = **правильная математика с неудобным результатом** (поток −55к при ремонте — это правда, пользователь реально тратит больше, чем зарабатывает)

**Педагогический аргумент (Ответ А vs Ответ Б):**
- Ответ А (мультиметрика): «Ты тратишь больше, чем зарабатываешь. Это ок — плановый ремонт. Посмотри на капитал и нагрузку — они в контроле.»
- Ответ Б (текущая модель): «СМО = +95к, всё в порядке.» ← ложь, пользователь тратит 255к при доходе 200к.

Школа финансовой грамотности должна показывать реальность, а не прятать.

**Также:** разделение на UX vs архитектурную задачу — UX можно итеративно улучшать, архитектурную нельзя.

**Предложены 4 блокера** для принятия гипотезы.

### Баланс после раунда 1
Claude Code признал: различие «ложь модели vs неудобная правда» — ключевое. Снял свой сильнейший аргумент на ~70%.

## Раунд 2 — Конкретные механизмы

### Claude Code: ответ на раунд 1
**Согласен:**
- Различие двойной счёт / аномалия — ключевое
- Ответ А педагогически сильнее Ответа Б
- Стоимость рефакторинга школы снижена (vault, не код)
- 4 блокера хорошо структурированы

**Не согласен:**
1. «UX-задача, решаемая после запуска» — слишком упрощённо. Если FI-score, алерты, школьные триггеры завязаны на поток — красный поток при плановом ремонте ломает всю систему доверия. Это архитектура.
2. Не ответил на вопрос доступных финансов конкретно
3. Тело/проценты — два варианта без выбора

### Cowork: ответ на раунд 2

**Принял поправку** про UX/архитектуру — если алерты и FI-score завязаны на метрику, это действительно архитектурное решение.

**Три конкретных механизма:**

**1. Алгоритм тело/проценты — ЗАКРЫТ:**
- Кредитные карты: `проценты = долг_конец − долг_начало + погашения − покупки`. Все данные из транзакций.
- Потребкредиты: аннуитетный график из ставки и срока (указывает пользователь при создании счёта).
- Рассрочки: проценты = 0, все платежи = переводы.

**2. Доступные финансы — ЗАКРЫТ:**
- Два числа: «На счетах: 600 000 ₽» и «Ваши деньги: 100 000 ₽» (= капитал).
- Запас = чистый запас (капитал / расходы). Показывает «сколько продержусь без ухудшения позиции».
- Физический запас — вторичный показатель.

**3. Механизм «Проект» для аномалий:**
- При credit_disbursement пользователь может пометить кредит как проект (сумма + срок). Одно действие.
- Система показывает два потока: **базовый** (без проекта) и **полный** (с проектом).
- Алерты и FI-score работают по базовому потоку.
- Школьный момент даёт контекст.

### Баланс после раунда 2
Claude Code признал: тело/проценты и доступные финансы закрыты. Баланс 65/35 в пользу мультиметрики.

## Раунд 3 — Проект как механизм

### Claude Code: анализ «Проекта»
**Согласен:** концепция правильная, одно действие при получении кредита.

**Критика:**
1. Пробел в механике: как система вычитает «проектные» расходы? Два варианта:
   - Фиксированное распределение (бюджет / срок = ежемесячная корректировка) — просто, но неточно помесячно
   - Привязка к транзакциям — точно, но = счётчик v2 / тегирование
2. Fallback: что если пользователь не создаст проект? Три красных месяца без контекста.
3. Наблюдение: Проект = counter v2 в другой шкуре. Обе архитектуры требуют пометки при получении кредита.

**Собственный прогноз:** фиксированное распределение достаточно для 80% случаев.

### Cowork: ответ на раунд 3
**Принял:** Проект = counter v2 в другой шкуре. Это не слабость — это природа задачи. Потребкредит на общие нужды смешивает деньги на одном счёте, ни одна система не различит автоматически.

**Разница с counter v2:** после пометки в мультиметрике — два честных числа, одна аналитика. В текущей — хирургическое исключение из СМО, расхождение аналитик, привязка категорий.

**Зафиксировано:**
- ✅ Фиксированное равномерное распределение (бюджет / срок)
- ✅ Промпт при credit_disbursement: «Это проект?»
- ✅ Школьное напоминание, если поток упал после получения кредита
