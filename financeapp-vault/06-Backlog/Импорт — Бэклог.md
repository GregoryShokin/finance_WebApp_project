## Задачи

### Б-01 — Visible error state в UI

**Приоритет:** Высокий | **Тип:** UX
**Зависит от:** Б-02 (рекомендуется)

Строки в статусе `error` сейчас «замерзают» без понятных действий. Юзер не знает что с ними делать.

**Что нужно:**
- В модераторе error-строки показываются в отдельной секции с понятным объяснением причины («не удалось определить сумму», «сломан формат даты», «transfer без счёта назначения»).
- Две кнопки на строку: **«Исключить»** (→ `excluded`) и **«Отложить»** (→ `parked`).
- Массовое действие на уровне секции: «Исключить все», «Отложить все».
- Коммит не блокируется наличием error-строк — они просто пропускаются. Но секция с ними видна.

**Связь со спеком:** §5.2 (triggers), §5.3 (переходы).

### Б-03 — Миграция legacy TransactionCategoryRule

**Приоритет:** Высокий | **Тип:** Миграция данных
**Блокирует:** чистую работу §6 спека

Существующие правила имеют поле `scope` со значениями `exact` / `bank` / `global` / `legacy_pattern`. В целевой модели — только `specific` (по идентификатору) и `general` (по скелетону).

**Что нужно:**
- `scope='exact'` с идентификатором → `specific`.
- `scope='global'` по скелетону → `general`.
- `scope='bank'` — переосмыслить: bank как контекст идёт в account_type metadata и LLM-контекст, а не в rule scope. Таким правилам — либо разжать в `specific`/`general`, либо пометить broken.
- `scope='legacy_pattern'` — пометить inactive, дать юзеру восстановить через подтверждение.
- Сохранить confirms/rejections при миграции.
- Alembic-миграция аддитивная (не ломает существующих юзеров).

**Порог доверия** при миграции `legacy_pattern` → `is_active = True` при `hit_count ≥ 5`, иначе `False`. Откалиброван по замеру БД 2026-04-21 (226 правил, 1 юзер): оставляет активными ~48% правил, отсекает пик `hit=3-4` (вероятный источник P-05).

**Связь со спеком:** §6.1, §6.8.

### Б-09 — Import Session Health Check

**Приоритет:** Высокий | **Тип:** UX / Наблюдаемость
**Зависимости:** нет

Четыре автоматические проверки при `build_preview`, результат пишется в `session.summary_json.health`. Показываются как dismissible banner вверху модератора. Не блокируют коммит.

**Что нужно — новый сервис `ImportSessionHealthService`:**

**1. Overlap detection**
```
Если date_range(session) пересекается с committed-транзакциями
того же account_id на >20%:
  → "Возможно этот период уже импортирован.
     Дубликаты будут помечены автоматически."
```

**2. Gap detection**
```
Если между двумя соседними строками >7 рабочих дней (нет операций):
  → "Возможно выписка неполная:
     нет операций с {date_from} по {date_to}"
```

**3. Format change signal**
```
Если error_rate текущей сессии > 15%
AND error_rate предыдущих 3 сессий того же счёта < 5%:
  → "Банк мог изменить формат выписки.
     Проверь маппинг колонок."
```

**4. Balance reconciliation** *(если баланс известен)*
```
Если account.balance известен на дату last_row
AND |computed_balance - account.balance| > 1%:
  → "Расхождение баланса на {diff}₽.
     Возможно пропущены строки."
```

**Реализация:** один вызов `health_service.check(session, rows)` в конце `build_preview`, пишет в `summary_json["health"]`. Frontend показывает как banner.

### Б-10 — Recurring Detection

**Приоритет:** Высокий | **Тип:** Фича — автоматизация
**Зависимости:** нет

Самый частый источник ошибки в Базовом Потоке — операции помечены `is_regular=False`, хотя юзер платит их каждый месяц. Зарплата, аренда, подписки, коммуналка. Plaid добивается 95% точности на salary/subscriptions именно через паттерн-матчинг по истории.

**Алгоритм — `RecurringDetectorService.check(row, user_id)`:**
```
Ищем в committed-транзакциях паттерн:
  (fingerprint совпадает ИЛИ skeleton совпадает)
  AND (amount ±10%)
  AND (day_of_month ±3)
  AND (direction совпадает)

Если ≥3 совпадений за последние 14 месяцев:
  → recurrence_detected = True
  → suggested_interval = monthly | biweekly | weekly (по медиане diff)
  → suggests is_regular = True
```

Результат пишется в `normalized_data.recurrence_hint` — **не применяется автоматически**, передаётся в модератор как suggestion. Юзер видит бейдж «↻ Повторяющаяся» и подтверждает одним кликом.

**Классификация по типу паттерна:**

| Паттерн | Признаки | Предлагаемая категория |
|---|---|---|
| **Зарплата** | income, 30k–500k, ±3 дня ежемесячно, один источник | «Зарплата», `is_regular=True` |
| **Подписка** | expense, точная сумма ≤3k, ежемесячно, один мерчант | «Подписки», `is_regular=True` |
| **Аренда** | expense, крупная сумма, ежемесячно, один контрагент | предложить «Аренда» |
| **Коммуналка** | expense, переменная сумма ±20%, ежемесячно, ЖКУ-маркеры | «Коммунальные», `is_regular=True` |
| **Кредит** | expense, фиксированная сумма, ежемесячно, банк-маркер | «Платёж по кредиту» + предложить split |

**СБП-кейс:** если income от одного phone_token повторяется ≥3 раза в ±3 дня каждого месяца на крупную сумму → дополнительно предложить `is_regular=True` (CounterpartyIdentifier уже запоминает контрагента, recurring detector добавляет только сигнал регулярности).

**Инварианты:**
- Только suggestion — никогда не применяется тихо.
- Не запускается для строк с `user_confirmed_at`.
- Паттерн ищется только в committed-транзакциях.

**Интеграция с Б-11:** если recurring detector нашёл паттерн с явными amount/day признаками → предложить создать Compound Rule с этими условиями.

### Б-11 — Compound Rules (составные правила)

**Приоритет:** Средний | **Тип:** Фича
**Связана с:** Б-10

Текущие правила — только по skeleton. Не позволяют поймать зарплату через СБП (нет стабильного skeleton, но есть паттерн суммы + дня + направления) или аренду (один контрагент, но описание каждый раз разное).

**Структура:**
```
CompoundRule:
  id, user_id, name, priority (int)

  Условия (все AND):
    description_contains: str | None
    amount_min: Decimal | None
    amount_max: Decimal | None
    direction: "income" | "expense" | None
    day_of_month_from: int | None  (1–31)
    day_of_month_to: int | None
    account_id: int | None
    counterparty_id: int | None

  Результат:
    category_id: int | None
    operation_type: str | None
    is_regular: bool | None
    debt_partner_id: int | None
    note: str | None
```

**Приоритет в pipeline:**
```
Compound rules (по priority DESC)
  → Specific identifier rules
  → Bank-scope rules
  → Legacy skeleton rules
  → LLM moderation
  → Manual
```

**Создание:**
- Автоматическое предложение от Recurring Detector (Б-10).
- Ручное из модератора: кнопка «Создать умное правило» с предзаполненными условиями из строки.

**Инварианты:**
- Compound rule не меняет facts (amount, date, account_id). Только decisions.
- `is_regular=True` через compound rule не применяется к историческим committed-транзакциям.

### Б-12 — Receipt Intelligence (ФНС API)

**Приоритет:** Средний | **Тип:** Большая фича
**Зависимости:** нет (opt-in канал)

Маркетплейсы и супермаркеты — крупнейшие статьи расходов, но выписка даёт одну строку: «WILDBERRIES 5 440₽». Невозможно отличить продукты от одежды. По закону 54-ФЗ все продавцы России обязаны регистрировать чеки в ФНС. ФНС API (`proverkacheka.nalog.ru`) — бесплатный и официальный, покрывает 100% продавцов.

**Каналы получения чеков:**

| Канал | Покрытие | Трение |
|---|---|---|
| Telegram бот — пересылает QR от WB/Ozon/ЯМ | ~70% МП-покупок | Минимальное |
| Email parser — автоимпорт писем от noreply@ozon.ru | ~50% при подключённом email | Нулевое после setup |
| QR-сканер — мобильное приложение (фаза 10+, **offline-first**) | ~30% бумажных чеков | Среднее |

**Пайплайн:**
```
1. Чек получен (любым каналом)
2. QR decode → {fn, fd, fp}
3. ФНС API: GET /check → {items: [{name, quantity, price}], totalSum, merchant}
4. Transaction matching: date ±2ч, amount ±0.01₽
5. Item classification (ReceiptItemClassifier):
   - Keyword rules: «молоко» → Продукты
   - LLM fallback (Haiku): «GLOV Face Mask Premium» → Уход за собой
6. Split proposal: каждая позиция → category_id + amount
7. Юзер подтверждает → commit создаёт N транзакций (split_items)
```

**Модели данных:**
```python
class Receipt(Base):
    id, user_id, fns_fn, fns_fd, fns_fp
    merchant_name, merchant_inn
    total_amount, receipt_date
    source: str  # telegram | email | qr_scan
    matched_transaction_id: int | None

class ReceiptItem(Base):
    id, receipt_id, name, quantity, price, sum
    category_id: int | None
    item_skeleton: str | None  # для rule learning
```

**Item-level rule learning:** после подтверждения каждая пара `(item_skeleton, category_id)` сохраняется как item-rule. При следующем появлении — автопредзаполнение.

**Инварианты:**
- Opt-in: без чека транзакция проходит обычный пайплайн.
- Сумма чека и транзакции обязана совпадать ±1₽.
- ФНС API недоступен → graceful fallback.
- Чеки хранятся только для matched-транзакций пользователя.

**Offline-first сканер (iOS, FinanceappIOS):**

Сценарий-мотивация: если приложение не попадёт в белые списки РФ и операторы заблокируют ему мобильный интернет, на кассе пользователь всё равно должен суметь отсканировать чек. Camera API работает офлайн без ограничений — сетевая часть выносится в фоновую очередь синхронизации.

- **Локальное хранилище** на устройстве (SwiftData, Application Support):
  ```swift
  @Model final class PendingReceipt {
      @Attribute(.unique) var clientId: UUID  // идемпотентный ключ
      var qrRaw: String                       // сырая строка QR: t=...&s=...&fn=...&i=...&fp=...&n=1
      var scannedAt: Date
      var status: PendingReceiptStatus        // pending | syncing | sent | failed
      var attempts: Int
      var lastError: String?
      var serverReceiptId: Int?
  }
  ```
  Сканирование = `INSERT` без сетевых вызовов; отклик мгновенный, никакой блокировки на кассе.

- **Sync-воркер** (`ReceiptSyncService`, синглтон) запускается на:
  - старт приложения и переход в foreground (`scenePhase`),
  - событие появления сети из `NWPathMonitor`,
  - системный `BGProcessingTask` / `BGAppRefreshTask` — iOS будит приложение даже когда оно закрыто.
  
  Берёт `status == .pending` батчем, шлёт POST, маркирует `.sent`. На ошибке — экспоненциальный бэкофф, `attempts++`.

- **Новый endpoint на бэке:**
  ```
  POST /api/v1/receipts/scan
  body: { client_id: "uuid", qr_raw: "...", scanned_at: "..." }
  202 Accepted → { receipt_id }
  ```
  `client_id` — идемпотентный ключ, повторная отправка одного чека не плодит дублей. Бэк ставит Celery-таск (тянет ФНС → парсит позиции → создаёт `Receipt` + `ReceiptItem`), iOS получает `receipt_id` сразу без ожидания ответа ФНС.

- **UI обратной связи**: рядом с pending-чеком бейдж «Ожидает синхронизации» / «Загружено» / «Ошибка ×3 — повторить вручную». Список pending-чеков виден отдельным экраном в разделе чеков.

**Инварианты офлайн-режима:**
- ФНС-вызовы делает только бэкенд, никогда iOS — там подписи, токены, нестабильное API.
- iOS хранит **только сырой QR + момент сканирования**, не парсит формат.
- `client_id` обязателен и генерируется на устройстве; бэк — единственный источник правды для статуса чека после успешной синхронизации.
- Локальная очередь не имеет лимита по количеству, но при `attempts >= 5` чек уходит в `.failed` и требует ручного действия пользователя (повтор/удаление).

**Объём (оценка):**
- iOS: `Features/Receipts/` — сканер `AVCaptureSession` (~150 LOC), SwiftData модель + `ReceiptSyncService` с `NWPathMonitor` и `BGTaskScheduler` (~300 LOC), список pending (~200 LOC). Итого 1–2 дня.
- Бэк: endpoint + Celery task; если парсер ФНС-чеков уже есть из других каналов (Telegram/email) — переиспользуем, иначе ~неделя на парсер.

**Зависимость:** offline-режим в iOS не зависит от вопроса дистрибуции (App Store / TestFlight / enterprise-сертификат) — это отдельная история про доставку самого бинарника.

### Б-13 — Яндекс Дебет bank_code detection

**Приоритет:** Средний | **Тип:** Фикс
**Зависимости:** нет

Текущий хардкод `bank_code='yandex'` в bank_mechanics предполагает, что оба счёта (Дебет и Сплит) имеют bank_code=yandex. Если у пользователя Яндекс Дебет и карта другого банка — правила §9.10 не сработают.

**Что нужно:**
- Explicit детектирование bank_code для Яндекс Дебет через pdf_extractor.
- Расширить условия срабатывания bank_mechanics rules: вместо проверки только bank_code — проверять признаки в заголовке выписки.

### Б-15 — Whitelist банков для импорта

**Приоритет:** Средний | **Тип:** Фича
**Решение принято:** 2026-04-27

### Б-17 — Активная перепривязка bindings при merge контрагентов

**Приоритет:** Средний | **Тип:** Рефакторинг
**Связь со спеком:** §11.1

Сейчас при merge A→B в контрагентах `CounterpartyFingerprint` и `CounterpartyIdentifier`, указывающие на A, остаются «висящими» — лениво перепривязываются при следующем матче.

**Что нужно:**
- В транзакции merge-операции: после обновления ссылок в Transaction'ах → пройти по всем `CounterpartyFingerprint` / `CounterpartyIdentifier` с `counterparty_id = A.id` и обновить на `B.id`.
- Атомарно в той же транзакции — если падает, откат всего merge.
- Тест: после merge новый импорт со старым fingerprint'ом корректно подставляет B.

Решение «этот fingerprint = платёж по кредиту, открыть split-форму» сейчас смешано с обычной category rule логикой через hardcode.

### Б-21 — Multi-dimensional LLM moderation

**Приоритет:** Низкий | **Тип:** Оптимизация

Паттерны типа «Линда + крупная сумма = аренда, Линда + мелкая = коммуналка», «120k от договора N = ЗП, 60–70k от того же договора = дивиденды». Текущая одноразмерная классификация такое не ловит.

**Что нужно:**
- Расширить input для LLM: массив (fingerprint, примеры сумм, типичный диапазон, частота, account type, сезонность).
- Prompt-template, который явно предлагает LLM искать многомерные паттерны.
- Хранение результата в модели — вероятно, rule с multi-condition.

**Приоритет низкий:** базовый случай «один fingerprint → одна категория» покрывает 95%+ трафика. Это оптимизация оставшихся 5%.

### Б-22 — LLM rescue subsystem

**Приоритет:** Низкий | **Тип:** Архитектура

Когда нормализация упала или вернула неполные decisions — LLM может обогатить decision-часть (но никогда не facts — §12.5 спека).

**Что нужно:**
- Отдельный сервис `LLMRescueService`, не смешивать с `ImportModeratorService`.
- Явное тегирование каждого LLM-вмешательства (source tag, prompt version, confidence).
- Кэширование ответов по `(prompt_hash, input_hash)` для воспроизводимости.
- Чёткое разделение: rescue работает на уровне decisions, moderator — на уровне предсказаний.

**Open question:** нужно ли это вообще? Возможно текущего модератора достаточно. Разобраться при реализации Б-02.
