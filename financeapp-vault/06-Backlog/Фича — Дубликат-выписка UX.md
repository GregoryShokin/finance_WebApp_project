# Фича — Дубликат-выписка UX
#бэклог #высокий-приоритет #импорт
> При повторной загрузке той же выписки система не должна молча создавать дубль-сессию — нужен явный экран выбора.
---
## Проблема (2026-05-03)
В `import_repository.py` уже есть `find_active_by_file_hash(user_id, file_hash)`, но в API/UI это не использовано полноценно. Если юзер второй раз загрузит ту же выписку:
- Сейчас: создаётся новая `ImportSession` рядом со старой.
- Риск: при коммите получаются дублирующие транзакции (защита по fingerprint спасает не во всех случаях, особенно для не-распознанных строк).

## Планируемое
### Backend
- В `POST /imports/upload` при совпадении `file_hash`:
  - Если у существующей сессии `status != "committed"` → вернуть ответ `{existing_session_id, status, created_at, action_required: "choose"}` вместо создания новой.
  - Если существующая `committed` → вернуть `{existing_session_id, status: "committed", action_required: "warn"}`.
- Новый необязательный параметр `force_new=true` в форме upload — игнорировать дубликат и создать новую сессию (для случая, когда банк выдал ту же выписку с обновлёнными данными).

### Frontend
- При получении `action_required = "choose"` — модалка:
  - «Эта выписка уже импортирована N дней назад (N строк, N% обработано)»
  - Действия: [Открыть существующую] [Перезаписать] [Отмена]
- При `action_required = "warn"` (committed) — мягкое предупреждение «уже импортировано — продолжить?» + опция [Отмена] / [Загрузить как новую].

## Критичность
**Высокий приоритет** — без этого после первого месяца использования будут случайные дубли при попытке «обновить» выписку.
