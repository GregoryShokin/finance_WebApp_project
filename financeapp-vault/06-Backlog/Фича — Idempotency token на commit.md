# Фича — Idempotency token на commit
#бэклог #средний-приоритет #импорт
> Защита `commit_import` от повторного выполнения при сетевых сбоях / ретраях.
---
## Проблема (2026-05-03)
`commit_import` создаёт десятки/сотни транзакций атомарно на уровне БД, но:
- Если фронт отправил POST → потерял ответ (timeout, потеря сети) → юзер нажал «коммит» ещё раз → создаются дубли (если БД успела зафиксировать первый) или коллизия (если первый ещё в процессе).
- При падении воркера в середине цикла часть rows получила `created_transaction_id`, часть — нет. Retry должен продолжить, не перепроцессить.

## Планируемое
### Backend
- Заголовок `Idempotency-Key: <uuid>` на `POST /imports/{session_id}/commit`.
- Redis storage: `commit:idempotency:{key} → {session_id, status, response_payload}` с TTL 24 часа.
- При повторном POST с тем же ключом → вернуть закешированный ответ (200 OK, не пытаться выполнить повторно).
- При retry без ключа: проверка `created_transaction_id` на каждой row — пропустить уже закоммиченные.
- Партиальный commit: если `commit_import` упал в середине, сессия остаётся в `preview_ready` с подмножеством rows уже committed; следующий retry продолжает с оставшихся.

### Frontend
- Генерация `Idempotency-Key` (uuid) перед первым commit, переиспользование при retry.
- Очистка ключа после успеха (новая попытка коммитить = новая операция).

## Критичность
**Средний приоритет** — до 100 пользователей можно жить без этого, после — обязательно.
