# Фича — Whitelist банков для импорта

#бэклог #импорт #средний-приоритет

> Список банков в форме создания счёта = whitelist банков, для которых у нас протестированный экстрактор. Остальные банки показываются как «импорт скоро будет», юзер может прислать образец выписки для добавления.

---

## Контекст и обоснование

Сейчас селектор банков в форме счёта содержит много банков, для которых у нас **нет** парсера импорта. Юзер выбирает свой банк → создаёт счёт → пытается импортировать выписку → импорт молча падает или принимает мусор. Это нарушает принцип честности из спеки импорта (§1.2): «лучше пометить warning и попросить подтверждение, чем тихо присвоить неверные данные».

Альтернатива — генерировать парсеры на лету через LLM (см. обсуждение от 2026-04-27) — отвергнута на этой стадии:
- Нет банковских интеграций, точность данных критична.
- Стоимость per-выписку × N юзеров заметна.
- Учётная стадия проекта — «первые своя сотня юзеров», хвост банков пока узкий.
- LLM может галлюцинировать суммы/даты — undetected коррупция аналитики.

LLM-парсер можно вернуть в горизонт позже, когда whitelist покрывает 80% юзеров и хвост непокрытых банков станет заметным.

---

## Решение

### Уровень UI

1. **Bank-picker в форме счёта** разделяется на две группы:
   - «Импорт поддерживается» — Сбер, Т-Банк, Озон Банк, Яндекс Банк (текущее покрытие) + по мере расширения.
   - «Импорт пока не поддерживается» — остальные банки, помечены бейджем «импорт скоро».
2. **Опция «Не нашёл свой банк»** в самом низу селектора. При выборе:
   - Поле для ввода названия банка.
   - Загрузка 1-3 примеров выписок (PDF/XLSX/CSV).
   - Сообщение «Мы добавим поддержку этого банка. Пока транзакции можно вводить вручную через форму новой транзакции».
3. **В мастере импорта** при попытке загрузить файл в счёт неподдерживаемого банка:
   - Блок «Импорт для этого банка пока не поддерживается».
   - Кнопка «Прислать образец выписки» (если ещё не присылал).
   - Возможность временно сменить банк счёта (если юзер ошибся при создании).

### Уровень модели

```python
class BankSupportRequest(Base):
    id: int
    user_id: int
    bank_name: str          # как ввёл юзер
    bank_code: str | None   # если выбрал из списка-расширения
    sample_files: list[str] # пути к загруженным файлам (S3 / local)
    note: str | None        # доп. комментарий юзера
    status: str             # pending / in_review / added / rejected
    created_at: datetime
    updated_at: datetime
    resolved_at: datetime | None
    notify_user_on_add: bool # уведомить когда добавим
```

Миграция Alembic — отдельная таблица, без изменений в существующих.

### Уровень API

- `POST /api/v1/imports/bank-support-request` — приём файлов + метаданных.
- `GET /api/v1/imports/bank-support-request` — список запросов юзера (для отображения статуса).
- Админский endpoint (или CLI `python -m scripts.list_bank_support_requests`) — выгрузка pending-запросов для разработчика.

### Расширение whitelist

После того как разработчик пишет парсер для банка X:
1. Добавляет банк в whitelist (переменная или таблица `supported_banks`).
2. Запускает скрипт нотификации `python -m scripts.notify_bank_support_added bank_name=X`.
3. Юзеры с `BankSupportRequest.bank_name=X AND notify_user_on_add=True` получают email/уведомление в приложении.
4. Сэмплы выписок остаются как тест-фикстуры в `tests/fixtures/statements/<bank_code>/`.

## Открытые вопросы

1. **Хранение образцов** — локально в репозитории (с анонимизацией) или в отдельном secured bucket? Файлы содержат ПДн.
2. **Анонимизация** — автоматическая (заменять номера карт/договоров на маски) или ручная разработчиком? Для тестов точные данные не нужны.
3. **UX уведомления** — email или только in-app banner? Если email — нужна валидация согласия юзера.
4. **Минимальный набор сэмплов** — сколько выписок достаточно для написания экстрактора? Зависит от вариативности форматов одного банка.
