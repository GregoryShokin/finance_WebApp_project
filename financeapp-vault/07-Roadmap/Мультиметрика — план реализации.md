# Мультиметрика — план реализации
#roadmap #в-работе
> Полный план перехода с СМО на мультиметрическую систему: модели, метрики, аналитика, дашборд, школа.

> [!important] Актуализация 2026-04-19
> План содержит этапы, сформулированные в апреле 2026 (v1.0–v1.3). По мере принятия решений часть формул и этапов обновилась. Ключевые изменения:
> - **Запас удалён из мультиметрики (v1.2)** → Этап 2г не выполняется; в FI-score и на дашборде вместо него **Буфер устойчивости** (вклады/расходы).
> - **Поток стал трёхслойным (v1.5)** → Базовый / Свободный / Полный с балансовым компенсатором. См. [[Поток]].
> - **Окно средних — 12 мес.**, не 3 мес.
> - **Метрика «Доступно для целей» отменена** — через связку показателей в школе.
> - **Этап 4 (дашборд)**: вместо четырёх карточек — виджет Потока с тремя табами + отдельные карточки DTI и Буфера, Капитал в секции 4.
> - **Архивные ссылки ниже** ([[Запас]], старые формулы Потока) оставлены для истории, но **не следовать им при реализации** — сверять с [[Спецификация — Целевое состояние системы]] и [[Поток]].

## Контекст

По результатам 8 раундов дебатов ([[Дебаты — Мультиметрическая система]], [[Дебаты — Раунды 4-8]]) и стресс-тестов ([[Мультиметрическая система — стресс-тест]], [[Стресс-тест — Классификация расходов]]) архитектура закрыта. Консенсус 80/20 в пользу мультиметрики. 14 отвергнутых подходов задокументированы ([[Отвергнутые подходы]]).

Суть перехода: вместо одного показателя СМО — набор метрик, каждая отвечает на один вопрос. Пирамида → радар. Школа по принципу «слабое звено».

**Актуальная структура метрик (v1.5):**

| Метрика | Вопрос | Формула |
|---------|--------|---------|
| **Поток — Базовый** | Могу ли я позволить текущий образ жизни? | рег. доходы − рег. расходы (проценты по кредитам внутри) |
| **Поток — Свободный** | Сколько ежемесячно освобождается? | Базовый − Σ обязательных тел кредитов |
| **Поток — Полный** | Что реально произошло с кэшем? | Δ(regular + cash + deposit), балансово |
| **Капитал** | Богатею или беднею? | Ликвидные активы + вклады − долги (траектория) |
| **Нагрузка (DTI)** | Сколько работаю на банк? | Σ платежей / рег. доход × 100% |
| **Буфер устойчивости** | На сколько месяцев хватит без дохода? | Σ(deposit balances) / avg(monthly расходы) |

~~Запас (ликвидный кэш / расход)~~ — **удалён из мультиметрики v1.2**. Причина: числитель включал заёмные средства (credit_disbursement), создавал компенсационную ловушку.

## Этап 1 — Классификация: регулярность на категориях

**Цель:** каждая категория получает режим regular / irregular / project. Транзакции фиксируют снимок при создании.

**Почему первым:** от этого зависит расчёт всех четырёх метрик. Базовый поток = только регулярные. DTI = только регулярный доход в знаменателе. Без классификации метрики не считаются.

- [ ] Добавить поле `regularity` в модель `Category` (enum: `regular`, `irregular`, `project`; default = `regular`)
- [ ] Добавить поле `is_regular: bool` в модель `Transaction` (снимок на момент создания)
- [ ] Миграция: `regularity` на Category + `is_regular` на Transaction
- [ ] Логика: при создании транзакции `is_regular = category.regularity == 'regular'`
- [ ] Обратная совместимость: существующие категории → `regular` по умолчанию; `exclude_from_planning=True` → `irregular`
- [ ] API: обновить схемы Category (CRUD) и Transaction (read)
- [ ] Фронтенд: переключатель «Регулярная / Нерегулярная / Проект» в настройках категории

**UX-решения (по ходу):**
- Где показывать переключатель — при создании категории, в редактировании, или оба
- Нужно ли подсказывать, какой режим выбрать (tooltip с примерами)
- Как визуально отличить нерегулярные категории в списке

**Зависимости:** нет, можно начинать сразу.

## Этап 2 — Метрики: Поток, Капитал, DTI, Запас

**Цель:** четыре метрики считаются в бэкенде и доступны через API.

### 2а — Поток (Flow, трёхслойный, v1.5)

- [ ] Базовый поток: `Σ(income where is_regular) − Σ(expense where is_regular)` (проценты по кредитам уже внутри как regular expense)
- [ ] Свободные средства: `Базовый − Σ(Account.monthly_payment × body_ratio)` по всем кредитным счетам (только тело, без процентов)
- [ ] Полный поток (балансово): `Δ(regular + cash + deposit)` за период
- [ ] Декомпозиция Полного в виджете: все категории + строка-компенсатор «Прирост долга по КК» = `Σ(CC_debt_end − CC_debt_start)`
- [ ] Показатель уровня жизни: `avg(базовый, 12 мес.) / avg(рег. доход, 12 мес.) × 100%`
- [ ] Зоны: 🟢 ≥ 20%, 🟡 0–19%, 🔴 < 0%
- [ ] Тренд: сравнение с прошлым месяцем / скользящее среднее

**Важно:**
- ~~credit_payment всегда в базовом потоке~~ — устарело. Тело кредита = `transfer`, НЕ в Базовом. Проценты = regular `expense`, уже учтены в Базовом.
- Вклады (`deposit`) считаются ликвидными и входят в состав расчёта Полного (решение 2026-04-19). Переводы на/с вклада = `transfer`, не участвуют в потоке как транзакция.

### 2б — Капитал (Capital)

- [ ] Формула: `Σ(regular.balance + cash.balance + deposit.balance) − Σ(credit.balance + credit_card.balance + installment_card.balance)`
- [ ] Траектория: `Δ Capital` за месяц (направление важнее абсолюта)
- [ ] Без нелеквидных активов (недвижимость, авто — отдельно на странице Активов)

### 2в — DTI (Нагрузка)

- [ ] Формула: `Σ credit_payment / avg(regular income, 3 мес.) × 100%`
- [ ] Источник платежей: 1) последний `credit_payment` из транзакций, 2) fallback: `monthly_payment` на счёте
- [ ] Знаменатель: только регулярный доход (`is_regular = true`)
- [ ] Зоны: 🟢 < 30%, 🟡 30–40%, 🔴 > 40%, 🔴🔴 > 60%

**Что меняется vs текущий DTI:** знаменатель переходит на `is_regular`, а не на `income_type = 'active'`.

### 2г — ~~Запас (Reserve)~~ → заменён на Буфер устойчивости (v1.2)

> **Запас удалён из мультиметрики (v1.2).** Причина: числитель (ликвидный кэш) включал заёмные средства (credit_disbursement), создавая ложный зелёный сигнал и компенсационную ловушку внутри FI-score.
>
> **Заменён на Буфер устойчивости:**

- [ ] Формула: `Σ(balance, accounts where type = 'deposit') / avg(monthly_expenses)`
- [ ] Только вклады (не дебет + наличные). Вклад = сознательное решение отложить, credit_disbursement туда не попадает
- [ ] Зоны: 🔴 < 1 мес., 🟡 1–3 мес., 🟢 3–6 мес., 💚 > 6 мес.
- [ ] Одновременно компонент FI-score (вес 0.25) и порог Этажа 2 пирамиды (≥ 3 мес.)

### Общее

- [ ] Создать / расширить `metrics_service.py` — четыре метода: `calculate_flow()`, `calculate_capital()`, `calculate_dti()`, `calculate_reserve()`
- [ ] API-эндпоинт: `GET /metrics/summary` — все 4 метрики + зоны + тренды
- [ ] Обновить FI-score: новые веса v1.4 (Норма сбережений 0.20, Капитал 0.30, DTI 0.25, Буфер устойчивости 0.25)
- [ ] Пересчитать `financial_health_service.py` — убрать старый СМО, подключить новые метрики

**UX-решения (по ходу):**
- Точные формулировки зон на русском для каждой метрики
- Формат тренда — стрелка, процент, абсолютное число
- Нужен ли «режим сравнения» (этот месяц vs прошлый)

**Зависимости:** Этап 1 (нужен `is_regular` для расчётов).

## Этап 7 — Образовательные моменты (школа)

**Цель:** приложение учит через подсказки в контексте метрик.

### 7а — Подсказки по рассрочкам

- [ ] Утилизация лимита > 70% (не требует покупок)
- [ ] `monthly_payment` > 15% дохода (не требует покупок)
- [ ] ≥ 3 активных покупки (требует InstallmentPurchase)
- [ ] DTI > 40% при наличии рассрочки

### 7б — Подсказки по метрикам

- [ ] Красная зона Базового потока: «Расходы превышают доход на X₽/мес»
- [ ] Красная зона Свободного потока: «Обязательные платежи забирают больше, чем освобождается. Фокус на DTI»
- [ ] Расхождение Свободного и Полного > compensator: «Ты держишься за счёт роста долга по кредитке»
- [ ] Предупреждение о нерегулярных: avg нерег. > 30% рег. дохода
- [ ] Буфер устойчивости < 1 мес.: «Одна задержка зарплаты = долг. Откладывай на вклад»
- [ ] Капитал падает при хорошем Потоке: «Скрытый долг»

### 7в — Пересмотр школьной программы

- [ ] Полная переработка: Урок 0.1 (пирамида → радар), Урок 1.3 (СМО → Поток)
- [ ] Частичная правка: Уроки 0.5, 1.5, 2.5
- [ ] Оценка: 3–4 дня на переписывание vault-контента

**UX-решения (по ходу):**
- Формат подсказок — toast, карточка, часть виджета
- Тон — нейтральный или мотивирующий
- Момент показа — дашборд, создание транзакции, оба

**Зависимости:** Этапы 3 + 6 (рассрочки + здоровье).

## Порядок работы

```
Этап 1 (классификация)
    ↓
Этап 2 (метрики) ←→ Этап 3 (рассрочки) — параллельно
    ↓                    ↓
Этап 4 (дашборд)    Этап 5 (аналитика)
    ↓
Этап 6 (здоровье)
    ↓
Этап 7 (школа)
```

Этапы 2 и 3 можно вести параллельно, потому что рассрочки = данные, а метрики = расчёты — пересекаются только в формулах.
