# Система ретроспективного распределения досрочных погашений

**Статус:** ❌ Отвергнута (апрель 2026)  
**Причина:** Простое исключение досрочки из СМО решает задачу проще и корректнее  
**Вместо неё:** [[Классификация типов поступлений]] — решает реальную проблему завышения СМО от нерегулярных доходов  
**Связанные файлы:** [[Текущее лучшее решение]], [[Отвергнутые подходы]], [[Рассматриваемые подходы]]

## Почему отвергнута

### Предпосылка оказалась ложной

Система проектировалась для решения проблемы: «досрочное погашение проваливает СМО в месяце платежа». Но в текущей согласованной модели `credit_early_repayment` **уже исключён из СМО** — досрочка не попадает в расходную аналитику, только в денежный поток.

Если досрочка не в СМО — нечего распределять. Проблема, которую система решала, уже решена проще.

### Проверка на 10 сценариях (апрель 2026)

Проверены все сценарии, в которых пользователь может использовать накопления для досрочного погашения:

| Сценарий | Исключение из СМО | Ретроспективное распределение |
|---|---|---|
| Копил на отдельный счёт → погасил | ✅ СМО корректен | ❌ Занижает прошлые месяцы до 0 — ложь |
| Копил «в уме» на том же счёте | ✅ СМО корректен | ❌ Та же проблема |
| Бонус/премия → сразу погасил | ⚠️ Завышает (но это проблема бонуса, не досрочки) | ❌ Распределяет по месяцам, не имеющим отношения к бонусу |
| Продал актив → погасил | ⚠️ Завышает (проблема классификации актива) | ❌ Неверный источник |
| Подарок от родственников → погасил | ⚠️ Завышает (проблема классификации поступления) | ❌ Неверный источник |
| Частичная досрочка | ✅ | Избыточно |
| Несколько досрочек в разные месяцы | ✅ | Избыточно + нарастающая сложность |
| Досрочка ипотеки (крупная сумма) | ✅ Эффект виден через снижение ежемесячного платежа | Избыточно |
| Отрицательный СМО + досрочка | ✅ | Бесполезно (нечего распределять по отрицательным месяцам) |
| Погашение кредитной карты (грейс) | Не применимо — это `credit_payment` | Не применимо |

### Три ключевых аргумента

**1. Ретроспективное распределение искажает прошлое.**

Пользователь с СМО 50к/мес действительно имел 50к свободных денег каждый месяц. То, что он решил сохранить их и потом направить на досрочку — его выбор. Если бы он потратил эти же накопления на отпуск, никто бы не предложил задним числом обнулить прошлые СМО.

Переписывание прошлого — это подмена фактов. В прошлом месяце у пользователя БЫЛО 50к свободных. Это факт. Ретроспективное распределение меняет факт на «у тебя было 0» — и это неправда.

**2. Сценарии с завышением СМО — не проблема досрочки.**

Когда пользователь получает бонус 300к и сразу гасит кредит, СМО завышается. Но не из-за досрочки (она исключена), а из-за бонуса. Тот же бонус завысит СМО и без досрочки. Это проблема **классификации типов поступлений**, а не обработки досрочных погашений.

**3. Сложность не оправдана.**

Система требует: таблицу корректировок, алгоритм LIFO, пересчёт каскадных показателей, UI для двух значений СМО, откат при удалении, обработку множественных досрочек. Всё это ради сценария, который уже покрыт одной строкой: `credit_early_repayment → не в СМО`.

## Известный компромисс: двойное исключение

Покупка на кредит — не расход (деньги банка). Досрочное погашение — не расход (исключено из СМО). Если кредит закрыт досрочно, покупка **целиком проходит мимо расходной аналитики**.

### Пример

Кредит 500к, плановый платёж 45к/мес. Пользователь закрывает досрочно через 2 месяца.

- В СМО попало: 2 × 45к = **90к**
- Реально потрачено: **~500к**
- Выпало из расходной аналитики: **~410к**

Крайний случай — погасил в первый же месяц: в расходной аналитике **0** из 500к.

### Почему это допустимо

Информация не потерялась — она распределена по слоям:

| Слой | 500к видно? |
|---|---|
| Категорийная аналитика («что купил») | **Да** — покупка 500к с категорией |
| Расходная аналитика / СМО («что ушло из кошелька в месяц») | Частично — только плановые платежи |
| Денежный поток (фактическое движение денег) | **Да** — досрочка −410к |
| Виджет «Долги» | **Да** — кредит закрыт, история платежей |
| Подушка безопасности | **Да** — сбережения уменьшились |
| Чистый капитал | **Да** — сбережения ↓, долг ↓ |

СМО отвечает на вопрос «сколько у меня остаётся каждый месяц», а не «во сколько мне обошёлся кредит». Для второго вопроса — категорийная аналитика + денежный поток + история кредита.

### Почему именно так: поток vs баланс

Три операции по кредиту — три разных экономических события:

| | Покупка на кредит | Ежемесячный платёж | Досрочное погашение |
|---|---|---|---|
| **Откуда деньги** | Деньги банка | Из дохода этого месяца | Из накоплений за прошлые месяцы |
| **Что это** | Появление долга | Расход из потока (заработал → отдал) | Перемещение актива в погашение долга |
| **Повторяется** | Разово | Каждый месяц | Разово |
| **Влияет на месячный поток** | Нет | Да | Нет |
| **В СМО** | Нет | Да | Нет |

**Ежемесячный платёж** — расход из потока: пользователь каждый месяц зарабатывает и часть отдаёт банку. Это уменьшает свободные деньги месяца. Корректно в СМО.

**Досрочное погашение** — перемещение из одного кармана в другой: из сбережений → в погашение долга. Ежемесячный доход и расходы не изменились. Изменился баланс: сбережений стало меньше, долга стало меньше. Аналогия: перевод 300к с вклада на другой вклад не меняет СМО. Досрочка — то же самое, только второй «вклад» — кредит со знаком минус.

**Покупка на кредит** — появление долга: пользователь получил вещь и обязательство. Из его дохода не ушло ничего. Расход появится позже через ежемесячные платежи.

СМО — метрика **потока** (сколько приходит и уходит за месяц). Досрочка и покупка на кредит — события **баланса** (перераспределение между активами и обязательствами). Поэтому они не в СМО.

### Почему альтернативы хуже

| Вариант | Что ломает |
|---|---|
| Включить покупку в СМО | Провал на сумму кредита (исходная проблема) |
| Включить досрочку в СМО | Аномальный месяц, искажает среднее на полгода |
| Ретроспективное распределение | Переписывает прошлое, концептуально неверно |
| Размазать досрочку на будущее | Фантомные расходы в месяцах, где ничего не было |

Двойное исключение — цена за стабильный СМО при кредитных операциях.

### Условие: другие слои должны работать

Компромисс допустим **только если** пользователь может увидеть полную картину:
- Категорийная аналитика — обязательна (есть в модели)
- Денежный поток — обязателен (пока не финализирован)
- При клике на закрытый кредит — полная история: сколько плановыми, сколько досрочкой, сколько процентов

Если эти три вещи работают — пользователь не в слепой зоне. Ответы на разные вопросы живут в разных местах.

### Статус

Принято как осознанный компромисс. Не требует решения, но требует реализации денежного потока и детализации по кредитам.

## Что решает реальную проблему

Сценарии 3–5 (бонус, продажа актива, подарок → досрочка) выявили настоящую проблему: **нерегулярные поступления искажают СМО**. Это не связано с досрочным погашением — бонус искажает СМО вне зависимости от того, что пользователь делает с деньгами.

Решение: **[[Классификация типов поступлений]]** — отделить регулярный доход (зарплата) от нерегулярного (бонус, продажа актива, подарок). Только регулярный доход формирует СМО.

## Исходная концепция (для истории)

<details>
<summary>Развернуть описание отвергнутой концепции</summary>

### Суть

Досрочное погашение распределяется ретроспективно по предыдущим месяцам алгоритмом LIFO — от ближнего к дальнему. Каждый месяц с положительным СМО отдаёт часть остатка в счёт досрочки, пока сумма не исчерпана.

### Экономическая логика

Деньги на досрочку были заработаны в предыдущих месяцах как положительный СМО. Распределение расхода обратно по этим месяцам отражает, «сколько реально можно было потратить свободно с учётом будущей досрочки». Аналогия: ретроспективный пересчёт в МСФО (IAS 8).

### Алгоритм

```
Остаток = сумма досрочки
Для каждого месяца от (T-1) до (T-6):
    Доступно = СМО месяца (до корректировки)
    Забираем = min(Доступно, Остаток)
    СМО_скорректированный = СМО - Забираем
    Остаток -= Забираем
    Если Остаток = 0 → стоп
Нераспределённый остаток → в текущий месяц
```

### Пример

Досрочное погашение 50 000 в июле при СМО ~45к/мес:
- Июнь: 45к → 0к (забрали 45к)
- Май: 45к → 40к (забрали 5к, остаток 0)
- Июль: 45к (чистый, без досрочки)

Сумма за период не менялась — менялось только распределение по месяцам.

### Угловые случаи

1. Накоплений не хватает — остаток в текущий месяц
2. Несколько досрочек — каждая создаёт корректировку поверх предыдущих
3. Удаление транзакции — откат всех связанных корректировок
4. Отрицательный месяц — пропускается
5. Первый месяц — нет истории, всё в текущий

### Модель данных (не реализована)

Планировалась таблица `smo_adjustments`: target_month, original_smo, adjustment_amount, adjusted_smo, source_transaction_id.

</details>
