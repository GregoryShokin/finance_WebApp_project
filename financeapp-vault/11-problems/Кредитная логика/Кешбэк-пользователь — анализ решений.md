# Кешбэк-пользователь — анализ решений

**Дата:** апрель 2026
**Проблема:** [[Стресс-тест кредитной модели#Сценарий А]]
**Серьёзность:** ВЫСОКАЯ — ломает ключевой виджет для самого массового сценария

---

## Суть проблемы

Пользователь оплачивает все покупки кредиткой ради кешбэка, каждый месяц гасит полностью в грейс-период. Текущая модель:

- Покупки на кредитку → **не расход** (деньги банка)
- Погашение кредитки → **«Кредитные платежи»**

Результат: все 100к расходов — один серый сектор «Кредитные платежи». Обязательные и второстепенные — пусто. Дашборд бесполезен.

Масштаб: кредитки для кешбэка — массовый сценарий в РФ, особенно в нашей ЦА (30–45 лет). Вероятно, большинство активных пользователей.

---

## Как делают конкуренты

| Приложение | Подход |
|---|---|
| **YNAB** | Покупка = расход по категории. Погашение кредитки = перевод. Кредитка прозрачна |
| **Тинькофф/Сбер** | Аналитика по категориям, не по способу оплаты. «Кредитный платёж» как понятия нет |
| **Copilot, Mint** | Расход привязан к покупке, а не к погашению |
| **Дзен-мани** | Аналогично — покупка = расход |

**Вывод:** все крупные приложения трактуют покупки с кредитки как обычные расходы. Наша модель — исключение, потому что принцип «деньги банка ≠ мои деньги» создавался для **реального кредитования**, но применяется и к **платёжному использованию** кредитки.

---

## Три подхода

### Подход 1 — «Прозрачная кредитка» (транзитный режим) ⭐ Рекомендован

**Суть:** Если кредитка гасится полностью в грейс-период → покупки идут в обязательные/второстепенные (как дебет), погашение = перевод между своими счетами.

**Логика:** Кредитка в грейс-периоде — это **не кредитный инструмент**, а **платёжный**. Пользователь не занимает деньги, а прогоняет свои деньги через банк ради кешбэка и 30-дневной отсрочки. Экономически = дебетовая карта с задержкой.

**Принцип «деньги банка ≠ мои деньги» не нарушается** — он уточняется. Когда банк не даёт деньги в долг, а обрабатывает платёж — это не кредит.

| Плюс | Минус |
|---|---|
| Совпадает с экономической реальностью | Нужен механизм определения режима |
| Дашборд работает правильно | Переходные ситуации (грейс нарушен) |
| Так делают все конкуренты | Два режима одного типа счёта |
| Итоговый СМО не меняется | |
| Понятно пользователю | |

**Числовой пример:**

| | Прозрачный режим | Текущая модель |
|---|---|---|
| Обязательные | 55к (продукты, коммуналка, транспорт) | ~0 |
| Второстепенные | 45к (рестораны, одежда) | ~0 |
| Кредитные платежи | 0 | 100к |
| **СМО** | **200к − 100к = 100к** | **200к − 100к = 100к** |

Итог одинаковый. Разница только в **разбивке** — а именно разбивка и есть ценность дашборда.

---

### Подход 2 — Атрибуция (распределение погашения по категориям)

**Суть:** При погашении 100к → система смотрит, что было куплено за месяц, и раскидывает: продукты 40к (обязательные), рестораны 15к (второстепенные) и т.д.

| Плюс | Минус |
|---|---|
| Не нужен «режим» карты | Сложная логика привязки платежей к покупкам |
| Работает задним числом | Тайминг: покупки в мае, погашение в июне |
| | По сути = подход 1, но реализован сложнее |
| | Непонятно при частичном погашении |

**Вывод:** даёт тот же результат, что подход 1, но через более сложный механизм. Не рекомендуется как основной.

---

### Подход 3 — Только детализация при клике

**Суть:** «Кредитные платежи» остаются 100к серым сектором. При клике → разбивка: обязательные 55к, второстепенные 45к.

| Плюс | Минус |
|---|---|
| Проще всего | Дашборд на первый взгляд врёт |
| Никаких изменений в логике | Теряется ценность «увидеть за секунду» |
| | Метка «Кредитные платежи» вводит в заблуждение |
| | Не решает проблему, а прячет |

**Вывод:** полезна как **дополнение** к подходу 1 (для пользователей с реальным долгом на кредитке), но **не как самостоятельное решение** кешбэк-проблемы.

---

## Рекомендация: подход 1 — «Прозрачная кредитка»

### Как определять режим

**Режим определяется не через абстрактный вопрос «как вы используете карту», а через конкретный факт: есть ли ежемесячный платёж.**

При создании кредитной карты система спрашивает:

> «Есть ли у вас ежемесячный платёж по этой карте?»

| Ответ | Что происходит | Режим |
|---|---|---|
| **Да, есть платёж** → указывает сумму | Платёж попадает в «Кредитные платежи» | **Кредитный** |
| **Нет платежа** → указывает грейс-период | Покупки = расходы, погашение = перевод | **Прозрачный** |

### Почему через платёж, а не через вопрос о режиме

1. **Конкретный факт, а не абстракция.** «Есть ли платёж?» — пользователь точно знает ответ. «Как вы используете карту?» — требует понимания концепций.
2. **Уже в модели данных.** Поле `monthly_payment` на счёте уже существует. Если заполнено → кредитный. Пусто → прозрачный. Режим **выводится** из данных, не хранится отдельно.
3. **Переходы автоматические.** Появился долг → пользователь добавил платёж → режим сменился. Погасил долг → убрал платёж → режим вернулся. Управление фактом, а не абстрактным переключателем.

### Грейс-период как защитный механизм

В прозрачном режиме пользователь указывает грейс-период карты (например, 55 дней). Система использует его для:
- **Напоминания:** «До конца грейс-периода 5 дней. Баланс карты: 95к»
- **Предупреждения:** «Грейс-период истёк. Если появился ежемесячный платёж — укажите его»

Как только пользователь укажет платёж → режим переключится на кредитный автоматически.

### Связь с уроками школы

Понимание разницы между «кредитка для покупок» и «кредитка с долгом» связано с модулем 2 (кредиты). Возможная логика:
- До модуля 2 → все кредитки по умолчанию без платежа (прозрачный режим) — подходит большинству
- Модуль 2 → урок о грейс-периоде и долговой спирали → пользователь осознанно проверяет, есть ли у него платёж

## Решения открытых вопросов

### 1. Смешанное использование: переход через добавление платежа

**Сценарий:** Кешбэк-пользователь (карта без платежа, прозрачный режим) совершает крупную покупку на 200к и не гасит вовремя. Банк назначает ежемесячный платёж.

**Решение: переход происходит естественно через факт появления платежа.**

1. Пользователь нарушает грейс-период
2. Система предупреждает: «Грейс-период истёк. Если появился ежемесячный платёж — укажите его»
3. Пользователь добавляет платёж на карте → режим автоматически становится кредитным
4. Со следующего месяца: покупки → не в расходную, платёж → «Кредитные платежи»

**Обратный переход:** пользователь погасил долг → убрал платёж → режим вернулся в прозрачный.

Никаких абстрактных «переключений режима» — пользователь управляет фактом (есть платёж / нет платежа), а режим выводится автоматически.

**Числовой пример:**

| Месяц | Платёж указан? | Режим | Покупки в расходную? |
|---|---|---|---|
| Январь | Нет | Прозрачный | ✅ Да |
| Февраль (нарушен грейс) | Нет → Да, 15к | Прозрачный → Кредитный | ✅ → ❌ (со след. мес.) |
| Март | Да, 15к | Кредитный | ❌ Нет |
| ... погасил долг | Да → убрал | Кредитный → Прозрачный | ❌ → ✅ (со след. мес.) |

---

### 2. Переходный месяц: режим вступает в силу со следующего месяца

**Решение: текущий месяц сохраняет прежний режим. Новый режим — с 1-го числа следующего месяца.**

**Почему:**
- СМО считается по **завершённым** месяцам. Разбивать один месяц на две части с разными правилами — сложность без пользы
- Пользователь видит чёткую границу: «с этого месяца карта работает иначе»
- Не нужно пересчитывать транзакции текущего месяца

**Исключение:** при первом добавлении карты режим применяется сразу (нет «прежнего» режима).

**Пример:**

| | Март (прозрачный) | → переключение 15 марта → | Апрель (кредитный) |
|---|---|---|---|
| Покупки на кредитку | В расходную ✅ | | В расходную ❌ |
| Погашение | Перевод (не в СМО) | | Кредитные платежи ✅ |

Март целиком считается по прозрачным правилам, даже если переключение было 15 марта. Апрель — целиком по кредитным.

---

### 3. Импорт: изменений не требуется

**Ключевое решение: импорт не меняется. Режим обрабатывается на уровне аналитики.**

Текущий импорт уже делает всё, что нужно:
- Импорт выписки кредитки → покупки попадают как расходы с категориями
- Импорт выписки дебетовой карты → погашение попадает как `credit_payment`

Типы транзакций остаются прежними. Разница в режимах — это вопрос **фильтрации при расчёте аналитики**:

| Транзакция | Прозрачный режим | Кредитный режим |
|---|---|---|
| Покупка на кредитку | Категорийная ✅ + Расходная ✅ (обязат./второстеп.) | Категорийная ✅, Расходная ❌ |
| Погашение (`credit_payment`) | Исключить из расходной (= перевод) | В расходную ✅ («Кредитные платежи») |
| **Итого в СМО** | **Покупки = расход** | **Погашение = расход** |
| **Сумма СМО** | **Одинаковая** | **Одинаковая** |

**Реализация:** режим выводится из поля `monthly_payment` на модели счёта. Если `monthly_payment > 0` → кредитный режим, если пусто → прозрачный. Отдельного поля `usage_mode` не нужно. Дополнительно: поле `grace_period_days` на счёте (только для кредитных карт без платежа). Сервис аналитики проверяет наличие платежа при расчёте. Импорт и типы транзакций не трогаем.

**Главный плюс:** если пользователь добавляет или убирает платёж — пересчёт аналитики автоматический. Данные те же, меняется только фильтрация. Никакой миграции данных.

---

### 4. Детализация «Кредитных платежей» — параллельная задача

Уже запланирована в [[Текущее лучшее решение#Слой 1]]: при клике на сектор «Кредитные платежи» → разбивка по категориям исходных покупок.

Работает **только для карт в кредитном режиме** и для потребительских кредитов. В прозрачном режиме сектора «Кредитные платежи» от этой карты вообще нет — покупки уже распределены по обязательным/второстепенным.

Не зависит от подхода «Прозрачная кредитка». Это параллельная задача с отдельным сроком.

---

## Совместимость с принципом «единый набор правил»

[[Текущее лучшее решение]] утверждает: «Один набор правил для кредитных карт, карт рассрочки, Яндекс Сплит, ипотеки, автокредита. Тип счёта не меняет логику.»

Прозрачный режим **не нарушает** этот принцип — он его уточняет:

- Принцип говорит: **тип счёта** не меняет логику
- Прозрачный режим говорит: **способ использования** меняет логику

Кредитная карта в прозрачном режиме экономически **не является кредитным инструментом** — она является платёжным инструментом. Логика «деньги банка ≠ мои деньги» по-прежнему работает для всех реальных кредитных ситуаций: карты с долгом, рассрочки, потребительские кредиты, ипотека.

Уточнённая формулировка принципа:

> **Логика определяется способом использования счёта (кредит vs платёж), а не его типом.** Все кредитные инструменты работают одинаково. Платёжное использование кредитной карты = дебет.
