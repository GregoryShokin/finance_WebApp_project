# Механизм «Проект»
#мультиметрика #кредитная-логика #архитектура
> Корректировка потока при растянутых расходах из потребкредита. Единственный дополнительный механизм мультиметрической системы.
---

## Статус
✅ Принято — по результатам дебатов ([[Дебаты — Мультиметрическая система]], [[Дебаты — Раунды 4-8]]).

## Зачем нужен

В мультиметрической системе все покупки = расходы, все погашения тела = переводы. Это работает для всех типов кредитов **кроме одного случая**: потребкредит на общие нужды (ремонт, свадьба, переезд), когда кредитные деньги приходят на дебет и тратятся постепенно.

Проблема: поток три месяца подряд красный, хотя базовый уровень жизни не изменился — пользователь тратит кредитные деньги на проект, а не на ежедневные расходы.

**Где проект НЕ нужен:**
- Кредитная карта — покупка = расход, погашение = перевод. Работает нативно.
- Рассрочка — покупка = расход (один удар), платежи = переводы. Работает нативно.
- Автокредит — покупка машины = один расход. Поток красный один месяц, потом восстанавливается.
- Ипотека — покупка квартиры = один расход. Один месяц красный.

Проект решает **только** растянутый расход кредитных денег через дебетовую карту.

---

## Пользовательский сценарий

### Шаг 1: Получение кредита

Система видит `credit_disbursement` — деньги пришли на дебет. Появляется карточка:

> **Вы получили кредит 500 000 ₽**
> Эти деньги пойдут на конкретную цель? Например, ремонт, учёба, переезд.
> Если да — укажите на что и на сколько месяцев. Так приложение будет точнее показывать ваш реальный уровень расходов.
>
> **[Да, это проект]** — **[Нет, пропустить]**

При нажатии «Да» — два поля:
- **Название** (текст, предзаполнено «Кредит от [дата]»)
- **Срок расходования** (месяцы, предзаполнено из срока кредита, можно изменить)

Бюджет = сумма кредита. Заполняется автоматически.

Одно действие. Два поля. Никаких дополнительных настроек.

### Шаг 2: Жизнь с проектом

Пользователь ничего не делает. Тратит деньги как обычно. Все покупки — в реальных категориях. Система ничего не спрашивает, транзакции не тегирует, категории не привязывает.

На дашборде карточка «Поток»:

```
ПОТОК
+95к ✅        ← базовый поток (главное число)
−55к всего     ← полный поток (мелко, серым)
📋 Ремонт     ← иконка активного проекта
```

### Шаг 3: Проект завершается

Три сценария:
- **Срок вышел** → система предлагает: «Проект "Ремонт" длился 3 месяца. Закрыть?»
- **Пользователь закрывает вручную** в настройках проекта
- **Расходы упали** → система спрашивает: «Похоже, проект "Ремонт" завершён. Закрыть?»

После закрытия — карточка Потока без разделения, одно число.

## Формула

**Базовый поток** = Полный поток + Σ(бюджет_i / срок_i) для всех активных проектов.

Пример: кредит 500к, срок проекта 3 мес. Коррекция = 500к / 3 = 167к/мес.
- Полный поток месяца: 200к − 105к − 150к(ремонт) = −55к
- Базовый поток: −55к + 167к = **+112к**
- «Истинный» lifestyle = +95к. Погрешность ±17к — допустимо для индикатора.

Базовый поток **точен в среднем** за весь срок проекта, но **шумит помесячно** при неравномерных расходах. Для алертов и FI-score при активном проекте — использовать скользящее среднее базового потока.

## Влияние на метрики

| Метрика | Влияние проекта |
|---|---|
| **Поток (главный на дашборде)** | Показывается базовый, полный — мелко |
| **FI-score** | Считается от базового потока |
| **Алерты** | Привязаны к базовому потоку (по скользящему среднему) |
| **Норма сбережений** | Считается от базового потока |
| **Капитал** | Без изменений — проект не влияет |
| **Нагрузка** | Без изменений — проект не влияет |
| **Запас** | Без изменений — проект не влияет |

Проект корректирует **только** Поток и его производные. Остальные три метрики — всегда честные.

## Учёт платежей по кредиту

Платежи по кредиту **не связаны с проектом**. Проект — про корректировку потока в месяцы расходования. Платежи идут по общим правилам:

| Часть платежа | Тип | В потоке? |
|---|---|---|
| Тело (основной долг) | Перевод | ❌ Нет |
| Проценты | Расход | ✅ Да |

**Пример полного цикла кредита:**

Кредит 500к на ремонт, платёж 40к/мес (30к тело + 10к проценты). Доход 150к, обычные расходы 80к.

| Период | Поток | На счёте реально | Разница |
|---|---|---|---|
| Месяцы 1–3 (ремонт) | Базовый: +60к | Зависит от расходов | Корректировка проекта |
| Месяцы 4+ (погашение) | 150к − 80к − 10к = **+60к** | 150к − 80к − 40к = **+20к** | 30к ушло на погашение долга |

**Почему поток +60к, а на счёте +20к?**

30к ушли банку на погашение тела. Это не расход — долг уменьшился на 30к. Деньги не пропали, они перешли из «наличных» в «минус долг». Капитал не изменился.

Нагрузка показывает бремя: 40к / 150к = **27%**. Пользователь видит: «Мой уровень жизни стоит 90к (80к + 10к процентов). Из 150к остаётся 60к свободных. Но 30к из них — обязательный платёж банку. Реально свободных: 20к.»

**Школьный момент** (триггер: первый месяц с кредитными платежами):

> «Ты заработал 150к. На жизнь ушло 90к — 80к расходы и 10к проценты. Остаток — 60к. Это твой поток.
> Но 30к из этих 60к ушли банку на погашение долга. Это не расход — ты уменьшил свой долг на 30к. Деньги не исчезли, просто перешли из наличных в уменьшение долга.
> Свободных — 20к. Через год, когда кредит закроешь, свободных станет 60к.»

## Что «Проект» НЕ делает

- ❌ Не привязывает транзакции к проекту (нет тегирования)
- ❌ Не следит за категориями (нет привязки категорий)
- ❌ Не считает «кредитных vs своих» (нет счётчика)
- ❌ Не меняет аналитику по категориям
- ❌ Не влияет на Капитал, Нагрузку, Запас

Делает **одну вещь**: корректирует главное число на дашборде и его производные.

---

## Edge cases

### Бюджет перерасходован
Проект не блокируется. Школьный момент: «Расходы на "Ремонт" превысили план на 50к. Обновить бюджет?» Кнопка «Обновить» → новая сумма. Базовый поток пересчитывается.

### Два проекта одновременно
Работают параллельно. Базовый поток = полный + Σ(бюджет_i / срок_i). Школьный момент при >2: «У тебя 3 активных проекта. Базовый поток может быть менее точным.»

### Пользователь не создал проект
Живёт с красным потоком. Система напоминает один раз через школьный момент, если видит паттерн: кредит получен → поток упал → красный 2 месяца подряд:

> «Твой поток отрицательный второй месяц подряд. Если это связано с кредитом на конкретную цель — отметь его как проект, и приложение покажет твой реальный уровень расходов отдельно.»

Больше не спрашивает.

### Проект без кредита
В MVP — только с привязкой к кредитному счёту. В будущем можно добавить «проект за свои» (ремонт из накоплений).

## Модель данных

```
Project:
    id: PK
    loan_id: FK → Account (тип credit)
    name: str
    budget: Decimal        ← = сумма кредита (автозаполнение)
    duration_months: int   ← пользователь указывает
    created_at: date
    closed_at: date | None
    status: active | closed
```

Одна сущность, 6 полей. Связь с кредитным счётом через `loan_id`.
