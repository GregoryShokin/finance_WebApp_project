# Анализ: можно ли просто встроить принятую кредитную логику в продукт

Дата: 2026-04-13  
Статус: черновик на основе текущего вывода, не финальное глубокое исследование

## Контекст

Исходим из того, что по текущей проблеме кредитной логики основные решения уже приняты:

- `СМО` остаётся узкой метрикой про свои регулярные деньги
- `DTI` не меняется
- для долговой спирали добавляется слой `здоровье долга`
- для потребкредита на общие нужды используется `тегирование + конверт`
- для целевого потребкредита действует `умная подсказка + галочка + страховка конвертом`
- досрочка живёт вне `СМО`
- типы поступлений разделяются, чтобы не ломать `СМО`

Вопрос: насколько вся эта логика сложна для целевой аудитории, и можно ли встроить её в приложение без перегруза.

## Главный вывод

Да, эту логику можно встроить в приложение просто для аудитории. Но только при одном жёстком принципе:

> Сложность должна жить в движке, а не в голове пользователя.

На уровне архитектуры система уже сложная. На уровне продукта она должна ощущаться как очень простая.

Пользователю не нужно понимать внутреннюю финансовую модель. Ему нужно понимать только:

1. `мои деньги` и `кредитные деньги` — не одно и то же
2. `что я купил`
3. `что реально давит на мой месяц`
4. `долг уменьшается или растёт`

Если приложение удержит именно эти 4 смысла, модель можно встроить мягко.

## Что для аудитории легко

Это хорошо ложится на ЦА и может быть подано бытовым языком:

- `долг растёт / долг уменьшается`
- `из 600 000 ₽ на счёте 500 000 ₽ — кредитные`
- `это обычный доход или разовое поступление`
- `сколько в этом месяце реально ушло на обязательства`
- `какие категории у меня съедают деньги`

Это понятные житейские конструкции, не требующие финансовой подготовки.

## Что для аудитории опасно

Здесь возникает риск перегруза и отторжения:

- если пользователь должен сам выбирать `режим кредитки`
- если нужно понимать, почему в одном месте покупка считается расходом, а в другом нет
- если нужно руками размечать много транзакций
- если на одном экране появляются несколько похожих сумм без ясного различия
- если на экран вывести внутренние термины вроде `FIFO`, `конверт`, `deferred`, `regular/irregular`

Ключевая опасность — не математика сама по себе, а лишние смыслы и лишние решения, которые перекладываются на пользователя.

## Оценка по решениям

### 1. Здоровье долга

Это одно из самых сильных решений для ЦА.

Почему:

- говорит человеческим языком
- отвечает на реальную тревогу
- не требует понимать формулы
- приводит к дейс��вию

Подача должна быть прямой:

- `долг растёт`
- `проценты выше платежа`
- `чтобы остановить рост, нужен платёж от X`

Вывод: решение очень хорошо встраивается в UX.

### 2. Тегирование кредитных денег

Как внутренняя механика — хорошее решение. Как пользовательский термин — плохое.

Плохо:

- `тегирование кредитной ликвидности`

Хорошо:

- `на счёте 600 000 ₽, из них 500 000 ₽ кредитные`

Если пользователь видит именно результат, а не механизм, логика становится понятной и даже полезной: снимает иллюзию ложного богатства.

Вывод: встраивается хорошо, если механизм скрыт.

### 3. Конверт для потребкр��дита на общие нужды

Это наиболее рискованное решение с точки зрения UX.

Не потому что оно плохое. Наоборот, оно выглядит самым жизнеспособным среди доступных компромиссов. Но это место, где пользователь реально включается в учёт.

Рабочий вариант:

- раз в месяц к��роткий вопрос
- одно число
- возможность пропустить

Нерабочий вариант:

- ручная разметка множества транзакций
- постоянная дисциплина
- сложная форма с большим числом подтверждений

Хорошая подача:

- `Похоже, в этом месяце был�� траты из кредита. Учесть их в аналитике?`
- `Введите сумму`
- `Пропустить`

Вывод: решение можно встроить, но только как мягкий редкий чек-ин, а не как постоянную бухгалтерию.

### 4. Классификация поступлений

Это сильное решение на уровне модели, но наружу его нужно радикально упрощать.

Для пользователя должно быть только два вопроса:

- `это ваш обычный доход?`
- `или это разовое поступление?`

Не стоит выводить сложную таксономию из пяти типов.

Вывод: решение хорошо встраивается, если пользователь видит бинарный выбор, а детальная классификация остаётся под капотом.

### 5. Логика кредитки

Какая бы модель ни была окончательно зафиксирована, пользователь не должен учить новую бухгалтерию кредитки.

Пользовательский уровень должен быть таким:

- что купил
- какая от этого нагрузка
- что происходит с долгом

Если для понимания экрана нужно объяснять лаг аналитики, тип признания расхода или внутренние правила расчёта, UX ещё не доварен.

Вывод: главная проверка качества здесь — не математическая красота, а то, насколько незаметно система объясняет реальность без обучения внутренней модели.

## Можно ли встроить это просто

Да, если держаться 5 правил.

### 1. Один экран = один вопрос

Не пытаться заставить один блок отвечать сразу на всё.

Лучше разделять:

- блок `покупки`
- блок `нагрузка месяца`
- блок `долг`

### 2. Не давать пользователю выбирать режимы

Не спрашивать:

- как учитывать кредитку
- включать ли что-то в `СМО`
- cash flow это или обязательство

Пользователь должен только подтверждать понятные факты.

### 3. Вмешательство только в аномалиях

Ручные вопросы нужны только там, где система сама не уверена:

- крупная покупка после выдачи кредита
- необычно большое поступление
- траты из кре��итного конверта

Остальное должно работать молча.

### 4. Внутренние механики не показывать

Не выводить пользователю:

- `FIFO`
- ретроспективу
- типы признания
- отложенные расходы

Показывать нужно только смысл:

- `эта покупка оплачивается частями`
- `часть денег на счёте кредитная`
- `долг растёт`

### 5. Каждое сложное правило должно объясняться одной человеческой фразой

Если правило нельзя объяснить одной бытовой фразой, его не надо выводить на экран.

Хорошие формулировки:

- `эти деньги не ваши, а кредитные`
- `это разовое поступление, поэтому оно не влияет на ваш обычный ритм`
- `долг растёт быстрее, чем вы платите`
- `эта покупка оплачивается постепенно`

## Честный итог

Оценка на текущий момент:

- архитектурная сложность: высокая
- UX-сложность для пользователя: может быть низкой
- главный риск: не математика, а перегрузка смыслами
- самое уязвимое решение: `конверт` по потребкредиту
- самые сильные решения: `здоровье д��лга`, отделение кредитных денег от своих, мягкая классификация поступлений

Итоговый вывод:

> Да, всю эту логику можно встроить в приложение просто для аудитории.
> Но только если пользователь будет вид��ть не систему кредитного учёта, а ограниченный человеческий интерфейс из нескольких сигналов и 1–2 редких подтверждений.
