# Концепция упрощения кредитной логики для пользователя

**Дата:** апрель 2026  
**Статус:** рабочий документ, основа для проектирования UX  
**Принцип:** ноль настроек при старте, умные вопросы по ситуации

---

## Главная идея

Вся сложная кредитная логика (6 механизмов) должна работать по одному принципу:

> **Приложение наблюдает → замечает аномалию → задаёт один точный вопрос.**

Пользователь не настраивает систему заранее. Он подтверждает понятные факты в момент, когда они возникают. Всё остальное — под капотом.

---

## Ограничения от ЦА

Целевая аудитория — люди 25–45, многие с низкой финансовой грамотностью. Из этого следует:

- Пользователь выдерживает **бытовые смыслы**, а не финансовую таксономию
- Может подтверждать **факты**, но не должен выбирать между абстрактными моделями
- Любой сценарий с регулярной ручной дисциплиной — по умолчанию рискован
- Любая логика, которую нельзя объяснить одной фразой — не должна всплывать наружу

**Проверочный вопрос для каждого UX-решения:**

> Поможет ли это семье с DTI 42%, низким СМО, без подушки, у которой нет сил разбираться в бухгалтерии?

## Механизм 1: Модель кредитной карты

### Принятое решение
Два подхода: «покупка = расход» и «платёж = расход». Пользователь выбирает.

### Как упрощаем

**Не спрашивать при создании счёта.** Дефолт — «покупка = расход» (подходит большинству: кешбэк-пользователи, гасящие в грейс).

**Вопрос возникает только по ситуации:** если система видит, что на карте накапливается непогашенный долг (баланс растёт 2+ месяца), появляется одно уведомление:

> «На карте [название] растёт долг. Хотите, чтобы аналитика учитывала ваши платежи по этой карте отдельно?»
> — Да → переключение на «платёж = расход»
> — Нет, я планирую погасить → остаётся «покупка = расход»

**Одно уведомление, один раз, по факту.** Не при онбординге, не в настройках — а когда данные говорят, что пора.

**В настройках:** переключатель доступен, но спрятан в расширенных настройках счёта. Для продвинутых пользователей, которые знают, чего хотят.

### Что скрыто
- Термины «модель учёта», «признание расхода», «СМО = кеш vs решение потратить»
- Математика переключения
- Пересчёт аналитики

### Что видно пользователю
- Покупки в реальных категориях (при дефолте)
- Один вопрос, если ситуация изменилась

## Механизм 2: Тегирование кредитных денег + подтверждение трат

### Принятое решение
При получении потребкредита деньги на дебете тегируются. При покупке система спрашивает «это из кредитных денег?».

### Проблема текущего решения
Спрашивать на **каждую** транзакцию — это 50+ вопросов за 3 месяца ремонта. Продукты, такси, кофе — на всё вопрос «это из кредитных денег?». Пользователь будет отвечать «нет» на 80% вопросов. Это пытка уведомлениями.

### Как упрощаем — три варианта (от простого к точному)

**Вариант А: Батч после импорта (рекомендуется для MVP)**

После импорта выписки за период — один экран:

> «У вас есть кредит "Ремонт" (осталось 350к кредитных).  
> В этом месяце расходы: 255к.  
> Сколько примерно ушло на ремонт?»  
> [_150 000_] [Пропустить]

Одно число, один раз после импорта. Не 30 вопросов на 30 транзакций.

**Вариант Б: Выбор категорий**

Вместо суммы — выбрать категории, которые относятся к кредиту:

> «Какие траты в этом месяце были из кредита "Ремонт"?»  
> ☑ Стройматериалы — 45к  
> ☑ Мебель — 32к  
> ☐ Продукты — 40к  
> ☐ Рестораны — 15к  
> **Итого из кредита: 77к** [Подтвердить]

Система сама суммирует. Пользователь только ставит галочки на очевидных категориях.

**Вариант В: Умный вопрос только на крупные**

Спрашивать не на каждую транзакцию, а только когда покупка похожа на кредитную:
- Сумма > 10к **И** на счёте есть тегированные деньги
- Или категория совпадает с назначением кредита (ремонт → стройматериалы)

Мелкие покупки (продукты, кофе, транспорт) — не спрашивать вообще. Нерешённый остаток — через батч в конце месяца (вариант А).

### Рекомендация

Комбинация: **вариант В (умные вопросы на крупные) + вариант А (батч в конце месяца на остаток)**. Минимум вопросов, максимум точности.

### Что скрыто
- Термин «тегирование», внутренняя механика тега
- Перемещение тега при переводах
- Формула корректировки СМО

### Что видно пользователю
- «На счёте 600к, из них 500к кредитные» — в виджете доступных финансов
- Редкие вопросы на крупные покупки
- Один батч-вопрос после импорта

## Механизм 3: Крупные покупки

### Принятое решение (пересмотрено)
~~Галочка «Крупная покупка» в форме транзакции~~ → **Упразднена (апрель 2026).**

Стресс-тест философии СМО показал: разделять покупки на «из текущего дохода» и «из накоплений» невозможно (деньги fungible) и опасно (маскирует истощение накоплений). См. [[Философия СМО — стресс-тест]].

### Новое решение: два пути

**Покупка за свои деньги (дебет, наличные):**

Никакого специального механизма. СМО отражает честный кэшфлоу. Пользователь анализирует аномалии через **фильтр исключения в аналитике**:

1. Видит красный месяц → заходит в аналитику → видит «80к — iPhone»
2. Нажимает «исключить» → видит картину без выброса
3. Понимает: «Без телефона я живу нормально»

Не нужно: галочка, автоподсказка, тип расхода «крупные покупки», амортизация.

**Покупка в кредит/рассрочку:**

Обрабатывается кредитной логикой — модель кредитной карты, FIFO-очередь для рассрочки, тегирование для потребкредита. «Крупная покупка» как отдельный механизм не нужна.

### Что скрыто
- FIFO-очередь для карт рассрочки
- Атрибуция платежей к покупкам
- Философия «честный СМО vs сглаженный» (см. [[Философия СМО — стресс-тест]])

### Что видно пользователю
- Фильтр исключения в аналитике (по транзакции или категории)
- Прогресс погашения (если на рассрочке): «Ноутбук 120к — оплачено 40к, осталось 80к»

> **Отвергнуто:** автоподсказка «Отметить как крупную?» по порогу суммы, тип расхода «крупные покупки», амортизация в СМО. Подробнее: [[Философия СМО — стресс-тест]].

## Механизм 4: Классификация поступлений

### Принятое решение
5 типов поступлений, из которых только `regular` влияет на СМО.

### Как упрощаем

**Не спрашивать на каждое поступление.** Спрашивать только при аномалии:

Зарплата 200к приходит каждый месяц → система распознаёт по паттерну (контрагент, сумма ±20%, дата ±5 дней). **Не спрашивает ничего.**

Поступление 300к от незнакомого контрагента → аномалия:

> «Поступление 300к — это ваш обычный доход или разовое?»  
> [Обычный доход] [Разовое поступление]

**Два варианта, не пять.** Детальная классификация (`irregular` / `asset_sale` / `gift` / `refund`) — под капотом, определяется автоматически по контексту или через расширенные настройки.

**Тип привязан к категории, не к транзакции.** Пользователь один раз создаёт категорию «Фриланс» → все будущие поступления автоматически `regular`. Не нужно размечать каждый перевод.

### Что скрыто
- Пять типов классификации
- Влияние на формулу СМО
- Термины `regular`, `irregular`, `asset_sale`

### Что видно пользователю
- Один вопрос при необычном поступлении
- «Доход: 200к» + «Другие поступления: +300к (бонус)» — в аналитике

## Механизм 5: Здоровье долга

### Принятое решение
Траектория, оценка процентов, проекция погашения, рекомендации.

### Как упрощаем

**Вообще не нужно упрощать — это уже просто.** Пользователь ничего не делает. Система считает из имеющихся данных и показывает результат.

На дашборде:

> **Долги: 355к** ↑🔴 *+5к за месяц*

На странице долга:

> Проценты: ~10к/мес  
> Ваш платёж: 5к/мес  
> **Платёж не покрывает проценты. Долг растёт.**  
> Чтобы остановить рост → от 10к/мес  
> Чтобы закрыть за 12 мес → 33к/мес

**Это самое сильное решение во всей кредитной логике.** Говорит человеческим языком, отвечает на реальную тревогу, не требует понимать формулы, приводит к действию. Не нужно ни одного вопроса к пользователю.

### Что скрыто
- Формула оценки процентов
- Декомпозиция изменения баланса (покупки / проценты / платёж)
- Алгоритм проекции

### Что видно пользователю
- Стрелка траектории на дашборде
- Простые фразы и конкретные числа на странице долга

## Механизм 6: Досрочное погашение

### Принятое решение
Исключено из СМО, только в денежный поток.

### Как упрощаем

**Автоопределение:** платёж на кредитный счёт, сумма значительно больше обычного ежемесячного платежа → система предлагает:

> «Платёж 300к по кредиту "Ремонт" — это досрочное погашение?»  
> [Да, досрочка] [Нет, обычный платёж]

Если обычный платёж 45к, а пришло 300к — очевидная аномалия. Один вопрос.

### Что скрыто
- Исключение из СМО
- Разница между потоком и балансом
- Почему ретроспектива не нужна

### Что видно пользователю
- Один вопрос при аномальном платеже
- Досрочка видна в денежном потоке и истории кредита
- Снижение долга в виджете «Долги»

---

## Сводка: сколько вопросов задаёт система

| Ситуация | Вопрос | Частота |
|---|---|---|
| Долг на кредитке растёт 2+ мес | «Хотите учитывать платежи отдельно?» | Разово |
| После импорта при активном кредите | «Сколько ушло на ремонт?» | ~1 раз/мес |
| Крупная покупка + тегированные деньги | «Это из кредитных денег?» | Редко |
| Необычное поступление | «Обычный доход или разовое?» | Редко |
| Аномально крупный платёж по кредиту | «Это досрочное погашение?» | Редко |

**Типичный месяц кешбэк-пользователя без кредитов:** **ноль вопросов.**

**Типичный месяц пользователя с ремонтным кредитом:** **1–2 вопроса** (после импорта + может быть крупная покупка на кредитные деньги).

**Типичный месяц пользователя с долгом на кредитке:** **ноль вопросов** (модель переключена, здоровье долга считается автоматически).

## Пять принципов UX (из исследования GPT)

### 1. Один экран = один вопрос
Не пытаться одним блоком отвечать на «что купил», «сколько ушло» и «что с долгом» одновременно.

### 2. Пользователь подтверждает факты, а не выбирает модели
Нельзя: «как учитывать кредитку?» Можно: «это покупка на кредитные деньги?»

### 3. Ручное участие — только в аномалиях
Чем реже система задаёт вопрос, тем сильнее он воспринимается как помощь.

### 4. Система полезна даже без идеальной разметки
Если пользователь пропустил вопрос — приложение не разваливается. Мягко напоминает, оставляет право не участвовать.

### 5. Каждое правило = одна бытовая фраза
Если нельзя объяснить одной фразой — не выводить на экран.

Хорошие формулировки:
- «Эти деньги не ваши, а кредитные»
- «Это разовое поступление, оно не влияет на ваш обычный ритм»
- «Долг растёт быстрее, чем вы платите»
- «Эта покупка оплачивается постепенно»

---

## Что обязательно скрыть

Пользователь **не должен видеть** как понятия:
- FIFO, конверт, deferred, regular/irregular
- Типы признания расходов
- Логику распределения платежа
- Разницу «СМО = кеш» vs «СМО = решение потратить»

Пользователь **должен видеть** результаты:
- «На счёте 600к, из них 500к кредитные»
- «Долг растёт / уменьшается»
- «Ваш платёж не покрывает проценты»
- Покупки в реальных категориях на дашборде

---

## Следующие шаги

1. Свести в таблицу: `решение → как показываем → что скрываем → когда спрашиваем`
2. Спроектировать экраны для самых рискованных точек:
   - Батч-вопрос после импорта (кредитные траты)
   - Фильтр исключения в аналитике (крупные покупки)
   - Виджет здоровья долга
3. Определить пороги для автоподсказок (% от средних расходов, сумма для тегированных покупок)
4. Протестировать на сценариях из стресс-теста: кешбэк, ремонт, импульсная покупка, долговая спираль
