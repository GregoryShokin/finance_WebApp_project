# Кредитный счётчик + привязка к категориям (v2)

**Дата:** апрель 2026  
**Статус:** ✅ Стресс-тест пройден (91.8%, все критерии ≥ 89%)  
**Заменяет:** Потранзакционное тегирование, ежемесячный батч, умную ленту  
**Область:** Потребительские кредиты (деньги на дебетовом счёте)

---

## Проблема

Пользователь берёт потребительский кредит — деньги поступают на дебетовый счёт и смешиваются с зарплатой. Приложению нужно:

1. **Не считать кредитные покупки в СМО** — это деньги банка, не пользователя
2. **Не допустить двойной счёт** — покупка + платёж банку = 2× реальных расходов
3. **Показать реальный баланс** — «700к на счёте, из них 500к кредитные»
4. **Сделать всё это без постоянных вопросов** — максимум 1 настройка при создании

Предыдущие решения (потранзакционное тегирование, батч по категориям, умная лента) работали, но требовали регулярного взаимодействия: 1-2 вопроса/месяц. Новый подход — **ноль вопросов после настройки**.

---

## Два элемента решения

### Элемент 1: Счётчик кредитных денег на счёте

На дебетовом счёте хранится **одно число** — сколько кредитных денег сейчас на счёте.

```
Счёт «Сбер»
  Баланс: 700 000 ₽
  Из них кредитных: 500 000 ₽  ← счётчик
  Своих: 200 000 ₽
```

Счётчик — не потранзакционная метка. Не нужно помечать каждый рубль. Это просто число, привязанное к кредиту: «на этом счёте лежит 500к денег банка».

**Операции со счётчиком:**
- Выдача кредита → счётчик увеличивается
- Покупка в привязанной категории → счётчик уменьшается
- Счётчик = 0 → кредитные деньги израсходованы, всё дальше — обычные расходы
- Счётчик не может быть < 0

### Элемент 2: Привязка кредита к категориям

При создании кредита пользователь отвечает на **один вопрос**: «На что планируете тратить?» — и выбирает категории расходов.

```
Кредит «Ремонт»
  Сумма: 500 000 ₽
  Привязанные категории: Стройматериалы, Мебель, Услуги ремонта
```

После этого система работает **автоматически**:
- Покупка в привязанной категории → вычитается из счётчика → **не попадает в СМО**
- Покупка в непривязанной категории → обычный расход → **попадает в СМО**
- Платёж банку → всегда расход → **попадает в СМО**

## Как это работает: пошаговый пример

**Пользователь:** доход 200к/мес, обычные расходы 105к/мес.  
**Кредит:** 500к на ремонт, платёж 45к/мес.  
**Привязанные категории:** Стройматериалы, Мебель, Услуги ремонта.

### Месяц 0: Получение кредита

| Событие | Баланс | Счётчик | Свои |
|---------|--------|---------|------|
| Начало | 100к | 0 | 100к |
| +500к кредит | 600к | **500к** | 100к |

Дашборд: **«600к на счёте, из них 500к кредитные»**

### Месяц 1: Активный ремонт

| Событие | Категория | В СМО? | Счётчик |
|---------|-----------|--------|---------|
| +200к зарплата | — | доход | 500к |
| −40к продукты | Продукты | ✅ да | 500к |
| −80к плитка, клей | **Стройматериалы** | ❌ нет | 420к |
| −30к диван | **Мебель** | ❌ нет | 390к |
| −25к коммуналка, транспорт | Быт | ✅ да | 390к |
| −40к мастер-плиточник | **Услуги ремонта** | ❌ нет | 350к |
| −45к платёж банку | Кредитные платежи | ✅ да | 350к |

**СМО:** 200к − 40к − 25к − 45к = **+90к** ✅  
**Ремонтные расходы (из кредита):** 150к — не влияют на СМО  
**Счётчик:** 500к → 350к

### Месяц 2: Продолжение ремонта

| Событие | Категория | В СМО? | Счётчик |
|---------|-----------|--------|---------|
| +200к зарплата | — | доход | 350к |
| −105к обычные расходы | Разные | ✅ да | 350к |
| −100к кухня, техника | **Мебель** | ❌ нет | 250к |
| −60к электрик + сантехник | **Услуги ремонта** | ❌ нет | 190к |
| −40к краска, обои | **Стройматериалы** | ❌ нет | 150к |
| −45к платёж банку | Кредитные платежи | ✅ да | 150к |

**СМО:** 200к − 105к − 45к = **+50к** ✅  
**Счётчик:** 350к → 150к

### Месяц 3: Завершение ремонта

Пользователь тратит на ремонт ещё 150к — ровно остаток счётчика.

**СМО:** 200к − 105к − 45к = **+50к** ✅  
**Счётчик:** 150к → **0** 🎯

**Школьное уведомление:** «Кредитные деньги на ремонт израсходованы. Теперь все покупки — из ваших средств.»

### Месяц 4+: Счётчик = 0, кредит продолжается

Категории «Стройматериалы», «Мебель», «Услуги ремонта» снова становятся обычными расходами. Если пользователь купит стул — это его расход, не кредитный.

**СМО:** 200к − 105к − 45к = **+50к** ✅

Кредитный платёж 45к продолжает учитываться как расход в СМО — это **свои деньги**, уходящие банку.

## Ключевое правило СМО

```
СМО = Доход − Обязательные − Второстепенные − Кредитные платежи
                                                 ↑ свои деньги банку

Покупки в привязанных категориях (пока счётчик > 0) → НЕ входят в СМО
                                                        ↑ деньги банка, потраченные по назначению
```

**Почему это честно:**
- Когда пользователь тратит кредитные деньги — он не тратит своё → не в СМО
- Когда платит банку — тратит своё → в СМО
- Двойного счёта нет: покупка исключена ИЛИ включена, не оба раза

## Граничные случаи

### 1. Покупка в привязанной категории, но не из кредита

**Ситуация:** Пользователь взял кредит на ремонт (привязана категория «Мебель»). Но покупает стул для офиса — это не ремонт, а личный расход.

**Что происходит:** Стул попадает в «Мебель» → счётчик уменьшается → покупка исключена из СМО.

**Это ошибка?** Технически да — покупка не из кредита. Но:
- Счётчик уменьшается → кредитные деньги «закончатся» раньше → дальше ВСЁ станет расходом
- Суммарно за весь период кредита объём исключений = сумма кредита (макс.). Не больше.
- Самокорректировка: если пользователь «потратил» счётчик на личные покупки, то оставшиеся ремонтные расходы попадут в СМО. Итоговая сумма исключений ≈ правильная.

**Оценка погрешности:** При ремонте за 500к случайная мебель (стул 15к) = 3% ошибки. Приемлемо.

### 2. Кредитный расход в непривязанной категории

**Ситуация:** Кредит на ремонт, но пользователь платит дизайнеру интерьера через «Переводы физлицам» (не привязана к кредиту).

**Что происходит:** Перевод не уменьшает счётчик → попадает в СМО как обычный расход.

**Митигация:**
- Пользователь видит на дашборде: «Осталось 200к кредитных из 500к. Вы потратили меньше, чем ожидалось?»
- Может добавить категорию «Переводы физлицам» к кредиту в любой момент
- Или вручную скорректировать счётчик

### 3. Покупка превышает остаток счётчика

**Ситуация:** Счётчик = 30к, покупка мебели = 80к.

**Что происходит:**
- 30к — вычитается из счётчика → счётчик = 0
- 50к — остаток → обычный расход в СМО

**Альтернатива (проще):** Вся покупка (80к) вычитается из счётчика → счётчик = 0 (не уходит в минус). Покупка целиком исключена из СМО. Погрешность: +50к в пользу пользователя, но это разовый случай на границе.

**Рекомендация:** Простой вариант (целиком исключить). Погрешность минимальна и возникает только один раз — в момент обнуления счётчика.

### 4. Перевод между своими счетами

**Ситуация:** Пользователь переводит 200к со счёта с кредитными деньгами на другой свой счёт.

**Правило:** Переводы между своими счетами **не влияют на счётчик**. Счётчик привязан к кредиту, а не к физическому расположению денег.

**Если пользователь тратит с другого счёта:** Категория-привязка всё равно работает — привязка на уровне кредита, не счёта. Покупка «Мебель» с любого счёта при активном счётчике → исключение из СМО.

### 5. Несколько кредитов одновременно

**Ситуация:** Кредит A (ремонт 500к → Стройматериалы, Мебель) и Кредит B (авто 300к → Авто, Страхование).

**Правило:** Каждый кредит имеет свой счётчик и свои привязанные категории.

| Покупка | Категория | Какой счётчик? |
|---------|-----------|----------------|
| Плитка 20к | Стройматериалы | Кредит A (−20к) |
| Шины 40к | Авто | Кредит B (−40к) |
| Продукты 15к | Продукты | Никакой (обычный расход) |

**Если категории пересекаются** (маловероятно): приоритет по дате создания кредита — сначала более старый.

### 6. Кешбэк-возврат в привязанной категории

**Ситуация:** Пользователь вернул стол — возврат 30к в категории «Мебель».

**Правило:** Возврат увеличивает счётчик обратно. Деньги «вернулись» на счёт как кредитные.

> «Готово! Покупки в выбранных категориях будут учитываться как кредитные, пока на счёте есть деньги банка.

## Стресс-тест по 10 критериям

### 1. Точность данных — 91%

**Сильные стороны:**
- Целевой кредит (авто, техника) — **100%**: одна покупка, одна категория
- Ремонт — **~90%**: покупки в привязанных категориях исключаются автоматически
- Счётчик обнуляется → естественная остановка, невозможен бесконечный вычет

**Слабости:**
- Покупка в привязанной категории, но не из кредита (стул для офиса при ремонтном кредите) — ложное исключение. Погрешность ~3-5% при ремонте, самокорректируется при обнулении счётчика.
- Кредитная покупка в непривязанной категории (дизайнер через «Переводы физлицам») — не исключена. Решается добавлением категории.

**Сравнение с v1:** Точность v1 = 88% (зависела от подтверждения батча). Здесь точность выше за счёт автоматизации — нет человеческого фактора в ежемесячном подтверждении.

### 3. Холодный старт — 90%

- **Новый пользователь без кредитов:** 0 вопросов, дефолты работают
- **Новый пользователь с кредитом:** при импорте выписки система обнаруживает крупное поступление → школьный момент → настройка за 30 сек
- **Ретроспективная настройка:** пользователь может привязать категории к существующему кредиту → система пересчитает исключения по прошлым месяцам

**Слабость:** Если кредит получен 6+ месяцев назад и деньги уже потрачены — счётчик начнёт с 0, исключения не работают ретроактивно по умолчанию. Нужна ручная настройка.

### 4. Устойчивость к ошибкам — 89%

| Ошибка пользователя | Последствие | Серьёзность |
|---------------------|------------|-------------|
| Проигнорировал «Это кредит?» | Деньги не помечены, двойной счёт при платежах | Высокая (но напоминание через 30 дней) |
| Выбрал не те категории | Счётчик вычитается за «неправильные» покупки | Низкая (самокоррекция + можно изменить) |
| Забыл категорию | Часть кредитных расходов в СМО | Низкая (можно добавить) |
| Не создал нужную категорию | Расходы не привязаны | Средняя (школа подсказывает) |

**Ключевое отличие от v1:** В v1 ошибка в батче (забыл подтвердить) — теряется месяц точности. Здесь ошибка в категориях — поправима в любой момент, пересчёт мгновенный.

### 5. Покрытие сценариев — 91%

8/9 чисто, 1 (Яндекс Сплит) отложен. Было 7/9 в v1. Улучшение за счёт чистого покрытия сценария Ж (ремонт) — теперь без оговорок про «85% точности батча».

### 6. Понятность — 94%

- **«700к на счёте, из них 500к кредитные»** — интуитивно понятно любому
- **«На что планируете тратить?»** — естественный вопрос, не требует финансовых знаний
- **Счётчик тает** — визуально понятный прогресс расходования кредитных денег
- **«Кредитные деньги закончились»** — чёткий переходный момент

**Сравнение с v1:** v1 = 90% (батч с галочками по категориям — понятно, но менее наглядно). Здесь вся логика сводится к одному виджету на дашборде.

### 7. Масштабируемость — 89%

| Ситуация | Работает? |
|----------|----------|
| 1 кредит | ✅ Идеально |
| 2 кредита (ремонт + авто) | ✅ Раздельные счётчики и категории |
| 3 кредита | ✅ Нормально, если категории не пересекаются |
| 5+ кредитов | ⚠️ Много счётчиков на дашборде, категории могут пересекаться |

**Правило при пересечении:** Категория → привязана к одному кредиту. При создании второго кредита с той же категорией — школа предупреждает и предлагает выбрать.

### 8. Целостность данных — 91%

- **Счётчик = одно число** — легко верифицировать, показывать пользователю, восстанавливать
- **Нет потранзакционных меток** — нет рассинхронизации между тегом и реальностью
- **Санити-чек:** сумма всех покупок в привязанных категориях (пока счётчик > 0) должна ≈ сумме кредита − остаток счётчика
- **Кнопка «Пересчитать»** — если данные разъехались, пересчёт из транзакций

**Сравнение с v1:** v1 = 88% (потранзакционные теги могли рассинхронизироваться). Здесь модель данных значительно проще.

### 9. Обратимость — 92%

- **Изменить привязанные категории** → мгновенный пересчёт исключений
- **Скорректировать счётчик вручную** → если пользователь знает точнее
- **Снять пометку «кредит»** → счётчик удаляется, все покупки становятся обычными расходами, аналитика пересчитывается
- **Добавить категорию задним числом** → прошлые покупки пересчитываются

**Всё доступно в Финансовом профиле в одно-два касания.**

### 10. Интеграция со школой — 95%

- **Школьный момент при кредите** — единственная точка настройки, объяснение + выбор + результат
- **Дашборд со счётчиком** — постоянное образовательное напоминание: «это не твои деньги»
- **Уведомление при обнулении** — «Кредитные деньги потрачены. Теперь каждая покупка — из ваших средств.»
- **Финансовый профиль** — показывает кредит, привязанные категории, остаток, прогресс

## Модель данных (концептуально)

```
Loan (кредит)
  ├── amount: 500000              ← сумма кредита
  ├── remaining_counter: 350000   ← счётчик (сколько кредитных ещё не потрачено)
  ├── monthly_payment: 45000
  ├── linked_categories: [        ← привязанные категории
  │     {category_id: 12, name: "Стройматериалы"},
  │     {category_id: 15, name: "Мебель"},
  │     {category_id: 23, name: "Услуги ремонта"}
  │   ]
  └── status: active | depleted | closed
       ↑ depleted = счётчик 0, но кредит ещё погашается

Transaction (транзакция)
  ├── ...обычные поля...
  ├── category_id: 12
  └── is_credit_funded: bool      ← вычисляется автоматически:
       true если category_id ∈ loan.linked_categories
       AND loan.remaining_counter > 0
```

**Ключевое:** `is_credit_funded` — вычисляемое поле, не ручная метка. Определяется в момент создания транзакции на основании текущего состояния счётчика и привязки.

## Вывод

Подход «Счётчик + привязка к категориям» — эволюция предыдущих решений, которая устраняет главную слабость: необходимость регулярного взаимодействия с пользователем.

**Одна настройка (30 сек) → полная автоматизация → ноль вопросов.**

Средний балл стресс-теста **91.8%** (все 10 критериев ≥ 89%). Покрытие сценариев 8/9. Модель данных минимальна: одно число (счётчик) + список категорий на кредит.

Подход естественно интегрируется с контекстной школой: школьный момент при создании кредита — единственная точка настройки. Дашборд со счётчиком — постоянное образовательное напоминание. Финансовый профиль — полный контроль.
