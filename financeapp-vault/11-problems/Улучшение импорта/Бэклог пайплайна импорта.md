## 1. Import Health — админский дашборд

**Спек:** §5 (error state, видимость проблем).
**Контекст.** При текущей реализации системные ошибки импорта (падение парсера, сбои LLM, нарушения инвариантов) не всплывают нигде — ни для админа, ни для юзера. Проблема обнаруживается только когда юзер сам заметит кривые данные.

**Что нужно.** Админская страница, где видна статистика импорт-сессий за период:
- Количество сессий по статусам (successful, partial, failed).
- Top-N банков по частоте error-строк.
- Top-N причин падения нормализации.
- Процент строк, ушедших в LLM-модерацию (косвенный индикатор качества правил).
- Алерты из §11.1 (сценарий fail системной целостности).

**Где.** Под административной ролью, отдельная страница `/admin/import-health`. Возможно, расширение Sentry-интеграции.

## 2. LLM rescue subsystem

**Спек:** §7 (LLM как верхний барьер). Выделено как отдельная подсистема, а не часть модератора.

**Контекст.** В обсуждении появилась идея «LLM rescue» — когда нормализация упала или вернула неполные decisions, LLM **не** применяется к fact-полям (запрещено §12.5), но может обогатить decision-часть. Например: skeleton получился слишком общий → LLM помогает достроить structural tokens.

**Что нужно.**
- Отдельный сервис `LLMRescueService`, не путать с `ImportModeratorService`.
- Явное тегирование каждого LLM-вмешательства (source tag, prompt version, confidence).
- Кэширование ответов по (prompt_hash, input_hash) для воспроизводимости.
- Клир разделение: rescue работает на уровне decisions, moderator — на уровне предсказаний.

**Open.** Нужно ли это вообще? Возможно текущего модератора достаточно. Разобраться при реализации #3.

## 3. Унифицированный normalization contract

**Спек:** §3, §4 (единая атомарная стадия, контракт facts vs decisions).

**Контекст.** Сейчас нормализация размазана: `ImportNormalizer` (v1), `ImportNormalizerV2`, плюс логика в `ImportRecognitionService`. Контракт нигде явно не зафиксирован. Поля facts и decisions физически в одной модели `ImportRow`, код может писать decisions в fact-поля.

**Что нужно.**
- Явный dataclass / Pydantic-модель `NormalizedRow` с разделёнными группами `facts` и `decisions`.
- Единственная функция `normalize(raw_row, settings) -> NormalizedRow`.
- Удалить параллельный v1/v2 путь — оставить один нормализатор.
- Правило: любое изменение fact-поля после нормализации — ValueError.
- `normalizer_version` как поле + transparent re-run на чтение (§4.3).

**Приоритет высокий** — без этого рефакторинга остальные задачи пайплайна болтаются на неустойчивом фундаменте.

## 4. Visible error state в UI

**Спек:** §5.2 (triggers), §5.3 (переходы).

**Контекст.** Строки статуса `error` сейчас либо молча исключаются из коммита, либо висят как «замороженные» без понятных действий. Юзер не знает что с ними делать.

**Что нужно.**
- В модераторе error-строки показываются в отдельной секции с понятным объяснением («не удалось определить сумму», «сломан формат даты»).
- Две кнопки на строку: **«Исключить»** (→ `excluded`) и **«Отложить»** (→ `parked`, переносится в следующую сессию).
- Массовое действие на уровне секции: «исключить все», «отложить все».
- Невозможность коммитить batch пока остались error-строки без решения.

**Зависимость.** Рекомендуется делать после #3 (единый контракт), иначе причины error'ов плохо структурированы.

## 5. Account type metadata field

**Спек:** §9.2 (три подкласса кредитных счетов), §7.2 (LLM контекст).

**Контекст.** Сейчас `Account.type` — свободное поле. Для §9 нужны конкретные подтипы кредитных счетов, а для §7 — широкий набор типов влияющих на контекст LLM.

**Что нужно.**
- Enum `AccountType`: `main` / `marketplace` / `loan` / `credit_card` / `installment_card` / `broker` / `savings` / `currency`.
- Миграция Alembic — добавить поле, смигрировать существующие счета на основе эвристик (можно вручную через админку).
- UI создания/редактирования счёта — селектор подтипа.
- Валидация: нельзя создать loan-счёт без параметров кредита (см. #8).
- Эвристики импорта по account_type (например, для `loan` — переводы всегда идут через split-форму).

**Блокирует.** #8, #9, #10, #11.

## 6. Multi-dimensional LLM moderation

**Спек:** §7 (раздел «Что НЕ делает LLM», бэклог.6).

**Контекст.** Обсуждение 2026-04-24 выявило паттерны типа «Линда + крупная сумма = аренда, Линда + мелкая = коммуналка», «120k от договора N = ЗП, 60-70k от того же договора = дивиденды». Это не fingerprint (один контрагент), но разная категория по другим осям. Текущая одноразмерная классификация такое не ловит.

**Что нужно.**
- Расширить input для LLM: массив (fingerprint, примеры сумм, типичный диапазон, частота, account type, сезонность).
- Prompt-template, который явно предлагает LLM искать многомерные паттерны.
- Сохранение в модели — вероятно, rule с multi-condition (сумма в диапазоне X-Y + скелетон Z → категория A).

**Приоритет низкий.** Базовый случай «один fingerprint → одна категория» покрывает 95%+ трафика. Это оптимизация последних 5%.

## 7. Миграция legacy TransactionCategoryRule

**Спек:** §6.1 (два типа правил: specific / general).

**Контекст.** Существующие правила имеют поле `scope` со значениями `exact` / `bank` / `global` / `legacy_pattern`. В новой модели — только `specific` (по идентификатору) и `general` (по скелетону). Нужна миграция.

**Что нужно.**
- Правила scope=`exact` с идентификатором → `specific`.
- Правила scope=`global` по скелетону → `general`.
- Правила scope=`bank` — переосмыслить: bank как контекст теперь входит в account_type metadata (#5) и идёт в LLM-контекст, а не в rule scope. Таким правилам — либо разжать в `specific`/`general` с сохранением уверенности, либо пометить broken и убрать.
- Правила `legacy_pattern` — скорее всего устаревшие, пометить inactive и дать юзеру возможность восстановить через подтверждение.
- Сохранить confidence-статистику (confirms / rejections) при миграции.

**Приоритет высокий** — блокирует чистую работу §6.

## 8. Параметры кредита как свойство loan-счёта

**Спек:** §9.4.

**Что нужно.**
- Новая таблица `loan_parameters` (или поля на `Account` для loan-подтипа): `principal_amount`, `interest_rate`, `term_months`, `start_date`, `loan_type` (annuity / differentiated), `monthly_payment` (расчётный).

## 9. Парсинг графика платежей + эволюция

**Спек:** §9.4.

**Что нужно.**
- Форма загрузки PDF/Excel-графика при создании loan-счёта (опционально).
- Парсер таблицы `(payment_date, total, interest, principal, remaining_balance)`. Разные банки дают разные форматы — нужно минимум 3-5 адаптеров для популярных (Сбер, ВТБ, Тинькофф, Альфа, Россельхоз).
- Хранение графика в новой таблице `loan_schedule` с FK на `Account`.
- При импорте платежа на loan — lookup строки графика по дате, подстановка split-дефолтов.
- После ручной правки юзером — **замена** строки графика фактическими значениями (§9.4, «эволюция графика»).

**Приоритет средний.** Без этого split-форма работает, но дефолты пустые.

## 10. Форма досрочного погашения

**Спек:** §9.5.

**Что нужно.**
- В split-форме (поверх #8, #9) — чекбокс «досрочное погашение».
- При активации — обязательный радиобаттон типа досрочного: «сокращение срока» / «сокращение ежемесячного платежа».

## 11. Warning при загрузке main без кредитного счёта

**Спек:** §9.8.

**Что нужно.**
- При анализе preview импорт-сессии — проверка: есть ли в строках переводы на счета типов `loan` / `credit_card` / `installment_card`.
- Если есть, и выписка с этого кредитного счёта не присутствует в текущей сессии (и не была загружена за перекрывающий период), — сессионный warning в UI.
- Текст: «ты загружаешь платёж на кредитный продукт [имя], но не загрузил выписку с него самого. Проценты не попадут в аналитику. Хочешь подгрузить сейчас или продолжить?»
- Не блокирует коммит (мягкое правило).

**Зависит от.** #5 (чтобы знать account_type).

## 12. Фоновая чистка inactive/broken правил

**Спек:** §11.3.

**Что нужно.**
- Celery-джоб, который раз в N дней проходит по правилам в статусе `inactive` (rejections > confirms) и по `broken` (swallow-результат §11.1).
- Для правил в `inactive` более M дней без реактивации → архивация (перенос в отдельную таблицу `archived_rules` или пометка флагом).
- Алерт админу если массовая архивация (более X% правил за период) — сигнал что что-то сломалось в пайплайне.

**Приоритет низкий.** Не блокирует ничего, но предотвращает деградацию БД со временем.

## 13. Активная перепривязка bindings при merge контрагентов

**Спек:** §11.1.

**Контекст.** Сейчас при merge A→B в контрагентах `CounterpartyFingerprint` и `CounterpartyIdentifier` указывающие на A остаются «висящими» — лениво перепривязываются (или не перепривязываются) при следующем матче.

**Что нужно.**
- В транзакции merge-операции: после обновления ссылки в Transaction'ах → пройти по всем `CounterpartyFingerprint` / `CounterpartyIdentifier` с `counterparty_id = A.id` и обновить на `B.id`.
- Атомарно в той же транзакции — если падает, откат всего merge.
- Тест: после merge, новый импорт со старым fingerprint'ом корректно подставляет B.

**Приоритет средний.** Баг скорее на уровне консистентности, чем видимый пользователю.

## 14. Мета-правила как отдельный тип

**Спек:** §10.5.

**Контекст.** Сейчас решение «этот fingerprint = платёж по кредиту, открыть split-форму» смешано с обычной category rule логикой. Нужно выделить.

**Что нужно.**
- Новая таблица `import_meta_rule` (или расширение существующей rule-таблицы с discriminator-полем).
- Типы мета-правил: пока один — `credit_payment_trigger`. Может добавиться в будущем (например, «этот fingerprint = refund, открыть pairing-форму»).
- Отдельные счётчики confirms/rejections для мета-правил.
- При матчинге — мета-правила применяются ДО category rules; если мета-правило сработало и определило что нужен split — category rule не применяется к этой исходной строке (применится к двум производным транзакциям после split'а, если применимо).

**Приоритет средний.** Сейчас split работает в лоб через hardcode; нормальная абстракция появится когда будет второй подтип мета-правил.
