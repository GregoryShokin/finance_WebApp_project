# План реализации И-08 (модератор импорта)

Пошаговая декомпозиция И-08 в инженерные задачи. Все дизайн-вопросы (Q1–Q7) закрыты — см. [[Идеи для импорта#И-08]]. Этот план — про **как делать**, не про **что делать**.

## Принципы исполнения

- **Итеративность.** Каждая фаза даёт работающий, развёрнутый инкремент. Не держать ветку месяцами — мерджить по фазам.
- **Обратная совместимость в пределах фазы.** Старые импорты не ломаются, пока не переехали на новую логику.
- **Retro-инвариант** (из [[../../../memory/project_import_moderator|project memory]]): прошлые транзакции не переклассифицируются никогда. Любая новая логика — только на новые импорты.
- **Без LLM должно работать.** Детерминированный слой (Phases 1–3) — самостоятельная ценность. Phase 4 (LLM) — усиление, не обязательная зависимость.
- **Покрытие тестами на реальных данных.** У пользователя есть живые выписки (Яндекс Сплит, Т-Банк с договорами, кредитки) — они и есть golden dataset.

## Общая последовательность

```
Phase 0 (чистка)
   ↓
Phase 1 (нормализация)   Phase 2 (модель правил)    ← можно параллельно
   ↓                            ↓
   └────────────┬───────────────┘
                ↓
        Phase 3 (кластеры + parked)
                ↓
        Phase 4 (LLM-модератор)
                ↓
        Phase 5 (UI wizard)
                ↓
        Phase 6 (калибровка, метрики)
                ↓
        Phase 7 (уборка legacy)
```

## Phase 1. Инфраструктура нормализации (Q1)

**Цель:** каждая строка импорта после нормализации имеет структурированные идентификаторы и текстовый `skeleton` для fingerprint-группировки.

### Задачи

- [x] **1.1** Модуль `app/services/import_normalizer_v2.py` (или рефактор существующего). Функции:
  - `extract_tokens(description) -> ExtractedTokens` — regex вырывает phone, contract, iban, card, person_name, amounts, dates.
  - `normalize_skeleton(description, extracted) -> str` — lowercase, удаление пунктуации/стоп-слов, схлопывание пробелов, замена вытащенных токенов плейсхолдерами в тексте.
  - `fingerprint(bank, account_id, direction, skeleton, contract) -> str` — стабильный хеш.
- [x] **1.2** Расширить JSON-структуру `ImportRow.normalized_data_json` новыми полями: `skeleton`, `tokens.{phone, contract, iban, card, person_name, counterparty_org}`, `fingerprint`. Схема — в [[../../14-Specifications/Import-Normalized-Data-Schema|отдельная спека]] (создать при необходимости).
- [x] **1.3** Интеграция в `ImportService`: после существующего normalizer вызывается новый `normalizer_v2`, результат пишется в `normalized_data_json`.
- [ ] **1.4** Тесты на golden dataset: каждый fixture-файл → ожидаемые extracted tokens + skeleton. Точные expected-значения пишем руками на 10–20 строках каждого банка.

**Acceptance:** прогон нормализатора на всех fixture'ах выдаёт корректные tokens и устойчивый skeleton. Две строки с одним и тем же номером договора получают одинаковый fingerprint (если остальные параметры совпадают), с разными — разный.

**Риски:** регулярки для ФИО — особенно хрупкие (русский язык, сокращения, инициалы). Стартовать с широкой регулярки, потом сужать по ошибкам на golden dataset.

## Phase 2. Модель правил со strength (Q3, Q4)

**Цель:** `TransactionCategoryRule` умеет confirms/rejections/scope/is_active; миграция существующих данных прошла с порогом `hit_count ≥ 5`.

  - Data migration: для всех существующих записей — `scope='legacy_pattern'`, `is_active = (hit_count >= 5)`, остальные поля null.
  - Переименовать `hit_count` в `confirms` (keep old as computed alias, если в коде много ссылок — решить при написании).
- [ ] **2.2** Обновить `TransactionCategoryRule` модель, `TransactionCategoryRuleRepository`.
- [ ] **2.3** Новый сервис `rule_strength_service.py`:
  - `on_rule_applied_and_confirmed(rule_id)` — `confirms++`, при пересечении порогов меняет `is_active` и `scope`.
  - `on_rule_applied_and_rejected(rule_id)` — `rejections++`, при пересечении порогов деактивирует.
  - Пороги вынесены в `settings` (`RULE_ACTIVATE_CONFIRMS=2`, `RULE_GENERALIZE_CONFIRMS=3`, `RULE_DEACTIVATE_REJECTIONS=3`, `RULE_ERROR_RATIO_CAP=0.3`).
- [ ] **2.4** Интеграция точек учёта confirm/reject:
  - Коммит импорта без правки → confirm для применённого правила.
  - Правка категории/типа/счёта в UI перед коммитом → reject.
  - Правка закоммиченной транзакции в окне ≤7 дней → reject.
- [ ] **2.5** Endpoint `GET /rules` с фильтрами по scope/is_active. Frontend страница «Мои правила» — отдельная задача в Phase 5, пока только API.

**Acceptance:** миграция применяется без потери данных; 48% правил активны; unit-тесты на переходы между состояниями strength проходят; учёт confirm/reject работает на E2E-сценарии «импорт → коммит → правка».

**Риски:** переименование `hit_count` → `confirms` может задеть много мест. Альтернатива — оставить `hit_count` как legacy-поле, синхронизировать с `confirms` в одну сторону.

## Phase 3. Кластеризация + parked-контейнер

**Цель:** импорт-сессия группирует строки в кластеры по fingerprint; неуверенные строки можно отложить в `parked`.

- [ ] **3.4** Расчёт cluster confidence: максимум из source-признаков (применённое правило, детерминированный rule-hit по ключевым словам, совпадение пары возвратов, совпадение transfer-кандидата).

**Риски:** многострочный перевод из Яндекс Сплит (17569 + 431 = 18000) — это отдельный матчер, не помещается в стандартную схему fingerprint. Либо доработка transfer_matcher для суммарных пар, либо отдельный `split_transfer_matcher`. В первой итерации — оставить нерешённым, записать в Phase 6 как расширение.

## Phase 4. LLM-модератор (Q5, Q6, Q7)

**Цель:** неуверенные кластеры (confidence < 0.7) обогащаются гипотезой LLM; работает асинхронно через Celery с per-cluster гранулярностью.

### Задачи

- [ ] **4.1** Рефакторинг `app/services/llm_service.py` в структуру `app/services/llm/`:
  - `llm/base.py` — ABC `LLMProvider` с методом `generate_structured(system, user, schema, *, max_tokens, cache_key) -> BaseModel | None`.
  - `llm/anthropic_provider.py` — текущая логика за вычетом бизнес-уровня.
  - `llm/registry.py` — `get_provider()` по `settings.LLM_PROVIDER`.
  - Наверху остаётся `llm_service.py` как domain-layer, про Anthropic не знает.
- [ ] **4.2** Новый сервис `import_moderator_service.py` с методом `moderate_cluster(cluster) -> ClusterHypothesis`. Строит промпт:
  - System: роль + формат вывода + правила confidence.
  - User: анонимизированные примеры строк (идентификаторы заменены плейсхолдерами), контекст счёта, список категорий пользователя, релевантные активные правила.
  - Output schema (Pydantic): `operation_type`, `direction`, `predicted_category_id`, `confidence`, `reasoning`, `follow_up_question` (если LLM сам неуверен — сформулирует вопрос пользователю).
- [ ] **4.3** Per-cluster хранение гипотезы: поле `moderation_status` + `llm_hypothesis` в runtime-кластере (Phase 3) или в `ImportRow.normalized_data_json` (один на первую строку кластера). Решение — при реализации.
- [ ] **4.4** Celery-task `moderate_import_session(session_id)`:
  - Проходит по кластерам с `confidence < 0.7`.
  - Для каждого вызывает `moderator.moderate_cluster(...)` и пишет результат.
  - Статус сессии: `moderation_status` = `pending` → `running` → `ready` / `failed`.
- [ ] **4.5** Endpoint `GET /imports/{id}/moderation-status`: `{status, progress, clusters: [{cluster_id, status, hypothesis?}]}`. Per-cluster гранулярность — уже заложена (из решения Q6).
- [ ] **4.6** Fallback-режим: LLM недоступен / сбой / `is_enabled=False` → кластеры получают `moderation_status='skipped'`, wizard всё равно открывается, гипотез нет — ручной разбор.
- [ ] **4.7** Анонимизация в промпте: использовать `skeleton` из Phase 1 + явные плейсхолдеры `<PHONE>`, `<CONTRACT>`, `<IBAN>`, `<CARD>`, `<PERSON_NAME>`. Юнит-тест: в payload, улетающем в LLM, нет сырых идентификаторов.
- [ ] **4.8** Prompt caching: `cache_control: ephemeral` на system-промпте + списке категорий (стабильная часть между сессиями пользователя).

**Acceptance:** Celery-task на реальной сессии с 20 кластерами завершается за < 30 секунд; гипотезы сохраняются per-cluster; при выключении LLM (`LLM_CLASSIFICATION_ENABLED=false`) импорт работает, кластеры без гипотез; в payload'е нет идентификаторов (проверяется тестом).

**Риски:** стоимость на пользователя. Надо в Phase 6 замерить: средний расход токенов за сессию. Заложить бюджетный gauge в метриках.

## Риски всего плана

- **Объём.** 7 фаз — это много. Может растянуться на месяц+. Держать фазы мерджабельными по одной.
- **Сдвиг скоупа.** По ходу всплывёт желание делать «ещё и это». Любое такое — записывать в И-09…И-XX, не размазывать по текущему плану.
- **LLM-стоимость.** Пока пользователь один (ты) — неважно. При масштабе — сможет стать узким местом. Phase 6 мониторит.
- **Голос LLM.** Промпт нужно будет доводить: чтобы LLM не был слишком самоуверенным (именно то, от чего лечимся). Тестировать на фикс-наборе на склонность к галлюцинациям.
