## И-08. Модератор импорта — диалоговый review на кластерах

**Проблема:** пересекает [[Импорт — проблемы и решения#P-01. Нестандартизированные выписки банков|P-01]], [[Импорт — проблемы и решения#P-03. Возвраты не определяются из контекста|P-03]], [[Импорт — проблемы и решения#P-05. Номера договоров игнорируются при внутрибанковских переводах|P-05]], [[Импорт — проблемы и решения#P-07. Ложная уверенность системы портит аналитику|P-07]].

**Статус:** accepted (концепция зафиксирована 2026-04-21, детали реализации — open).

### Суть

Перед коммитом импорт-сессии пользователь проходит **review-диалог на кластерах**: система группирует похожие строки и задаёт один вопрос на группу, а не на каждую транзакцию. Без отдельного UI-режима — встроено в текущий import-wizard как стадия между preview и commit.

Философия: **не угадывать молча, а честно спросить один раз и запомнить**. Согласуется с принципом «наставник, не трекер».

### Архитектура: гибрид (принято 2026-04-21)

Модератор — **двухслойная система**: детерминированный слой + LLM-слой. Чистый rules-движок не понимает контекст, чистый LLM стоит денег и недетерминирован; гибрид берёт лучшее из обоих.

**Детерминированный слой** (всегда работает, без внешних зависимостей):
- Извлечение классов токенов через regex (Q1): договор, телефон, IBAN, сумма, дата.
- Нормализация описания → `skeleton` (Q1).
- Кластеризация по fingerprint = банк + счёт + направление + `skeleton` (+ `contract`).
- Поиск парных возвратов в окне ±14 дней с плавающей confidence (Q2).
- Применение `TransactionCategoryRule` со strength — персональная память пользователя.
- Расчёт confidence кластера.

**LLM-слой** подключается **только там, где детерминированный не справился** (confidence < 0.7, неизвестный паттерн, неоднозначность transfer / refund / regular):
- Вход: 3–5 примеров строк из кластера + контекст счёта (банк, тип дебет/кредит) + список категорий + релевантные правила пользователя.
- Выход: структурированный JSON `{operation_type, direction, predicted_category_id, confidence, reasoning}`.
- **LLM не пишет в базу.** Её ответ — гипотеза, показывается пользователю в карточке кластера; пользователь подтверждает или отклоняет, это становится confirm/reject для правила.
- Если LLM недоступен (сбой API, нет ключа, оффлайн) — система работает на одном детерминированном слое, просто больше строк уходит в warning / ручной разбор.

**Провайдер модели:**
- Сейчас: Claude Haiku через Anthropic API ([app/services/llm_service.py](../../../app/services/llm_service.py) — уже интегрирован для классификации категорий, ключ зашит в проект, prompt caching включён).
- Будущее: локальная open-source модель (Qwen / Llama / Gemma через ollama или эквивалент) для удешевления и приватности. Переход — через адаптер под тем же интерфейсом, без перелома логики.
- **Политика пользователя:** пользователь **никогда** не подключает свой API-ключ и не платит за LLM напрямую. Стоимость на операторе приложения, отсюда и план на open-source.

**Почему гибрид экономит деньги:**
- LLM вызывается на 20–30% кластеров (только неуверенные). Остальное обрабатывается детерминированно бесплатно.
- Со временем `TransactionCategoryRule` накапливается → LLM вызывается всё реже → стоимость падает.
- Переход на локальную модель — нулевая стоимость за запрос, только фиксированная стоимость инфры.

### Что решает

- **P-07 (ложная уверенность):** в серой зоне confidence система не угадывает, а спрашивает. Пользователь не перестаёт доверять — потому что система не лжёт о том, что знает.
- **P-01 (неявное направление):** вопрос задаётся на реальном примере из данных пользователя, не на абстрактных правилах.
- **P-03 (возвраты):** кандидатные пары (одинаковая сумма, противоположные направления, близкая дата) показываются одной карточкой «это возврат?».
- **P-05 (номера договоров):** идентификатор становится частью fingerprint'а кластера — похожие «Внутрибанковский перевод» с разными договорами становятся разными кластерами.

### Ключевые механизмы

**1. Fingerprint строки = банк + счёт + нормализованное описание + направление.**
Нормализация: вырезать даты, суммы, одноразовые токены, оставить скелет. Это и есть «сигнал vs мусор» — механически, без магии.

**2. Регулярки по классам токенов.**
Номер договора (`№\d{6,}`, `contract_id=…`), телефон (`+7\d{10}`), IBAN/счёт (20 цифр), ФИО контрагента — каждый класс в отдельное поле `normalized_data`, не склеивается в общее описание.

**3. Частотный анализ внутри банка.**
Токен, встречающийся в N разных транзакциях с РАЗНЫМ окружением — структурный тег (шум для группировки). Токен, встречающийся редко — специфика (сигнал).

**4. Confidence-gate на кластерах.**
- `≥ 0.9` — проходит молча, помечается «авто», видно в preview.
- `0.7–0.9` — «подтвердить одним кликом».
- `< 0.7` — обязательный вопрос модератора.

**5. Правила с «силой» (strength).**
- Создано из 1 примера → `strength=1`, применяется только к ТОЧНОМУ совпадению признака (этот конкретный номер телефона → жильё).
- Обобщение включается, только когда подтверждено ≥2–3 разных примера.
- Никакое правило не применяется **кросс-банком**. Scope по умолчанию — банк+счёт.

Это лечит корень болезни «запомнила номер как жильё → все неизвестные номера стали жильём».

**6. Cold start vs ongoing — разные режимы.**
- **Первый импорт банка:** много вопросов, строим словарь паттернов.
- **Повторный импорт:** вопросы только по новым/неоднозначным кластерам. Известные паттерны — молча через правила.

Без этой развилки модератор превратится в «те же вопросы каждый раз».

### Парковочный контейнер

«Пропустить, потом разберусь» → строки уходят в **глобальный** контейнер `parked` (across sessions, не привязан к конкретной импорт-сессии).

Жёстко исключён из всей аналитики: Поток, FI-score, Капитал, DTI, Health-метрики. Отдельная страница «Недоразобранное» с счётчиком на главной.

Статус отдельный от `warning` (warning — готово к коммиту с пометкой; parked — сознательно отложено).

### Retro-политика

**Прошлые транзакции не переклассифицируются никогда.**
- Ужесточение правил → применяется только к новым импортам.
- Изменение ответа пользователя → применяется только к новым строкам.

Причина: дрейф прошлых месяцев подрывает доверие к аналитике сильнее, чем старая ошибка. Пользователь открывает прошлый месяц и видит те же цифры, что и вчера — это инвариант.

### UI-управление scope правила

В момент подтверждения кластера — **без выбора scope**, по умолчанию «точный признак, этот банк+счёт».

Расширение scope (обобщить на банк / на все счета / добавить синонимы) — только через отдельную страницу **«Мои правила»**. Не перегружает момент подтверждения.

### Голос

Текстовые вопросы, не чат. Кластер → карточка → один клик-ответ. Голос нейтральный «системный», не наставнический (школа — отдельный канал, здесь нужна сухость и скорость).

### Плюсы

- Прямо решает P-01, P-03, P-05, P-07 одним механизмом.
- Использует уже существующие сущности (`TransactionCategoryRule`, `Counterparty`, `ImportRow.status`).
- Обучается на пользователе — точность растёт со временем без ML.
- Строит персональный словарь паттернов по банкам.

### Минусы / риски

- **Инфраструктура нормализации** — нетривиальна. Регулярки по токенам, fingerprint, частотный анализ — всё это надо написать и поддерживать.
- **Первый импорт тяжелее.** Пользователь получит много вопросов сразу. Лечится: агрессивная группировка + защитный счётчик «осталось N кластеров».
- **Правила со strength — новая модель.** Текущие `TransactionCategoryRule` этого не поддерживают, нужна миграция или расширение.
- **Парковочный контейнер** — новая сущность, UI, фильтрация в аналитике. Не забыть исключения во всех агрегациях.

### Открытые технические вопросы

- ~~Алгоритм нормализации описания~~ → **решено 2026-04-21: гибрид**. Regex вырезают классы токенов (дата, сумма, договор, телефон, IBAN, номер счёта) в отдельные поля `normalized_data`. Остаток проходит простую текстовую нормализацию (lowercase, схлопывание пробелов, удаление пунктуации) и становится `skeleton`. Fingerprint кластера = банк + счёт + направление + `skeleton` (+ `contract`, если есть). Частотный фильтр — отложено, добавим при необходимости.
- ~~Окно парных возвратов~~ → **решено 2026-04-21: плавающая confidence в окне ±14 дней**. 0–2 дня → 0.95 (молча, видно в preview), 3–7 дней → 0.8 (подтверждение одним кликом), 8–14 дней → 0.6 (обязательный вопрос модератора). Старше 14 дней — автоматическая пара не строится, только ручное сведение в wizard. Повышающие сигналы (независимо от даты): совпадающий контрагент/магазин, ключевые слова `возврат / refund / отмена / reversal`, совпадение `source_reference`. Экономическая трактовка: возврат уменьшает расход того же месяца — логика уже работает, отдельно не реализуем.
- ~~Strength правила~~ → **решено 2026-04-21: вариант B (confirms + rejections)**. Поля: `confirms: int`, `rejections: int`, `scope: enum('exact','bank','global')`, `is_active: bool`. Пороги (стартовые, могут калиброваться): применяется автоматически при `confirms ≥ 2` и `rejections = 0`; обобщается при `confirms ≥ 3` и доле ошибок `≤ 10%`; деактивируется при `rejections ≥ 3` или доле ошибок `> 30%`. Деактивация обратима — правило не удаляется, видно в «Мои правила» с меткой «отключено системой». Confirm = коммит без правки предложения / подтверждение в модераторе. Reject = правка категории/типа/счёта в UI импорта / отказ в модераторе / правка закоммиченной транзакции в окне ≤7 дней.
- ~~Миграция существующих `TransactionCategoryRule`~~ → **решено 2026-04-21: вариант D (миграция с порогом доверия по `hit_count`).** Alembic-миграция аддитивно добавляет поля: `rejections: int = 0`, `scope: enum`, `is_active: bool`. У существующих записей: `confirms = hit_count` (старое поле `hit_count` оставляем как alias/backref для обратной совместимости или сносим в той же миграции), `scope = 'legacy_pattern'` (новый scope — описание-паттерн без привязки к банку), `is_active = True` при `hit_count ≥ 5`, иначе `False`. Порог откалиброван по замеру БД 2026-04-21 (226 правил, 1 пользователь): отсекает 52% правил (включая подозрительный пик на `hit=3-4` — 82 правила, вероятный источник P-05), оставляет активными 108 правил (48%) с `hit_count ≥ 5` — зарплата, постоянные платежи, стабильные категории. При следующем успешном применении через модератор правило со scope=`legacy_pattern` конвертируется в `scope='bank'` с привязкой к банку подтверждения — техдолг временный, самосамо рассасывается. Старые мусорные rules (типа «Внутрибанковский перевод → Доставка еды» из P-05): низкий hit_count → автоматически неактивны; высокий → покажутся в модераторе и быстро деактивируются через rejections.
- ~~Что LLM видит~~ → **решено 2026-04-21: refined C (частичная анонимизация).** Идентификаторы (телефон, номер договора, IBAN, номер карты, ФИО физлица) извлекаются в структурированные поля `normalized_data` и живут в детерминированном слое — там они работают интенсивнее, чем в LLM (fingerprint-группировка, rule matching, counterparty-связи, парные возвраты по `source_reference`). В LLM-промпт уходит только семантический остаток: названия компаний/магазинов, ключевые слова, суммы, даты, банк, направление. Приватные идентификаторы заменяются плейсхолдерами `<PHONE>`, `<CONTRACT>`, `<IBAN>`, `<CARD>`, `<PERSON_NAME>` (тип сохраняется — это сам по себе сигнал: «перевод физлицу» vs «оплата компании»). Хранение идентификаторов на старте — открытый вид (для отображения пользователю); хеширование добавим при переходе в SaaS/multi-tenant режим.
- ~~Когда LLM вызывается~~ → **решено 2026-04-21: вариант B (асинхронно через Celery) с архитектурной готовностью к C.** После preview запускается Celery-таск `moderate_import_session(session_id)`. Frontend опрашивает `/imports/{id}/moderation-status` (polling ~2с) до `ready`, затем открывает wizard-стадию review. При сбое LLM — статус `failed`, wizard всё равно открывается, просто без гипотез (fallback на ручной разбор). **Обязательное условие реализации:** гранулярность на уровне кластера, не сессии. У каждого кластера свой `moderation_status` и `llm_hypothesis`, Celery-таск обновляет их по одному. В режиме B frontend ждёт `all-ready`, но сами данные уже обновляются per-cluster — апгрейд до C — это переключение одной условной проверки на фронте, без новых эндпоинтов и миграций данных. **Триггеры апгрейда до C:** (а) P95 времени модерации регулярно > 20 сек на реальных выписках; (б) явные жалобы пользователей на ожидание; (в) выход из закрытой беты с ростом нагрузки.
- ~~Абстракция провайдера LLM~~ → **решено 2026-04-21: вариант B (порт + адаптер).** Новая папка `app/services/llm/`: `base.py` (`LLMProvider` ABC, `LLMRequest`/`LLMResponse` dataclasses), `anthropic_provider.py` (нынешний код `LLMService` за вычетом бизнес-логики), `registry.py` (`get_provider() → LLMProvider` по `settings.LLM_PROVIDER`). Единственный метод интерфейса — `generate_structured(system, user, schema, *, max_tokens, cache_key) → BaseModel | None`; покрывает и Anthropic-нативный structured output, и open-source text-completion + postprocess + Pydantic-validate. `LLMService` наверху остаётся бизнес-слоем (доменные задачи: «классифицируй категорию», «угадай operation_type», «сгруппируй кластер»), про Anthropic не знает. При появлении open-source провайдера — новый файл `ollama_provider.py`, флаг в `.env`. LiteLLM/LangChain не используем: потеряем нативный Anthropic structured output и `cache_control: ephemeral`, выиграем мало.
