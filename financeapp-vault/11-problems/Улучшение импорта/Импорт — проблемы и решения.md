# Импорт — проблемы и решения

Живой реестр проблем импорта банковских выписок и решений по ним. Идеи и гипотезы — в соседнем файле [[Идеи для импорта]].

## Контекст

Цель: **точность автоматического распознавания = 100%** (или явный `warning` со снижением confidence). Ошибочная уверенность хуже, чем честное «не знаю» — пользователь перестаёт проверять и аналитика портится.

Принцип: лучше пометить строку `warning` и попросить подтверждение, чем тихо присвоить неверную категорию.

## Проблемы

### P-01. Нестандартизированные выписки банков

**Описание.** Банки кодируют направление транзакции по-разному:
- знаки `+/-` (например);
- отдельные колонки `Дебет/Кредит`;
- **неявно** — только по контексту описания (Яндекс Банк).

Пример Яндекс Банка: направление нужно выводить из описания.
- `Отмена ...` → поступление (reversal).
- `Погашение задолженности` на **кредитной карте** → поступление; на **дебетовой** → списание.

**Статус:** **closed** (2026-04-28). Для всех активных банков пользователя (Сбер, Т-Банк, Альфа, Озон, Яндекс) проблема закрыта детерминированной классификацией:
- Generic prefix-matching по raw_type («приход/зачис/пополн» → income; «расход/спис/оплата» → expense) в `ImportNormalizer._resolve_direction`.
- Account-type-aware hinting в `ImportClusterService._apply_account_context_hints`: для credit_card строки «погашение» → transfer income, «отмена» → refund.
- Bank mechanics (`bank_mechanics_service.py`) для Яндекс Сплит/Дебет: «погашение основного долга / процентов / по договору» классифицируется по комбинации account_type × skeleton × direction (см. P-02).

Открытым остаётся только теоретический случай добавления нового нестандартизованного банка, у которого нет ни знака суммы, ни raw_type, ни поддержанных bank-mechanics правил. Решение для будущих банков — обучение системы на реальных выписках (расширение bank_mechanics_service.py + account_context hints) по мере появления у пользователей.

**Related:** [[Идеи для импорта#И-01. Контекст-распознавание направления по описанию]]

### P-02. Разбивка одного перевода на несколько транзакций (Яндекс Сплит и т.п.)

**Описание.** Один внутрибанковский перевод приходит несколькими строками:
- перевёл 18 000 ₽ с дебетового счёта на Сплит → в выписке 17 569 ₽ + 431 ₽.
- матчер переводов ищет ровную сумму и не срабатывает.

**Статус:** **closed** (2026-04-27). Решение — детерминированная классификация по типу счёта + описанию + номеру договора (без multi-row matching). Три изменения:

1. **bank_mechanics_service.py** — новые income-direction правила для Яндекс Сплит (credit/installment_card):
   - «погашение основного долга» income → `suggest_exclude=True` (покрыто phantom-доходом от Дебет-перевода; если коммитить — баланс Сплит задвоится)
   - «погашение процентов» income → `operation_type='regular'`, category=«Проценты по кредитам» (расход пользователя за пользование кредитом)
   - Деbit «погашение / оплата по договору» expense → `operation_type='transfer'`, `suggest_target_by_contract=True` + резолвинг `target_account_id` из `tokens.contract` через `account_repo.find_by_contract_number`
   - Confidence boosts подняты с 0.05–0.08 до 0.12–0.18

2. **import_cluster_service.py** — `bank_mechanics.apply()` получает `identifier_key`/`identifier_value`; `Cluster` содержит `bank_mechanics_suggest_exclude` и `bank_mechanics_resolved_target_account_id`

3. **import_service.py** — новый `_apply_bank_mechanics_yandex_post_process()`: после refund-passes обходит строки сессии, применяет auto-exclude для Сплит-phantom-строк и проставляет `target_account_id` для Дебет-transfer-строк.

**Схема балансов:**
- Дебет −38 transfer → phantom +38 на Сплит; Дебет −162,61 transfer → phantom +162,61 на Сплит
- Сплит +200,61 «погашение процентов» → expense −200,61 (Проценты по кредитам)
- Net Сплит: +38+162,61−200,61 = 0 ✓ (проценты пришли и сразу ушли банку)

Регрессия: [tests/test_p02_yandex_split_mechanics.py](tests/test_p02_yandex_split_mechanics.py) — 5 тестов.

**Related:** [[Идеи для импорта#И-02. Многострочные переводы (сплит-суммы)]]

### P-03. Возвраты не определяются из контекста

**Описание.** Возврат (refund) распознаётся по ключевым словам, но не использует контекст пары:
- две строки с одинаковой суммой и противоположными направлениями за короткое время — это возврат, а не пара «расход + доход».

**Статус:** **closed** (2026-04-27). В пайплайне работают два дополняющих механизма:
1. [`RefundMatcherService`](app/services/refund_matcher_service.py) находит пары `expense ↔ income` внутри сессии: одинаковая сумма, противоположные направления, окно ±14 дней. Confidence 0.95 при совпадении контракта/person_hash/counterparty_org или keyword+brand; 0.75 при совпадении только brand; ниже `MIN_CONFIDENCE=0.60` (только keyword без brand или только amount-window) — не эмитим, чтобы не клеить случайные совпадения сумм между разными мерчантами. Подключён в `build_preview` через `_apply_refund_matches` ([app/services/import_service.py:2254](app/services/import_service.py#L2254)).
2. `_apply_refund_cluster_overrides` ([app/services/import_service.py:2264](app/services/import_service.py#L2264)) для refund-кластеров (income со словами «возврат»/«отмена»/«refund») наследует counterparty + категорию из истории покупок у того же бренда, даже если pair не нашёлся.

Регрессия: [tests/test_p03_refund_pair_recognition.py](tests/test_p03_refund_pair_recognition.py) — покрывает brand-only match, keyword+brand match, amount-coincidence false-positive guard, окно > 14d.

**Related:** [[Идеи для импорта#И-03. Парное распознавание возвратов]]

### P-04. Кредитные операции (погашения, платежи, досрочки) не распознаются

**Описание.** Погашения кредита, плановые платежи, досрочные погашения не отличаются от обычных переводов/расходов. По CLAUDE.md кредитный платёж = **две транзакции** (проценты + тело), это усложняет автоматику.

**Статус:** **closed** (2026-04-27). Три изменения:

1. **Backend — keyword detection** ([app/services/import_service.py](app/services/import_service.py)): добавлен `_CREDIT_PAYMENT_KEYWORDS` (погашение кредита / оплата по кредиту / ежемесячный платёж по кредиту / …). В `build_preview` — если `operation_type='transfer'` + description/skeleton совпадает → `requires_credit_split=True`. Покрывает Т-Банк, Сбер и любой банк, у которого `_classify_yandex_raw_type` ещё не написан.

2. **Backend — loan target detection** ([app/services/import_service.py](app/services/import_service.py), `update_row`): когда юзер вручную ставит `target_account_id` на loan-тип счёта → `requires_credit_split=True`. Покрывает случай когда описание не содержит ключевых слов, но счёт говорит за себя.

3. **Frontend — split-форма открывается автоматически** ([frontend/components/import/import-moderation-panel.tsx](frontend/components/import/import-moderation-panel.tsx)): добавлено чтение `requires_credit_split` из `row.normalized_data`. Когда флаг выставлен + `operation_type='transfer'`, `mainOp` инициализируется как `'credit_operation'` / `creditKind='payment'` — форма с полями «Проценты» и «Тело долга» открывается без ручного переключения типа. Исправлены имена полей `credit_principal_amount` / `credit_interest_amount` (ранее читались как `principal_amount` / `interest_amount`).

Регрессия: [tests/test_p04_credit_payment_recognition.py](tests/test_p04_credit_payment_recognition.py) — 12 тестов (keyword detection × 9, loan target, non-loan guard, two-transaction split).

**Related:** [[Идеи для импорта#И-04. Распознавание кредитных операций]]

### P-05. Номера договоров игнорируются при внутрибанковских переводах

**Описание.** Т-Банк при каждом переводе указывает номер договора. Система их **не извлекает** и не использует.
Последствие: похожие строки «Внутрибанковский перевод» в тестах массово получают категорию **Доставка еды** (победивший majority-класс для шаблона без различающих признаков).

**Статус:** **closed** (2026-04-28). Все три аспекта проблемы закрыты:
1. **Извлечение контракта** — `extract_tokens()` в `import_normalizer_v2.py` через `CONTRACT_RX` извлекает номер договора в `tokens.contract`.
2. **Контракт в fingerprint** — `pick_transfer_identifier()` для transfer-like строк выбирает контракт как identifier (`kind="contract"`); `fingerprint()` включает его в сыром виде, благодаря чему переводы по разным договорам попадают в разные кластеры. СБП-переводы и обычные внутрибанковские переводы корректно расходятся по фингерпринтам.
3. **Контракт в transfer matcher** — `TransferMatcherService._score_pair` даёт +0.05 бонус за совпадение `contract_number` между сторонами пары.
4. **Защита transfer от перетирания категорией** — orphan transfer (transfer-like строка без найденной пары) демотируется в `regular` через `_escalate_orphan_transfers` (v1.9 от 2026-04-26), поэтому category-правила не применяются к строкам, которые остались transfer'ом. Дополнительно работают три барьера P-06.

Регрессия: `tests/test_import_normalizer_v2.py` (контракт в fingerprint), `tests/test_transfer_matcher_simple_keyword_filter.py` (transfer matching).

**Related:** [[Идеи для импорта#И-05. Извлечение номеров договоров]]

### P-06. Категория перетирает тип операции при коммите

**Описание.** Пользователь в UI явно выставляет `operation_type = transfer`, но при коммите транзакция получает категорию «Кафе и рестораны».

**Гипотеза.** В pipeline коммита правила категоризации применяются **после** пользовательских изменений и перезаписывают их. Нужно: если строка отредактирована вручную — правила не применять; transfer в принципе не должен иметь категории расхода.

**Статус:** **closed** (2026-04-27). В текущем коде три независимых барьера защищают user choice:
1. `ImportService._validate_manual_row` ([app/services/import_service.py:1540](app/services/import_service.py#L1540)) — при `operation_type='transfer'` обнуляет `category_id` в `normalized_data`.
2. `ImportService.build_preview` ([app/services/import_service.py:2083-2084](app/services/import_service.py#L2083-L2084)) — после resolve operation_type обнуляет category_id для transfer и `NON_ANALYTICS_OPERATION_TYPES`.
3. `TransactionService._validate_payload` ([app/services/transaction_service.py:670](app/services/transaction_service.py#L670)) — defense-in-depth: payload с `operation_type='transfer'` + `category_id` отбрасывается с ошибкой.

Регрессия: [tests/test_p06_user_transfer_choice_survives_commit.py](tests/test_p06_user_transfer_choice_survives_commit.py).

**Related:** [[Идеи для импорта#И-06. Приоритет пользовательских правок]]

### P-07. Ложная уверенность системы портит аналитику

**Описание.** Автораспознавание обычно право → пользователь перестаёт проверять → редкие ошибки проникают в данные → ломается аналитика (особенно FI-score, Поток).

**Целевое состояние.** Либо 100% точность, либо явный `warning` с понятным объяснением «почему я не уверен». Честный отказ лучше уверенной ошибки.

**Статус:** **partially closed** (2026-04-28). Базовая калибровка confidence реализована и работает корректно:
- Три trust-зоны (green ≥0.85 / yellow ≥0.65 / red <0.65) в `import_cluster_service.py`.
- Singleton penalty (-0.05) в формуле confidence.
- Identifier match factor (matched=1.0, unmatched=0.60).
- Error ratio (`confirms / (confirms + rejections)`).
- Auto-trust gate строгий: identifier-rule + ≥5 confirms + 0 rejections + matched identifier + confidence ≥0.99.
- Fallback-категория НЕ применяется: при отсутствии правила/паттерна/LLM `category_id` остаётся `None`, статус → warning с issue «Категория не определена — укажи вручную».

**Остаётся открытым (переформулировано) — слишком широкие generic-правила без identifier'а.**

Реальное наблюдение: система присваивает категории «Здоровье», «Подписки на сервисы» нетипичным операциям. Источник — пользовательские правила (`TransactionCategoryRule`) с широким скелетом и без `identifier_key`, которые накопили confirms и проходят rule-matching по любой строке с похожим скелетом. Это не проблема калибровки confidence — это проблема **скоупа правил**.

**Гипотеза решения.**
1. Ужесточить auto-trust gate: запретить молчаливое применение rule без `identifier_key` (т.е. generic skeleton-правил), даже при высоком confirm count. Такие правила должны давать категорию-предложение со статусом warning, а не ready.
2. Singleton + generic rule (без identifier'а) → forced warning независимо от confidence.
3. Detection «правило-захватчик»: если одно правило начинает матчиться к слишком разному набору контрагентов/банков, помечать его как подозрительное и предлагать ревизию.

## Решённые / закрытые

- **P-06** (2026-04-27) — категория не перетирает `operation_type=transfer` при коммите. Подтверждено тремя независимыми барьерами в коде + регрессионным тестом.
- **P-02** (2026-04-27) — Яндекс Сплит/Дебет: phantom-mirror строки auto-excluded, target_account резолвится по договору, проценты → расход.
- **P-04** (2026-04-27) — кредитные погашения распознаются и открывают split-форму автоматически.
- **P-03** (2026-04-27) — refund-пары распознаются по контексту (`RefundMatcherService` + `_apply_refund_cluster_overrides`). False-positive guard на совпадение сумм между разными мерчантами на месте.
- **P-01** (2026-04-28) — направление по контексту описания работает для всех активных банков (Сбер, Т-Банк, Альфа, Озон, Яндекс) через generic prefix-matching + account-type hinting + bank mechanics. Открытым остаётся только теоретический случай нового нестандартизованного банка — решение через постепенное обучение системы по мере появления выписок.
- **P-05** (2026-04-28) — номера договоров извлекаются (`CONTRACT_RX`), включаются в fingerprint в сыром виде для transfer-like строк, дают бонус в transfer matcher. Защита transfer от category-rule перетирания работает через orphan demotion (v1.9) + три барьера P-06.
