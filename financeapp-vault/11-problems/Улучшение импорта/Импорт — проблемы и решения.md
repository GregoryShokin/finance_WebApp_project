# Импорт — проблемы и решения

Живой реестр проблем импорта банковских выписок и решений по ним. Идеи и гипотезы — в соседнем файле [[Идеи для импорта]].

## Контекст

Цель: **точность автоматического распознавания = 100%** (или явный `warning` со снижением confidence). Ошибочная уверенность хуже, чем честное «не знаю» — пользователь перестаёт проверять и аналитика портится.

Принцип: лучше пометить строку `warning` и попросить подтверждение, чем тихо присвоить неверную категорию.

## Проблемы

### P-01. Нестандартизированные выписки банков

**Описание.** Банки кодируют направление транзакции по-разному:
- знаки `+/-` (например);
- отдельные колонки `Дебет/Кредит`;
- **неявно** — только по контексту описания (Яндекс Банк).

Пример Яндекс Банка: направление нужно выводить из описания.
- `Отмена ...` → поступление (reversal).
- `Погашение задолженности` на **кредитной карте** → поступление; на **дебетовой** → списание.

**Статус:** open.
**Related:** [[Идеи для импорта#И-01. Контекст-распознавание направления по описанию]]

---

### P-02. Разбивка одного перевода на несколько транзакций (Яндекс Сплит и т.п.)

**Описание.** Один внутрибанковский перевод приходит несколькими строками:
- перевёл 18 000 ₽ с дебетового счёта на Сплит → в выписке 17 569 ₽ + 431 ₽.
- матчер переводов ищет ровную сумму и не срабатывает.

**Гипотеза.** Допускать сумму как **сумму N строк** в пределах короткого временного окна, при совпадении контрагента/контракта.

**Статус:** open.
**Related:** [[Идеи для импорта#И-02. Многострочные переводы (сплит-суммы)]]

---

### P-03. Возвраты не определяются из контекста

**Описание.** Возврат (refund) распознаётся по ключевым словам, но не использует контекст пары:
- две строки с одинаковой суммой и противоположными направлениями за короткое время — это возврат, а не пара «расход + доход».

**Статус:** open.
**Related:** [[Идеи для импорта#И-03. Парное распознавание возвратов]]

---

### P-04. Кредитные операции (погашения, платежи, досрочки) не распознаются

**Описание.** Погашения кредита, плановые платежи, досрочные погашения не отличаются от обычных переводов/расходов. По CLAUDE.md кредитный платёж = **две транзакции** (проценты + тело), это усложняет автоматику.

**Статус:** open.
**Related:** [[Идеи для импорта#И-04. Распознавание кредитных операций]]

---

### P-05. Номера договоров игнорируются при внутрибанковских переводах

**Описание.** Т-Банк при каждом переводе указывает номер договора. Система их **не извлекает** и не использует.
Последствие: похожие строки «Внутрибанковский перевод» в тестах массово получают категорию **Доставка еды** (победивший majority-класс для шаблона без различающих признаков).

**Гипотеза.**
1. Экстрактор должен вытаскивать номер договора в отдельное поле.
2. Матчер переводов должен учитывать совпадение номера договора как сильный сигнал.
3. Правила категорий не должны срабатывать, если строка уже классифицирована как transfer.

**Статус:** open.
**Related:** [[Идеи для импорта#И-05. Извлечение номеров договоров]]

---

### P-06. Категория перетирает тип операции при коммите

**Описание.** Пользователь в UI явно выставляет `operation_type = transfer`, но при коммите транзакция получает категорию «Кафе и рестораны».

**Гипотеза.** В pipeline коммита правила категоризации применяются **после** пользовательских изменений и перезаписывают их. Нужно: если строка отредактирована вручную — правила не применять; transfer в принципе не должен иметь категории расхода.

**Статус:** open.
**Related:** [[Идеи для импорта#И-06. Приоритет пользовательских правок]]

---

### P-07. Ложная уверенность системы портит аналитику

**Описание.** Автораспознавание обычно право → пользователь перестаёт проверять → редкие ошибки проникают в данные → ломается аналитика (особенно FI-score, Поток).

**Целевое состояние.** Либо 100% точность, либо явный `warning` с понятным объяснением «почему я не уверен». Честный отказ лучше уверенной ошибки.

**Гипотеза.** Калибровка confidence + агрессивное понижение в `warning` при любой неоднозначности (конфликт правил, незнакомый паттерн, низкая похожесть на историю).

**Статус:** open.
**Related:** [[Идеи для импорта#И-07. Калибровка уверенности и honest-warning]]
