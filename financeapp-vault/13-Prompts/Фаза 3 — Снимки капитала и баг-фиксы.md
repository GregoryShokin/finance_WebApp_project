## БЛОК А. Снимки капитала (GAP #2)

### А.1 Модель и миграция

Создай модель `CapitalSnapshot` в `app/models/capital_snapshot.py`:

```python
class CapitalSnapshot(Base):
    __tablename__ = "capital_snapshots"

    id: Mapped[int] = mapped_column(primary_key=True, index=True)
    user_id: Mapped[int] = mapped_column(ForeignKey("users.id", ondelete="CASCADE"), nullable=False, index=True)

    # Период снимка — первое число месяца, за который фиксируем состояние
    snapshot_month: Mapped[date] = mapped_column(Date, nullable=False, index=True)

    # Компоненты на конец месяца
    liquid_amount: Mapped[Decimal] = mapped_column(Numeric(14, 2), nullable=False)    # regular + cash
    deposit_amount: Mapped[Decimal] = mapped_column(Numeric(14, 2), nullable=False)   # все deposit
    credit_debt: Mapped[Decimal] = mapped_column(Numeric(14, 2), nullable=False)      # сумма долгов по credit/credit_card/installment_card
    capital: Mapped[Decimal] = mapped_column(Numeric(14, 2), nullable=False)          # liquid + deposit − credit_debt
    net_capital: Mapped[Decimal] | None = mapped_column(Numeric(14, 2), nullable=True)  # capital + real_assets (для страницы Assets)

    created_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), server_default=func.now(), nullable=False)

    __table_args__ = (
        UniqueConstraint("user_id", "snapshot_month", name="uq_capital_snapshots_user_month"),
    )

    user = relationship("User")
```

Миграция `alembic/versions/0038_add_capital_snapshots.py` — стандартная: таблица, индексы, unique constraint.

### А.3 Celery-джоб ежемесячного снимка

В `app/jobs/` создай файл `monthly_capital_snapshot.py`:

```python
from celery.schedules import crontab

from app.core.celery_app import celery_app
from app.core.db import SessionLocal
from app.services.capital_snapshot_service import CapitalSnapshotService
from app.models.user import User


@celery_app.task(name="monthly_capital_snapshot")
def run_monthly_capital_snapshot() -> dict:
    """
    Запускается 1-го числа каждого месяца в 03:00 UTC.
    Фиксирует снимок за ЗАВЕРШЁННЫЙ прошлый месяц для всех пользователей.
    """
    db = SessionLocal()
    try:
        service = CapitalSnapshotService(db)
        # Прошлый месяц
        today = date.today()
        prev = date(today.year, today.month, 1) - timedelta(days=1)
        snapshot_month = date(prev.year, prev.month, 1)

        user_ids = [u.id for u in db.query(User.id).all()]
        created = 0
        for user_id in user_ids:
            try:
                service.create_snapshot_for_month(user_id, snapshot_month)
                created += 1
            except Exception as exc:
                logger.exception("Snapshot failed for user %s: %s", user_id, exc)

        db.commit()
        return {"month": snapshot_month.isoformat(), "users": len(user_ids), "created": created}
    finally:
        db.close()
```

Зарегистрируй периодическое расписание в `app/core/celery_app.py`:

```python
celery_app.conf.beat_schedule = {
    "monthly-capital-snapshot": {
        "task": "monthly_capital_snapshot",
        "schedule": crontab(day_of_month="1", hour="3", minute="0"),
    },
}
```

Если в `docker-compose.yml` ещё нет celery-beat сервиса — добавь его (рядом с worker, с командой `celery -A app.core.celery_app beat --loglevel=info`).

### А.4 Ретро-скрипт для исторических снимков

В `_calc_fi_score`:
- `capital_score` на базе trend_3m: > 0 → 10, трактовать нулевой тренд = 5.0, отрицательный → 0 (пропорционально)

## БЛОК Б. Фикс DTI: включить тело кредитных платежей

### Б.1 Проблема

В `app/services/metrics_service.py` метод `calculate_dti` сейчас считает числитель только по процентам:

```python
all_credit_txns = self.db.query(Transaction).filter(
    Transaction.type == "expense",
    Transaction.operation_type == "regular",   # только проценты!
    Transaction.credit_account_id.isnot(None),
).all()
```

После Фазы 1 тело кредитного платежа идёт как `operation_type='transfer'` с `target_account_id=credit`. В числитель DTI оно должно попадать тоже, потому что по спецификации DTI = `Σ(тело + проценты) / рег. доход × 100%`.

**Симптом:** DTI занижен на сумму тела. Для «Первого шага» (ипотека + потребкредит, тело ~18к/мес) реальный DTI 42% показывается как ~35%, зона смещается с «опасной» на «допустимую».

### Б.2 Фикс в metrics_service.py

Изменить сбор транзакций платежей в `calculate_dti`:

```python
# Было: только проценты
all_credit_txns = self.db.query(Transaction).filter(
    Transaction.type == "expense",
    Transaction.operation_type == "regular",
    Transaction.credit_account_id.isnot(None),
).all()

# Стало: проценты + тело
interest_txns = self.db.query(Transaction).filter(
    Transaction.user_id == user_id,
    Transaction.type == "expense",
    Transaction.operation_type == "regular",
    Transaction.credit_account_id.isnot(None),
).all()

body_txns = self.db.query(Transaction).filter(
    Transaction.user_id == user_id,
    Transaction.operation_type == "transfer",
    Transaction.target_account_id.in_(credit_account_ids),
).all()

all_credit_txns = interest_txns + body_txns
```

Дальше логика группировки по account_id и фильтрации по окну не меняется — но должна учитывать оба типа.

**Ключевой момент для фильтра:** для `body_txns` привязка к кредитному счёту идёт через `target_account_id`, а не через `credit_account_id`. В цикле собирай `account_id` так:

```python
for tx in all_credit_txns:
    if tx.operation_type == "transfer":
        account_id = tx.target_account_id
    else:
        account_id = tx.credit_account_id
    if account_id is None or account_id not in credit_account_ids:
        continue
    ...
```

## БЛОК В. Фикс IncomeStructureWidget: фильтрация регулярности

### В.1 Проблема

В `frontend/components/dashboard/income-structure-widget.tsx` виджет считает структуру дохода по правилу 50/30/20 (обязательные / второстепенные / остаток от дохода). Но:

- **Доходы** суммируются без фильтра по `is_regular` → бонусы и подарки раздувают income, искажают пропорции
- **Расходы** попадают в `essential`/`secondary` без фильтра по `is_regular` → мебель (irregular) падает в secondary и завышает эту долю
- Фильтр `credit_payment` устарел (после Фазы 1 типа не существует). `transfer` не исключён → тело кредита попадает в расходы как secondary

Правило 50/30/20 — про регулярную структуру жизни. Нерегулярные всплески (крупные покупки, бонусы) не должны её искажать.

### В.2 Фикс

В `frontend/components/dashboard/income-structure-widget.tsx`, цикл `for (const transaction of analyticsTransactions)` (примерно строки 196-215):

```typescript

    if (!bucket) continue;

    const amount = Number(transaction.amount);

    // Доходы: только регулярные
    if (transaction.type === 'income') {
        if (!transaction.is_regular) continue;
        if (transaction.operation_type === 'credit_disbursement') continue;
        bucket.income += amount;
        continue;

    // Расходы: только регулярные; переводы и досрочные исключаем

    if (!transaction.is_regular) continue;  // нерегулярные не в структуре

    if (priority === 'expense_essential') {
        bucket.essential += amount;
    } else {
        bucket.secondary += amount;
    }
}
```

Убрать фильтр `'credit_payment'` — тип упразднён.

- Если это **фильтр** `operation_type === 'credit_payment'` (позитивный — «это кредитный платёж»): удалить логику ИЛИ заменить на проверку, подходящую для новой модели (например, для виджета кредитов — брать `transfer` на `credit_account_id` в кредитный счёт)

- Если это **строка в справочнике/селекторе** (`review/page.tsx`): убрать пункт «Кредитная операция» из selectable-операций. В UI создания транзакции пользователь больше не выбирает этот тип — сервис сам генерирует пару expense+transfer при импорте

2. **Оптимизация `compute_components`.** Наивная версия «пересчитать баланс по всем транзакциям до as_of» может быть медленной для активных пользователей. Оптимизация: использовать текущий `account.balance` и откатить назад через `Σ(post-as_of транзакций)`. Оба варианта должны давать один и тот же результат; выбирай тот, что быстрее на данных.

3. **Celery-beat в dev-окружении.** При `docker compose up` beat должен стартовать автоматически. Проверь, что он не создаёт второй worker-процесс (обычно beat — отдельный контейнер).

4. **Блок Г и `credits-widget.tsx`.** Может оказаться, что виджет полагается на старую модель глубже, чем просто фильтр. Если рефакторинг выходит за рамки Фазы 3 — помечай `credits-widget.tsx` как техдолг и оставь `TODO(phase5)` комментарий.

5. **Регулярный vs нерегулярный в расходах с КК.** После Фазы 1 покупка продуктов кредиткой — это `expense` с `operation_type='regular'` (не transfer!) и `is_regular=true`. В Блоке В фильтр `is_regular` её оставит в структуре. Это корректно.
