### Шаг 2. Миграция `credit_payment` → `expense` + `transfer`

**Критичная часть. Пиши миграцию как отдельный скрипт с dry-run режимом.**

2.1. Создай `scripts/migrate_credit_payments.py` (НЕ alembic — это data migration, нужна выполняется отдельно через `docker compose exec api python -m scripts.migrate_credit_payments`):

Логика:
```
FOR каждой транзакции t WHERE t.operation_type = 'credit_payment':
    principal = t.credit_principal_amount
    interest = t.credit_interest_amount

    # Валидация
    IF principal IS NULL OR interest IS NULL:
        log warning: "Transaction {id}: missing principal/interest, skipping"
        continue

    IF abs((principal + interest) - t.amount) > 0.01:
        log warning: "Transaction {id}: principal + interest != amount, skipping"
        continue

    # Найти системную категорию процентов для этого user
    interest_category = SELECT id FROM categories WHERE user_id = t.user_id AND is_system = true AND name = 'Проценты по кредитам'

    # Создать две новые транзакции в одной сессии
    t_interest = Transaction(
        user_id=t.user_id,
        account_id=t.account_id,  # с какого счёта платили (дебет)
        credit_account_id=t.credit_account_id,  # привязка к кредиту (для DTI)
        category_id=interest_category.id,
        amount=interest,
        currency=t.currency,
        type='expense',
        operation_type='regular',
        description=f"Проценты · {t.description or ''}".strip(' ·'),
        transaction_date=t.transaction_date,
        is_regular=True,
        affects_analytics=t.affects_analytics,
    )

    t_transfer = Transaction(
        user_id=t.user_id,
        account_id=t.account_id,  # списание с дебета
        target_account_id=t.credit_account_id,  # поступление на кредитный счёт (уменьшение долга)
        amount=principal,
        currency=t.currency,
        type='expense',  # для transfer это направление списания
        operation_type='transfer',
        description=f"Тело кредита · {t.description or ''}".strip(' ·'),
        transaction_date=t.transaction_date,
        is_regular=True,
        affects_analytics=t.affects_analytics,
    )

    # Удалить исходную транзакцию
    DELETE t

    # Логировать: какой id был, какие два id созданы
```

2.2. Скрипт должен поддерживать `--dry-run` (по умолчанию) и `--execute`. В dry-run — только печатать SQL-план и количество затронутых строк, не коммитить. В execute — оборачивать всё в одну транзакцию БД, чтобы при ошибке откатиться целиком.

2.3. В начале execute-режима: сделать SQL-дамп таблицы `transactions` в файл `backup_transactions_before_migration_YYYYMMDD.sql` (для быстрого восстановления).

2.4. Логировать итоговую статистику:
- Всего обработано credit_payment: N
- Успешно разделено: M
- Пропущено (missing/mismatch): N-M
- Список пропущенных transaction_id для ручного разбора

### Шаг 3.5. Исключить `credit_disbursement` из аналитики доходов (GAP #8)

Параллельно с обходом 6 сервисов — одноразовое уточнение: `credit_disbursement` **не доход**, а балансовая операция (деньги банка пришли на дебет, долг вырос ровно на эту сумму). В аналитике доходов и в формулах Потока он не должен участвовать.

3.5.1. В каждом сервисе, где считаются доходы/расходы (`metrics_service.py`, `financial_health_service.py`, `analytics_service.py`, любые отчёты), убедись, что фильтр транзакций исключает `operation_type='credit_disbursement'`:

Типовой SQL/ORM фильтр для агрегации доходов должен содержать:
```python
Transaction.type == 'income'
AND Transaction.operation_type NOT IN ('credit_disbursement', 'transfer', 'credit_early_repayment')
AND Transaction.affects_analytics == True
```

3.5.2. В `analytics_service.py` (если есть разбивка доходов на активные/пассивные/прочие) — проверь, что `credit_disbursement` не попадает ни в одну из групп.

3.5.3. В API-ответах (например, `/transactions/summary`, если такой есть) — убедись, что сумма входящих поступлений не включает `credit_disbursement`.

## Подводные камни

1. **Транзакции с missing principal/interest.** Могут быть исторические credit_payment без разбивки. Миграция их пропускает, логирует. В конце миграции пользователю показывается отчёт — что пропустили. Решение: эти транзакции надо разобрать вручную или удалить.

2. **`credit_interest` как отдельный тип.** В enum есть `credit_interest`. Проверь — используется ли он реально (grep), или это мёртвый код. Если используется — такой же перевод: в `regular expense` с категорией процентов.

3. **Импорт выписок.** `import_service.py` раньше создавал `credit_payment`. После Фазы 1 он должен создавать пару `expense+transfer` сразу при распознавании кредитного платежа. Проверь логику на реальной выписке.

4. **Несогласованность с `credit_principal_amount` на Transaction.** После разделения эти поля на новых `expense` и `transfer` не нужны (принципал и процент разделены через разные записи). На `expense` (проценты) оставь NULL в этих полях. На `transfer` (тело) — тоже NULL. Поля оставляем на модели для обратной совместимости, но новые записи их не используют.

5. **Откат.** Если что-то пошло не так после execute — восстанавливаем из `backup_transactions_before_migration_*.sql`. Не пытаемся делать downgrade через alembic — data-миграция это не умеет.
