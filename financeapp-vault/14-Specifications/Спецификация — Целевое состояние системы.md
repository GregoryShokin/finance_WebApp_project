## 1. Философия и принципы

### 1.1 Миссия

FinanceApp — не трекер, а **наставник**. Каждый виджет и показатель содержит конкретный совет: «что делать дальше». Разница между «получить результаты анализов» и «получить результаты с рекомендациями врача».

### 1.2 Четыре принципа ДНК

| # | Принцип | Суть |
|---|---------|------|
| 1 | **Снизу вверх** | Порядок священен: стабилизация → рост → инвестиции. Проверочный вопрос для каждой фичи: «Поможет ли это семье с DTI 42%, потоком +5 000 и без подушки?» |
| 2 | **За каждой цифрой — эмоция** | Не морализировать, не стыдить, не пугать. Финансы — это страх, стыд, импульсы. Тон: поддерживающий наставник |
| 3 | **Приложение растёт с пользователем** | Виджеты открываются по мере прохождения уроков: сначала знание, потом инструмент |
| 4 | **Сложность — в движке, не в голове** | Пользователь подтверждает факты, а не выбирает модели. Кредитная логика, метрики, классификация — всё внутри |

### 1.3 Пирамида финансовых приоритетов

```
    ╭───────────╮
    │  Вершина  │  Финансовая свобода (инвестиции)
    ├───────────┤
    │  Этаж 3   │  Финансовые цели
    ├───────────┤
    │  Этаж 2   │  Устойчивость (вклады ≥ 3 мес. расходов)
    ├───────────┤
    │  Этаж 1   │  Контроль долгов (DTI < 40%)
    ├───────────┤
    │ Фундамент │  Осознанность (Поток, мультиметрика)
    ╰───────────╯
```

**Порог финансовой устойчивости:** DTI < 40% (Этаж 1) **И** деньги на вкладах/накопительных счетах ≥ 3 мес. среднемесячных расходов. Тип счёта — естественный фильтр намерения: перевод на вклад = сознательное решение отложить деньги. Кредитные средства (credit_disbursement) поступают на дебетовые карты, а не на вклады, поэтому показатель не обманывается. До достижения порога — фокус на стабилизацию, инвестиционный блок заблокирован. Источник: [[Дебаты — Показатель финансовой устойчивости]].

### 1.4 Целевая аудитория

| Профиль | Доход | DTI | Поток | Вклады | Фокус |
|---------|-------|-----|-------|--------|-------|
| **Первый шаг** | 120к на двоих | 42% ❌ | +5к | 0 (нет вкладов) | Выживание, разрыв цикла кредитов |
| **Рост** | 200к | 25% | +30к | 1.5 мес. расходов | Закрыть ипотеку, нарастить буфер |
| **Прорыв** | 300к | 20% | +80к | 6+ мес. расходов | Инвестиции, финансовая свобода |

## 2. Мультиметрическая система

> Заменяет единый СМО (архивирован апрель 2026). Три независимые метрики + композитный FI-score.

### 2.1 Поток (Flow)

> **Полная действующая спецификация:** [[Поток]] (`01-Metrics/Поток.md`). Этот раздел даёт консолидированную выжимку для целей спецификации системы; детали формул и методологические заметки — в [[Поток]].

**Отвечает на вопрос:** «Могу ли я позволить себе текущий уровень жизни?»

**Три слоя (трёхслойная модель, решение v1.5 от 2026-04-19):**

```
Базовый поток       = регулярные доходы − регулярные расходы
Свободные средства  = Базовый поток − Σ обязательных тел кредитных платежей
Полный поток        = Δ кэша на ликвидных счетах за период
```

**Ликвидные счета** для расчёта Полного потока: `regular + cash + deposit`. Вклады включены — они работают как инструмент ликвидности, не как инвестиция (решение 2026-04-19, см. § 3.1 и [[Поток]]).

> **Принцип: кредит = отдельный мир.** Все операции по телу кредита (disbursement, платёж тела, досрочное погашение) — это перемещения между счетами, не доход и не расход, они не в Базовом/Свободном. Единственное исключение — **проценты**: регулярный `expense` в системной категории «Проценты по кредитам», входит в регулярные расходы Базового потока.

**Базовый поток** — устойчивый операционный уровень. Транзакционная метрика, отвечает на вопрос о структуре жизни.
- Включает только транзакции с `is_regular = true`
- Проценты по кредитам в системной категории `regular` → уже в формуле
- Тело кредита = `transfer` → НЕ в Базовом
- Покупки на КК/рассрочку = `expense` → в Базовом, если категория регулярная

**Свободные средства** — скорость генерации. Ежемесячный остаток, который можно направить на цели, инвестиции, досрочное погашение.
- `= Базовый − Σ(Account.monthly_payment × body_ratio)` по всем кредитным счетам
- При отсутствии кредитов: Свободные = Базовый
- **Свободный не равен Полному при активном использовании КК и других балансовых эффектах** — это сознательный поведенческий сигнал (см. «Почему компенсатор только в Полном» в [[Поток]]).

**Полный поток** — балансовая метрика: `Δ(regular + cash + deposit)` за период. Замкнутость дана по построению.

**Декомпозиция Полного в виджете** (путь Б, решение 2026-04-19):

```
Зарплата и другие доходы        +200 000
Повседневные расходы             −157 000   (включая покупки с КК)
Тело кредитных платежей            −3 000
Покупка инвестиций                −10 000
Поступление кредитных средств           0
Прирост долга по кредиткам        +15 000   ← строка-компенсатор
─────────────────────────────────────────
Δ кэша (Полный поток)             +45 000
```

Строка «Прирост долга по кредиткам» = `Σ(долг_КК_конец − долг_КК_начало)`. Замыкает арифметику декомпозиции с балансовым числом, не требуя парных проводок на каждую КК-покупку.

**Почему компенсатор только в Полном:** Базовый и Свободный — транзакционные, отвечают на вопросы о структуре и скорости. Полный — балансовый, обязан сводиться с фактическим остатком на счетах. Применение компенсатора к Базовому/Свободному стёрло бы поведенческий сигнал «живу за счёт роста долга». Подробнее — в [[Поток]].

**Связь трёх слоёв** (пример — типовой месяц без КК-активности):
```
Базовый поток:        +80 000   ← устойчивый уровень жизни
Тело обязательных:     −5 000
                      ─────────
Свободные средства:   +75 000   ← скорость генерации
Нерегулярные:              0
                      ─────────
Полный поток:         +75 000   ← реальный Δ кэша
```

**Показатель уровня жизни** (нормализованный базовый поток):
```
Показатель = avg(базовый, 12 мес) / avg(регулярный доход, 12 мес) × 100%
```
Скользящее окно в **12 завершённых месяцев** (единое окно для всех средних в системе, решение 2026-04-19). Если данных менее 12 мес. — используются доступные, виджет сигнализирует (см. § 9.4 UX-гипотезы ДДС).

Математически: `Показатель = 1 − ExpenseRatio` (проценты по кредитам внутри ExpenseRatio как регулярный расход).

**Зоны:**
| Зона | Значение | Цвет |
|------|----------|------|
| Здоровый | ≥ 20% | 🟢 |
| Узкий | 0–19% | 🟡 |
| Дефицит | < 0% | 🔴 |

**Виджет на дашборде:** один донат с тремя табами (Базовый / Свободный / Полный). Сегментированные кольца по цветовой палитре (доходы/расходы/тело/инвестиции/компенсатор). Подсказка-наставник в отрицательной зоне — с планом выхода и ссылкой на модуль школы. Прогрессивное открытие табов:
- Базовый — Модуль 0
- Свободный — Модуль 2 (Кредитная нагрузка)
- Полный — Модуль 3 (Финансовая устойчивость)

Детальная UX-спецификация: [[Гипотеза — UX виджета ДДС]] (название исторически, описывает трёхслойную модель после актуализации 2026-04-19).

**Отменено:** ранее рассматривалась отдельная метрика «Доступно для целей» (одно число, «сколько я могу направить на цели»). После разбора 2026-04-19 — отменена. Причина: disbursement ломает любую формулу такого вида (кредит 500к → метрика скачет на +500к, пользователь чувствует себя королём). Вместо этого — объяснение через связку показателей в рамках школы (Свободный поток + DTI + Капитал + Буфер устойчивости).

**Школьные моменты (триггеры):**
1. Резкое падение потока → предложение пометить категорию нерегулярной
2. Нерегулярные расходы > 30% дохода → «Проверь, это плановые траты?»
3. Нерегулярная категория появляется 3+ мес. подряд → «Может, она регулярная?»
4. Компенсатор > 30% расходов месяца → «Ты много тратишь в долг. Смотри риск кредитной ловушки»
5. Свободный в красной зоне 3+ мес. подряд → «Фокус на DTI, не на инвестициях»

### 2.2 DTI / Кредитная нагрузка

**Отвечает на вопрос:** «Сколько дохода уходит банку?»

**UI-название:** «Кредитная нагрузка» (не DTI).

```
DTI = Σ ежемесячных платежей по всем кредитам (тело + %) / средний регулярный доход × 100%
```

**Числитель:** полный платёж (тело + проценты) по всем кредитным счетам, включая рассрочки. Источник: последний фактический `credit_payment` → fallback: `Account.monthly_payment`.

**Знаменатель:** ТОЛЬКО регулярный доход (средний за 12 завершённых мес., при нехватке данных — все доступные). Нерегулярные поступления (бонусы, продажа активов) исключены.

**Зоны:**
| Зона | Значение | Метафора |
|------|----------|----------|
| Норма | < 30% | — |
| Допустимо | 30–40% | «2 из 5 дней — на банк» |
| Опасно | > 40% | — |
| Критично | > 60% | — |

### 2.3 Капитал (Capital)

**Отвечает на вопрос:** «Я богатею или беднею?»

```
Капитал = Σ ликвидные активы (regular, cash) + Σ вклады (deposit) − Σ долги по кредитам
```

**Что НЕ входит:** квартира, машина, прочие неликвидные активы. Причина: для ЦА с DTI ~42% неликвид — шум, его нельзя продать чтобы заплатить за продукты.

**Главная ценность — траектория, не позиция.** Пользователь видит не «−150к», а «растёт на 8к/мес ↑ 🟢».

**Расширенный вариант:** «Чистый капитал» = Капитал + оценка неликвидных активов. Только на странице Assets.

**Что ловит:** долговая спираль (капитал падает при хорошем потоке), lifestyle inflation (стоит на месте при хорошем доходе), истощение накоплений.

### 2.4 FI-score (композитный)

**Определение:** показатель финансового здоровья, 0–10.

**Целевые веса (мультиметрическая система v1.4):**

| Компонент | Вес | Источник | Нормализация |
|-----------|-----|----------|-------------|
| Норма сбережений | 0.20 | avg(базовый поток) / avg(регулярный доход) | 0% → 0, ≥30% → 10 |
| Капитал (траектория) | 0.30 | Тренд капитала за 3–12 мес. | Падает → 0, стагнация → 5, растёт → 10 |
| DTI (inverse) | 0.25 | Чем ниже DTI, тем выше балл | ≥60% → 0, 0% → 10 |
| Буфер устойчивости | 0.25 | Σ(deposit balances) / avg(monthly_expenses) | 0 мес. → 0, ≥6 мес. → 10 |

> **v1.2:** Запас (Reserve) удалён из FI-score. Причина: числитель (ликвидный кэш) включал заёмные средства (credit_disbursement), что создавало компенсационную ловушку (сценарий F3).
>
> **v1.4:** Буфер устойчивости (на базе deposit-счетов) **возвращён в FI-score**. Вклады не подвержены credit_disbursement (кредитные деньги поступают на дебет, не на вклад) → компенсационная ловушка исключена. Буфер одновременно является компонентом FI-score и порогом Этажа 2 пирамиды (≥ 3 мес.). Источник: [[Дебаты — Показатель финансовой устойчивости]].

**Зоны:**
| Зона | Диапазон |
|------|----------|
| Риск | 0–3 |
| Рост | 3–6 |
| Путь | 6–8 |
| Свобода | 8–10 |

**Правила:**
- Использует `avg_savings_rate` (среднее за завершённые месяцы), не текущий месяц
- Требование чувствительности: переключение одной средней категории (~15к/мес) с regular на irregular не должно менять FI-score > 0.5–1 балл

### 2.5 Свободные средства (Free Cash)

**Отвечает на вопрос:** «Сколько я реально могу направить на цели, инвестиции или досрочное погашение?»

```
Свободные средства = Базовый поток − Σ обязательных тел кредитных платежей
```

**Числитель тела:** `Account.monthly_payment − credit_interest` (проценты уже учтены в базовом потоке как расход). Для рассрочки 0%: тело = полный платёж.

**Проверка на сценариях:**

| Сценарий | Базовый п��ток | Тело платежей | Свободные | Полный поток |
|----------|:---:|:---:|:---:|:---:|
| Нет кредитов | +80к | 0 | **+80к** | +80к |
| Рассрочка 0% (5к/мес) | +80к | 5к | **+75к** | +75к |
| Потребкредит (40к тело + 5к %) | +75к | 40к | **+35к** | +35к |
| Ипотека (3к тело + 42к %) | +38к | 3к | **+35к** | +35к |
| Кешбэк (КК, полное погашение) | +80к | 0 | **+80к** | +80к |

> При отсутствии нерегулярных аномалий: **Свободные средства = Полный поток**. Расхождение между ними = нерегулярные расходы/доходы мес��ца.

**Где исп��льзуется:**
- Виджет Потока на дашборде (третье число в декомпозиции)
- Калькулятор целей (ориентир «сколько можно откладывать»)
- Стратегии погашения кредитов («свободно X₽ → рекомендуемый доп. взнос Y₽»)

## 4. Кредитная логика

> Источники: дебаты «Овердрафтная модель vs Покупка-расход», «Карта рассрочки», «Операционные слабости модели рассрочки», стресс-тест кредитной модели, анализ импульсных покупок, кешбэк-пользователь.

### 4.1 Единое правило (решение апрель 2026)

**Покупка на кредитные средства = расход в реальной категории. Всегда. Для всех типов кредитов.**

**Принцип: кредит = отдельный мир.** Только два типа событий создают настоящий расход:
1. **Покупка** — решение потратить деньги (в категории)
2. **Проценты** — плата банку за услугу (регулярный расход)

Всё остальное (получение кредита, возврат тела, досрочное погашение) — **перемещения между счетами**, не доход и не расход.

| Шаг | Операция | Тип | В потоке? |
|-----|---------|-----|:---------:|
| Покупка | Расход в категории | `expense` | Да (по регулярности категории) |
| Поступление кредита на дебет | Баланс (не доход) | `credit_disbursement` | Нет |
| Погашение тела | Перевод | `transfer` | Нет |
| Уплата процентов | Расход (регулярный) | `expense` | Да |
| Досрочное погашение | Перевод | `credit_early_repayment` | Нет |

**Никакого выбора режима пользователем.** Режим «платёж = расход» отвергнут — ломает кешбэк-сценарий (DTI 60% при нулевом долге, категории невидимы за «Кредитными платежами»).

### 4.2 Применение к типам кредитов

#### Кредитная карта

```
Покупка 5 000 «Продукты» на кредитку:
  → Расход −5 000 на счёте КК, категория «Продукты»
  → Долг на КК: +5 000
  → Никакого disbursement

Погашение кредитки 5 000 с дебета:
  → Перевод: дебет → КК
  → Долг КК: −5 000
  → Не расход, не в потоке
```

Кешбэк-пользователь (все 120к/мес через кредитку, полное погашение):
- Все расходы видны по категориям ✅
- DTI = 0% (погашение = перевод, нет credit_payment) ✅
- Базовый поток = 200к − 120к = 80к ✅

#### Карта рассрочки

```
Покупка телефона 60к на рассрочку (12 мес, 0%):
  → Расход −60 000, категория «Электроника» (нерегулярная)
  → Долг: +60 000

Ежемесячный платёж 5 000:
  → Перевод: дебет → installment_card (тело 5 000)
  → Если есть проценты: расход (проценты)
  → Долг: −5 000
```

#### Потребительский кредит

```
Получение 500к на дебетовый счёт:
  → credit_disbursement +500 000 (НЕ доход, НЕ в аналитике доходов)
  → Кредитный счёт: долг +500 000

Расходование (ремонт 150к/мес, 3 месяца):
  → Обычные расходы в категориях («Ремонт» — нерегулярная)

Ежемесячный платёж 45к:
  → Перевод (тело ~40к) + расход (проценты ~5к)
```

**credit_disbursement** — специальный тип транзакции:
- НЕ доход (ни регулярный, ни нерегулярный)
- НЕ появляется в аналитике доходов
- Влияет на балансы счетов (дебет ↑, долг ↑)
- В Капитале: нетто = 0 (дебет +500, долг +500)

#### Автокредит / ипотека

```
Ипотека 5М:
  → credit_disbursement +5 000 000 на дебет
  → Покупка квартиры −5 000 000 (расход, нерегулярный)
  → Нетто на дебете: 0. Долг: +5 000 000

Ежемесячный платёж 45к:
  → Перевод (тело ~3к) + расход (проценты ~42к)
```

### 4.3 Сводная таблица: кредиты в метриках

| Тип кредита | Покупка | Disbursement | Тело платежа | Проценты | Тело в «Свободных средствах» |
|-------------|---------|:---:|:---:|:---:|:---:|
| Кредитная карта | Расход (категория) | Нет | Перевод | Расход (рег.) | Нет (погашение = разовый перевод) |
| Карта рассрочки | Расход (категория) | Нет | Перевод | Расход (= 0) | **Да** (ежемесячный обязательный) |
| Потребкредит | Расход (из кредитных денег) | credit_disbursement | Перевод | Расход (рег.) | **Да** |
| Автокредит | Расход (нерег.) | credit_disbursement | Перевод | Расход (рег.) | **Да** |
| Ипотека | Расход (нерег.) | credit_disbursement | Перевод | Расход (рег.) | **Да** |

> **Примечание по кредитной карте:** при полном ежемесячном погашении КК — тело не вычитается из свободных средств, т.к. расход уже учтён в базовом потоке (покупка = расход). Погашение = перевод, компенсирующий этот расход. Если же на КК есть revolving-долг с минимальным платежом — тело минимального платежа вычитается из свободных.

### 4.4 Правило «каждый рубль — один раз»

Если покупка уже учтена как расход → тело погашения = перевод (не дублируется в потоке).

Механика: при `credit_payment` с привязкой к кредитному счёту, где `purchase_is_expense = true` (всегда) → тело записывается как `transfer`, проценты как `expense`.

### 4.5 Типы транзакций (operation_type)

| Тип | Описание | В расходной аналитике | В базовом потоке | В «Свободных средствах» |
|-----|---------|:---:|:---:|:---:|
| `income` | Любой доход | — | Если is_regular | Через базовый поток |
| `expense` | Любой расход (включая покупки на кредит и проценты) | Да | Если is_regular | Через базовый поток |
| `transfer` | Перевод между счетами (в т.ч. тело кредита) | Нет | Нет | **Тело вычитается** (если обязательный кредитный платёж) |
| `credit_disbursement` | Получение кредитных средств на счёт. **НЕ доход.** | Нет | Нет | Нет |
| `credit_early_repayment` | Досрочное погашение | Нет | Нет | Нет (это решение пользователя из свободных средств) |
| `investment_buy` | Покупка актива | Нет | Нет | Нет |
| `investment_sell` | Продажа актива | Нет | Нет | Нет |
| `debt_give` / `debt_take` / `debt_return` / `debt_collect` | Личные долги | Нет | Нет | Нет |

> **Упрощение:** `credit_payment` как отдельный operation_type **упразднён**. Проценты = обычный `expense` в категории «Проценты по кредитам» (регулярная). Тело = `transfer`. Это устраняет необходимость специального типа.

### 4.6 Досрочное погашение

`credit_early_repayment` — всегда перевод (тело). Не попадает в расходную аналитику, не влияет на поток. Влияет на Капитал (долг ↓, кэш ↓ → нетто ≈ 0) и на DTI (при полном закрытии кредита → DTI падает).

### 4.7 Конвертация покупки КК → рассрочка

Сценарий: купил ТВ 80к на кредитке, банк конвертировал в рассрочку.

1. Исходная транзакция расхода помечается `converted_to_installment = true`
2. Создаётся `InstallmentPurchase` с параметрами рассрочки
3. Если конвертация в том же месяце — чисто
4. Если в прошлом месяце — ручная коррекция (MVP)

### 4.8 Тегирование кредитных денег (потребкредит)

Для потребительских кредитов, где деньги приходят на дебет и смешиваются с зарплатой:

**Три линии защиты от двойного учёта:**
1. Умная подсказка при импорте (совпадение суммы disbursement ± 10%)
2. Ручная галочка «оплачено в кредит» на транзакции
3. Кредитный конверт — месячный батч-ввод: «сколько ушло на ремонт?»

**Данные:** `Account.credit_envelope_remaining`, `Account.credit_source_id`, `Account.tagged_credit_balance`.

### 4.9 Здоровье долга (планируемый виджет)

Автоматический сигнал при нарастании долга:
- Траектория баланса: ↓ гасится (зелёный), → стабилен (жёлтый), ↑ растёт (красный)
- При росте 2+ мес. подряд → школьный момент: «Долг растёт. Платёж не покрывает проценты. Для остановки → X₽/мес.»

## 5. Категории и классификация

> Источники: стресс-тест классификации расходов (12 сценариев), дебаты по базовому потоку.

### 5.1 Приоритеты категорий (5 типов)

| Приоритет | Тип | Описание |
|-----------|-----|-----------|
| `expense_essential` | Расход | Обязательные: аренда, продукты, транспорт |
| `expense_secondary` | Расход | Второстепенные: кафе, развлечения, подписки |
| `expense_target` | Расход | Целевые: привязаны к финансовой цели |
| `income_active` | Доход | Активный доход: зарплата, фриланс |
| `income_passive` | Доход | Пассивный доход: проценты, дивиденды |

### 5.2 Флаг регулярности (Category.regularity)

Три значения: `regular` | `irregular` | `project`

| Значение | В базовом потоке | В полном потоке | Типичные категории |
|----------|:---:|:---:|---|
| `regular` | Да | Да | Продукты, зарплата, аренда, транспорт |
| `irregular` | Нет | Да | Электроника, мебель, годовой бонус, подарки |
| `project` | Нет (с коррекцией через Проект) | Да | Ремонт, переезд, свадьба |

**Критерий регулярности — природа источника, не стабильность суммы:**
- Зарплата с волатильной суммой → регулярная
- Годовой бонус с фиксированной суммой → нерегулярная

### 5.3 Снапшот регулярности (Transaction.is_regular)

При создании транзакции значение `is_regular` копируется из `Category.regularity` и **замораживается**. Смена флага на категории не переписывает историю (нерретроактивность).

Это обеспечивает:
- Стабильность исторических метрик
- Контролируемость: пользователь видит, что пометка «регулярный» действует с момента переключения

### 5.4 Правило регулярности для кредитных покупок

**Расход на кредитные средства следует регулярности своей КАТЕГОРИИ, а не способу оплаты.**

| Покупка | Категория | Регулярность | В базовом потоке? |
|---------|-----------|:---:|:---:|
| Продукты на кредитку | Продукты (regular) | Регулярный | Да |
| MacBook на кредитку | Электроника (irregular) | Нерегулярный | Нет |
| Ремонт из потребкредита | Ремонт (project) | Проект | Нет (с коррекцией) |

### 5.5 Три архитектурных правила классификации

1. **Проект + нерегулярность = приоритет флага.** Если все расходы проекта уже в нерегулярных категориях → Проект = только UX (иконка, прогресс), без метрической коррекции
2. **Тело платежа = перевод** если покупка уже была расходом. Тело = расход только если покупка не была расходом (legacy-сценарий, не используется в текущей модели)
3. **Transaction.is_regular = snapshot** флага категории на момент создания. Нерретроактивность

### 5.6 Graceful degradation

Если пользователь не настраивает флаги регулярности — все расходы помечаются как регулярные (дефолт). Базовый поток = полный поток. Менее информативно, но не ломается.

## 6. Целевые накопления

> Источник: дебаты по целевым накоплениям (5 раундов, балл 85/15). Модель упрощена в v1.2 по итогам стресс-теста.

### 6.1 Модель «Цель = трекер» (v1.2)

> **v1.2:** Модель «откладывание = расход» **отвергнута.** Причина: при весе Капитала 0.40 в FI-score откладывание на expenditure-цель создавало двойной удар (расход в Потоке + скрытие из Капитала), наказывая пользователя за накопление. Заменена на простую модель: цель = трекер.

**Принцип: все добровольные финансовые действия = перевод.**

Откладывание на цель, инвестиции, досрочные погашения, переводы на вклады — всё это **перемещения между своими счетами**. Они не расход, не в Потоке, не в Свободных средствах. Единственное, что вычитается из Свободных средств — **обязательные** тела кредитных платежей (их нельзя не заплатить).

| Операция | Тип | В Потоке? | В Свободных? | В Капитале? |
|----------|-----|:---------:|:------------:|:-----------:|
| Перевод на целевой счёт | `transfer` | Нет | Нет | Видим (оба счёта) |
| Трата целевых денег | `expense` (по категории) | По регулярности | Через базовый | Уменьшает |
| Досрочное погашение | `credit_early_repayment` | Нет | Нет | Нетто ≈ 0 |
| Инвестиция | `investment_buy` | Нет | Нет | Бэклог |
| Перевод на вклад | `transfer` | Нет | Нет | Видим |

### 6.2 Единый тип целей

> **v1.2:** Разделение на `expenditure` / `accumulation` **упразднено.** Все цели = трекер.

| Пример | Модель | Влияние на метрики |
|--------|--------|-------------------|
| Отпуск, машина, техника | Трекер: привязка к счёту | Нет (до траты) |
| Подушка, пенсионный фонд | Трекер: привязка к счёту | Нет |

### 6.2.1 Жизненный цикл цели

**Фаза копления:** Пользователь переводит деньги на целевой счёт (transfer). Метрики не затронуты — деньги перемещаются между видимыми счетами, Капитал неизменен.

**Фаза траты:** Пользователь расходует накопленные средства. Расходы идут в категории по своей природе:
- Крупные целевые расходы (перелёт, отель, стройматериалы, мебель) → **нерегулярная** категория (например, «Отпуск», «Ремонт») → не в Базовом потоке
- Мелкие бытовые расходы (рестораны, продукты, такси в отпуске) → в тех же **регулярных** категориях → в Базовом потоке, но эффект минимален (замещают домашние расходы, а не добавляются к ним)

**Влияние на метрики при трате:**
| Метрика | Эффект |
|---------|--------|
| Базовый поток | ≈ 0 (регулярные расходы замещают домашние, дельта ~5%) |
| Свободные средства | ≈ 0 |
| Полный поток | Снижается на сумму нерегулярных трат |
| Капитал | Снижается на потраченную сумму |
| FI-score | Умеренное влияние через Капитал-траекторию (смягчено 12-мес окном) |

**Пример:** Пользователь накопил 400к на отпуск, потратил за 2 недели. Базовый поток: +80к → +76к (−4к, шум). Капитал: −400к. Через 5 месяцев Капитал восстанавливается. Три из четырёх метрик зелёные — пользователь видит, что фундамент стабилен.

**Трата не по назначению:** Если пользователь снял деньги с целевого счёта и потратил на другое — система не блокирует. Расходы учитываются как обычно. Прогресс цели уменьшается. Это выбор пользователя.

### 6.3 Модель данных

```
Goal:
  id, name, target_amount, account_id,
  planned_monthly, target_date, status, is_system, system_key,
  created_at, closed_at
```

> **Удалено в v1.2:** поле `goal_type` (expenditure/accumulation) — больше не нужно. Таблица `GoalSpendingAllowance` — больше не нужна (нет двойного учёта).

**Привязка:** Goal → Account (один счёт = одна цель в MVP). Прогресс = Account.balance / target_amount.

### 6.4 Подушка безопасности (рекомендованная цель)

> **Изменение v1.4:** Подушка больше не является системной целью и не участвует в пороге устойчивости. Порог устойчивости (Этаж 2 пирамиды) вычисляется автоматически из баланса deposit-счетов (§1.3). Подушка — **первая рекомендованная цель** на Этаже 3 (Goals).

Подушка безопасности предлагается пользователю после достижения Этажа 2 пирамиды:
- Рекомендация: «Устойчивость достигнута! Следующий шаг — подушка безопасности: 3–5 мес. расходов на отдельном вкладе»
- Рекомендуемый таргет: `avg_monthly_expenses × 5`
- Пользователь создаёт цель и привязывает deposit-счёт
- Подушка — обычная цель-трекер. Откладывание = transfer, деньги видны в Капитале

**Связь с порогом устойчивости:** порог проверяет `Σ(deposit balances) / avg(expenses) ≥ 3` автоматически, без привязки к цели. Подушка как цель — дополнительная мотивация, а не условие порога.

### 6.5 Норма сбережений

```
Норма сбережений = базовый поток / регулярный доход × 100%
```

> **v1.2:** Коррекция `(Поток + целевые_взносы) / Доход` **больше не нужна** — целевые взносы не являются расходом и не уменьшают Поток.

## 7. Проект

> Источник: дебаты мультиметрической системы (раунд 5). Роль упрощена в v1.3.

### 7.1 Назначение (v1.3)

> **v1.3:** Проект = **UX-обёртка для группировки нерегулярных расходов**, а не метрический корректор. Формула коррекции базового потока (`+ бюджет/срок`) **упразднена** — при использовании нерегулярных категорий коррекция не нужна.

Проект объединяет разнородные расходы под одной шапкой: ремонт (стройматериалы + мебель + работы), отпуск (перелёт + отель + рестораны + развлечения), свадьба и т.д.

**Проект создаётся:**
- При планировании крупной траты (накопление на цель → трата)
- При получении потребкредита (credit_disbursement)
- Вручную пользователем

### 7.2 Что даёт Проект

- **Группировка:** все расходы проекта видны в одном месте с разбивкой по категориям
- **Бюджет:** прогресс-бар «Потрачено X из Y»
- **Аналитика:** можно оценить, сколько ушло на каждую составляющую (стройматериалы: 150к, мебель: 200к, работы: 100к)
- **Школьный момент** при завершении или превышении бюджета

### 7.3 Влияние на метрики

**Никакого.** Все расходы проекта идут в нерегулярные категории → не попадают в Базовый поток и Свободные средства. Влияют только на Полный поток и Капитал — как обычные нерегулярные расходы.

Если пользователь ошибочно записал проектный расход в регулярную категорию — это проблема классификации, Проект её не исправляет. Школьный момент: «Расход на ремонт в категории "Хозтовары" (регулярная). Пометить как нерегулярный?»

### 7.4 Модель данных

```
Project:
  id, name, budget, goal_id (FK → Goal, опционально),
  created_at, closed_at, status (active|closed)

ProjectCategory:
  id, project_id (FK → Project), category_id (FK → Category)
```

**Связь с целями:** Проект может быть привязан к цели (Goal). Цель = «сколько накопить», Проект = «как потратить». Одна цель → один проект (или без проекта).

**Категории проекта:** Проект содержит список категорий, расходы в которых автоматически группируются в проект. Все эти категории должны быть `irregular`.

## 10. Импорт данных

### 10.1 Каналы импорта

1. **Веб-интерфейс** — drag&drop файлов (PDF/Excel/CSV)
2. **Telegram-бот** `@financeapp_import_bot` — отправка файлов в чат

### 10.2 Конвейер обработки

```
Загрузка → Нормализация → Распознавание → Автокатегоризация → Дедупликация → Ревью
```

4. **Дедупликация** — поиск дублей по дате + сумме + описанию
5. **Ревью** — очередь `needs_review: true` на странице Review

### 10.3 ImportWizard (4 шага)

1. Определение формата и банка
2. Маппинг колонок (если нестандартный формат)
3. Предпросмотр транзакций
4. Проверка и подтверждение

### 10.4 Свежесть данных

**AccountFreshnessBlock** — вверху страницы Import: дата последнего обновления по каждому счёту. Стимулирует регулярный импорт.

## 11. GAP-анализ

### 11.1 Критические GAP-ы

| # | GAP | Целевое | Текущее | Критичность |
|---|-----|---------|---------|:-----------:|
| 1 | **Две параллельные FI-score системы** | Единый FI-score v1.4 (веса: Норма 0.20, Капитал 0.30, DTI 0.25, Буфер 0.25) | `metrics_service.py` и `financial_health_service.py` с разными формулами и весами | 🔴 Высокая |
| 2 | **Капитал: тренд всегда null** | Траектория (главная ценность: «растёт/падает») | `capital.trend = None`, `capital_score = 5.0` по дефолту. Нет таблицы снапшотов | 🔴 Высокая |
| 3 | **Кредитные платежи: тело/проценты не разделены** | Проценты = `expense` (рег. расход). Тело = `transfer`. Нет `credit_payment` как отдельного типа | Весь кредитный платёж = `credit_payment` (одна транзакция). Поля `credit_principal_amount`/`credit_interest_amount` есть, но не используются для разделения на `expense` + `transfer` | 🔴 Высокая |
| 4 | **Метрика «Свободные средства» не существует** | Базовый поток − Σ тел кредитных платежей | Нет ни расчёта, ни API-поля, ни виджета | 🔴 Высокая |

| 6 | ~~GoalSpendingAllowance~~ — УДАЛЕНО v1.2: модель «цель = трекер» не требует allowance | — | — | — |
| 7 | **Project как сущность** | Таблица Project + ProjectCategory (UX-обёртка, без метрической коррекции) | Категория `project` в enum есть, отдельной сущности нет | 🟡 Средняя |
| 8 | **credit_disbursement не классифицирован как не-доход** | Специальный тип: не доход, не в аналитике | `credit_disbursement` есть в operation_type, но поведение в аналитике не реализовано | 🟡 Средняя |

6. GAP #8: credit_disbursement → исключить из аналитики доходов

7. GAP #7: Таблица Project при реализации Механизма Проект

8. ~~GAP #6: GoalSpendingAllowance~~ — УДАЛЕНО v1.2 (модель «цель = трекер»)
