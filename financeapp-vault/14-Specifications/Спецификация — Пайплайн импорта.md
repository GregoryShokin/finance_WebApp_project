## 1. Философия и инварианты

### 1.1 Иерархия барьеров

Строка проходит через убывающую по автоматичности лестницу. Каждый следующий уровень подхватывает то, что не осилил предыдущий:

```
парсинг → нормализация → fingerprint/skeleton → правила → LLM-модератор → пользователь
```

**LLM — самый верхний автоматизированный барьер.** Всё, что LLM не смог классифицировать уверенно, уходит на ручную проверку юзером. Юзер — гарантированно проходимый барьер.

### 1.2 Честность важнее автоматизации

Принцип: **лучше пометить строку `warning` и попросить подтверждение, чем тихо присвоить неверную категорию.** Ложная уверенность хуже честного «не знаю» — юзер перестаёт проверять и аналитика портится.

Этот принцип наследуется из [[Импорт — проблемы и решения#Контекст|живого реестра проблем]] и является сквозным для всего пайплайна.

### 1.3 Детерминизм и идемпотентность

Одни и те же входные данные при одних и тех же настройках юзера должны давать один и тот же результат. Нормализация не зависит от времени, случайности, порядка обработки строк внутри батча.

### 1.4 Соответствие ключевым архитектурным решениям

Спек совместим с:
- [[Спецификация — Целевое состояние системы#10. Импорт данных]] — общее место импорта в архитектуре;
- решением об отмене `credit_payment` как `operation_type` (см. CLAUDE.md, раздел «Credit Payment Model»);
- решением о разделении `Counterparty` и `DebtPartner` (см. CLAUDE.md, раздел «Counterparty vs DebtPartner»).

## 2. Архитектура пайплайна

### 2.1 Стадии

| # | Стадия | Что делает | Может ли повторяться |
|---|---|---|---|
| 1 | Парсинг | Читает файл, строит сырые строки (raw rows) | На каждую загрузку |
| 2 | Распознавание | Определяет банк, счёт, колоночные роли | На каждую загрузку |
| 3 | Нормализация | Единая атомарная стадия: создаёт facts + decisions-группы (см. §3) | Один раз при построении preview |
| 4 | Кластеризация | Группирует по fingerprint | Один раз при построении preview |
| 5 | Матчинг правил | Применяет category/counterparty rules | Один раз при построении preview |
| 6 | Модерация (LLM) | Классифицирует кластеры низкой уверенности | По кластеру, один раз |
| 7 | Ручная проверка | Юзер в модераторе подтверждает / правит | Сколько угодно до коммита |
| 8 | Коммит | Создаёт Transaction-записи, обновляет статистику правил | Один раз (терминально) |

### 2.2 Границы ответственности

- **API layer** — только маршрутизация запросов, не содержит бизнес-логики.
- **Service layer** — оркестрация: `ImportService` вызывает `ImportNormalizer`, `ImportClusterService`, `ImportModeratorService`, `RuleEngine` в детерминированном порядке.
- **Repository layer** — все запросы к БД.
- **Pydantic schemas** — граница данных между слоями.

## 3. Нормализация: единая стадия

### 3.1 Единственная точка входа

Нормализация — **одна атомарная стадия** на строку. Внутри может быть сколько угодно проходов (v1-basic, v2-tokens, skeleton, fingerprint), но снаружи это чёрный ящик: raw row → normalized row.

### 3.2 Запуск и перезапуск

**Запускается один раз** — при построении preview импорт-сессии. Не перезапускается при:
- ручном редактировании юзером любых полей строки (описание нередактируемо, но другие поля — да);
- переключении счёта — юзер не может менять счёт строки после preview;
- коммите — данные уже готовы.

**Прозрачный перезапуск на чтение** — если версия нормализатора изменилась между preview и открытием сессии, при отображении строка пересчитывается «на лету» из raw row + текущих settings. Это не пишется в БД, сохраняется только для консистентности отображения.

**UI-инвариант (v1.5):** компоненты модератора, отображающие строку (`AttentionCardImpl`, `defaultRowState` в bulk-карточке, любые будущие row-cards), обязаны инициализировать локальный state из `row.normalized_data` приоритетно над cluster-предсказаниями (`cluster.candidate_category_id`, `cluster.hypothesis.operation_type`, и т. п.). Это касается всех полей, которые юзер может менять и/или которые сохраняются на бэке: `category_id`, `operation_type`, `counterparty_id`, `debt_partner_id`, `target_account_id`, `credit_account_id`, `principal_amount`, `interest_amount`, `debt_direction`, `invest_direction`, `credit_kind`. Cluster — это рекомендация модели, она пересчитывается на каждом `/moderation-status`-fetch'е и **не учитывает user-edit'ы по дизайну**; persisted-state в `normalized_data` — единственный источник истины после Apply / bulk-apply / commit. Без этого инварианта F5 (или любой unmount/mount) восстанавливает state из cluster, визуально откатывает user-выбор, и следующий клик «Применить» отправляет stale state, перезаписывая корректную БД-row.

**UI-инвариант про bucket'ы (v1.6):** фильтр между «Готово к импорту» и «Требуют твоего внимания» в модераторе обязан учитывать `normalized_data.user_confirmed_at` приоритетно над cluster-уровневым trust_zone. Backend ставит этот timestamp на Apply (см. §5.4 — individual confirm = full-touch signal). Если фильтр смотрит **только** на `cluster.trust_zone !== 'red'` (как в §1.2 honesty gate), то любая ready-строка с low-confidence rule после refresh снова уезжает в attention — даже если юзер её уже явно подтвердил. Юзер вынужден пере-подтверждать каждый F5. **Правило:** `user_confirmed_at` непустой ⇒ строка всегда considered confirmed (попадает в «Готово к импорту», бейдж «✓ Проверено»), независимо от trust_zone. Honesty gate §1.2 продолжает работать только для ready-строк, которые юзер **не** трогал индивидуально — для них cluster.trust_zone остаётся источником истины.

**UI-инвариант про bulk-card бейджи (v1.7):** аналогично §3.2 v1.5, но для bulk-карточек — `bulkCategoryId` и `bulkCounterpartyId` в `cluster-card.tsx` обязаны fallback'ать на consensus row-уровневых `normalized_data.category_id` / `counterparty_id`, если cluster-уровневый `candidate_category_id` пуст. После bulk-apply backend записывает выбор юзера в каждую row + создаёт `TransactionCategoryRule`, но при следующем `/moderation-status` fetch'е cluster.candidate_category_id может вернуться null (если confidence свежесозданного правила ниже cluster-rule-порога), и бейдж «Транспорт» исчезает с карточки, хотя каждая row внутри её имеет category_id=Y. **Правило consensus:** перебрать все member row, считать значение через `normalized_data.<field>`. Если все непустые значения совпадают — это и есть ответ кластера, использовать его. Если расходятся — fallback на null (юзеру дать выбрать вручную). Это не противоречит cluster-recompute'у (он остаётся для зон trust), просто дополняет его персистентностью user-edit'ов на UI-слое.

### 3.3 Причины падения нормализации

- Некорректный raw row (битая строка выписки).
- Невозможно извлечь обязательные факты (сумма, дата).
- Нет распознанного банка/формата.

Любое падение → строка в статус `error` (см. §5).

### 3.4 Часовой пояс naive datetime

Парсеры русских банков (Т-Банк, Я-Банк, Озон) отдают время операций в МСК, но без явной TZ-метки. Нормализатор обязан помечать такие naive datetime как **`Europe/Moscow`**, не как `UTC`.

Историческая причина фикса (v1.3): naive→UTC давал две каскадные проблемы.
1. **UI-сдвиг в следующий день** — операция «13.03 23:14 МСК» сохранялась как `2026-03-13T23:14:00+00:00`, в локальной TZ юзера (тоже МСК) фронт рендерил её как «14.03 02:14». В live-выписке это выглядело как «парсер ошибся датой», хотя данные исходно были корректны.
2. **Матчер не сводил трансферы между банками** — две стороны одной transfer-пары (Т-Банк 23:14 МСК и Я-Банк 23:14 МСК) сохранялись с одинаковым UTC-time, но при этом если хоть одна сторона приходила без time-компонента (некоторые экстракторы дают только `DD.MM.YYYY`), её сохраняли как `00:00 UTC` → diff между сторонами 23 ч → выходило за `MAX_DATE_DIFF_HOURS=12`.

Когда появятся не-российские банки, TZ нужно поднять на уровень `session.mapping_json.bank_timezone` (см. бэклог §14.15). До тех пор хардкод `Europe/Moscow` в нормализаторе закрывает 100% реальных выписок.

**Aware datetime не перезаписывается.** Если raw row уже содержит TZ-info (ISO с offset), нормализатор сохраняет её как есть.

## 4. Контракт нормализованных данных

### 4.1 Две группы полей

Результат нормализации делится на **строго две непересекающиеся группы**:

**Факты (facts).** Что произошло. Извлекаются из raw row однозначно, не зависят от интерпретации:
- `amount` — сумма со знаком / абсолютное значение.
- `currency` — валюта операции.
- `booking_date` — дата проведения.
- `raw_description` — исходный текст, как в выписке.
- `direction` — дебет/кредит (нормализованный).
- `source_file_row` — ссылка на исходную строку файла.
- `bank_code`, `account_id` — откуда прилетело.

Факты **не меняются** после нормализации. Даже если юзер ошибся в счёте при загрузке — он перезагружает файл в правильный счёт, это другая импорт-сессия.

**Решения (decisions).** Как интерпретировать. Вычисляются на базе facts и могут меняться при обучении системы:
- `skeleton` — нормализованное описание с плейсхолдерами (`<PHONE>`, `<CONTRACT>`, `<IBAN>`, `<CARD>`, `<PERSON>`, `<ORG>`).
- `tokens` — извлечённые идентификаторы (номера договоров, IBAN, телефоны, карты).
- `fingerprint` — хеш `(bank_code, account_id, direction, skeleton, contract_token, transfer_identifier)`.
- `predicted_category_id` — что предложили правила или LLM.
- `predicted_counterparty_id` / `predicted_debt_partner_id`.
- `predicted_operation_type`.
- `is_self_transfer_candidate` — только если оба конца известны.

### 4.2 Инвариант разделения

Код не должен смешивать эти группы в одной структуре данных. Факт не может быть перезаписан результатом правила; решение не может попасть в поле, предназначенное для факта.

### 4.3 Версионирование

`normalizer_version` хранится на уровне импорт-сессии (или отдельно по строке). При рассогласовании — прозрачный перезапуск decisions-части при чтении. Facts никогда не пересчитываются.

## 5. Статусы строк и переходы

### 5.1 Семь статусов

| Статус | Что означает | Блокирует коммит |
|---|---|---|
| `ready` | Всё распознано с высокой уверенностью, готова к коммиту | Нет |
| `warning` | Применено предсказание, но требует внимания юзера. Коммитится только после того, как юзер коснулся кластера строки — индивидуальным подтверждением или cluster-level bulk-ack (§5.4) | Да, пока кластер не подтверждён |
| `error` | Нормализация упала, данные некорректны или нарушена целостность (нераспознанный счёт, transfer без counter-account) — строка не может быть закоммичена как есть | Да (пока не перевели в `excluded` или `parked`) |
| `duplicate` | Совпадает с уже закоммиченной транзакцией, авто-скип | Нет (скипается автоматически) |
| `parked` | Юзер временно отложил («разберусь позже») | Да, но допускается перенос в следующую сессию |
| `excluded` | Юзер решил не коммитить эту строку | Нет (просто не коммитится) |
| `committed` | Строка закоммичена, создала Transaction(s) | Терминал |

### 5.2 Триггеры warning vs error

Два разных класса проблем, две разные реакции.

**`warning` — качество предсказания.** Данные целостны, мы просто не уверены в классификации. Юзер должен взглянуть, но коммит технически возможен (после cluster-level подтверждения, §5.4).

1. Правило применено, но его confidence ниже порога «молчаливого применения» (см. §6).
2. LLM не уверен в ответе (см. §7).
3. Tokens извлечены неоднозначно (два contract_token-кандидата, два transfer_identifier-кандидата).
4. Направление операции выведено контекстно, не из явного знака/колонки.

**`error` — целостность данных.** Строку нельзя закоммитить как есть, даже при согласии юзера — получится мусорная транзакция. Требуется явное решение (fix / park / exclude).

5. Счёт не распознан или распознан с низкой уверенностью (нельзя создать транзакцию без account_id).
6. Похоже на transfer, но counter-account не определён (см. §12 — transfer с одним концом запрещён).

*Было в v1.0:* триггеры 5 и 6 квалифицировались как `warning`. Переквалифицированы в `error` в v1.1 — строка, которую физически нельзя привязать к счёту, не должна иметь лёгкого пути в коммит (даже через bulk-ack).

### 5.3 Переходы

```
[parsing] ──ok──> ready | warning
          ──fail─> error

ready    ──юзер правит──> warning (если правка касается категории/контрагента — применится новое правило)
warning  ──юзер подтвердил──> ready (опционально, при явном confirm click)
error    ──юзер──> excluded | parked
error (orphan transfer) ──matcher свёл пару──> ready | duplicate (см. ниже)
parked   ──следующая сессия──> ready | warning | error (ре-нормализация)
excluded ──(терминал до коммита)──> не коммитится
duplicate ──(терминал)──> не коммитится (юзер НЕ может override)
ready|warning ──commit──> committed
```

**Допуск orphan-transfer error rows в matcher pool.** Триггер 6 в §5.2 ставит `error`, когда классификатор уверен, что строка — transfer, но counter-account не определён. Это терминальный data-integrity error для юзера, но **не** для самого matcher'а: при следующем проходе (например, после загрузки выписки контр-счёта) matcher обязан включать такие строки в pool кандидатов и попытаться свести пару. При успехе `_apply_assignments` переписывает `status` в `ready` (primary side) или `duplicate` (secondary side cross-session pair) и очищает `error_message`. Если пара по-прежнему не найдена — статус остаётся `error`, никаких регрессий.

Технически: фильтр загрузки активных кандидатов в matcher включает `status IN ('ready','warning','error')`, но пускает в pool только error-строки c `operation_type='transfer'` и пустым `transfer_match`. Прочие error-строки (нет счёта, битый amount) продолжают игнорироваться.

### 5.4 Массовые действия

**Философский принцип:** каждый кластер — отдельная единица внимания. Сквозной «commit everything» запрещён, потому что после кластеризации 600 строк превращаются в 30 кластеров — это уже не массовая операция, а 30 коротких решений. Юзер обязан коснуться каждого.

В модераторе доступно:
- **Cluster-level bulk-ack** — подтвердить предложенную категорию/контрагента для всех `warning`-строк **внутри одного кластера** (например, 50 строк «Пятёрочки» одним кликом). Засчитывается как **confirm с весом 0.5** для применённого правила (см. §10).
- **Cross-cluster bulk-ack ЗАПРЕЩЁН.** Нет кнопки «подтвердить все warning'и в выписке сразу». Юзер обязан пройти по кластерам.
- **Exclude all errors** — перевести все `error` в `excluded` (одним действием, потому что `error`-строки уже не дадут транзакций, коммит для них невозможен — это чисто операция очистки от шума).
- **Park all parked-candidates** — убрать отложенные строки из текущего коммита.

**Кнопка «коммит всей сессии» не трогает warning-строки, которые не были подтверждены на уровне кластера.** Они переносятся в `parked` или остаются в сессии до следующего захода.

**Bulk-кластеры transfer'ов** (например, серия внутрибанковских переводов на одном договоре) — особый случай: они представлены **в двух местах UI одновременно** — как обычная bulk-карточка с кнопкой массового подтверждения И как строки в виджете «Переводы и дубли» для сводной картины. Bulk-кластеры с обычными покупками (Пятёрочка, Вкусно и Точка) показываются ТОЛЬКО в bulk-секции — повторение в виджете для них шум. Различение по `operationType === 'transfer' && targetAccountId != null`. См. [import-moderation-panel.tsx#isBulkNonTransferRow](file:///d:/projects/financeapp/frontend/components/import/import-moderation-panel.tsx).

**Bulk-кластеры исключают `status=duplicate`.** Терминальные duplicate-строки (§8.3) уже выводятся в виджет «Переводы и дубли» и не могут быть закоммичены. Тащить их в bulk-карточку на ack — пустой шум, ломающий счётчик «9/9». Фильтр в `build_bulk_clusters` отрезает их **по двум независимым признакам**:
1. `status == 'duplicate'` — покрывает строки, которые `_find_transfer_pair_duplicate` пометил duplicate в build_preview БЕЗ записи `transfer_match` (находит уже-импортированную парную транзакцию по `account_id`/`amount`/`transaction_date` ±2 дня).
2. `transfer_match` is set — покрывает строки, помеченные cross-session matcher'ом или `_detect_committed_duplicates`.

Один признак не покрывает другой — нужны оба, иначе duplicate-строки протекают в bulk UI (наблюдалось в кластере «Внутрибанковский перевод с договора 0504603705» 9/9 на скрине от 2026-04-25).

## 6. Правила и confidence

### 6.1 Два типа правил

**Specific** — по идентификатору. Привязано к concrete token: номер договора, IBAN, terminal ID, ФИО контрагента. Высокая точность, низкое recall.

**General** — по скелетону. Привязано к нормализованному шаблону описания. Среднее recall, вариативная точность.

*Устаревшие scope-варианты (`bank`, `legacy_pattern`) мигрируют в одну из двух категорий. См. бэклог §14.7.*

### 6.2 Правила на категорию vs на контрагента

**Независимые сущности с независимыми счётчиками.** Обоснование: категория меняется от операции к операции (один магазин — разные категории), контрагент может быть одинаковым для десятков разных fingerprint'ов.

- **Category rule** — per-fingerprint. Один fingerprint → одно правило категории.
- **Counterparty binding** — per-identifier. Один идентификатор → один контрагент. Реализуется через `CounterpartyFingerprint` / `CounterpartyIdentifier`.

**Rejection по одному не трогает другое.** Юзер поменял категорию → rejection только category rule. Юзер поменял контрагента → rejection только counterparty binding.

### 6.3 Кластер vs контрагент: два уровня группировки с разной политикой по направлению

Кластеры и контрагенты — **два разных уровня агрегации**, и они трактуют направление операций по-разному.

**Кластер (low-level, fingerprint-based)** — делится по направлению. Fingerprint включает `direction` (см. §4.1), поэтому один и тот же скелетон в income и в expense даёт **два разных fingerprint'а**, два разных кластера, два разных правила категории. Это технически правильно: «Внутрибанковский перевод с договора N» income (приход) и expense (расход) — разные паттерны, могут попадать в разные категории.

**Контрагент (high-level, user-bound)** — НЕ делится по направлению. Один контрагент = одна карточка, независимо от того, в каком направлении идут операции с ним. Кейс-якорь: «Отец» как и отправитель денег (income), и получатель денег (expense) — это **одна и та же родственная связь**. Делить «Отец-приходы» от «Отец-расходы» на отдельные карточки в bulk-section было бы пользовательской бессмыслицей.

Технически, в `_group_by_counterparty` bucket-ключ — **только `counterparty_id`** (раньше был `(counterparty_id, direction)`). Все fingerprint-кластеры, привязанные к одному контрагенту через `CounterpartyFingerprint` или `CounterpartyIdentifier`, схлопываются в одну группу. Поля карточки:

| Поле | Семантика |
|---|---|
| `direction` | Доминирующее направление по count членов. При равенстве — `expense` (исторический дефолт; refund-карточки до v1.4 рендерились как expense). |
| `is_mixed_direction` | `True`, если есть хотя бы по одному кластеру в обеих направлениях. UI рендерит mixed-mode badge. |
| `count` | Сумма count всех members. |
| `total_amount` | Net по доминирующему направлению: `expense_sum - income_sum` (если `direction='expense'`) или `income_sum - expense_sum` (если `direction='income'`). Может быть отрицательным, если non-dominant сторона больше — это сигнал юзеру, что фактический баланс по контрагенту обратный «средней по count» картине. |

**Refund — частный случай, не отдельная ветка.** Раньше (v1.0–v1.3) refund-income специально схлопывался в expense-карточку через отдельную проверку `all(is_refund)` в членах. С v1.4 refund-income — просто income-кластер, привязанный к тому же контрагенту, что и expense-кластер. Получает то же поведение, что и любой другой income-binding: попадает в один bucket, считается в `income_sum`, выгружается из чистой суммы expense. Old refund-fold blob удалён.

**Direction-binding в правилах.** Counterparty binding (`CounterpartyFingerprint`) — это `fingerprint → counterparty_id`. Fingerprint содержит direction, поэтому биндинг автоматически направленный (один на income-fingerprint, другой на expense-fingerprint). Когда юзер вручную добавляет income-кластер «Отец-приходы» в существующего контрагента «Отец», создаётся отдельный `CounterpartyFingerprint` для income-fingerprint'а — и оба биндинга (income + expense) при следующем сборе кластеров `_group_by_counterparty` слепляются в одну карточку.

**Что НЕ меняется** этим разделом: §6.1 (specific vs general rules), §6.4 (confirms/rejections counters), §10 (commit-time updates), все инварианты §12. Это чисто UI/grouping-уровень изменения.

### 6.3 Confidence: что учитывается

Confidence правила определяется соотношением **confirms** и **rejections**, а также абсолютным размером выборки. Возраст правила сам по себе НЕ декеит confidence — реальность проверяется на следующем применении.

### 6.4 Что считается confirm / rejection

**Confirm** (с весом 1.0 или 0.5):
- Строка закоммичена со статусом `ready` и финальная категория совпадает с предсказанной → **confirms += 1.0**.
- Строка закоммичена со статусом `warning` через cluster-level bulk-ack (§5.4), финальная категория совпадает с предсказанной → **confirms += 0.5**. Вес 0.5 отражает, что юзер коснулся кластера, но не индивидуальной строки.

**Rejection:**
- Строка закоммичена, финальная категория отличается от предсказанной → **rejections += 1** для старого правила **и** формируется/обновляется новое правило с новой категорией (confirms = 1.0 для нового).

**Не сигнал:**
- Строки в статусе `excluded`, `parked`, `duplicate`, `error` не влияют на счётчики.

### 6.5 Три уровня поведения по confidence

| Уверенность | Статус строки | Поведение |
|---|---|---|
| Высокая | `ready` | Применяем молча |
| Средняя | `warning` | Применяем, но подсвечиваем |
| Низкая | — | НЕ применяем, отправляем в LLM-модератор |

**Отдельное состояние правила — `inactive`** (rejections > confirms). Такое правило не участвует в матчинге, но хранится в истории для возможного ревью.

### 6.6 Органическое разрешение конфликтов

Если юзер многократно правит предсказание одного и того же fingerprint — формируется конкурирующее правило. Оба живут параллельно:
- старое R с растущими rejections,
- новое R' с растущими confirms.

Побеждает то, у кого выше confidence на следующем применении. Со временем R уходит в `inactive`, R' становится каноничным. Специальная логика «разрешения конфликтов» не нужна.

## 7. Модератор и LLM

### 7.1 LLM — верхний барьер автоматизации

В LLM попадают кластеры, где:
- Правила не нашли совпадения.
- Правила нашли, но confidence ниже порога применения.

LLM предлагает: категория, контрагент, operation_type. Решение применяется **к целому кластеру** (один fingerprint → один ответ LLM → применяется ко всем строкам кластера).

### 7.2 Доверие самооценке LLM

**Частичное.** Самооценка LLM используется как **один из** сигналов, комбинируется с:
- размером кластера,
- наличием пересекающихся правил,
- согласованностью контрагента между банками,
- account type (см. §9.4) и другим контекстом.

Полное доверие опасно (LLM галлюцинирует с высокой confidence); полное недоверие — выбрасывает реальный сигнал.

### 7.3 Поведение при низкой уверенности LLM

Все три сценария ведут к **ручной проверке юзером** (статус `warning`), но с разной начинкой UI:

| Сценарий | Что делаем |
|---|---|
| LLM ответил, но неуверен | Применяем самый вероятный ответ как дефолт. Форма строки: «подтверди/измени предложение» |
| LLM выдал невалидный ответ (категория не существует, пустой контрагент, unparseable) | Поля пустые. Форма строки: «заполни сам» |
| LLM не ответил (таймаут, API упал) | Retry. После ретраев — как и выше, поля пустые, юзер заполняет |

### 7.4 Кластеризация: один ответ на fingerprint

LLM видит кластер целиком, не отдельные строки. Если fingerprint одинаковый → строки **должны** получать одинаковый ответ. Расщепление внутри кластера = баг fingerprint'а, лечится на нижнем уровне, не апелляцией к LLM.

### 7.5 Независимость ответов между кластерами

LLM обрабатывает кластеры **независимо**. Не имеет памяти между ответами внутри одной сессии. Консистентность имён контрагентов обеспечивается **пост-процессингом** (fingerprint → canonical name), не памятью LLM.

### 7.6 Что НЕ делает LLM

- Не принимает решения по фактам (сумма, дата, счёт) — только по decisions.
- Не рассчитывает split credit payment (см. §9).
- Не декеит правила по возрасту.

### 7.7 Симметричная классификация transfer-keyword'ов

Если в описании одной из сторон выписки есть однозначный transfer-маркер — `«перевод между счетами»`, `«перевод на свой счёт»`, `«пополнение своего счёта»`, **`«внутрибанковский перевод»`**, **`«внутренний перевод»`**, **`«межбанковский перевод»`**, `«с карты на карту»` — классификатор обязан вернуть `operation_type='transfer'` с **STRONG**-уверенностью (≥0.9), не WEAK.

Жирные фразы в списке выше — банковские термины. До v1.2 они лежали в WEAK (0.7) и валились ниже порога downgrade'а 0.88, из-за чего одна сторона перевода (например, Тинькоф Дебет: «Внутренний перевод на договор …») классифицировалась как `regular`, а вторая (Тинькоф Сплит: «Внутрибанковский перевод с договора …») — как `transfer`. Ассиметрия ломала [transfer_matcher_service.py](file:///d:/projects/financeapp/app/services/transfer_matcher_service.py): он мог свести только две стороны, у которых обе уже размечены transfer'ом. Теперь обе стороны симметричны.

Контр-индикаторы (явно НЕ transfer): «оплата покупки», «погашение кредита», «оплата услуг», имена операторов (Megafon/MTS/…) — см. список `_ANTI_TRANSFER_KEYWORDS`. Они переводят строку обратно в regular даже при наличии «перевод» в описании. См. §12.8.

## 8. Дедупликация

### 8.1 Определение дубликата

Строка помечается `duplicate`, если существует уже закоммиченная транзакция с совпадением по ключу:

```
(account_id, booking_date, amount, skeleton)
```

**Важно:** ключ включает **skeleton**, не raw description. Это значит, что «СБП ВХОДЯЩИЙ» и «ПЕРЕВОД ОТ ИВАНОВ И. И.» — разные skeleton → не дубликаты друг друга (даже если это одна и та же логическая операция). Случаи «апгрейд описания банком» — известное ограничение, лечится юзером руками.

### 8.2 Дедуп проверяется ТОЛЬКО против коммитов

- ❌ Не против других строк **в этом же импорте** — внутри импорта дублей не бывает.
- ✅ Только против уже закоммиченных `Transaction`-записей того же юзера.

### 8.3 Терминальный авто-скип

Статус `duplicate` — **терминальный**. Юзер НЕ может его override'нуть в модераторе. При коммите строки со статусом `duplicate` скипаются автоматически.

Если юзер обнаружил что дубликат ложный (редкий кейс ретроактивного добавления банком) — он добавляет операцию через обычную форму «новая транзакция», минуя импорт.

### 8.4 Визуальное поведение

Дубликаты по умолчанию свёрнуты в UI модератора (не засоряют поток). Юзер может раскрыть и увидеть каких закоммиченных транзакций они зеркалят — только для информации, не для действий.

### 8.5 Дубликат vs Зеркало пары

В виджете «Переводы и дубли» каждая строка со статусом `duplicate` принадлежит к одной из двух категорий:

| Категория | Признак | Поведение в UI | Поле `transfer_match.is_secondary` |
|---|---|---|---|
| **Mirror match** | committed-tx живёт на **другом** счёте, наш счёт совпадает с её `target_account_id` | «Партнёр → этот счёт» с именем партнёрского счёта | `True` |
| **Real duplicate** | committed-tx живёт на **том же** счёте, что row | «Дубль · другая сессия» (приглушённо) | `False` |

Без этого различения UI отрисовывает self-loop вида «Тинькоф Дебет → Тинькоф Дебет» — это всегда баг и означает, что matcher неверно определил, какая из двух сторон transfer-пары является «партнёрской» для активной row.

**Реализация** ([transfer_matcher_service._detect_committed_duplicates](file:///d:/projects/financeapp/app/services/transfer_matcher_service.py)):
- Если `tx.account_id != row.account_id` → mirror, partner = `tx.account_id`.
- Если `tx.account_id == row.account_id` → real duplicate, partner = `tx.target_account_id` (может быть None).

### 8.6 Skeleton обязателен в дедуп-ключе для duplicate-detection

Логика `_detect_committed_duplicates` ищет по индексу `(account_id, direction, amount)` в окне ±2 дня. Без обязательной проверки **совпадения skeleton** matcher ложно склеивал две независимые операции, случайно совпавшие по этим четырём параметрам:

- СБП-приход 5000 ₽ от внешнего отправителя (skeleton: «пополнение система быстрых платежей»)
- внутрибанковский перевод 5000 ₽ с другой карты в тот же день (skeleton: «внутренний перевод на договор <CONTRACT>»)

Суммы и даты совпали, направление противоположное счетам — но это разные операции. Skeleton'ы у них разные, и совпадение skeleton'а — то, что отличает повторный импорт от случайной коллизии сумм.

**Правило:** active-row может быть помечена `duplicate` через `_detect_committed_duplicates` ТОЛЬКО если у обеих сторон `skeleton` непустой и совпадает (case-sensitive). Если хотя бы у одной стороны skeleton отсутствует — fallback на `normalized_description`. См. §12.6.

### 8.7 Окно матчера трансферов: ±1 календарный день в МСК

Матчер cross-session трансферов сравнивает стороны не по фиксированному часовому бюджету, а по **разнице календарных дней в `Europe/Moscow`**. Допуск — `MAX_DATE_DIFF_DAYS = 1` (тот же день или соседний).

**Почему дни, а не часы.** Часовой бюджет (`MAX_DATE_DIFF_HOURS = 12` в v1.2 и раньше) регулярно отрезал легитимные пары вида «23:50 одного дня → 02:10 следующего», где обе стороны — одна и та же transfer-операция, но банки записали их в разные календарные сутки. Сравнение по `.date()` в МСК закрывает этот кейс (diff_days == 1) без расширения окна до 36-48 часов, которое тащило бы amount-coincidence false positives на сутки шире.

**Скоринг (внутри окна):**

| Условие | Score |
|---|---|
| `diff_seconds == 0` | 1.00 |
| `diff_seconds ≤ 60` | 0.97 |
| `diff_seconds ≤ 1 ч` | 0.93 |
| Тот же день, > 1 ч | 0.88 |
| Соседний день | 0.80 |
| `diff_days > 1` | 0.0 (отбрасывается до скоринга) |

Часы остались в шкале как тай-брейкер: при равном амаунте/направлении/keyword'ах побеждает пара, у которой стороны ближе по времени.

**Конвертация в `Europe/Moscow` обязательна** перед `.date()` — committed-tx могут хранить TZ-aware datetime в UTC; без явного `astimezone(_BANK_TZ)` UTC-дата может на 3 часа отличаться от МСК-даты, и пара случайно вылетит из окна.

## 9. Credit payment при импорте

### 9.1 `credit_payment` как operation_type отменён

Не создаём транзакции с `operation_type='credit_payment'`. См. CLAUDE.md, раздел «Credit Payment Model».

### 9.2 Три подкласса кредитных счетов

В метаданных счёта (`account_type`, см. §14.5):

| Подкласс | Поведение |
|---|---|
| `loan` (кредит наличными, ипотека, автокредит) | Перевод на счёт = **платёж по кредиту**, требует split на проценты + тело |
| `credit_card` (кредитная карта) | Перевод на счёт = **обычный transfer**, без split. Проценты — отдельной строкой на выписке карты |
| `installment_card` (карта рассрочки) | Аналогично `credit_card` |

### 9.3 Split для loan-счетов

Платёж на loan-счёт в модераторе открывает **split-форму** с двумя обязательными полями:
- `interest_amount` — сумма процентов
- `principal_amount` — сумма тела долга
- валидация: `interest_amount + principal_amount = total_amount`

При коммите из **одной импорт-строки создаются две транзакции**:

**(1) Interest expense**
- `type = 'expense'`
- `operation_type = 'regular'`
- `category_id` = системная категория «Проценты по кредитам»
- `credit_account_id` = loan-счёт
- `is_regular = True`
- идёт в Базовый Поток

**(2) Principal transfer**
- `type = 'expense'`
- `operation_type = 'transfer'`
- `target_account_id` = loan-счёт
- `affects_analytics = False`
- снижает долг, НЕ в расходах

### 9.4 Источник дефолтов — график платежей

**Параметры кредита** (срок, ставка, сумма, тип, дата выдачи, ежемесячный платёж) юзер указывает при создании loan-счёта. Справочно — для отображения «ожидаемого остатка долга».

**График платежей** (опционально) юзер загружает PDF'ом из банка при создании loan-счёта. Парсер извлекает таблицу: 36+ строк `(payment_date, total, interest, principal, remaining_balance)`.

**Дефолты в split-форме** берутся из соответствующей строки графика для даты платежа.

**Эволюция графика.** Если фактический split от юзера отличается от предсказанного графиком — система **заменяет строку графика на фактические значения**. График самоочищается, постепенно замещая предсказания правдой.

### 9.5 Досрочное погашение

Split-форма содержит чекбокс «досрочное погашение». При активации:
- Обязательное поле: **тип досрочного** — «сокращение срока» ИЛИ «сокращение ежемесячного платежа».
- Система пересчитывает оставшиеся строки графика по стандартной формуле аннуитета.
- Дефолты для следующих месяцев строятся из пересчитанного графика.

### 9.6 Формулы для split'а не используются

Автоматический split без ручного ввода **не реализуется**. Причины:
- Банки не публикуют помесячную методологию расчёта.
- Стандартная формула аннуитета расходится с реальными цифрами даже для чистых случаев без досрочных, страховок, плавающих ставок.
- Эмпирически: в бизнес-учёте требуется ручная проверка каждый месяц.

Что точно — **годовая сумма процентов** (она детерминирована параметрами). Помесячный split — приблизительный с правкой при необходимости.

### 9.7 LLM не используется для расчёта split'а

Может использоваться только для **парсинга таблицы графика платежей**, если PDF-формат сложный. Не для вывода методологии банка.

### 9.8 Warning при загрузке main без кредитного счёта

Если в выписке main-счёта есть перевод на счёт любого из трёх кредитных подклассов, но выписка с самого кредитного счёта в текущем импорте отсутствует → на уровне импорт-сессии показываем warning: «ты загружаешь платёж по кредитному продукту, но не загрузил выписку с него самого. Проценты не попадут в аналитику».

Это **мягкое правило**, не блокирует коммит.

## 10. Обновление статистики правил

### 10.1 Применение ≠ подтверждение

На этапе preview генерации правило применяется → в строке сохраняются:
- `applied_rule_id`
- `predicted_category_id`
- `predicted_counterparty_id` (если соответствующий binding сработал)

**Счётчики правил НЕ обновляются на этапе применения.** Только на коммите.

### 10.2 Четыре кейса на коммите

Для каждой коммитящейся строки:

| Кейс | Условие | Действие |
|---|---|---|
| A | Строка была `ready`, final_category = predicted | R.confirms += 1.0 |
| B | Строка была `warning`, подтверждена через cluster-level bulk-ack (§5.4), final_category = predicted | R.confirms += 0.5 |
| C | Строка имела applied_rule_id, final_category ≠ predicted | R.rejections += 1, создаём/обновляем R' для (fingerprint → final_category), R'.confirms = 1.0 |
| D | Строка не имела applied_rule_id, юзер указал final_category | Создаём новое R'' для (fingerprint → final_category), R''.confirms = 1.0 |

**Прим.:** индивидуальный commit warning-строки без правки (юзер раскрыл строку, посмотрел, подтвердил предсказание) засчитывается как кейс A с весом 1.0 — это полноценный контакт с конкретной строкой, а не скользящее подтверждение.

Для counterparty binding — полностью аналогичная схема, отдельными счётчиками.

### 10.3 Что НЕ меняет счётчики

- Строки `excluded`, `parked`, `duplicate`, `error` — не трогают счётчики.
- Промежуточные правки юзера («поменял → вернул как было») — не трогают счётчики. Учитывается **только финальное состояние на момент коммита**.
- Изменение других полей (сумма, дата) — не влияет на rule stats. Правила предсказывают только категорию/контрагента.

### 10.4 Новое правило R' получает confirms = 1.0

Стартует с сигналом, а не с нуля. Юзер явно выбрал эту категорию при коммите — это валидное первое подтверждение.

### 10.5 Мета-правила отдельный тип

Правило «этот fingerprint = credit payment, открыть split-форму» — **отдельный тип** (meta-rule), не смешивается с category rule. Его confirms/rejections обновляются на коммите split'а: реально заполнил — confirm, отменил и выбрал другую трактовку — rejection.

### 10.6 Линковка active-стороны к committed orphan вместо создания дубля

Сценарий: пользователь сначала закоммитил выписку счёта A. На момент того коммита второй стороны (счёт B) ещё не было, и transfer-транзакции A ушли как **orphan transfer** — `target_account_id IS NULL` и `transfer_pair_id IS NULL` — либо как `regular income/expense`, если классификатор не распознал. Затем пользователь импортирует выписку счёта B, transfer_matcher свёл строки B с committed-транзакциями A и записал `matched_tx_id` в `normalized_data.transfer_match`.

**Правило коммита:** при коммите active-стороны с `transfer_match.matched_tx_id is not None` система **линкует** активную транзакцию к существующей committed-транзакции вместо вызова `_create_transfer_pair`. Без этой защиты обычная commit-логика создаёт ВТОРУЮ зеркальную транзакцию на target-счёте, и баланс target-счёта получает double-credit.

**Алгоритм** ([import_service._link_transfer_to_committed_pair](file:///d:/projects/financeapp/app/services/import_service.py)):
1. Создаётся ТОЛЬКО active-сторона (B) как `transfer`.
2. Достраивается committed-сторона (A): `target_account_id`, `transfer_pair_id`, `operation_type='transfer'`, `affects_analytics=False`.
3. Balance применяется ТОЛЬКО к active-счёту (B). Баланс A не трогается — он уже был учтён при предыдущем коммите orphan'а.

**Защиты:** если `committed_tx.transfer_pair_id` уже не NULL (уже спарен), или сумма не совпадает, или направление одинаковое — линковка отказывает (`return None`), и caller делает fallback на обычное создание пары через `_create_transfer_pair`.

См. §12.1 (transfer требует оба счёта — после линковки оба известны) и §12.4 (committed orphan, который был `regular income`, после достройки получает `affects_analytics=False`, чтобы не двоиться в income-аналитике).

## 11. Ошибки привязки

### 11.1 Три стратегии

| Стратегия | Когда |
|---|---|
| **Swallow** (проглотить) | Удалённая категория / удалённый счёт в binding-е. Правило молча идёт в `inactive`, строка идёт дальше по пайплайну (LLM/юзер). Самоисцеление через обычный путь обучения |
| **Retry** (попытка починить) | Mерж контрагентов. При merge A→B все `CounterpartyFingerprint`/`CounterpartyIdentifier` указывающие на A **активно** переставляются на B в той же транзакции — не лениво, не при следующем матче |
| **Fail** (явная ошибка) | Системная категория недоступна (баг миграции). Строка в `error`, алерт в админку, юзер не должен чинить |

### 11.2 Принцип разделения

- **Ошибки конфигурации/данных юзера → swallow** + самоисцеление.
- **Ошибки системной целостности → fail** + алерт, невидимо для юзера.
- **Retry только с явным target'ом** для ретрая.

### 11.3 Фоновая чистка

Правила в статусе `inactive` (rejections > confirms, или помеченные broken в swallow-сценарии) чистятся фоновым джобом. Не удаляются жёстко сразу — хранятся для возможного ревью юзером.

## 12. Жёсткие инварианты

### 12.1 Transfer требует оба счёта

Транзакция с `operation_type = 'transfer'` обязана иметь:
- `source_account_id` — известный счёт юзера,
- `target_account_id` — известный счёт юзера.

Строка импорта, похожая на transfer, но где хотя бы один из счетов не определён → **`error`** (см. §5.2, триггер 6), не автоприменяется. Юзер решает: `excluded` / `parked` / ручная правка с явным указанием счёта.

### 12.2 Debt vs Counterparty disjoint

`operation_type = 'debt'` ⇒ `debt_partner_id` обязателен, `counterparty_id` запрещён. Любой другой `operation_type` ⇒ `debt_partner_id` запрещён. См. CLAUDE.md, раздел «Counterparty vs DebtPartner».

### 12.3 credit_payment не создаётся

`operation_type = 'credit_payment'` — запрещено. См. §9.1.

### 12.4 credit_disbursement не в доходах

`operation_type = 'credit_disbursement'` ⇒ `affects_analytics = False`. Все агрегации доходов дополнительно фильтруют `operation_type != 'credit_disbursement'`.

### 12.5 Facts vs decisions

Поля группы facts (см. §4.1) никогда не перезаписываются результатами decisions-части. Код, пишущий в amount/date/raw_description/direction от имени правила или LLM — баг.

### 12.6 Дедуп ключ включает skeleton

Не «any description», а именно skeleton. См. §8.1.

### 12.7 Нормализация не перезапускается при правках юзера

См. §3.2. Версионная миграция — через transparent re-run на чтение, не write.

### 12.8 Симметричная классификация двух сторон одного перевода

Если две stmts (выписки) описывают **одну и ту же** межсчётную операцию (например, Тинькоф Дебет «Внутренний перевод на договор N» и Тинькоф Сплит «Внутрибанковский перевод с договора N» — обе на одну сумму, в одну дату, противоположные направления), их `operation_type` обязан быть **одинаковым** = `transfer`.

Ассиметричная классификация ломает любой автоматический matcher: он может свести только две стороны, у которых обе уже размечены transfer'ом. Поэтому keyword-список STRONG-классификатора в §7.7 должен быть достаточно полным, чтобы покрыть формулировки описаний для обоих банков пары.

См. §7.7. Список STRONG-фраз — расширяется по мере появления новых форм выписок; всякая ассиметрия двух сторон одного логического перевода — баг классификатора, не данных.

### 12.9 Linking, не creation, при наличии matched_tx_id

При коммите active-row с `transfer_match.matched_tx_id` запрещено создавать новую зеркальную транзакцию на target-счёте. Линковка — единственная корректная операция. См. §10.6.

Нарушение даёт double-credit на балансе target-счёта и две независимые orphan-транзакции вместо одной правильной пары.

## 13. Глоссарий

| Термин | Определение |
|---|---|
| **Raw row** | Одна строка исходной выписки до нормализации |
| **Normalized row** | Результат нормализации — факты + decisions |
| **Facts** | Данные, полученные напрямую из raw row. Не меняются |
| **Decisions** | Выводы пайплайна: skeleton, tokens, fingerprint, предсказания. Могут обновляться |
| **Skeleton** | Нормализованное описание с плейсхолдерами (`<PHONE>`, `<CONTRACT>`, …) |
| **Token** | Извлечённый идентификатор — номер договора, IBAN, телефон, карта |
| **Fingerprint** | Хеш `(bank_code, account_id, direction, skeleton, contract_token, transfer_identifier)` |
| **Cluster** | Группа строк с одинаковым fingerprint |
| **Rule** | category rule ИЛИ counterparty binding. См. §6 |
| **Meta-rule** | Правило типа «этот fingerprint = credit payment». См. §10.5 |
| **Barrier / барьер** | Уровень пайплайна который может классифицировать. См. §1.1 |
| **Bulk-ack** | Коммит warning-строк пачкой без индивидуального review. Даёт confirm с весом 0.5 |
| **Loan / credit_card / installment_card** | Три подкласса кредитных счетов. См. §9.2 |
