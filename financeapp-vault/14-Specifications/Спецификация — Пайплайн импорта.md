- v1.20 (2026-05-03, продолжение 2) — **четырёхветочная модель `_detect_committed_duplicates` + reset legacy-меток.**

  Триггер: live-кейс 2026-05-03. Юзер импортировал новую выписку Т Банк кредитки (session 150), и 10 строк (включая «Оплата услуг mBank.MegaFon» и «Внутренний перевод на договор 5452737298») были помечены как `duplicate · уже есть в Т банк дебет`, хотя ни одна из них не является дублем — ни по контракту (5452737298 ≠ существующего 0504603705), ни по описанию (MegaFon — не перевод). SQL-аудит показал: все 10 строк имеют `transfer_match.match_branch IS NULL` и `match_source='committed_tx_duplicate'`, причём один и тот же `tx_id` фигурирует partner'ом для двух разных row — нарушение greedy 1-to-1, которое в v1.20 уже защищено через `used_tx_ids`. Это **отпечаток до-v1.20 матчера**: метки записаны старой логикой и зависают навсегда, потому что текущий `_reset_cross_session_secondary_duplicates` фильтрует только по `match_source='cross_session'` и не трогает legacy `committed_tx_duplicate` без `match_branch`.

  **§8.5 (расширен) — четырёхветочная модель.** Полностью описана в §8.5 (была введена ранее, но история версий её явно не фиксировала). Ключевое отличие от v1.19: skeleton перестал быть obligatory-key и стал per-branch confirmation-сигналом (§8.6); income-mirror отбракован безусловно (ветка C); mirror-expense сужена до credit-target счетов с credit-keyword (ветка D, §8.9).

  **§8.12 (новый) — reset legacy duplicate-меток.** При rollout v1.20 в БД остаются строки с `status='duplicate'` + `transfer_match.match_source='committed_tx_duplicate'` + `transfer_match.match_branch IS NULL`. Это маркер «помечено старой логикой, не переоценено новой». Расширенный метод `_reset_legacy_committed_duplicates` (запускается в начале `match_transfers_for_user`, до `_detect_committed_duplicates`) сбрасывает такие row в `status='warning'` и удаляет `transfer_match` из `normalized_data_json` — следующий проход matcher'а оценит их по актуальной четырёхветочной модели. Без этого метода live-юзеры с старыми session'ами видят залипшие ложные дубли.

  **§20.3 (новый) — known risk о rollout-меток.** Зафиксирован сам класс проблемы: при апгрейде detection-логики (4-branch model, greedy 1-to-1, identifier guard) старые duplicate-метки в БД не переоцениваются автоматически, потому что статус `duplicate` — терминальный (§8.3). Решение для будущих rollout'ов — добавлять explicit reset-метод по новому маркеру в `match_branch` или версии алгоритма.

- v1.20 (2026-05-03, продолжение) — **закрытые счета + history-based orphan-transfer hint.**

  Триггер: бизнес-кейс с закрытыми счетами. Когда юзер импортирует выписку Сбера 1 апреля – 1 мая, в которой есть строка «Поступление с карты Тинькофф +50 000 ₽», но Тинькоф-счёт уже закрыт (выписку не получить) — раньше matcher не находил пару, `_escalate_orphan_transfers` слепо демоутил строку до `regular`, и она шла в аналитику как обычный доход, портя savings rate.

  **Новый §13 — Account state machine.** `Account.is_closed: bool` + `Account.closed_at: date | None` (миграция 0057). Закрытый счёт остаётся в БД с историей, но скрыт из активных списков (sidebar, dashboard, форма создания транзакции). Виден в моде- раторе (target/source dropdown), фильтрах транзакций, странице «Все счета» в свёрнутой секции «Закрытые счета». Endpoints `POST /accounts/{id}/close` и `POST /accounts/{id}/reopen`; `GET /accounts?include_closed=true` для включения в выдачу.

  **§5.2 (расширен) — history-based orphan-transfer hint.** Перед демоутом orphan transfer'а в `regular` (текущая v1.9 логика) `_escalate_orphan_transfers` загружает до 20 последних committed-транзакций с тем же fingerprint. Если **≥3** таких транзакций И **≥80%** из них имеют `operation_type='transfer'` — оставляет `operation_type='transfer'` и подсказывает наиболее частый target из истории через `suggested_target_account_id`/`suggested_target_account_name`/`suggested_target_is_closed`/`suggested_reason`. Иначе — fall through на v1.9 demote-path. Закрытый target (`is_closed=true`) — валидное значение подсказки.

  **§4.1 (расширен) — поля decisions.** Добавлены `suggested_target_account_id`, `suggested_target_account_name`, `suggested_target_is_closed`, `suggested_reason` (формат `transfer-history N/M`). Появляются ТОЛЬКО при выполнении условий §5.2 history-check; очищаются при `update_row(action='confirm')`.

  **Денормализация fingerprint** (миграция 0058): `Transaction.fingerprint` (indexed) скопирован из `ImportRow.normalized_data_json.fingerprint` на бэкфилле и заполняется на коммите импорта. Закрывает потребность history-hint в быстром lookup'е по fingerprint без JOIN через `created_transaction_id`.

  **Frontend.** `OrphanTransferHint` (`frontend/components/import-redesign/orphan-transfer-hint.tsx`) — баннер «По истории это перевод с/на «X (закрыт)»» с кнопками «Подтвердить / Изменить счёт / Это не перевод». Bulk-кнопка в `ClusterModal` для случая, когда все строки кластера разделяют один `suggested_target_account_id`. Account-selector в моде- раторе (`AttentionFeed`, `EditTxRazvorot`, `ClusterModal`, `SplitModal`) подгружают закрытые счета через `getAccounts({ includeClosed: true })`.

  **Тесты.** `tests/test_account_close.py` (9), `tests/test_orphan_transfer_history_hint.py` (8), `tests/test_bulk_apply_target_account.py` (3). Покрыты: закрытие/открытие, валидации, list filtering, history-ratio threshold ≥3/0.8, weak history demote, no history demote, target deleted edge case, closed target works, user_confirmed_at skip, bulk-apply schema, row editor target-stamp.

- v1.20 (2026-05-03) — **переписан `_detect_committed_duplicates` под четырёхветочную модель + добавлен fee-aware suspect-pair pass.**

  Триггер: live-аудит сессий 153/154 у pavel.shokin1991@yandex.ru обнаружил три класса false-positive дубликатов (ATM-снятие 42 000₽ ложно как mirror transfer'а; «Досрочное погашение 8845» 20 840₽ ложно как mirror transfer'а другого договора 5422638063; два ряда 200₽ ложно matched к одному и тому же phantom'у вместо closest-time tie-break). Корень — content-blind mirror-индекс с conditional-on-`is_mirror` guard'ами, наслоившимися пятью слоями реакции на bug-репорты.

  **Архитектурная замена (см. §8.5–§8.10).** Вместо одного индекса `(account, direction, amount)` с теневыми mirror-ключами и conditional-цепочкой guard'ов теперь чёткая ветвящаяся модель: same-account vs mirror, mirror отдельно для income (универсально reject) и для expense (узко активируется только для credit-target с credit-keyword). Identifier-discrimination как first-class сигнал (mismatch одного типа → hard reject в любой ветке). Anti-transfer keyword guard (`atm`, `банкомат`, `cash withdrawal`, `комиссия`, `inkasso`) → hard reject mirror-ветки. Greedy 1-to-1 assignment по сортировке `(closest-time ASC, identifier-match DESC, skeleton-similarity DESC)` с `used_tx_ids` set'ом — закрывает кейс «один phantom как partner для двух разных row'ов».

  **Новый второстепенный pass — `FeeMatcherService` (§8.10).** После основного matcher'а проходит по неcмэтченным парам и предлагает «transfer with fee» suspect-pair, когда `b.amount < a.amount`, дельта в пределах `min(5%, 500₽)`, time ≤ 60 sec, transfer-keyword'ы с обеих сторон. Не auto-trust — обязательное подтверждение юзером в UI; на confirm создаёт transfer-пару на меньшую сумму + отдельную expense-транзакцию комиссии в категорию «Банковские комиссии».

  **Spec-инварианты:** §8.6 (skeleton в дедуп-ключе) переформулирован — skeleton остаётся **confirmation-сигналом**, не obligatory-key. §12.6 актуализирован под новую модель. Новые §8.9 (mirror-expense только для credit-target с credit-keyword'ом) и §8.10 (fee-aware suspect-pair). Удалена «income-mirror reject» как избыточная (новая модель не индексирует mirror-income в принципе).

- v1.19 (2026-05-03) — **LLM-модерация полностью отключена + удалён keyword-fallback категорий**.

  **§7 (полностью переписан) — Модератор без LLM.** LLM-модератор был верхним автоматизированным барьером с момента v1.0. По итогам live-тестирования принято решение его исключить: токены сгорали без ощутимого выигрыша confidence над детерминистическим стэком (правила + bank_mechanics + counterparty/identifier-биндинги + history). LLM также вносил недетерминизм в воспроизводимость импорта. После отключения «верхним барьером» становится **history-based категоризация** в `_resolve_category` (Counter по истории counterparty / по нормализованному описанию / системные категории). Что не классифицировано детерминистически → строка идёт в attention-bucket для ручного выбора (юзер — гарантированный финальный барьер).

  **API-эндпоинты модерации возвращают 410 Gone:** `GET /imports/{id}/moderation-status`, `POST /imports/{id}/moderate`, `GET /imports/moderation-metrics`. Сервисы `ImportModeratorService` / `LLMService` / `ModerationMetricsService` остаются в репозитории как мёртвый код (Этап 3 cleanup-PR), но не вызываются ни UI, ни Celery. Celery-task `moderate_import_session` исключена из `include` в `celery_app.py`. Frontend: `getModerationStatus` / `startModeration` — стабы; `moderationQuery` в `use-active-session.ts` заменён на null-stub; polling `tmStatus` удалён.

  **§6.5 (фикс) — Удалён keyword-fallback категорий.** Код в `_resolve_category` имел финальный fallback: `CATEGORY_KEYWORD_LIBRARY` маппил подстроки в названиях категорий ("продукт" → ["market", "grocery", ...]) и автоматически назначал категорию по совпадению одного токена в описании. Из-за этого `Оплата товаров YANDEX*5399*MARKET` получала категорию «Продукты» от подстроки `market`, давая иллюзию что LLM-модератор всё ещё работает после его отключения. Fallback удалён вместе с LLM-fallback'ом — категория теперь только из правил/истории/системных маркеров. `CATEGORY_KEYWORD_LIBRARY` и `_build_category_keywords` помечены deprecated, удаление в cleanup-PR.

- v1.18 (2026-05-01) — три UI/поведенческих изменения по итогам live-тестирования модератора.

  **§6.10 + §3.2 — CounterpartyGroup без минимума строк.** Counterparty-карточка эмитируется при ≥1 live-строк в любом из связанных fingerprint'ов. Singleton-строки от известного контрагента (например, единственный перевод от «Брат» по номеру телефона) сразу показываются под карточкой «Брат» в ClusterGrid — без ожидания второго совпадения. Singleton fingerprint попадает в `aux_clusters` и возвращается в `fingerprint_clusters` — `inClusterRowIds` в AttentionFeed корректно перехватывает такую строку. Порог ≥2 строк остаётся только для standalone fingerprint-кластеров и brand-кластеров.

  **§3.2 — dominantOpType в collapsed card (UI-инвариант).** Для collapsed-карточек кластеров/контрагентов, где ВСЕ строки имеют один нон-категориальный `operation_type` (debt / transfer / investment_buy / investment_sell / credit_disbursement / credit_early_repayment), карточка показывает синий тип-бейдж («Долг», «Перевод», «Инвестиция» и т.д.) вместо «Выбери категорию». Эти операции не имеют категории по дизайну, показывать пустой слот — вводит в заблуждение. Смешанные типы (часть regular + часть debt) → показывается только категориальная логика.

  **§3.2 — attachRowToCounterparty для non-eligible строк (UI-инвариант).** Когда в cluster-modal'е все строки non-eligible (debt / transfer / credit — не участвуют в bulk-apply), но пользователь выбрал контрагента в заголовке и нажал «Готово», `handleApplyBulk` вызывает `attachRowToCounterparty` для каждой non-eligible строки. Это создаёт CounterpartyFingerprint-binding на уровне fingerprint'а (для группировки в UI), **не** выставляет `counterparty_id` в `normalized_data` строки (запрещено для debt per §12.2). После этого карточка контрагента появляется в ClusterGrid для будущих выписок с тем же fingerprint'ом.

- v1.17 (2026-05-01) — два изменения по итогам live-тестирования импорта: пороги кластеризации и поведение bulk-split.

  **§5.4 — пороги снижены.** `MIN_BULK_CLUSTER_SIZE` и `MIN_BRAND_CLUSTER_SIZE` снижены с 3 до 2. UX-обоснование: при коротких или месячных выписках мерчант с 2 транзакциями образовывал sub-threshold кластер → строки падали в attention-ленту как одиночки, лишая пользователя one-click bulk-confirm. Два одинаковых расхода у одного мерчанта — уже выраженный паттерн, который стоит одной карточки.

  **§5.4 — Cluster-Split при bulk-apply (новый инвариант).** Когда пользователь снимает галочки с части строк в bulk-модальном окне и нажимает «Подтвердить»:
  - **1–2 unchecked строки** → `detach-from-cluster` вызывается для каждой → строки попадают в attention-ленту как одиночки (прежнее поведение).
  - **≥3 unchecked строк** → `detach-from-cluster` **не вызывается** → строки остаются со своим fingerprint'ом и при следующем пересчёте кластеров образуют собственный fingerprint-кластер (если `count ≥ MIN_BULK_CLUSTER_SIZE = 2`) или brand-кластер. Это позволяет пользователю разделить смешанный кластер (напр. «Яндекс.Такси» + «Яндекс.Плюс» → два отдельных кластера) без ручного разбора каждой строки.

  Реализовано в `cluster-grid.tsx: handleApplyBulk` (проверка `unchecked.length < 3`).

- v1.15 (2026-04-30) — фиксация фактического состояния кластеризатора и confidence-механики после декомпозиции `import_service.py` (коммит 6d613fe, ветка `feature/import-moderator`). Спек ранее описывал философию confidence/правил абстрактно («высокая / средняя / низкая»), но не зафиксировал точные пороги, формулу и многоуровневую иерархию, которые уже год живут в коде. Изменения только документационные — поведение кода не меняется.

  **§5.4 (расширен) — точные пороги bulk-агрегации.** Зафиксированы константы `MIN_BULK_CLUSTER_SIZE=3` (с v1.15, ранее 5 — снижено для типичной выписки, где у мерчанта 2-4 транзакции), `MIN_TRANSFER_IDENTIFIER_CLUSTER_SIZE=2` (transfer-кластеры с phone/contract/iban/card-идентификатором — два совпадения уже паттерн), `MIN_BRAND_CLUSTER_SIZE=3` (синхронно с bulk), `MIN_FINGERPRINT_COUNT_FOR_BRAND=2` (брендовая агрегация требует ≥2 разных fingerprint'ов).

  **§6.5 (расширен) — точная формула confidence.** `confidence = base × error_ratio × evidence × identifier_match`, где `base` зависит от типа найденного правила (identifier-proven 0.99 / identifier-active 0.92 / bank 0.85 / legacy 0.78), `error_ratio = confirms/(confirms+rejections)`, `evidence` имеет singleton drag 0.05 для одиночных кластеров (отключается для proven identifier-rules через `skip_singleton_drag`), `identifier_match ∈ {MATCHED:1.0, ABSENT:1.0, UNMATCHED:0.60}`. Сводная таблица confidence-band'ов и UI-зон зелёный/жёлтый/красный приведена там же.

  **§6.7 (новый) — Auto-trust gate.** Двухкорзинная UX-модель «Готово к импорту / Требуют твоего внимания». Кластер попадает в auto-trust ТОЛЬКО когда выполнены все 5 условий: rule_source=`identifier`, identifier_match=MATCHED, rule.confirms ≥ 5, rule.rejections == 0, confidence ≥ 0.99. Bank-scope и legacy-rules никогда не дают auto-trust по дизайну — для перевода в зелёное нужна явная история подтверждений конкретного идентификатора.

  **§6.8 (новый) — Трёхуровневый поиск правил.** Кластеризатор пробует 3 типа правил в порядке убывания доверия: (1) Exact identifier rule по `(identifier_kind, identifier_value)`; (2) Bank-scope rule по `(bank_code, skeleton)`; (3) Legacy rule только по skeleton. Идентификатор-mismatch (cluster имеет idn, у rule никогда не было такого idn) мультиплицирует confidence на 0.60 — защита от «новый contract ДГ-99999 наследует категорию confirmed ДГ-12345».

  **§6.9 (новый) — Multi-layer hints.** Поверх confidence-движка работают три независимых слоя подсказок: Layer 1 `account_context` (детерминированные правила по `account_type` ∈ {credit_card, deposit, credit, installment_card} — «Пополнение кредитной карты», «Проценты по вкладу», «Погашение тела кредита»), Layer 2 `bank_mechanics` (специфические правила по `bank_code` — Яндекс Сплит/Дебет из §9.10), Layer 3 `global_patterns` (кросс-юзерный сигнал: skeleton подтверждённый ≥N юзерами в одну категорию даёт consensus-boost confidence на 0.04 за каждого юзера до 5).

Также: bulk-кластеры transfer-типа теперь дополнительно отображаются в виджете «Переводы и дубли» (UI) — §5.4.
- v1.1 (2026-04-25) — уточнение модели warning/error: триггеры «целостности данных» переквалифицированы из `warning` в `error` (блокируют коммит); bulk-ack теперь работает только внутри одного кластера, не сквозной. См. §5.1, §5.2, §5.4, §6.4, §10.2.
- v1.0 (2026-04-25) — первая утверждённая версия. Охватывает 10 узлов логики пайплайна, зафиксированных в обсуждении после разведки текущей реализации.

## 1. Философия и инварианты

### 1.1 Иерархия барьеров

Строка проходит через убывающую по автоматичности лестницу. Каждый следующий уровень подхватывает то, что не осилил предыдущий:

```
парсинг → нормализация → fingerprint/skeleton → правила/история/bank_mechanics → пользователь
```

**Верхний автоматизированный барьер — детерминистический стэк** (с v1.19): exact-identifier-rule, bank-scope-rule, history-Counter по counterparty, history по нормализованному описанию, системные категории по точному keyword-маркеру (только «Проценты по кредитам»). Всё, что не классифицировано детерминистически, уходит на ручную проверку юзером. Юзер — гарантированно проходимый барьер.

**LLM-модератор был отключён в v1.19.** Раньше он стоял между «правилами» и «пользователем»; см. §7 для деталей решения.

### 1.2 Честность важнее автоматизации

Принцип: **лучше пометить строку `warning` и попросить подтверждение, чем тихо присвоить неверную категорию.** Ложная уверенность хуже честного «не знаю» — юзер перестаёт проверять и аналитика портится.

Этот принцип наследуется из [[Импорт — проблемы и решения#Контекст|живого реестра проблем]] и является сквозным для всего пайплайна.

### 1.3 Детерминизм и идемпотентность

Одни и те же входные данные при одних и тех же настройках юзера должны давать один и тот же результат. Нормализация не зависит от времени, случайности, порядка обработки строк внутри батча.

### 1.4 Соответствие ключевым архитектурным решениям

Спек совместим с:
- [[Спецификация — Целевое состояние системы#10. Импорт данных]] — общее место импорта в архитектуре;
- решением об отмене `credit_payment` как `operation_type` (см. CLAUDE.md, раздел «Credit Payment Model»);
- решением о разделении `Counterparty` и `DebtPartner` (см. CLAUDE.md, раздел «Counterparty vs DebtPartner»).

## 3. Нормализация: единая стадия

### 3.1 Единственная точка входа

Нормализация — **одна атомарная стадия** на строку. Внутри может быть сколько угодно проходов (v1-basic, v2-tokens, skeleton, fingerprint), но снаружи это чёрный ящик: raw row → normalized row.

### 3.2 Запуск и перезапуск

**Запускается один раз** — при построении preview импорт-сессии. Не перезапускается при:
- ручном редактировании юзером любых полей строки (описание нередактируемо, но другие поля — да);
- переключении счёта — юзер не может менять счёт строки после preview;
- коммите — данные уже готовы.

**Прозрачный перезапуск на чтение** — если версия нормализатора изменилась между preview и открытием сессии, при отображении строка пересчитывается «на лету» из raw row + текущих settings. Это не пишется в БД, сохраняется только для консистентности отображения.

**UI-инвариант (v1.5):** компоненты модератора, отображающие строку (`AttentionCardImpl`, `defaultRowState` в bulk-карточке, любые будущие row-cards), обязаны инициализировать локальный state из `row.normalized_data` приоритетно над cluster-предсказаниями (`cluster.candidate_category_id`, `cluster.hypothesis.operation_type`, и т. п.). Это касается всех полей, которые юзер может менять и/или которые сохраняются на бэке: `category_id`, `operation_type`, `counterparty_id`, `debt_partner_id`, `target_account_id`, `credit_account_id`, `principal_amount`, `interest_amount`, `debt_direction`, `invest_direction`, `credit_kind`. Cluster — это рекомендация модели, она пересчитывается на каждом `/moderation-status`-fetch'е и **не учитывает user-edit'ы по дизайну**; persisted-state в `normalized_data` — единственный источник истины после Apply / bulk-apply / commit. Без этого инварианта F5 (или любой unmount/mount) восстанавливает state из cluster, визуально откатывает user-выбор, и следующий клик «Применить» отправляет stale state, перезаписывая корректную БД-row.

**UI-инвариант про bucket'ы (v1.6):** фильтр между «Готово к импорту» и «Требуют твоего внимания» в модераторе обязан учитывать `normalized_data.user_confirmed_at` приоритетно над cluster-уровневым trust_zone. Backend ставит этот timestamp на Apply (см. §5.4 — individual confirm = full-touch signal). Если фильтр смотрит **только** на `cluster.trust_zone !== 'red'` (как в §1.2 honesty gate), то любая ready-строка с low-confidence rule после refresh снова уезжает в attention — даже если юзер её уже явно подтвердил. Юзер вынужден пере-подтверждать каждый F5. **Правило:** `user_confirmed_at` непустой ⇒ строка всегда considered confirmed (попадает в «Готово к импорту», бейдж «✓ Проверено»), независимо от trust_zone. Honesty gate §1.2 продолжает работать только для ready-строк, которые юзер **не** трогал индивидуально — для них cluster.trust_zone остаётся источником истины.

**UI-инвариант: confirm с контрагентом = два вызова (v1.16):** подтверждение строки с выбранным контрагентом в `TxRow` обязано вызвать **и** `updateImportRow` (сохранить `counterparty_id` в normalized_data), **и** `attachRowToCounterparty` (создать `CounterpartyFingerprint` binding). Только наличие binding'а заставляет `build_bulk_clusters` включить fingerprint строки в `counterparty_groups` — без этого строка остаётся в AttentionFeed после рефреша, даже если `counterparty_id` корректно сохранён. После обоих вызовов обязательна инвалидация `bulk-clusters` query. Одного `updateImportRow` недостаточно. См. §6.3.

**UI-инвариант про bulk-card бейджи (v1.7):** аналогично §3.2 v1.5, но для bulk-карточек — `bulkCategoryId` и `bulkCounterpartyId` в `cluster-card.tsx` обязаны fallback'ать на consensus row-уровневых `normalized_data.category_id` / `counterparty_id`, если cluster-уровневый `candidate_category_id` пуст.

**UI-инвариант: категориальный слот collapsed-карточки (v1.18).** Свёрнутая карточка кластера/контрагента показывает категорию в правом бейдже по следующему приоритету:
1. **Consensus категория** (все строки с одной category_id) → фиолетовый бейдж `«Зарплата»`.
2. **Смешанные категории** (строки имеют разные category_id) → два серых бейджа `«Коммуналка» · «+1»` (первая по алфавиту + счётчик остальных).
3. **Нет категорий, но единый нон-категориальный `operation_type`** (все строки debt / transfer / investment_buy / investment_sell / credit_disbursement / credit_early_repayment) → синий бейдж типа операции: `«Долг»` / `«Перевод»` / `«Инвестиция»` / `«Получение кредита»` / `«Досрочное погашение»`. Эти типы не требуют категории — показывать «Выбери категорию» вводит в заблуждение.
4. **Нет ни категорий, ни типа** → серый `«Выбери категорию»`.

**UI-инвариант: контрагент для non-eligible строк в bulk-apply (v1.18).** Когда в cluster-modal'е все строки non-eligible (debt / transfer / credit) и пользователь выбрал контрагента в заголовке и нажал «Готово», `handleApplyBulk` вызывает `attachRowToCounterparty(sessionId, rowId, cpId)` для каждой non-eligible строки. Эффект: CounterpartyFingerprint-binding создаётся на уровне fingerprint'а — следующий вызов `/bulk-clusters` покажет строки под карточкой контрагента. `counterparty_id` в `normalized_data` строки **не** изменяется (для debt: запрещено §12.2; для transfer: не применимо). Это чисто UI-организационная привязка, не транзакционная. После bulk-apply backend записывает выбор юзера в каждую row + создаёт `TransactionCategoryRule`, но при следующем `/moderation-status` fetch'е cluster.candidate_category_id может вернуться null (если confidence свежесозданного правила ниже cluster-rule-порога), и бейдж «Транспорт» исчезает с карточки, хотя каждая row внутри её имеет category_id=Y. **Правило consensus:** перебрать все member row, считать значение через `normalized_data.<field>`. Если все непустые значения совпадают — это и есть ответ кластера, использовать его. Если расходятся — fallback на null (юзеру дать выбрать вручную). Это не противоречит cluster-recompute'у (он остаётся для зон trust), просто дополняет его персистентностью user-edit'ов на UI-слое.

### 3.3 Причины падения нормализации

- Некорректный raw row (битая строка выписки).
- Невозможно извлечь обязательные факты (сумма, дата).
- Нет распознанного банка/формата.

Любое падение → строка в статус `error` (см. §5).

### 3.4 Часовой пояс naive datetime

Парсеры русских банков (Т-Банк, Я-Банк, Озон) отдают время операций в МСК, но без явной TZ-метки. Нормализатор обязан помечать такие naive datetime как **`Europe/Moscow`**, не как `UTC`.

Историческая причина фикса (v1.3): naive→UTC давал две каскадные проблемы.
1. **UI-сдвиг в следующий день** — операция «13.03 23:14 МСК» сохранялась как `2026-03-13T23:14:00+00:00`, в локальной TZ юзера (тоже МСК) фронт рендерил её как «14.03 02:14». В live-выписке это выглядело как «парсер ошибся датой», хотя данные исходно были корректны.
2. **Матчер не сводил трансферы между банками** — две стороны одной transfer-пары (Т-Банк 23:14 МСК и Я-Банк 23:14 МСК) сохранялись с одинаковым UTC-time, но при этом если хоть одна сторона приходила без time-компонента (некоторые экстракторы дают только `DD.MM.YYYY`), её сохраняли как `00:00 UTC` → diff между сторонами 23 ч → выходило за `MAX_DATE_DIFF_HOURS=12`.

Когда появятся не-российские банки, TZ нужно поднять на уровень `session.mapping_json.bank_timezone` (см. бэклог §14.15). До тех пор хардкод `Europe/Moscow` в нормализаторе закрывает 100% реальных выписок.

**Aware datetime не перезаписывается.** Если raw row уже содержит TZ-info (ISO с offset), нормализатор сохраняет её как есть.

## 4. Контракт нормализованных данных

### 4.1 Две группы полей

Результат нормализации делится на **строго две непересекающиеся группы**:

**Факты (facts).** Что произошло. Извлекаются из raw row однозначно, не зависят от интерпретации:
- `amount` — сумма со знаком / абсолютное значение.
- `currency` — валюта операции.
- `booking_date` — дата проведения.
- `raw_description` — исходный текст, как в выписке.
- `direction` — дебет/кредит (нормализованный).
- `source_file_row` — ссылка на исходную строку файла.
- `bank_code`, `account_id` — откуда прилетело.

Факты **не меняются** после нормализации. Даже если юзер ошибся в счёте при загрузке — он перезагружает файл в правильный счёт, это другая импорт-сессия.

**Решения (decisions).** Как интерпретировать. Вычисляются на базе facts и могут меняться при обучении системы:
- `skeleton` — нормализованное описание с плейсхолдерами (`<PHONE>`, `<CONTRACT>`, `<IBAN>`, `<CARD>`, `<PERSON>`, `<ORG>`).
- `tokens` — извлечённые идентификаторы (номера договоров, IBAN, телефоны, карты).
- `fingerprint` — хеш `(bank_code, account_id, direction, skeleton, contract_token, transfer_identifier)`.
- `predicted_category_id` — что предложили правила или history-Counter (раньше также LLM, отключён в v1.19).
- `predicted_counterparty_id` / `predicted_debt_partner_id`.
- `predicted_operation_type`.
- `is_self_transfer_candidate` — только если оба конца известны.

### 4.2 Инвариант разделения

Код не должен смешивать эти группы в одной структуре данных. Факт не может быть перезаписан результатом правила; решение не может попасть в поле, предназначенное для факта.

### 4.3 Версионирование

`normalizer_version` хранится на уровне импорт-сессии (или отдельно по строке). При рассогласовании — прозрачный перезапуск decisions-части при чтении. Facts никогда не пересчитываются.

## 5. Статусы строк и переходы

### 5.1 Семь статусов

| Статус | Что означает | Блокирует коммит |
|---|---|---|
| `ready` | Всё распознано с высокой уверенностью, готова к коммиту | Нет |
| `warning` | Применено предсказание, но требует внимания юзера. Коммитится только после того, как юзер коснулся кластера строки — индивидуальным подтверждением или cluster-level bulk-ack (§5.4) | Да, пока кластер не подтверждён |
| `error` | Нормализация упала, данные некорректны или нарушена целостность (нераспознанный счёт, transfer без counter-account) — строка не может быть закоммичена как есть | Да (пока не перевели в `excluded` или `parked`) |
| `duplicate` | Совпадает с уже **закоммиченной** транзакцией того же счёта — повторный импорт или зеркало уже-закоммиченной transfer-пары. **Две строки из разных активных сессий — НЕ дубли, они обе `ready`.** | Нет (скипается автоматически) |
| `parked` | Юзер временно отложил («разберусь позже») | Да, но допускается перенос в следующую сессию |
| `excluded` | Юзер решил не коммитить эту строку | Нет (просто не коммитится) |
| `committed` | Строка закоммичена, создала Transaction(s) | Терминал |

### 5.2 Триггеры warning vs error

Два разных класса проблем, две разные реакции.

**`warning` — качество предсказания.** Данные целостны, мы просто не уверены в классификации. Юзер должен взглянуть, но коммит технически возможен (после cluster-level подтверждения, §5.4).

1. Правило применено, но его confidence ниже порога «молчаливого применения» (см. §6).
2. ~~LLM не уверен в ответе~~ — отключено в v1.19 (см. §7).
3. Tokens извлечены неоднозначно (два contract_token-кандидата, два transfer_identifier-кандидата).
4. Направление операции выведено контекстно, не из явного знака/колонки.

**`error` — целостность данных.** Строку нельзя закоммитить как есть, даже при согласии юзера — получится мусорная транзакция. Требуется явное решение (fix / park / exclude).

5. Счёт не распознан или распознан с низкой уверенностью (нельзя создать транзакцию без account_id).
6. Похоже на transfer, но counter-account не определён (см. §12 — transfer с одним концом запрещён).

*Было в v1.0:* триггеры 5 и 6 квалифицировались как `warning`. Переквалифицированы в `error` в v1.1 — строка, которую физически нельзя привязать к счёту, не должна иметь лёгкого пути в коммит (даже через bulk-ack).

### 5.3 Переходы

```
[parsing] ──ok──> ready | warning
          ──fail─> error

ready    ──юзер правит──> warning (если правка касается категории/контрагента — применится новое правило)
warning  ──юзер подтвердил──> ready (опционально, при явном confirm click)
error    ──юзер──> excluded | parked
error (orphan transfer) ──matcher свёл пару──> ready (обе стороны)
cross-session pair ──matcher──> ready (ОБЕ стороны — и income, и expense)
parked   ──следующая сессия──> ready | warning | error (ре-нормализация)
excluded ──(терминал до коммита)──> не коммитится
duplicate ──(терминал)──> не коммитится (юзер НЕ может override)
          только для совпадений с уже-committed TX (§8.8)
ready|warning ──commit──> committed
```

**Cross-session transfer pair (v1.11):** когда matcher находит пару между двумя активными import rows из разных сессий (одна income, другая expense, одинаковая сумма, ±1 день), обе стороны получают `status='ready'`, `operation_type='transfer'`, `target_account_id` = счёт противоположной стороны. `is_secondary=False` для обеих. UI показывает: expense-сторону оранжевой стрелкой «А → Б», income-сторону синей стрелкой «А ← Б» в виджете «Переводы и дубли». Первая закоммиченная сторона создаёт пару через `_create_transfer_pair`. Вторая — линкуется через `_link_transfer_to_committed_cross_session_pair` к уже-существующему phantom TX (§10.7).

**Допуск orphan-transfer error rows в matcher pool.** Триггер 6 в §5.2 ставит `error`, когда классификатор уверен, что строка — transfer, но counter-account не определён. Это терминальный data-integrity error для юзера, но **не** для самого matcher'а: при следующем проходе (например, после загрузки выписки контр-счёта) matcher обязан включать такие строки в pool кандидатов и попытаться свести пару. При успехе `_apply_assignments` переписывает `status` в `ready` (primary side) или `duplicate` (secondary side cross-session pair) и очищает `error_message`. Если пара по-прежнему не найдена — статус остаётся `error`, никаких регрессий.

Технически: фильтр загрузки активных кандидатов в matcher включает `status IN ('ready','warning','error')`, но пускает в pool только error-строки c `operation_type='transfer'` и пустым `transfer_match`. Прочие error-строки (нет счёта, битый amount) продолжают игнорироваться.

### 5.4 Массовые действия

**Философский принцип:** каждый кластер — отдельная единица внимания. Сквозной «commit everything» запрещён, потому что после кластеризации 600 строк превращаются в 30 кластеров — это уже не массовая операция, а 30 коротких решений. Юзер обязан коснуться каждого.

В модераторе доступно:
- **Cluster-level bulk-ack** — подтвердить предложенную категорию/контрагента для всех `warning`-строк **внутри одного кластера** (например, 50 строк «Пятёрочки» одним кликом). Засчитывается как **confirm с весом 0.5** для применённого правила (см. §10).
- **Cross-cluster bulk-ack ЗАПРЕЩЁН.** Нет кнопки «подтвердить все warning'и в выписке сразу». Юзер обязан пройти по кластерам.
- **Exclude all errors** — перевести все `error` в `excluded` (одним действием, потому что `error`-строки уже не дадут транзакций, коммит для них невозможен — это чисто операция очистки от шума).
- **Park all parked-candidates** — убрать отложенные строки из текущего коммита.

**Кнопка «коммит всей сессии» не трогает warning-строки, которые не были подтверждены на уровне кластера.** Они переносятся в `parked` или остаются в сессии до следующего захода.

**Bulk-кластеры transfer'ов** (например, серия внутрибанковских переводов на одном договоре) — особый случай: они представлены **в двух местах UI одновременно** — как обычная bulk-карточка с кнопкой массового подтверждения И как строки в виджете «Переводы и дубли» для сводной картины. Bulk-кластеры с обычными покупками (Пятёрочка, Вкусно и Точка) показываются ТОЛЬКО в bulk-секции — повторение в виджете для них шум. Различение по `operationType === 'transfer' && targetAccountId != null`. См. [import-moderation-panel.tsx#isBulkNonTransferRow](file:///d:/projects/financeapp/frontend/components/import/import-moderation-panel.tsx).

**Bulk-кластеры исключают `status=duplicate`.** Терминальные duplicate-строки (§8.3) уже выводятся в виджет «Переводы и дубли» и не могут быть закоммичены. Тащить их в bulk-карточку на ack — пустой шум, ломающий счётчик «9/9». Фильтр в `build_bulk_clusters` отрезает их **по двум независимым признакам**:
1. `status == 'duplicate'` — покрывает строки, которые `_find_transfer_pair_duplicate` пометил duplicate в build_preview БЕЗ записи `transfer_match` (находит уже-импортированную парную транзакцию по `account_id`/`amount`/`transaction_date` ±2 дня).
2. `transfer_match` is set — покрывает строки, помеченные cross-session matcher'ом или `_detect_committed_duplicates`.

Один признак не покрывает другой — нужны оба, иначе duplicate-строки протекают в bulk UI (наблюдалось в кластере «Внутрибанковский перевод с договора 0504603705» 9/9 на скрине от 2026-04-25).

**Точные пороги bulk-агрегации** ([import_cluster_service.py:351-360](file:///d:/projects/financeapp/app/services/import_cluster_service.py)):

| Константа | Значение | Что означает |
|---|---|---|
| `MIN_BULK_CLUSTER_SIZE` | 2 | Минимальный count fingerprint-кластера, чтобы попасть в bulk-секцию (был 5 до v1.15 → 3 до v1.17 → 2). Два одинаковых расхода у мерчанта — уже паттерн, достойный карточки. |
| `MIN_TRANSFER_IDENTIFIER_CLUSTER_SIZE` | 2 | Снижение порога для transfer-кластеров с явным identifier (phone/contract/iban/card). Два перевода на один телефон — уже паттерн «перевод маме». |
| `MIN_BRAND_CLUSTER_SIZE` | 2 | Минимальное суммарное количество рядов для эмиссии `BrandCluster` (синхронно с bulk, снижено с 3 до 2 в v1.17). |
| `MIN_FINGERPRINT_COUNT_FOR_BRAND` | 2 | Минимум разных fingerprint'ов для брендовой агрегации. Один fingerprint бренда уже представляет себя fingerprint-кластером, агрегировать нечего. |

**Cluster-Split при bulk-apply** (`cluster-grid.tsx: handleApplyBulk`). Когда пользователь снимает галочки с части строк:
- **1–2 unchecked** → `detachImportRowFromCluster` → строки уходят в attention-ленту как одиночки.
- **≥3 unchecked** → detach не вызывается → строки остаются в fingerprint-кластерах и при следующем `/bulk-clusters` формируют отдельную карточку (если count ≥ `MIN_BULK_CLUSTER_SIZE`). Используется для разделения смешанных brand-кластеров (напр. «Яндекс.Такси» + «Яндекс.Плюс» в одном кластере «Яндекс»).

Transfer-кластер без identifier (skeleton содержит «перевод», но phone/contract отсутствуют) **никогда** не попадает в bulk и брендовую агрегацию — без identifier'а получатели не различаются, агрегация дала бы false grouping разных людей в одну карточку.

### 5.5 Автоматическое распознавание возвратов (refund-matching)

`RefundMatcherService` (v1.12) находит пары expense↔income внутри одной сессии по следующим сигналам (в порядке убывания уверенности):

| Сигнал | Confidence |
|---|---|
| Совпадение contract / person_hash / counterparty_org | 0.95 |
| Refund keyword («возврат», «отмена», «refund») + совпадение brand | 0.95 |
| Совпадение skeleton | 0.80 |
| Совпадение brand (без keyword) | 0.75 |
| Только amount + direction + ±14 дней | 0.60 |
| Keyword без brand | 0.50 → отбрасывается |

**Порог отсечки `MIN_CONFIDENCE = 0.60`.** Пары ниже порога не эмитируются — защита от ложных коллизий сумм между разными мерчантами (кейс: «Оплата POPLAVO 700₽» и «Отмена KOFEMOLOKO 700₽» — разные brand, keyword есть только у одной стороны → 0.50, не матч).

**Greedy 1-to-1 assignment** — одна строка участвует максимум в одной паре. Высокая уверенность выигрывает при конкуренции.

`_apply_refund_matches` запускается в `build_preview` после нормализации. Результат — `refund_match` метаданные на обеих строках пары (side: expense/income, partner_row_id, confidence, reasons).

`_apply_refund_cluster_overrides` (также в `build_preview`) для income-кластеров с `is_refund=True` наследует counterparty + доминирующую категорию из истории покупок пользователя у того же brand за последние 365 дней. Если history не найдена — строка уходит в attention без категории (не в silent-auto). Результат refund-кластера: `operation_type='refund'`, `type='income'`, `category_id` и `counterparty_id` из истории.

## 6. Правила и confidence

### 6.1 Два типа правил

**Specific** — по идентификатору. Привязано к concrete token: номер договора, IBAN, terminal ID, ФИО контрагента. Высокая точность, низкое recall.

**General** — по скелетону. Привязано к нормализованному шаблону описания. Среднее recall, вариативная точность.

*Устаревшие scope-варианты (`bank`, `legacy_pattern`) мигрируют в одну из двух категорий. См. бэклог §14.7.*

### 6.2 Правила на категорию vs на контрагента

**Независимые сущности с независимыми счётчиками.** Обоснование: категория меняется от операции к операции (один магазин — разные категории), контрагент может быть одинаковым для десятков разных fingerprint'ов.

- **Category rule** — per-fingerprint. Один fingerprint → одно правило категории.
- **Counterparty binding** — per-identifier. Один идентификатор → один контрагент. Реализуется через `CounterpartyFingerprint` / `CounterpartyIdentifier`.

**Rejection по одному не трогает другое.** Юзер поменял категорию → rejection только category rule. Юзер поменял контрагента → rejection только counterparty binding.

### 6.3 Кластер vs контрагент: два уровня группировки с разной политикой по направлению

Кластеры и контрагенты — **два разных уровня агрегации**, и они трактуют направление операций по-разному.

**Кластер (low-level, fingerprint-based)** — делится по направлению. Fingerprint включает `direction` (см. §4.1), поэтому один и тот же скелетон в income и в expense даёт **два разных fingerprint'а**, два разных кластера, два разных правила категории. Это технически правильно: «Внутрибанковский перевод с договора N» income (приход) и expense (расход) — разные паттерны, могут попадать в разные категории.

**Контрагент (high-level, user-bound)** — НЕ делится по направлению. Один контрагент = одна карточка, независимо от того, в каком направлении идут операции с ним. Кейс-якорь: «Отец» как и отправитель денег (income), и получатель денег (expense) — это **одна и та же родственная связь**. Делить «Отец-приходы» от «Отец-расходы» на отдельные карточки в bulk-section было бы пользовательской бессмыслицей.

Технически, в `_group_by_counterparty` bucket-ключ — **только `counterparty_id`** (раньше был `(counterparty_id, direction)`). Все fingerprint-кластеры, привязанные к одному контрагенту через `CounterpartyFingerprint` или `CounterpartyIdentifier`, схлопываются в одну группу. Поля карточки:

| Поле | Семантика |
|---|---|
| `direction` | Доминирующее направление по count членов. При равенстве — `expense` (исторический дефолт; refund-карточки до v1.4 рендерились как expense). |
| `is_mixed_direction` | `True`, если есть хотя бы по одному кластеру в обеих направлениях. UI рендерит mixed-mode badge. |
| `count` | Сумма count всех members. |
| `total_amount` | Net по доминирующему направлению: `expense_sum - income_sum` (если `direction='expense'`) или `income_sum - expense_sum` (если `direction='income'`). Может быть отрицательным, если non-dominant сторона больше — это сигнал юзеру, что фактический баланс по контрагенту обратный «средней по count» картине. |

**Refund — частный случай, не отдельная ветка.** Раньше (v1.0–v1.3) refund-income специально схлопывался в expense-карточку через отдельную проверку `all(is_refund)` в членах. С v1.4 refund-income — просто income-кластер, привязанный к тому же контрагенту, что и expense-кластер. Получает то же поведение, что и любой другой income-binding: попадает в один bucket, считается в `income_sum`, выгружается из чистой суммы expense. Old refund-fold blob удалён.

**Direction-binding в правилах.** Counterparty binding (`CounterpartyFingerprint`) — это `fingerprint → counterparty_id`. Fingerprint содержит direction, поэтому биндинг автоматически направленный (один на income-fingerprint, другой на expense-fingerprint). Когда юзер вручную добавляет income-кластер «Отец-приходы» в существующего контрагента «Отец», создаётся отдельный `CounterpartyFingerprint` для income-fingerprint'а — и оба биндинга (income + expense) при следующем сборе кластеров `_group_by_counterparty` слепляются в одну карточку.

**Frontend-флоу создания binding'а (из AttentionFeed).** Когда юзер подтверждает строку с выбранным контрагентом в `TxRow` (AttentionFeed), фронтенд делает **два последовательных вызова**:
1. `PATCH /imports/rows/{id}` (`updateImportRow`) — сохраняет `counterparty_id` в `normalized_data` строки.
2. `POST /imports/{sessionId}/rows/{id}/attach-to-cluster` (`attachRowToCounterparty`) — создаёт `CounterpartyFingerprint` binding.

Только после шага 2 `build_bulk_clusters` начинает включать fingerprint строки в `counterparty_groups`. Без биндинга запись `counterparty_id` в `normalized_data` не меняет результат кластеризации — строка остаётся в AttentionFeed при следующем рефреше.

После обоих вызовов инвалидируется `bulk-clusters` query — ClusterGrid перерисовывается с новой counterparty-карточкой, строка исчезает из AttentionFeed.

**Aux-кластеры и AttentionFeed.** Малые fingerprint-кластеры (count < `MIN_BULK_CLUSTER_SIZE`) принадлежащие counterparty group возвращаются бэкендом в составе `fingerprint_clusters` (вместе с eligible-кластерами). `AttentionFeed.inClusterRowIds` строится по `fingerprint_clusters.row_ids` и корректно перехватывает строки из counterparty groups, включая singleton'ы — при условии что binding создан.

**Что НЕ меняется** этим разделом: §6.1 (specific vs general rules), §6.4 (confirms/rejections counters), §10 (commit-time updates), все инварианты §12. Это чисто UI/grouping-уровень изменения.

### 6.3 Confidence: что учитывается

Confidence правила определяется соотношением **confirms** и **rejections**, а также абсолютным размером выборки. Возраст правила сам по себе НЕ декеит confidence — реальность проверяется на следующем применении.

### 6.4 Что считается confirm / rejection

**Confirm** (с весом 1.0 или 0.5):
- Строка закоммичена со статусом `ready` и финальная категория совпадает с предсказанной → **confirms += 1.0**.
- Строка закоммичена со статусом `warning` через cluster-level bulk-ack (§5.4), финальная категория совпадает с предсказанной → **confirms += 0.5**. Вес 0.5 отражает, что юзер коснулся кластера, но не индивидуальной строки.

**Rejection:**
- Строка закоммичена, финальная категория отличается от предсказанной → **rejections += 1** для старого правила **и** формируется/обновляется новое правило с новой категорией (confirms = 1.0 для нового).

**Не сигнал:**
- Строки в статусе `excluded`, `parked`, `duplicate`, `error` не влияют на счётчики.

### 6.5 Три уровня поведения по confidence

| Уверенность | Статус строки | Поведение |
|---|---|---|
| Высокая | `ready` | Применяем молча |
| Средняя | `warning` | Применяем, но подсвечиваем |
| Низкая | — | НЕ применяем — строка идёт в attention-bucket для ручного выбора (раньше отправлялось в LLM-модератор, отключён в v1.19) |

**Отдельное состояние правила — `inactive`** (rejections > confirms). Такое правило не участвует в матчинге, но хранится в истории для возможного ревью.

**Точная формула confidence** ([import_cluster_service.py:1221-1258](file:///d:/projects/financeapp/app/services/import_cluster_service.py)):

```
confidence = base × error_ratio × evidence × identifier_match
```

Все четыре множителя в [0, 1], результат clamp'ится в [0, 1].

**Base — стартовая уверенность по типу найденного правила** (см. §6.8):

| Тип правила | Base | Условие |
|---|---|---|
| Identifier rule, proven | **0.99** | confirms ≥ 5 AND rejections == 0 |
| Identifier rule, active | **0.92** | иначе |
| Bank-scope rule | **0.85** | — |
| Legacy normalized-description rule | **0.78** | — |
| Не найдено правило | **0.0** | rule_source=`none` |

**Error ratio — устойчивость правила:**

```
error_ratio = confirms / (confirms + rejections)   if (confirms + rejections) > 0
            = 1.0                                  if нет истории (новое правило)
```

Идеальная история (0 rejections) даёт 1.0 (не давит base). Любой rejection начинает тянуть к 0.

**Evidence — размер кластера:**

```
evidence = 1.0   если cluster.count > 1
         = 0.95  если cluster.count == 1   (singleton drag = 0.05)
```

**Исключение для proven identifier-rules** — флаг `skip_singleton_drag=True`: 10 прошлых подтверждений конкретного contract'а сильнее, чем «всего 1 ряд в текущем кластере». Bank/legacy rules singleton drag всегда применяется.

**Identifier match factor — защита от ложного наследования** (Phase 7 fix):

| Состояние | Factor | Когда |
|---|---|---|
| `MATCHED` | 1.0 | у кластера и rule одинаковый identifier_value |
| `ABSENT` | 1.0 | ни у кластера, ни у rule нет identifier'а — нормальный skeleton-match |
| `UNMATCHED` | **0.60** | у кластера есть identifier, rule никогда его не видела (или у них разные значения) |

Сценарий, который закрывает UNMATCHED-факор: правило confirmed на «договор ДГ-12345 → Связь», в новой выписке прилетает «договор ДГ-99999 → ?» с тем же skeleton'ом. Без UNMATCHED-drag'а ДГ-99999 наследует категорию ДГ-12345 без явного подтверждения. С drag'ом 0.92 × 1.0 × 1.0 × 0.60 = 0.55 — кластер выпадает из green/yellow зон в attention.

**Confidence-зоны UI** (legacy 3-tier, информационно):

| Зона | Порог | Семантика |
|---|---|---|
| `green` | ≥ 0.85 | «можно не смотреть, AI точно знает» |
| `yellow` | ≥ 0.65 | «подтверди одним кликом — похоже, но не 100%» |
| `red` | < 0.65 | «требует выбора категории или ответа AI» |

Авторитетная двухкорзинная классификация (Phase 7+) — auto-trust vs attention — описана в §6.7.

### 6.6 Органическое разрешение конфликтов

Если юзер многократно правит предсказание одного и того же fingerprint — формируется конкурирующее правило. Оба живут параллельно:
- старое R с растущими rejections,
- новое R' с растущими confirms.

Побеждает то, у кого выше confidence на следующем применении. Со временем R уходит в `inactive`, R' становится каноничным. Специальная логика «разрешения конфликтов» не нужна.

### 6.7 Auto-trust gate

**Двухкорзинная UX-модель.** В модераторе кластеры распределяются по двум bucket'ам:

- **Готово к импорту (auto-trust)** — юзер просматривает список и нажимает «Применить всё». Не требует ни одного клика на каждой строке.
- **Требуют твоего внимания (attention)** — юзер обязан выбрать категорию / подтвердить кластер / отказаться.

**Жёсткий gate auto-trust** ([import_cluster_service.py:123-152](file:///d:/projects/financeapp/app/services/import_cluster_service.py)). Кластер попадает в auto-trust ТОЛЬКО когда выполнены **все** условия:

| # | Условие | Почему |
|---|---|---|
| 1 | `rule_source == 'identifier'` | Bank/legacy rules не доказывают identifier-level стабильность |
| 2 | `identifier_match == MATCHED` | Cluster.identifier == rule.identifier (точное совпадение) |
| 3 | `rule.confirms ≥ AUTO_TRUST_MIN_CONFIRMS` (= 5) | Юзер подтвердил ровно этот identifier ≥5 раз |
| 4 | `rule.rejections == 0` | Любое rejection вносит сомнение |
| 5 | `confidence ≥ AUTO_TRUST_THRESHOLD` (= 0.99) | Только base=0.99 (proven exact) проходит |

**Bank-scope и legacy rules никогда не дают auto-trust**, даже при confirms=100 и rejections=0 — по дизайну. Generalisation across identifier'ов всегда оставляет неизвестный риск, который требует одного клика подтверждения.

**Math sanity check:** обычная identifier-active confidence = 0.92 × 1.0 × 1.0 × 1.0 = 0.92 < 0.99 ⇒ не проходит. Только промоушен base до 0.99 (proven branch) даёт шанс пересечь threshold. Это намеренно — gate привязан к истории confirms ≥ 5, а не просто к «сейчас уверен».

**UI поведение.** Кластер с `auto_trust=True` рендерится в секции «Готово к импорту» с уже выбранной категорией (зелёная карточка). Юзер может развернуть, но не обязан. На commit'е кластер засчитывается как Case A (`confirms += 1.0`) для каждой member-строки — индивидуальный confirm, не bulk-ack.

### 6.8 Трёхуровневый поиск правил

`ImportClusterService._apply_rule` ([import_cluster_service.py:1030-1120](file:///d:/projects/financeapp/app/services/import_cluster_service.py)) пробует три типа правил по убыванию доверия. Первое match'нувшееся правило используется — никаких комбинаций.

**Уровень 1 — Exact identifier rule.**

```
get_active_rule_by_identifier(user_id, identifier_kind, identifier_value)
```

Срабатывает только если у кластера есть identifier (priority order: contract → phone → iban → card → person_hash). Если у этого user'а есть `TransactionCategoryRule` с тем же identifier_kind+identifier_value — это самое сильное доказательство. Base = 0.99 (proven, при confirms ≥ 5 и 0 rejections) или 0.92 (active иначе).

**Уровень 2 — Bank-scope rule.**

```
get_active_rule_by_bank(user_id, bank_code, normalized_description)
```

Generalisation across identifier'ов в рамках одного банка. База — 0.85. Если у кластера есть identifier, проверяется `_classify_identifier_match`: если rule никогда не видела такой identifier_value — `UNMATCHED` × 0.60.

**Уровень 3 — Legacy normalized-description rule.**

```
get_active_legacy_rule(user_id, normalized_description)
```

Старая модель (до identifier binding'а), исключаются rule'ы с заполненным `identifier_value` — те принадлежат уровню 1 и не должны утечь сюда через skeleton coincidence. Base = 0.78. Identifier mismatch также вешает `UNMATCHED` × 0.60, если у кластера есть identifier.

> **Терминология:** поле `Cluster.rule_source` для этого уровня содержит строку `"normalized_description"` (не `"legacy"`). Авто-trust gate проверяет `rule_source == "identifier"` — остальные значения (`"bank"`, `"normalized_description"`, `"none"`) не различаются gate'ом и никогда не дают auto-trust.

**Если ни одно правило не нашлось** → rule_source=`none`, confidence = 0.0, кластер уходит в attention.

### 6.9 Multi-layer hints (account_context / bank_mechanics / global_patterns)

Поверх confidence-движка работают **три независимых слоя подсказок**. Они не пишут в `predicted_category_id`, а кладут параллельные поля кластера, которые UI использует как additional signals (никогда не override'ят rule layer). До v1.19 эти же подсказки потреблял LLM-модератор; после его отключения (v1.19) единственный потребитель — UI и `_resolve_category` history-pass.

**Layer 1 — Account context** ([import_cluster_service.py:965-1028](file:///d:/projects/financeapp/app/services/import_cluster_service.py)). Детерминированные правила по `account_type` и keyword'ам в skeleton'е:

| account_type | Direction | Keyword в skeleton | Hint |
|---|---|---|---|
| `credit_card` | income | «погашение», «пополнение», «внесение» | `transfer` + label «Пополнение кредитной карты» |
| `credit_card` | income | «отмена», «возврат», «refund», «chargeback» | `refund` + label «Возврат на кредитную карту» |
| `credit` / `installment_card` | expense | «погашение процентов», «проценты пользование/договору» | `regular` + cat «Проценты по кредитам» |
| `credit` / `installment_card` | expense | «погашение основного долга», «погашение тела» | `transfer` + label «Погашение тела кредита» |
| `credit` / `installment_card` | income | «отмена/возврат по операции» | `refund` + label «Возврат по кредитной операции» |
| `deposit` | income | «начисление процентов», «капитализация», «проценты по вкладу» | `regular` + cat «Проценты от вклада» |
| `deposit` | income | «пополнение», «взнос» | `transfer` + label «Пополнение вклада» |
| `deposit` | expense | «частичное снятие», «снятие», «закрытие вклада» | `transfer` + label «Снятие с вклада» |

Поля кластера: `account_context_operation_type`, `account_context_category_id`, `account_context_label`. Не влияют на confidence — это чистая UI-подсказка.

**Layer 2 — Bank mechanics** ([bank_mechanics_service.py](file:///d:/projects/financeapp/app/services/bank_mechanics_service.py)). Bank-specific patterns, использующие `bank_code` + skeleton + account-type. На текущий момент покрывает Yandex Сплит/Дебет (§9.10). Каждое правило, кроме классификации, может выдать:

- `confidence_boost` — бамп confidence кластера (0.12-0.18)
- `cross_session_warning` — текстовый алерт в UI
- `suggest_exclude=True` — пометка строки `excluded` (phantom-mirror)
- `resolved_target_account_id` — авто-разрешение target_account по contract token (§9.10)

Поля кластера: `bank_mechanics_operation_type`, `bank_mechanics_category_id`, `bank_mechanics_label`, `bank_mechanics_cross_session_warning`, `bank_mechanics_suggest_exclude`, `bank_mechanics_resolved_target_account_id`.

**Применение к строкам — Pass 4 `build_preview`.** Результаты bank_mechanics остаются в кластере до конца `build_preview`, затем `ImportPostProcessor.apply_bank_mechanics(session)` проходит по всем кластерам с ненулевыми bank_mechanics полями и записывает результаты в `ImportRow`:
- `bank_mechanics_suggest_exclude=True` → `row.status = 'excluded'` (если нет `user_confirmed_at` и статус не committed/parked).
- `bank_mechanics_resolved_target_account_id is not None` → `normalized_data.target_account_id = <id>` + `operation_type = 'transfer'` (если target_account_id ещё не выставлен вручную).

**Layer 3 — Global patterns** ([global_pattern_service.py](file:///d:/projects/financeapp/app/services/global_pattern_service.py)). Кросс-юзерный consensus: если skeleton (`bank_code` + `skeleton`) подтверждён ≥N разными юзерами с одной и той же категорией — это коллективный сигнал, которому стоит верить даже у нового юзера без личной истории.

Confidence-boost: `0.04 × min(user_count, 5)` — максимум +0.20 при ≥5 юзерах. Поля кластера: `global_pattern_category_id`, `global_pattern_category_name`, `global_pattern_user_count`, `global_pattern_total_confirms`.

**Refund override** ([import_cluster_service.py:540-567](file:///d:/projects/financeapp/app/services/import_cluster_service.py)). Когда кластер `is_refund=True` (см. §5.5), отдельная ветка перекрывает rule chain: ищется counterparty + доминирующая категория из истории expense-транзакций за 365 дней по `LIKE %brand%` в `counterparty.name` / `description` / `normalized_description`. Найденная категория override'ит candidate_category_id, confidence boost'ится до 0.95 (без 0.99 — refund по определению не может пройти proven gate, у refund-fingerprint'ов нет identifier rule с историей confirms).

### 6.10 Брендовая агрегация и счёт направления

`ImportClusterService.build_bulk_clusters` возвращает три параллельных уровня агрегации:

**Уровень 1 — `Cluster` (per fingerprint).** Базовая единица. Включается в bulk-секцию UI при `count ≥ MIN_BULK_CLUSTER_SIZE` (= 2) или при `count ≥ MIN_TRANSFER_IDENTIFIER_CLUSTER_SIZE` (= 2) для transfer-кластеров с identifier'ом. Иначе уходит в attention.

**Уровень 2 — `BrandCluster` (per brand + direction).** Эмитируется когда ≥`MIN_FINGERPRINT_COUNT_FOR_BRAND` (= 2) разных fingerprint'ов извлекают одинаковый brand-токен через `brand_extractor_service.extract_brand` И суммарный `count ≥ MIN_BRAND_CLUSTER_SIZE` (= 2). Direction в брендовой группировке **разделяется** (income и expense — разные карточки, разное user intent).

Transfer-skeleton'ы исключаются из брендовой агрегации полностью — они должны различаться по identifier'у получателя, не по бренду.

**Уровень 3 — `CounterpartyGroup` (per counterparty_id).** Эмитируется когда ≥1 fingerprint'ов привязан к одному `Counterparty.id` через `CounterpartyFingerprint` или `CounterpartyIdentifier`. Direction **не разделяется** (см. §6.3) — один контрагент = одна карточка, `is_mixed_direction=True` если есть и income, и expense members.

**Нет минимума строк для CounterpartyGroup.** Counterparty-карточка появляется даже при одной строке в сессии (singleton). Сценарий: пользователь установил binding «телефон +7916… = Брат» в прошлом импорте. Новый импорт содержит один перевод от этого телефона → в ClusterGrid сразу появляется карточка «Брат · 1 операция». Singleton fingerprint включается в `aux_clusters` (§ ниже) и возвращается в `fingerprint_clusters` ответа. Порог `MIN_BULK_CLUSTER_SIZE = 2` применяется **только** к standalone fingerprint-кластерам без counterparty binding.

**Исключение для transfer-строк.** Строки с `operation_type='transfer'` или `transfer_match` исключаются из `live_clusters` на этапе `build_bulk_clusters` и не участвуют в CounterpartyGroup — они уходят в виджет «Переводы и дубли». Это не баг: transfer-пары имеют собственный UI для категоризации.

**Иерархия preemption:**

```
CounterpartyGroup ⊃ BrandCluster ⊃ FingerprintCluster
```

Если все fingerprint'ы бренда привязаны к counterparty — `BrandCluster` не эмитируется (counterparty wins). Если только часть — бренд-карточка остаётся с оставшимися fingerprint'ами, остальные уходят в counterparty-карточку.

**Aux-кластеры.** Маленькие fingerprint-кластеры (count < MIN_BULK_CLUSTER_SIZE), являющиеся участниками brand- или counterparty-группы, возвращаются в массиве `aux_clusters` — UI разворачивает их при раскрытии бренд/counterparty-карточки. Без aux фронт не смог бы резолвить row_ids member-fingerprint'ов в кликабельные строки.

## 7. Модератор без LLM (v1.19)

### 7.0 Решение и обоснование (2026-05-03)

LLM-модерация полностью отключена в v1.19. Причины:

1. **Слабый дельта-сигнал.** Детерминистический стэк (правила + bank_mechanics + counterparty/identifier-биндинги + history) уже покрывает 90%+ кейсов на typical-выписке. LLM добавлял ≤ 5% точности, но обрабатывал 100% оставшихся кластеров.
2. **Стоимость токенов** растёт линейно по количеству неизвестных fingerprint'ов; на каждой новой выписке банк-агрегатор добавлял десятки таких. Цена не оправдывает дельту.
3. **Недетерминизм.** Один и тот же кластер мог получать разные ответы при разных запусках — это нарушало §1.3 (детерминизм/идемпотентность).
4. **«Иллюзия LLM» после отключения.** Пока был активен `CATEGORY_KEYWORD_LIBRARY`-fallback в `_resolve_category`, юзер не мог отличить «применилась эвристика по подстроке» от «ответил LLM». Симптом сильно бил по доверию: после явного «LLM выключен» строки всё равно получали category. Fallback также удалён.

### 7.1 Что заменило LLM-модератора

«Верхний автоматизированный барьер» — `TransactionEnrichmentService._resolve_category`. Источники категории, в порядке убывания приоритета:

1. **`transaction_category_rules`** — явные правила, обученные на прошлых коммитах.
2. **`Counter` по counterparty-history** — ≥1 прошлая транзакция у этого counterparty с непустой category → берём dominant.
3. **`Counter` по нормализованному описанию** — fuzzy-история по skeleton/normalized_description.
4. **Системные категории по точному маркеру** — только «Проценты по кредитам» при явных фразах "погашение процентов" / "проценты по кредиту" и т.п.
5. **Multi-layer hints** — bank_mechanics (Layer 2) и global_patterns (Layer 3) дают **дополнительный сигнал** confidence, но не override'ят rule layer.

Если ничего не сработало → `category_id=None`, строка → status='warning', → attention-bucket для ручного выбора. Юзер — гарантированно проходимый барьер.

### 7.5 Симметричная классификация transfer-keyword'ов

Если в описании одной из сторон выписки есть однозначный transfer-маркер — `«перевод между счетами»`, `«перевод на свой счёт»`, `«пополнение своего счёта»`, **`«внутрибанковский перевод»`**, **`«внутренний перевод»`**, **`«межбанковский перевод»`**, `«с карты на карту»` — классификатор обязан вернуть `operation_type='transfer'` с **STRONG**-уверенностью (≥0.9), не WEAK.

Жирные фразы в списке выше — банковские термины. До v1.2 они лежали в WEAK (0.7) и валились ниже порога downgrade'а 0.88, из-за чего одна сторона перевода (например, Тинькоф Дебет: «Внутренний перевод на договор …») классифицировалась как `regular`, а вторая (Тинькоф Сплит: «Внутрибанковский перевод с договора …») — как `transfer`. Ассиметрия ломала [transfer_matcher_service.py](file:///d:/projects/financeapp/app/services/transfer_matcher_service.py): он мог свести только две стороны, у которых обе уже размечены transfer'ом. Теперь обе стороны симметричны.

Контр-индикаторы (явно НЕ transfer): «оплата покупки», «погашение кредита», «оплата услуг», имена операторов (Megafon/MTS/…) — см. список `_ANTI_TRANSFER_KEYWORDS`. Они переводят строку обратно в regular даже при наличии «перевод» в описании. В `transfer_signal()` (нормализатор) anti-transfer keywords умножают результат на **×0.3**. См. §12.8.

> **Anti-transfer penalty в матчере (отдельный механизм).** `TransferMatcherService._score_pair` применяет самостоятельный множитель **×0.4** при обнаружении `_ANTI_TRANSFER_KEYWORDS` в описании одной из сторон пары — это penalty для скоринга transfer-пар, не для классификации отдельных строк. Константы намеренно разные: при классификации строки (0.3) цель — убедительно заглушить слабый transfer-сигнал; при скоринге пары (0.4) цель — снизить оценку, но не отбросить пару совсем, поскольку другая сторона может содержать сильный transfer-маркер.

## 8. Дедупликация

### 8.1 Определение дубликата (v1.20)

Строка помечается `duplicate`, если существует уже закоммиченная транзакция, проходящая через четырёхветочную модель `_detect_committed_duplicates`:

```
GATE: same (account, direction, amount), окно ±2 дня

ВЕТКА A — same-account real-duplicate (is_mirror=False, нормальная tx)
ВЕТКА B — same-account phantom-transfer-income (is_mirror=False, tx — phantom)
ВЕТКА C — cross-account mirror INCOME (is_mirror=True, direction=income) → reject (никогда не дубль)
ВЕТКА D — cross-account mirror EXPENSE (is_mirror=True, direction=expense) → активна узко, см. §8.9
```

**Skeleton — confirmation-сигнал, не obligatory-key.** Раньше §8.6 (v1.11) делал skeleton обязательной частью ключа, что неверно работало для phantom-транзакций (skeleton phantom'a наследует banking-описание отправителя, а не получателя). Новая модель: skeleton проверяется как guard в каждой ветке отдельно, с правильной семантикой ветки.

**Identifier-discrimination — first-class сигнал.** Любая ветка делает hard reject если у row и tx одного типа identifier (contract / phone / iban) и они **разные значения**. Это закрывает кейс «Досрочное погашение 8845» vs «Внутренний перевод на договор 5422638063» — оба expense на 20 840₽, но разные contract'ы.

**Anti-transfer keyword guard для mirror-веток.** ATM/банкомат/cash withdrawal/inkasso/комиссия в `description` row → mirror reject независимо от других сигналов. Закрывает «ATM-снятие 42 000₽ ложно как mirror transfer'а».

**Greedy 1-to-1 assignment.** Множественные кандидаты в одной bucket сортируются по `(closest |diff_seconds| ASC, identifier-match-score DESC, skeleton-similarity DESC)`. Walk через сортировку с `used_tx_ids` set'ом — один committed-tx не используется как partner дважды. Закрывает кейс «два row 200₽ ложно matched к одному phantom'у».

### 8.2 Дедуп проверяется ТОЛЬКО против коммитов

- ❌ Не против других строк **в этом же импорте** — внутри импорта дублей не бывает.
- ✅ Только против уже закоммиченных `Transaction`-записей того же юзера.

### 8.3 Терминальный авто-скип

Статус `duplicate` — **терминальный**. Юзер НЕ может его override'нуть в модераторе. При коммите строки со статусом `duplicate` скипаются автоматически.

Если юзер обнаружил что дубликат ложный (редкий кейс ретроактивного добавления банком) — он добавляет операцию через обычную форму «новая транзакция», минуя импорт.

### 8.4 Визуальное поведение

Дубликаты по умолчанию свёрнуты в UI модератора (не засоряют поток). Юзер может раскрыть и увидеть каких закоммиченных транзакций они зеркалят — только для информации, не для действий.

### 8.5 Четырёхветочная модель `_detect_committed_duplicates` (v1.20)

В виджете «Переводы и дубли» каждая строка со статусом `duplicate` приходит из одной из четырёх веток. Каждая ветка имеет explicit вход (по типу tx и совпадению счёта), explicit guard'ы, explicit `is_secondary` для UI:

| Ветка | Условие входа | Guard'ы | UI | `is_secondary` |
|---|---|---|---|---|
| **A. Same-account real-duplicate** | `tx.account_id == row.account_id` AND tx — обычная (не phantom transfer-income) | Skeleton-match (или Jaccard ≥ 0.8) ИЛИ identifier-match. Mismatch одного типа identifier'а → reject. | «Дубль · другая сессия» | `False` |
| **B. Same-account phantom-transfer-income** | `tx.account_id == row.account_id` AND `tx.type='income'` AND `tx.operation_type='transfer'` (т.е. phantom-сторона уже-committed transfer'а) | Skeleton **не** проверяется (phantom наследует skeleton отправителя). Contract guard обязателен: row.contract != phantom.contract → reject. | «Дубль · другая сессия» (зеркало уже-committed transfer'а на этот же счёт) | `False` |
| **C. Mirror INCOME** | `tx.account_id != row.account_id` AND `row.direction == 'income'` | Hard reject. Income-операция приходит на один конкретный счёт; income на другом счёте — другая операция. | — | — (никогда не помечается дублем) |
| **D. Mirror EXPENSE** | `tx.account_id != row.account_id` AND `row.direction == 'expense'` | Узко активируется: `target.account_type ∈ {credit_card, installment_card, loan}` AND `description` row содержит credit-keyword («погашение», «проценты», «закрытие договора»). Plus identifier guard, plus anti-transfer keyword guard. | «Партнёр → этот счёт» (mirror) | `True` |

**Time window** — ±2 календарных дня (буфер для разногласий booking_date между банками).

**Greedy 1-to-1 assignment** — внутри одной bucket кандидатов сортируем по `(|diff_seconds| ASC, identifier-match-score DESC, skeleton-similarity DESC)`, проходим с `used_tx_ids` set'ом. Один committed-tx используется максимум как partner для одного row.

**Удалённое (vs v1.19):**
- Mirror-индекс с теневыми ключами `(target_acc, both_directions, amount)` упразднён. Теперь ветки D и C проходят прямой lookup по committed-tx с `target_account_id == row.account_id`, без conditional-on-`is_mirror`.
- Conditional skeleton-guard (раньше: skeleton-match только для is_mirror=False, exception для phantom) разнесён по веткам.
- Income-mirror reject (раньше: одна строчка `if is_mirror and direction='income': continue`) теперь explicit ветка C.

**Diagnostic поле `transfer_match.match_branch`** — каждая помеченная row несёт значение `'A'` / `'B'` / `'D'` (C никогда не записывается — ветка-reject). `NULL` в этом поле — маркер «помечено логикой до v1.20» и триггер для §8.10 reset.

### 8.6 Skeleton как confirmation-сигнал, не obligatory-key (v1.20)

**Раньше (v1.11–v1.19):** skeleton был обязательной частью дедуп-ключа `(account, direction, amount, skeleton)`. Это ломалось на phantom-кейсах, потому что phantom'ы наследуют skeleton банка-отправителя, а row из новой выписки имеет skeleton банка-получателя — даже для одной и той же логической операции.

**Сейчас (v1.20):** skeleton используется **per-branch** как confirmation-сигнал:

| Ветка | Использование skeleton |
|---|---|
| **A. Same-account real-duplicate** | Skeleton-match (равенство ИЛИ Jaccard ≥ 0.8 на token-overlap) подтверждает дубль. |
| **B. Same-account phantom-transfer-income** | Skeleton **не** проверяется (phantom skeleton ≠ row skeleton по дизайну). Contract guard заменяет skeleton-проверку. |
| **C. Mirror income** | N/A — ветка отбракована безусловно. |
| **D. Mirror expense** | Skeleton-match boost'ит confidence, но не обязателен. Credit-keyword + identifier-match — основные сигналы. Если skeletons radically различны (Jaccard < 0.2) AND identifier'a нет одного типа AND anti-transfer keyword присутствует → soft reject. |

**Кейс-якорь.** СБП-приход 5000 ₽ skeleton «пополнение система быстрых платежей» И внутрибанковский перевод 5000 ₽ skeleton «внутренний перевод на договор <CONTRACT>» — одна сумма, один день, противоположное direction между счетами. Контракт-токен на одной стороне есть, на другой нет → identifier guard не запускается. Anti-transfer keyword отсутствует. Skeleton-mismatch + skeleton-Jaccard < 0.2 → ветка D даёт soft reject. Не дубль. ✓

См. §8.5 (full ветки), §8.9 (mirror-expense узкая активация), §12.6 (актуализированный инвариант).

### 8.8 Cross-session pair — не дубль

Две строки из **разных активных сессий** с противоположными направлениями и одинаковой суммой — это **transfer-pair**, не дубликат. `status='duplicate'` присваивается ТОЛЬКО при совпадении с уже-закоммиченной транзакцией (`_detect_committed_duplicates`). Старая реализация ошибочно помечала income-сторону cross-session пары как `is_secondary=True` → `status='duplicate'`, из-за чего входящие переводы отображались как «Дубль · зеркало пары» вместо синей стрелки.

**Правило (v1.11):** `_apply_assignments` устанавливает `is_secondary=False` для обеих сторон cross-session пары. Обе строки `status='ready'`.

**Cleanup:** при каждом старте `match_transfers_for_user` метод `_reset_cross_session_secondary_duplicates` сбрасывает строки с `status='duplicate'` + `match_source='cross_session'` обратно в `status='warning'`, чтобы их подхватил свежий проход matcher'а.

### 8.7 Окно матчера трансферов: ±1 календарный день в МСК

Матчер cross-session трансферов сравнивает стороны не по фиксированному часовому бюджету, а по **разнице календарных дней в `Europe/Moscow`**. Допуск — `MAX_DATE_DIFF_DAYS = 1` (тот же день или соседний).

**Почему дни, а не часы.** Часовой бюджет (`MAX_DATE_DIFF_HOURS = 12` в v1.2 и раньше) регулярно отрезал легитимные пары вида «23:50 одного дня → 02:10 следующего», где обе стороны — одна и та же transfer-операция, но банки записали их в разные календарные сутки. Сравнение по `.date()` в МСК закрывает этот кейс (diff_days == 1) без расширения окна до 36-48 часов, которое тащило бы amount-coincidence false positives на сутки шире.

**Скоринг (внутри окна):**

| Условие | Score |
|---|---|
| `diff_seconds == 0` | 1.00 |
| `diff_seconds ≤ 60` | 0.97 |
| `diff_seconds ≤ 1 ч` | 0.93 |
| Тот же день, > 1 ч | 0.88 |
| Соседний день | 0.80 |
| `diff_days > 1` | 0.0 (отбрасывается до скоринга) |

Часы остались в шкале как тай-брейкер: при равном амаунте/направлении/keyword'ах побеждает пара, у которой стороны ближе по времени.

**Конвертация в `Europe/Moscow` обязательна** перед `.date()` — committed-tx могут хранить TZ-aware datetime в UTC; без явного `astimezone(_BANK_TZ)` UTC-дата может на 3 часа отличаться от МСК-даты, и пара случайно вылетит из окна.

### 8.9 Mirror-expense ветка: узкая активация для credit-target (v1.20)

Ветка D из §8.5 (mirror-expense) — это случай, когда **одна и та же логическая** transfer-операция booking'ится **обоими банками как expense**. Это специфика credit-учёта: банк-отправитель пишет «перевод на договор N» (expense), банк-получатель (credit account) пишет «погашение основного долга» — и это **тоже** expense с точки зрения credit-аккаунта (debt reduction = credit "возвращён"). Income-сторона на target-счёте отсутствует в выписке банка-получателя.

**Активация ветки D:**

```
ВСЕ условия должны выполняться:

1. tx.account_id != row.account_id  (cross-account)
2. tx.target_account_id == row.account_id  (committed transfer указывает на наш счёт)
3. row.direction == 'expense' AND tx.type == 'expense'  (оба expense)
4. target_account.account_type ∈ {credit_card, installment_card, loan}  (получатель — credit)
5. row.description содержит credit-keyword из списка:
   {'погашение', 'погашение основного долга', 'погашение тела',
    'погашение процентов', 'проценты по кредиту', 'проценты пользование',
    'закрытие договора', 'досрочное погашение', 'плановый платёж по кредиту'}
6. Identifier guard: row.contract && tx.contract && row.contract != tx.contract → reject
7. Identifier guard: row.phone && tx.phone && row.phone != tx.phone → reject
8. Anti-transfer keyword guard: row.description содержит {'atm', 'банкомат',
   'cash withdrawal', 'inkasso', 'комиссия', 'снятие наличных'} → reject
```

Если ВСЕ условия выполнены и ничто не reject'ит → row.status = 'duplicate', is_secondary=True.

**Контр-кейс — ATM-снятие 42 000₽ Сбер.** target_account=Сбер, account_type='main' (не credit). Условие 4 не выполняется → ветка D не активируется. ✓ Закрывает bug 16216.

**Контр-кейс — «Досрочное погашение 8845» 20 840₽.** target_account=Тинькоф дебет, account_type='main' (не credit). Условие 4 не выполняется → ветка D не активируется. ✓ Закрывает bug 16312.

**Дополнительно**: если target — credit, но contract'ы у row и tx есть и разные — guard 6 reject'ит. Двойная защита.

**Yandex Сплит/Дебет (см. §9.10) — основной use-case ветки D.** target_account=Yandex Сплит (account_type='credit_card'), row.description = «Погашение основного долга» (credit-keyword) → ветка активируется. Phantom-mirror корректно помечается дублем.

### 8.10 Fee-aware suspect-pair (v1.20)

После основного matcher'а `FeeMatcherService` запускает **второстепенный pass** на pool'е неcмэтченных строк, у которых `transfer_match` пустой и transfer-keyword присутствует в skeleton'е.

**Цель:** обнаружить случаи cross-bank transfer'ов, где банк-отправитель удержал комиссию, и приход на стороне получателя меньше, чем списание — например, «−37 000 ₽ Сбер → +36 943 ₽ Тинькоф, удержано 57 ₽ за СБП».

**Условия эмиссии suspect-pair:**

```
ВСЕ условия:

1. row_a.direction == 'expense', row_b.direction == 'income'
2. row_a.account_id != row_b.account_id  (разные счета)
3. row_b.amount < row_a.amount  (приход СТРОГО меньше списания —
   защита от мульти-источникового СБП, который наоборот даёт +дельту;
   такие кейсы вне scope, см. §8.11)
4. delta = row_a.amount - row_b.amount
   delta_pct = delta / row_a.amount
   delta_pct ≤ 0.05  (≤ 5%)
   delta ≤ 500  (≤ 500₽)
   ОБА порога одновременно
5. abs(timestamp_a - timestamp_b) ≤ 60 sec  (то же время до минуты)
6. row_a.skeleton ИЛИ row_a.description содержит transfer-keyword
   AND row_b.skeleton ИЛИ row_b.description содержит transfer-keyword
7. anti-transfer keyword отсутствует с обеих сторон
```

**Поведение при матче:**

`FeeMatcherService` НЕ создаёт transfer-пару автоматически. Записывает в обе строки:

```
normalized_data.fee_suspect_pair = {
    "partner_row_id": row_b.id (или row_a.id для другой стороны),
    "partner_account_id": ...,
    "delta_amount": delta,
    "delta_pct": delta_pct,
    "confidence": ... (0.8-0.95 в зависимости от identifier-match),
    "suggested_action": "transfer_with_fee"
}
```

Status строк остаётся `warning`. UI показывает виджет «Возможный перевод с комиссией» рядом с обычным виджетом «Переводы и дубли»:

> «Похоже на перевод с комиссией: −37 000 ₽ с Сбера → +36 943 ₽ на Тинькоф, 12.03 14:00. Удержано 57 ₽. [Подтвердить как перевод и записать комиссию] [Не связаны]»

**На confirm:**

1. Создать transfer-пару на меньшую сумму (`min(amount_a, amount_b)` = 36 943 ₽) через `TransferLinkingService.create_transfer_pair`.
2. Создать дополнительную expense-транзакцию на счёте-источнике на сумму дельты (57 ₽) в системную категорию «Банковские комиссии», `affects_analytics=True`.
3. Обе строки → `status='committed'`, `created_transaction_id` указывают на свои транзакции (главная для transfer-стороны; row-источник также имеет fee-tx ассоциацию).

**На «Не связаны»:** обе строки остаются `warning`, `fee_suspect_pair` стирается, identifier-флаг ставится чтобы fee-matcher повторно не предлагал ту же пару.

**Никогда не auto-trust** — Auto-trust gate (§6.7) для fee-suspect-pair не применяется. Пользователь обязан подтвердить.

**Системная категория «Банковские комиссии»** (`is_system=True`) автоматически создаётся через `CategoryService.ensure_default_categories()` — аналогично «Проценты по кредитам» (§9.3). Не удаляется и не переименовывается через API.

### 8.11 Out-of-scope: расхождения суммы вне fee-pattern

Случаи, которые `FeeMatcherService` сознательно НЕ покрывает:

- **Приход больше списания** (мульти-источниковое СБП, кэшбек получателю). Физически такое возможно, но статистически редко и неотличимо от случайного совпадения. Юзер обязан загрузить выписки всех счетов-источников.
- **Дельта > 5% или > 500₽**. Если потерялось больше 500₽ или больше 5% от перевода — это не банковская комиссия, а data quality issue (FX-конверсия, частичный возврат). Юзер разбирается вручную.
- **Time delta > 60 sec.** Без exact-time-сигнала amount-tolerance даёт слишком много false positives.
- **Cross-session pair с разными суммами.** `TransferMatcherService` (основной pass) использует exact-amount equality — это специально оставлено как hard rule, чтобы не загрязнять confidence-историю transfer-rules.

### 8.12 Reset legacy duplicate-меток при rollout (v1.20)

**Проблема.** Статус `duplicate` — терминальный (§8.3), и записанная `transfer_match`-метка живёт навсегда. При апгрейде detection-логики (v1.19 → v1.20) старые метки, записанные старой моделью (без `match_branch`, без greedy 1-to-1, без identifier guard), не переоцениваются автоматически. Существующий `_reset_cross_session_secondary_duplicates` фильтрует строго по `match_source='cross_session'` и legacy-метки не трогает.

**Симптом (live, 2026-05-03).** Session 150 (Т Банк кредитка): 10 строк помечены `duplicate · уже есть в Т банк дебет`, хотя contract row (5452737298) ≠ contract committed-tx (0504603705), а одна из строк («Оплата услуг mBank.MegaFon») вообще не содержит transfer-keyword'ов. SQL-аудит выявил у всех 10 строк `match_branch IS NULL` + один tx_id (5016) partner'ом для двух разных row — отпечаток pre-v1.20 матчера, не пойманный greedy 1-to-1.

**Решение.** Метод `_reset_legacy_committed_duplicates(user_id)` запускается в начале `match_transfers_for_user`, до основного прохода `_detect_committed_duplicates`. Логика:

```python
WHERE status = 'duplicate'
  AND transfer_match.match_source = 'committed_tx_duplicate'
  AND transfer_match.match_branch IS NULL
→ status = 'warning'
→ удалить transfer_match из normalized_data_json
```

После reset свежий проход matcher'а оценит row по четырёхветочной модели — корректные дубли будут помечены заново (с `match_branch` set), ложные останутся `warning` и пройдут через обычный пайплайн.

**Идемпотентность.** Метод можно запускать многократно — он трогает только row без `match_branch`, а свежезаписанные после v1.20 метки этим маркером не обладают.

**Альтернатива через миграцию.** Один SQL в alembic-миграции `0058_reset_legacy_duplicate_marks.py`:

```sql
UPDATE import_rows
SET status = 'warning',
    normalized_data_json = (normalized_data_json::jsonb - 'transfer_match')::json
WHERE status = 'duplicate'
  AND normalized_data_json->'transfer_match'->>'match_source' = 'committed_tx_duplicate'
  AND normalized_data_json->'transfer_match'->>'match_branch' IS NULL;
```

Миграция запускается один раз при deploy v1.20, после чего метод `_reset_legacy_committed_duplicates` работает как defensive guard для row, импортированных в окне между deploy миграции и инвалидацией кешей celery worker'ов.

**Связь с §20.3.** Класс проблемы — rollout detection-логики не переоценивает терминальные `duplicate`-метки автоматически — зафиксирован как known risk на будущие апгрейды (v1.21+).

## 9. Credit payment при импорте

### 9.1 `credit_payment` как operation_type отменён

Не создаём транзакции с `operation_type='credit_payment'`. См. CLAUDE.md, раздел «Credit Payment Model».

### 9.2 Три подкласса кредитных счетов

В метаданных счёта (`account_type`, см. §14.5):

| Подкласс | Поведение |
|---|---|
| `loan` (кредит наличными, ипотека, автокредит) | Перевод на счёт = **платёж по кредиту**, требует split на проценты + тело |
| `credit_card` (кредитная карта) | Перевод на счёт = **обычный transfer**, без split. Проценты — отдельной строкой на выписке карты |
| `installment_card` (карта рассрочки) | Аналогично `credit_card` |

### 9.3 Split для loan-счетов

Платёж на loan-счёт в модераторе открывает **split-форму** с двумя обязательными полями:
- `interest_amount` — сумма процентов
- `principal_amount` — сумма тела долга
- валидация: `interest_amount + principal_amount = total_amount`

При коммите из **одной импорт-строки создаются две транзакции**:

**(1) Interest expense**
- `type = 'expense'`
- `operation_type = 'regular'`
- `category_id` = системная категория «Проценты по кредитам»
- `credit_account_id` = loan-счёт
- `is_regular = True`
- идёт в Базовый Поток

**(2) Principal transfer**
- `type = 'expense'`
- `operation_type = 'transfer'`
- `target_account_id` = loan-счёт
- `affects_analytics = False`
- снижает долг, НЕ в расходах

### 9.4 Источник дефолтов — график платежей

**Параметры кредита** (срок, ставка, сумма, тип, дата выдачи, ежемесячный платёж) юзер указывает при создании loan-счёта. Справочно — для отображения «ожидаемого остатка долга».

**График платежей** (опционально) юзер загружает PDF'ом из банка при создании loan-счёта. Парсер извлекает таблицу: 36+ строк `(payment_date, total, interest, principal, remaining_balance)`.

**Дефолты в split-форме** берутся из соответствующей строки графика для даты платежа.

**Эволюция графика.** Если фактический split от юзера отличается от предсказанного графиком — система **заменяет строку графика на фактические значения**. График самоочищается, постепенно замещая предсказания правдой.

### 9.5 Досрочное погашение

Split-форма содержит чекбокс «досрочное погашение». При активации:
- Обязательное поле: **тип досрочного** — «сокращение срока» ИЛИ «сокращение ежемесячного платежа».
- Система пересчитывает оставшиеся строки графика по стандартной формуле аннуитета.
- Дефолты для следующих месяцев строятся из пересчитанного графика.

### 9.6 Формулы для split'а не используются

Автоматический split без ручного ввода **не реализуется**. Причины:
- Банки не публикуют помесячную методологию расчёта.
- Стандартная формула аннуитета расходится с реальными цифрами даже для чистых случаев без досрочных, страховок, плавающих ставок.
- Эмпирически: в бизнес-учёте требуется ручная проверка каждый месяц.

Что точно — **годовая сумма процентов** (она детерминирована параметрами). Помесячный split — приблизительный с правкой при необходимости.

### 9.7 Расчёт split'а — детерминистический

Split рассчитывается формулой Δ кредитного баланса между периодами (см. §9.6), а не выводится моделью. **С v1.19 LLM в импорте не используется в принципе** (см. §7) — этот раздел сохранён исторически. Если в будущем понадобится распарсить нестандартный PDF графика платежей, это будет отдельный одноразовый парсер, не часть импорт-пайплайна.

### 9.8 Warning при загрузке main без кредитного счёта

Если в выписке main-счёта есть перевод на счёт любого из трёх кредитных подклассов, но выписка с самого кредитного счёта в текущем импорте отсутствует → на уровне импорт-сессии показываем warning: «ты загружаешь платёж по кредитному продукту, но не загрузил выписку с него самого. Проценты не попадут в аналитику».

Это **мягкое правило**, не блокирует коммит.

### 9.9 Расширенное распознавание кредитных погашений (v1.12)

**Путь 1 — keyword detection в `build_preview`.** `_CREDIT_PAYMENT_KEYWORDS` — набор узких фраз, содержащих слово «кредит» рядом с глаголом платежа: «погашение кредита», «оплата по кредиту», «ежемесячный платёж по кредиту», «loan payment» и др. Если `operation_type` уже resolved в `transfer` AND description/skeleton совпадает → `requires_credit_split=True`. Срабатывает для Т-Банка, Сбера и любого банка без кастомного raw_type-экстрактора.

**Путь 2 — loan target detection в `update_row`.** Когда юзер вручную указывает `target_account_id` типа `loan` для transfer-строки → `requires_credit_split=True` проставляется автоматически. Покрывает случай когда описание не содержит ключевых слов, но тип целевого счёта говорит за себя.

**UI (v1.12).** `import-moderation-panel.tsx` читает `requires_credit_split` из `normalized_data`. При флаге + `operation_type='transfer'` — `mainOp` инициализируется как `credit_operation`, `creditKind='payment'`: split-форма с полями «Проценты» и «Тело долга» открывается без ручного переключения типа. Исправлены имена полей персистентного состояния: `credit_principal_amount` / `credit_interest_amount` (ранее читались как `principal_amount` / `interest_amount`, что вызывало обнуление формы при F5).

### 9.10 Яндекс Сплит / Яндекс Дебет — bank-mechanics pattern (v1.12)

Специфика платёжного цикла Яндекс:
- **Яндекс Дебет**: исходящие строки «погашение [X] по договору №КС...» — переводы на Сплит-счёт, идентифицированный по номеру договора.
- **Яндекс Сплит**: входящие строки разбиваются на «погашение основного долга» (phantom-mirror Дебет-перевода) и «погашение процентов» (расход на проценты банку).

**Семь детерминированных правил** в `bank_mechanics_service._YANDEX_RULES` (bank_code=`yandex`):

| Строка | Тип счёта | Direction | Действие | Boost |
|---|---|---|---|---|
| «погашение основного долга / тела / просроченной» | credit/installment | income | `suggest_exclude=True` — phantom-mirror Дебет-перевода; коммит задвоит баланс Сплит | 0.18 |
| «погашение процентов / проценты пользование / уплата процентов» | credit/installment | income | `operation_type='regular'`, category=«Проценты по кредитам» | 0.18 |
| «оплата товаров / услуг» | credit/installment | expense | `operation_type='regular'` (покупка через BNPL) | 0.05 |
| «погашение процентов / проценты пользование» | credit/installment | expense | `operation_type='regular'`, category=«Проценты по кредитам» | 0.15 |
| «погашение основного долга / тела» | credit/installment | expense | `operation_type='transfer'` (погашение тела) | 0.15 |
| «погашение / оплата по договору / перевод по договору» | main/savings | expense | `operation_type='transfer'` + `resolved_target_account_id` из `tokens.contract` | 0.12 |
| «отмена по операции / отмена операции / возврат по операции» | любой | income | `operation_type='refund'` | 0.06 |

**Постпроцессинг** — `ImportPostProcessor.apply_bank_mechanics(session)` запускается в `build_preview` после refund-passes (Pass 4). Применяет bank_mechanics результаты кластеров к индивидуальным строкам:
- Строки с `bank_mechanics_suggest_exclude=True` → `row.status='excluded'` (если юзер не выставил `user_confirmed_at` и статус не committed/parked)
- Строки с `bank_mechanics_resolved_target_account_id is not None` → `normalized_data.target_account_id = <id>` + `operation_type='transfer'` (если target_account_id ещё не выставлен вручную)

**Баланс-схема** (только-проценты кейс, 200,61 ₽):
```
Дебет −38 transfer → phantom +38 на Сплит
Дебет −162,61 transfer → phantom +162,61 на Сплит
Сплит +200,61 «погашение процентов» → expense −200,61 (Проценты по кредитам)
Net Сплит: 0 ✓    P&L: −200,61 ₽ ✓
```

**Баланс-схема** (тело + проценты кейс, 18 000 ₽ = 17 500 тело + 500 %):
```
Дебет −18000 transfer → phantom +18000 на Сплит
Сплит +17500 «погашение основного долга» → excluded
Сплит +500 «погашение процентов» → expense −500
Net Сплит: +17500 ✓    P&L: −500 ₽ ✓
```

## 10. Обновление статистики правил

### 10.1 Применение ≠ подтверждение

На этапе preview генерации правило применяется → в строке сохраняются:
- `applied_rule_id`
- `predicted_category_id`
- `predicted_counterparty_id` (если соответствующий binding сработал)

**Счётчики правил НЕ обновляются на этапе применения.** Только на коммите.

### 10.2 Четыре кейса на коммите

Для каждой коммитящейся строки:

| Кейс | Условие | Действие |
|---|---|---|
| A | Строка была `ready`, final_category = predicted | R.confirms += 1.0 |
| B | Строка была `warning`, подтверждена через cluster-level bulk-ack (§5.4), final_category = predicted | R.confirms += 0.5 |
| C | Строка имела applied_rule_id, final_category ≠ predicted | R.rejections += 1, создаём/обновляем R' для (fingerprint → final_category), R'.confirms = 1.0 |
| D | Строка не имела applied_rule_id, юзер указал final_category | Создаём новое R'' для (fingerprint → final_category), R''.confirms = 1.0 |

**Прим.:** индивидуальный commit warning-строки без правки (юзер раскрыл строку, посмотрел, подтвердил предсказание) засчитывается как кейс A с весом 1.0 — это полноценный контакт с конкретной строкой, а не скользящее подтверждение.

Для counterparty binding — полностью аналогичная схема, отдельными счётчиками.

### 10.3 Что НЕ меняет счётчики

- Строки `excluded`, `parked`, `duplicate`, `error` — не трогают счётчики.
- Промежуточные правки юзера («поменял → вернул как было») — не трогают счётчики. Учитывается **только финальное состояние на момент коммита**.
- Изменение других полей (сумма, дата) — не влияет на rule stats. Правила предсказывают только категорию/контрагента.

### 10.4 Новое правило R' получает confirms = 1.0

Стартует с сигналом, а не с нуля. Юзер явно выбрал эту категорию при коммите — это валидное первое подтверждение.

### 10.5 Мета-правила отдельный тип

Правило «этот fingerprint = credit payment, открыть split-форму» — **отдельный тип** (meta-rule), не смешивается с category rule. Его confirms/rejections обновляются на коммите split'а: реально заполнил — confirm, отменил и выбрал другую трактовку — rejection.

### 10.6 Линковка active-стороны к committed orphan вместо создания дубля

Сценарий: пользователь сначала закоммитил выписку счёта A. На момент того коммита второй стороны (счёт B) ещё не было, и transfer-транзакции A ушли как **orphan transfer** — `target_account_id IS NULL` и `transfer_pair_id IS NULL` — либо как `regular income/expense`, если классификатор не распознал. Затем пользователь импортирует выписку счёта B, transfer_matcher свёл строки B с committed-транзакциями A и записал `matched_tx_id` в `normalized_data.transfer_match`.

**Правило коммита:** при коммите active-стороны с `transfer_match.matched_tx_id is not None` система **линкует** активную транзакцию к существующей committed-транзакции вместо вызова `_create_transfer_pair`. Без этой защиты обычная commit-логика создаёт ВТОРУЮ зеркальную транзакцию на target-счёте, и баланс target-счёта получает double-credit.

**Алгоритм** ([import_service._link_transfer_to_committed_pair](file:///d:/projects/financeapp/app/services/import_service.py)):
1. Создаётся ТОЛЬКО active-сторона (B) как `transfer`.
2. Достраивается committed-сторона (A): `target_account_id`, `transfer_pair_id`, `operation_type='transfer'`, `affects_analytics=False`.
3. Balance применяется ТОЛЬКО к active-счёту (B). Баланс A не трогается — он уже был учтён при предыдущем коммите orphan'а.

**Защиты:** если `committed_tx.transfer_pair_id` уже не NULL (уже спарен), или сумма не совпадает, или направление одинаковое — линковка отказывает (`return None`), и caller делает fallback на обычное создание пары через `_create_transfer_pair`.

См. §12.1 (transfer требует оба счёта — после линковки оба известны) и §12.4 (committed orphan, который был `regular income`, после достройки получает `affects_analytics=False`, чтобы не двоиться в income-аналитике).

### 10.7 Cross-session pair: вторая сторона линкуется к phantom TX

Сценарий: обе сессии активны (не закоммичены). Пользователь коммитит сессию A первой. `_create_transfer_pair` создаёт:
- TX_A на счёте A (та сторона, которую закоммитили)
- TX_B (phantom) на счёте B (`transfer_pair_id = TX_A.id`)

Затем пользователь коммитит сессию B. Строка B имеет `transfer_match.matched_row_id = row_A.id`. Метод `_link_transfer_to_committed_cross_session_pair`:
1. Находит row_A → `created_transaction_id = TX_A.id`
2. Из TX_A получает `transfer_pair_id = TX_B.id`
3. Определяет, какой из {TX_A, TX_B} живёт на счёте B → это TX_B (phantom)
4. Возвращает TX_B → строка B линкуется к нему, новый TX не создаётся

**Защита от дублей:** если бы строка B просто вызвала `_create_transfer_pair`, получилась бы ещё одна пара TX_C + TX_D, и баланс счёта B получил бы двойной зачёт. Линковка к phantom предотвращает это.

**Инвариант (§12.12):** при наличии `transfer_match.matched_row_id`, если matched_row уже закоммичен (`created_transaction_id IS NOT NULL`), линковка к phantom обязательна. Создание новой пары запрещено.

## 11. Ошибки привязки

### 11.1 Три стратегии

| Стратегия | Когда |
|---|---|
| **Swallow** (проглотить) | Удалённая категория / удалённый счёт в binding-е. Правило молча идёт в `inactive`, строка идёт дальше по пайплайну (LLM/юзер). Самоисцеление через обычный путь обучения |
| **Retry** (попытка починить) | Mерж контрагентов. При merge A→B все `CounterpartyFingerprint`/`CounterpartyIdentifier` указывающие на A **активно** переставляются на B в той же транзакции — не лениво, не при следующем матче |
| **Fail** (явная ошибка) | Системная категория недоступна (баг миграции). Строка в `error`, алерт в админку, юзер не должен чинить |

### 11.2 Принцип разделения

- **Ошибки конфигурации/данных юзера → swallow** + самоисцеление.
- **Ошибки системной целостности → fail** + алерт, невидимо для юзера.
- **Retry только с явным target'ом** для ретрая.

### 11.3 Фоновая чистка

Правила в статусе `inactive` (rejections > confirms, или помеченные broken в swallow-сценарии) чистятся фоновым джобом. Не удаляются жёстко сразу — хранятся для возможного ревью юзером.

## 12. Жёсткие инварианты

### 12.1 Transfer требует оба счёта

Транзакция с `operation_type = 'transfer'` обязана иметь:
- `source_account_id` — известный счёт юзера,
- `target_account_id` — известный счёт юзера.

Строка импорта, похожая на transfer, но где хотя бы один из счетов не определён → **`error`** (см. §5.2, триггер 6), не автоприменяется. Юзер решает: `excluded` / `parked` / ручная правка с явным указанием счёта.

### 12.2 Debt vs Counterparty disjoint

`operation_type = 'debt'` ⇒ `debt_partner_id` обязателен, `counterparty_id` запрещён. Любой другой `operation_type` ⇒ `debt_partner_id` запрещён. См. CLAUDE.md, раздел «Counterparty vs DebtPartner».

### 12.3 credit_payment не создаётся

`operation_type = 'credit_payment'` — запрещено. См. §9.1.

### 12.4 credit_disbursement не в доходах

`operation_type = 'credit_disbursement'` ⇒ `affects_analytics = False`. Все агрегации доходов дополнительно фильтруют `operation_type != 'credit_disbursement'`.

### 12.5 Facts vs decisions

Поля группы facts (см. §4.1) никогда не перезаписываются результатами decisions-части. Код, пишущий в amount/date/raw_description/direction от имени правила (или раньше — LLM, отключён в v1.19) — баг.

### 12.6 Дедуп — четырёхветочная модель + identifier guard (v1.20)

`_detect_committed_duplicates` обязан использовать четыре explicit ветки (§8.5) вместо conditional-on-`is_mirror`-цепочки. В каждой ветке свой набор guard'ов:

- **Identifier-mismatch одного типа (contract / phone / iban) → hard reject** во всех ветках.
- **Skeleton — confirmation-сигнал**, не obligatory-key (§8.6). Per-branch использование.
- **Anti-transfer keyword guard** для mirror-веток.
- **Mirror-expense (ветка D)** активируется ТОЛЬКО при `target.account_type ∈ credit-set` AND credit-keyword в `description` (§8.9).
- **Greedy 1-to-1 assignment** с `used_tx_ids` set'ом и сортировкой кандидатов по `(closest-time, identifier-strength, skeleton-similarity)`.

Mirror-индекс с теневыми ключами `(target_acc, both_directions, amount)` упразднён. Прямой lookup committed-tx с `target_account_id == row.account_id`.

### 12.7 Нормализация не перезапускается при правках юзера

См. §3.2. Версионная миграция — через transparent re-run на чтение, не write.

### 12.8 Симметричная классификация двух сторон одного перевода

Если две stmts (выписки) описывают **одну и ту же** межсчётную операцию (например, Тинькоф Дебет «Внутренний перевод на договор N» и Тинькоф Сплит «Внутрибанковский перевод с договора N» — обе на одну сумму, в одну дату, противоположные направления), их `operation_type` обязан быть **одинаковым** = `transfer`.

Ассиметричная классификация ломает любой автоматический matcher: он может свести только две стороны, у которых обе уже размечены transfer'ом. Поэтому keyword-список STRONG-классификатора в §7.7 должен быть достаточно полным, чтобы покрыть формулировки описаний для обоих банков пары.

См. §7.7. Список STRONG-фраз — расширяется по мере появления новых форм выписок; всякая ассиметрия двух сторон одного логического перевода — баг классификатора, не данных.

### 12.10 Cross-session pair не является дублем

Две строки из разных **активных** (не закоммиченных) сессий, образующие transfer-пару (противоположные направления, одинаковая сумма) — обе `status='ready'`. Ни одна не `status='duplicate'`. `duplicate` присваивается только при совпадении с **уже закоммиченной** транзакцией. См. §8.8.

### 12.11 Card identifier binding — только для transfer-строк

`CounterpartyIdentifier` с `identifier_kind='card'` создаётся **только** для строк с `operation_type='transfer'`. В российских банковских выписках токен card в описании покупки (`«Операция по карте ****7123»`) идентифицирует карту ПЛАТЕЛЬЩИКА (т.е. собственный счёт пользователя), а не карту контрагента. Биндинг такого card-токена к контрагенту приведёт к тому, что все транзакции с этой картой окажутся под одним контрагентом. Для transfer-строк card-токен идентифицирует получателя перевода и является корректным cross-account ключом.

### 12.12 Cross-session pair: вторая сторона линкуется к phantom, не создаёт новую пару

При коммите строки с `transfer_match.matched_row_id`, если matched row уже имеет `created_transaction_id IS NOT NULL` (закоммичена первой), метод `_link_transfer_to_committed_cross_session_pair` обязан найти phantom TX на стороне текущей строки и линковаться к нему. Создание новой пары через `_create_transfer_pair` запрещено — это даст double-credit/debit на балансах. См. §10.7.

### 12.9 Linking, не creation, при наличии matched_tx_id

При коммите active-row с `transfer_match.matched_tx_id` запрещено создавать новую зеркальную транзакцию на target-счёте. Линковка — единственная корректная операция. См. §10.6.

Нарушение даёт double-credit на балансе target-счёта и две независимые orphan-транзакции вместо одной правильной пары.

## 13. Глоссарий

| Термин | Определение |
|---|---|
| **Raw row** | Одна строка исходной выписки до нормализации |
| **Normalized row** | Результат нормализации — факты + decisions |
| **Facts** | Данные, полученные напрямую из raw row. Не меняются |
| **Decisions** | Выводы пайплайна: skeleton, tokens, fingerprint, предсказания. Могут обновляться |
| **Skeleton** | Нормализованное описание с плейсхолдерами (`<PHONE>`, `<CONTRACT>`, …) |
| **Token** | Извлечённый идентификатор — номер договора, IBAN, телефон, карта |
| **Fingerprint** | Хеш `(bank_code, account_id, direction, skeleton, contract_token, transfer_identifier)` |
| **Cluster** | Группа строк с одинаковым fingerprint |
| **Rule** | category rule ИЛИ counterparty binding. См. §6 |
| **Meta-rule** | Правило типа «этот fingerprint = credit payment». См. §10.5 |
| **Barrier / барьер** | Уровень пайплайна который может классифицировать. См. §1.1 |
| **Bulk-ack** | Коммит warning-строк пачкой без индивидуального review. Даёт confirm с весом 0.5 |
| **Loan / credit_card / installment_card** | Три подкласса кредитных счетов. См. §9.2 |
| **Auto-trust** | Двухкорзинная классификация UX: «Готово к импорту». Жёсткий 5-условный gate — см. §6.7 |
| **Identifier match** | Состояние сравнения cluster.identifier_value vs rule.identifier_value: MATCHED / ABSENT / UNMATCHED. Влияет на confidence-множитель (§6.5) |
| **Brand cluster** | UI-агрегация ≥2 fingerprint'ов одного бренда — см. §6.10 |
| **Counterparty group** | UI-агрегация всех fingerprint'ов, привязанных к одному `Counterparty.id` — см. §6.3, §6.10 |
| **Multi-layer hint** | Параллельная подсказка от account_context / bank_mechanics / global_patterns. Не override'ит rule layer — см. §6.9 |

## 13. Закрытые счета

**Введено 2026-05-03 (миграция 0057, v1.20).**

### 13.1 Концепция

`Account.is_closed: bool` + `Account.closed_at: date | None` — первоклассное состояние «счёт существовал, операции были, банковский договор закрыт». Отделено от `is_active=False` (временно скрыт, может вернуться).

| Состояние | `is_active` | `is_closed` | `closed_at` | Поведение |
|---|:-:|:-:|:-:|---|
| Активный | `True` | `False` | `NULL` | Видим в активных списках |
| Временно скрыт | `False` | `False` | `NULL` | Скрыт из списков, но не "закрыт исторически" |
| Закрыт | `False` | `True` | `<date>` | Скрыт из активных списков; виден в моде- раторе и истории |

При закрытии через `POST /accounts/{id}/close` — атомарно `is_closed=True`, `is_active=False`, `closed_at=<payload>`. Открытие через `POST /accounts/{id}/reopen` — обратно `is_closed=False`, `closed_at=NULL`, `is_active=True`.

### 13.2 Валидации

`AccountService.close()`:
- `closed_at` обязателен при `is_closed=True`.
- `closed_at` не в будущем (`> today`).
- `closed_at ≥ MAX(transaction_date)` для всех committed-транзакций счёта.
- Повторное закрытие уже-закрытого → `CloseAccountValidationError`.

`reopen()`:
- Открытие активного счёта → `CloseAccountValidationError`.

Удаление закрытого счёта блокируется FK (`ON DELETE RESTRICT` на `transactions.account_id`).

### 13.3 Видимость

| Слой | Закрытые счета |
|---|---|
| `GET /accounts` (default) | Скрыты |
| `GET /accounts?include_closed=true` | Видимы |
| Sidebar / dashboard / форма создания транзакции (frontend) | Скрыты |
| Страница «Все счета» (секция «Закрытые счета», свёрнута) | Видимы |
| Дропдаун target/source в моде- раторе (AttentionFeed, EditTxRazvorot, ClusterModal, SplitModal) | Видимы (с пометкой `(закрыт)`) |
| Фильтры транзакций / история | Видимы |
| `_detect_committed_duplicates` mirror-индекс | Видимы (закрытые target — валидные partner-аккаунты для §8.5 ветки D) |

### 13.4 Mirror-транзакции на закрытых счетах

Когда orphan transfer привязывается к закрытому счёту (через `OrphanTransferHint` или ручной выбор юзером), `TransferLinkingService.create_transfer_pair` создаёт **обе** ноги transfer'а как обычно. Mirror-tx на закрытом счёте — обычная `Transaction`-запись без специального флага. Баланс закрытого счёта изменяется. Юзер сам решает, нужно ли потом скорректировать баланс закрытого счёта вручную (мы НЕ обнуляем автоматически при закрытии).

### 13.5 Денормализация fingerprint на Transaction

Миграция 0058 добавляет `Transaction.fingerprint` (indexed). На коммите импорта `_prepare_transaction_payloads` копирует `normalized_data.fingerprint` в новое поле, и `TransferLinkingService` пробрасывает его на обе ноги transfer-пары. Backfill из `ImportRow.normalized_data_json.fingerprint` через `created_transaction_id`. Цель — быстрый lookup по fingerprint в `_load_history_by_fingerprint` (§5.2 v1.20) без JSON-traversal.

## 19. Жёсткое правило: счёт обязан иметь банк

**Введено 2026-04-29 (миграция 0055).**

### 19.1 Инвариант

`Account.bank_id` — `NOT NULL`. Создание / обновление счёта без `bank_id` отвергается на всех слоях:

| Слой | Защита |
|---|---|
| Pydantic | `AccountCreateRequest.bank_id: int = Field(ge=1)` |
| Service | `AccountService._require_bank()` проверяет существование банка → `BankRequiredError` |
| API | `BankRequiredError` → HTTP 422 |
| БД | `accounts.bank_id NOT NULL` + `FOREIGN KEY ... ON DELETE RESTRICT` |
| Frontend | Поле «Банк *» в `account-form.tsx` валидируется перед submit |

### 19.2 Заглушка «Неопределённый банк»

Для счетов без реального банка (наличные, тестовые брокерские, кредиты/вклады по которым не загружаются выписки, малоизвестные банки без extractor'а) предусмотрен **системный банк-заглушка**:
- `code='unknown'`
- `name='Неопределённый банк'`
- `is_popular=False`
- Создан миграцией 0055; присутствует автоматически в любой БД.

Юзер выбирает его в `BankPicker` так же, как любой другой банк. Skeleton/fingerprint и весь детерминистический стэк работают под `bank_code='unknown'` — теряются только bank-specific mechanics (Yandex Сплит/Дебет, Sberbank PDF quirks), что и ожидаемо для неизвестного банка. (До v1.19 здесь упоминался LLM-модератор — он отключён.)

### 19.3 Почему это инвариант

Без банка ломаются:
- `bank_mechanics_service` (Yandex/Sberbank/Tinkoff правила).
- Скоп правил `(bank_code, account_id)`.
- Recognition / extractor выбор по bank_code.
- Refund cluster overrides, использующие историю по тому же `bank_code`.

Раньше `bank_id` был nullable, и `bank_code='unknown'` падал в эти места молчаливым нулём, не возвращая warning юзеру. Теперь либо реальный банк, либо явная заглушка — никаких NULL.

## 20. Известные риски (информационно, без срочного исправления)

### 20.1 Защита от двойного счёта статистики правил

**Что:** теоретически при `bulk_apply_cluster` строка получает `+0.5` к `confirms` правила (Case B), затем индивидуальный `confirm` той же строки добавляет `+1.0` (Case A). Защита держится на флаге `_already_counted_at_bulk_ack` в `normalized_data_json`. Если коммит сорвётся между `cluster_bulk_acked_at` stamp'ом и `on_confirmed()` — счётчик может рассинхронизироваться.

**Почему откладываем:** на текущем объёме нет наблюдаемого дрифта. Логика guard'а покрывает 99% сценариев. Структурный фикс (отдельная таблица `rule_confirm_log` с unique-key `(rule_id, row_id)` + миграция существующих счётчиков) дороже наблюдаемого риска.

**Когда возвращаться:** при первой жалобе на «правило слишком быстро/медленно растёт» — провести SQL-аудит `confirms vs ожидаемое`, и при подтверждении дрифта внедрить `rule_confirm_log`.

### 20.2 Facts vs decisions в одном JSON

**Что:** §4.2 спека требует «факты и решения никогда не смешиваются в одной структуре». На уровне типов разделено корректно (`ParsedRow` vs `DecisionRow`), но в БД оба сериализуются в одно поле `import_rows.normalized_data_json`. Защита держится **дисциплиной разработчика** — никто кроме нормализатора не должен писать в facts-ключи.

**Почему откладываем:** ни одного зафиксированного бага по этому пункту. Все вызовы `update_row` / refund-overrides / Yandex post-process пишут только в decisions-поля по convention, и code review это ловит.

**Когда возвращаться:** при первом инциденте, когда rule/post-processor случайно затрёт `amount` или `transaction_date` (LLM-pathway убран в v1.19, поэтому одна категория источников риска отпала). Тогда мигрировать на две jsonb-колонки (`facts_json` + `decisions_json`) или хотя бы линтер-инвариант (тест-проверка, что только `import_normalizer.py` пишет в facts-ключи).

### 20.3 Rollout detection-логики не переоценивает старые метки

**Что:** статус `duplicate` — терминальный (§8.3). При апгрейде логики `_detect_committed_duplicates` (например, v1.19 → v1.20 четырёхветочная модель) старые метки, записанные предыдущей версией, остаются в БД нетронутыми, потому что текущий reset-механизм фильтрует только по `match_source='cross_session'`.

**Симптом (зафиксирован 2026-05-03):** session 150 содержала 10 строк с `match_branch IS NULL`, ложно помеченных дублями (см. §8.12 кейс-якорь и решение).

**Текущее решение:** §8.12 — explicit reset legacy-меток через маркер `match_branch IS NULL`. Работает для текущего rollout v1.19 → v1.20.

**Почему остаётся как known risk:** маркер `match_branch IS NULL` одноразовый. При следующем апгрейде detection-логики (v1.21+) понадобится новый маркер — например, версия алгоритма в `transfer_match.algo_version`. Без этого rollout'ы будут продолжать оставлять залипшие ложные дубли.

**Когда возвращаться:** при разработке v1.21+ изменений в `_detect_committed_duplicates` — добавить `algo_version` в записываемую `transfer_match`-метку и расширить reset-метод фильтром по version mismatch.
